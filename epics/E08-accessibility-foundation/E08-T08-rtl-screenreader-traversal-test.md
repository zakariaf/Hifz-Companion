## Goal

A per-locale widget test proves that screen-reader focus and reading order run **right-to-left, top-to-bottom** on the shell's key screens in fa, ckb, and ar — matching the visual order, the §8 (A8) invariant. The test pumps the E07 shell (`HomeShell` + its inert placeholder cards) under the locale-derived `Directionality.rtl` with the **real bundled UI fonts loaded via `FontLoader`** (never `Ahem`, so the Sorani extra letters پ چ ژ گ ڤ ڕ ڵ ۆ ێ ە and the Persian/Arabic digit blocks are actually shaped and traversed), walks the merged, labeled semantics tree E08-T02 produced, and asserts the ordered sequence of focusable nodes is the right-to-left visual sequence — not LTR, not source order. This is the co-owned RTL focus/reading-order invariant shared with E09; E08 owns its screen-reader/semantics half. The harness reuses E08-T07's audit wiring so the traversal proof runs PR-blocking in the same lane.

## Implementation notes

**TEST-FIRST (correctness-critical — this *is* a test deliverable):** the traversal-order assertion is the artifact. Write the failing/red form first by asserting the *visual* right-to-left node order against an intentionally LTR/source-order expectation, confirm it fails, then encode the correct RTL expectation; the test must distinguish RTL traversal from "happens to look right under LTR." There is no production code to write under this task beyond a small shared traversal-extraction test helper.

1. **File.** `packages/features/test/a11y/shell_traversal_rtl_test.dart` (alongside E08-T02's `features/test/a11y/` suite), `flutter test`, widget tier. Pull the real-font `setUpAll`/offline-`HttpOverrides` bootstrap and the locale-pump helper from the shared E08-T02/E08-T07 test bootstrap (e.g. `features/test/a11y/_a11y_test_bootstrap.dart`) rather than re-declaring them; add only the traversal-order extraction here.

2. **Real bundled fonts, never `Ahem` (sacred rule extends to chrome here).** In `setUpAll`, load the real bundled UI fonts via `FontLoader` (the same registration the E08-T02 RTL golden uses) so the three scripts shape genuinely — Sorani's extra letters and the Extended-Arabic-Indic (fa/ckb, `۰۱۲`) / Arabic-Indic (ar, `٠١٢`) digit blocks are actually laid out and traversed. `Ahem` (every glyph a solid square) would defeat the point: ordering of identical squares proves nothing about real-script reading order (eng-write-dart-test §6; eng-rtl rule 10).

3. **Direction is locale-derived (eng-rtl §2).** Drive the shell through `MaterialApp(locale: Locale('fa'/'ckb'/'ar'), localizationsDelegates: …, supportedLocales: …)` and let `GlobalWidgetsLocalizations` resolve `TextDirection.rtl` — do **not** hardcode an app-wide `Directionality(TextDirection.rtl)`. (A hardcoded root direction hides physical-side bugs and is the named refusal.) For a narrowly-scoped single-widget probe, pumping under `Directionality.rtl` is acceptable, but the shell-level traversal run goes through the locale so the test exercises the same path the app does.

4. **Pump the key screens.** Pump `HomeShell` with a representative placeholder child and the four inert placeholder cards (E07-T04) — the §8 "Today, Muṣḥaf, Progress" surfaces, as they exist in the E07 slice. The cards are the merged nodes from E08-T02; the nav bar is the five `labeled(button: true, …)` destinations (Today rightmost under RTL).

5. **Extract the traversal order, assert RTL.** Read the ordered list of focusable/labeled semantics nodes (via the semantics tree — `tester.binding.pipelineOwner.semanticsOwner` / `SemanticsTester`-style traversal, or `tester.getSemantics` over the candidate nodes), then assert the sequence matches the **visual** right-to-left, top-to-bottom order: for a horizontal row (the nav bar), the first traversed node is the one at the greatest `x` (right edge) and the last is at the least `x`; within a column the order is increasing `y`; an embedded "Juz N · page M" run reads its tokens in logical order (FSI/PDI-isolated, so "page 7 of 30" never reads "30 of 7"). Compute the expectation from each node's rendered rect under RTL, not from source/child order — that is what makes the test catch an LTR traversal regression.

6. **Three locales, the same assertion (eng-write-dart-test §7).** Parameterize over `['fa', 'ckb', 'ar']` and run the identical ordering assertion for each, so a locale-specific regression (e.g. ckb's longer strings reflowing a row, or ar's digit block) is caught in its own voice — never English-only with an "fa/ckb/ar inherit it" assumption (ds-09 §10 anti-pattern). Tag the real-font run `@Tags(['golden'])` if it shares the pinned-OS golden lane wiring, or keep it in the PR-blocking widget lane alongside E08-T07; match whichever lane E08-T07 established.

7. **The sacred-text boundary holds (ds-09 §7).** No node whose order is asserted is a muṣḥaf glyph codepoint or a re-typeset āyah; where a page is referenced the traversed label is its **reference** (surah/ayah range/juz / "Page N · Juz M"), exactly as E08-T02 produced it. The test imports only the `a11y/` chrome and the shell — never the muṣḥaf/glyph layer.

8. **Pitfalls to avoid.** Asserting source/child order instead of the rendered right-to-left rect order (the test would pass under a broken LTR traversal — the central trap); rendering the traversal test with `Ahem`/a placeholder font (forbidden — the scripts must be real); hardcoding an app-wide `Directionality.rtl` instead of selecting the locale (eng-rtl §2 refusal); testing only `ar` and assuming `fa`/`ckb` inherit (each locale runs); `pumpAndSettle()` on an intentionally-indefinite indicator (pump explicit durations); letting a number inside a traversed label arrive as raw ASCII rather than the locale numeral + FSI/PDI isolate (the ASCII-digit grep + this order assertion both guard it); reaching into the muṣḥaf glyph layer; any network/microphone/model dependency creeping into the harness (the offline `HttpOverrides` throws if so).

- [ ] Any number/page-reference run inside a traversed label is the locale numeral (Extended-Arabic-Indic for fa/ckb, Arabic-Indic for ar) and FSI/PDI-isolated, so it reads in order; the ASCII-digit grep and the physical-side grep stay green.
- [ ] No traversed node is a muṣḥaf glyph codepoint or a re-typeset āyah; where a page is referenced the label is the page reference (surah/ayah/juz / "Page N · Juz M").

## Tests

This task *is* the test deliverable; it adds no production code beyond a small shared traversal-extraction helper. Dart tests run in CI on every PR (`flutter test`); the real bundled UI fonts are loaded via `FontLoader` (never `Ahem`) and the runner is pinned. **Write the ordering assertion first (red before green).**

- `packages/features/test/a11y/shell_traversal_rtl_test.dart` — **widget (write first)**:
  - **Nav-bar RTL traversal** (per `fa`/`ckb`/`ar`): pump `HomeShell`; extract the ordered focusable nav semantics nodes; assert the first traversed node is the rightmost-rendered (Today, at the greatest `x`) and the sequence decreases in `x`, top-to-bottom for any wrapped rows — i.e. matches the visual RTL order, not source order.
  - **Placeholder-cards column traversal** (per `fa`/`ckb`/`ar`): pump the four inert placeholder cards (each a single `mergedItem` node from E08-T02); assert traversal runs top-to-bottom (increasing `y`) and each merged node is reached as one node in order, not as fragments out of order.
  - **Embedded-run reads in order** (per `fa`/`ckb`/`ar`): where a traversed label carries a "Page N · Juz M"/"Juz N" run, assert the locale-numeral, FSI/PDI-isolated tokens read in logical order (no "30 of 7" reorder).
  - **Discriminator (the red proof)**: an asserted-LTR/source-order expectation fails; switching to the rendered-RTL expectation passes — demonstrated in the diff so the gate is proven to check something, not nothing.
- **Offline / no-network guard**: the shared throwing `HttpOverrides` bootstrap is installed (reused from E08-T02/E08-T07); a banned-import grep over `features/test/a11y` finds no `http`/`dio`/`HttpClient`/`drift`/glyph-layer symbol; no microphone/ASR/model dependency.
- **Localization gates stay green**: `app_ar.arb` key-coverage compiles (`nullable-getter: false`); the `features/**` hardcoded-string grep, the physical-side grep, and the ASCII-digit grep stay green; fa/ckb/ar values present for every label the traversal reads (ckb provisional where flagged). No test asserts a coverage percentage; every case carries a meaningful `expect`.

## Goal

The chrome honors the OS text-size setting **automatically** through the E06 `type.*` styles: every visible string flows through Flutter's built-in `MediaQuery.textScalerOf` path with no per-widget scale code, and stays legible and usable to **200%+** (and the five iOS AX Dynamic-Type sizes) on the real E07 shell screens ŌĆö no clipped buttons, no truncated labels, no horizontal scroll. Where a genuinely dense row would clip, the **only** sanctioned remedy is `MediaQuery.withClampedTextScaling` applied as a documented ceiling on that one subtree ŌĆö never a fixed-pixel font, never `TextScaler.noScaling` on user-facing text. The contract is proved by a per-locale (`fa`/`ckb`/`ar`, with `ckb` as the longest-string binding case) golden suite that pumps the shell screens at the unscaled, 200%, and AX-extreme scales on the real bundled Vazirmatn/Estedad fonts and asserts no overflow.

## Implementation notes

TEST-FIRST (correctness-adjacent): a clipped button or truncated label at 200% is a *visible* WCAG 1.4.4/1.4.10 defect, so write the at-scale overflow widget tests and the per-locale at-scale golden stubs (see Tests) **before** the clamp helper and any layout adjustment ŌĆö the unscaled ŌåÆ 200% ŌåÆ AX goldens and the "no `RenderFlex` overflow at 200%" assertions must exist and fail on a clipping layout before the fix lands.

1. **Convention doc (the rule the feature epics inherit).** Add a short "Dynamic text scale" section to the design-system README / accessibility convention notes (wherever E08-T01 records the conformance claim) stating, in one place: (a) chrome scales **automatically** via the E06 `type.*` styles + `MediaQuery.textScalerOf` ŌĆö never read or hard-code a raw `fontSize`/`TextScaler` on user-facing text; (b) `MediaQuery.withClampedTextScaling(maxScaleFactor: ŌĆ”)` is the **only** permitted ceiling, used solely where a dense row provably clips, on the smallest possible subtree, with a `// why:` comment; (c) `TextScaler.noScaling` and fixed-pixel fonts on user-facing text are **banned**; (d) the muß╣ŻßĖźaf glyph layer is out of scope here (E08-T04 owns it). No new copy strings ŌĆö this is engineering convention text, not user-facing.

2. **Verify the automatic path, add nothing where nothing is needed.** The E06 `type.*` styles already flow through `MediaQuery.textScalerOf`; the bulk of this task is **proving** the shell reflows, not adding code. Where a screen already reflows cleanly at 200%/AX, the only deliverable is its at-scale golden. Resist the urge to wrap every screen in a clamp ŌĆö clamping where it is not needed is itself an accessibility regression (the epic's "a clamp becomes a disable" risk).

3. **The clamp helper (the one sanctioned ceiling).** Add a tiny, documented wrapper under `packages/features/lib/src/design_system/a11y/` (e.g. `clamped_text_scaling.dart`) that re-exposes `MediaQuery.withClampedTextScaling` with a named, design-system-sourced default ceiling (e.g. `textScaleCeilingDense`), so a dense row caps its scale via one auditable call site rather than ad-hoc inline magic. The helper takes a `maxScaleFactor` and a `child`; it never reads a wall clock, never touches the network, and is chrome-only. `///` doc states it is a *ceiling* (scaling continues up to the cap), names the WCAG SC it serves (1.4.4 / 1.4.10), and forbids passing it the muß╣ŻßĖźaf reader subtree. Export through the `features` barrel under the `design_system`/`a11y` namespace.

4. **Apply the ceiling only to the genuinely-dense shell rows that clip ŌĆö surgically.** Audit the E07 shell at 200% + AX in `ckb` (the longest strings) first; for any row that clips after honest reflow attempts (logical `start`/`end`, `Wrap`/`Flexible`, single-`Text` labels), prefer **reflow/pagination** (per design-system 09 ┬¦5); reach for the clamp helper **only** if reflow cannot keep the row usable, and cap as high as the layout tolerates (a ceiling near the AX range, not a low cap that defeats large text). Candidate sites are the page-card chip row and the grade band (E07-T08) and the bottom-nav labels (E07-T04) ŌĆö apply only where the test proves a clip, with the `// why:` comment naming the screen and the binding `ckb` string.

5. **Keep each label a single `Text`/`TextSpan`; reflow, never truncate.** When fixing a clip, do **not** fragment a localized label to shrink part of it (diacritic-clip/wrap bug) and do **not** add `overflow: TextOverflow.ellipsis`/`maxLines: 1` to load-bearing chrome to "hide" a clip ŌĆö that is a 1.4.4 failure. Convert fixed-width chips/rows to logical, reflowing layout (`Wrap`, `Flexible`, `IntrinsicHeight` only where needed); the layout is already logical `start`/`end` (E07), so reflow needs no RTL-specific codepath.

6. **Numerals stay on the locale path under scale.** Any digit in a scaled row still renders via `numberFormatFor(locale)` (Extended Arabic-Indic for fa/ckb, Arabic-Indic for ar) and stays bidi-isolated ŌĆö scaling changes size, never the digit block or isolation; the at-scale goldens render fa/ckb/ar so a numeral regression under scale is caught (eng-rtl ┬¦5, ┬¦7).

7. **Pitfalls to avoid:**
   - `MediaQuery(data: ŌĆ”copyWith(textScaler: TextScaler.noScaling))` or a fixed `fontSize`/`.copyWith(fontSize:)` on user-facing chrome to "protect" a layout ŌĆö **banned**; reflow or clamp-with-ceiling instead.
   - Using the deprecated `textScaleFactor` API anywhere ŌĆö all scaling is `TextScaler` (Android 14 non-linear scaling respected).
   - A low `maxScaleFactor` that caps text below the AX range ŌĆö the ceiling exists to prevent *clipping*, not to shrink large text; cap as high as the layout allows.
   - Wrapping a whole screen (or the app) in the clamp helper ŌĆö it is a per-dense-subtree tool; over-clamping is a disable in disguise.
   - Passing the muß╣ŻßĖźaf reader / any QPC glyph subtree to the clamp helper or any chrome scale path ŌĆö the glyph layer is never OS-scaled (E08-T04 seam; the ┬¦5 anti-pattern).
   - Rendering the at-scale goldens with `Ahem`/a placeholder font ŌĆö fa/ckb would not actually reflow; load the **real** Vazirmatn/Estedad via `FontLoader`.
   - `maxScaleFactor` or a clamp magic literal inlined in a widget ŌĆö source the ceiling from the design-system token / helper default, named, never a bare number.
   - Hiding a clip with `ellipsis`/`maxLines: 1` on load-bearing text, or fragmenting a label across `TextSpan`s ŌĆö both are 1.4.4 failures.

## Goal

A documented, tested boundary exists asserting that the mu·π£·∏•af glyph layer is **never** OS-scaled, re-typeset, or reflowed for any accessibility goal: the only sanctioned enlargement is the dedicated zoom transform of the *already-rendered* layer, which E13 builds. The seam is proven against a page/placeholder stub ‚Äî given the same `MushafLayout` page, the assembled layout data (line grouping, glyph-code strings, page geometry) is **byte-identical** under any `TextScaler` the OS supplies, including the AX extremes; and the screen reader is fed the page's *reference* (surah ¬∑ ayah range ¬∑ juz) from the bundled structure data, never a reconstruction of the QPC glyph codepoints. This task ships the **contract surface E13 is built against** ‚Äî it does not build the reader, its zoom control, or any real page rendering.

## Implementation notes

This task is correctness-critical for the existential R1 covenant, so it is **TEST-FIRST**: write the seam assertions below before (or alongside) the tiny seam helper they pin; the layout-identity and reference-not-glyphs cases must exist and express the contract before any reader code in E13 leans on it.

1. **Where the seam lives.** The mu·π£·∏•af layout types (`MushafLayout`/`MushafPage`/`MushafWord`, `GlyphLine`, `PageGeometry`, `assemblePage`, `qpcFontFamily`) live in the `packages/quran` data package (per domain-mushaf-text-integrity and `docs/engineering/02-project-structure.md` ¬ß4). The seam is **not a new subsystem** ‚Äî it is (a) a one-line documented invariant on the page-assembly path and (b) a small `PageReference` value the reader hands to assistive tech. Add the seam contract as a `///` doc comment on `assemblePage` (and on the page-build provider) stating verbatim: *the assembled layout is a pure function of `(pageNumber, MushafLayout)` and does not read `MediaQuery`, `TextScaler`, or any OS scale ‚Äî enlargement is E13's zoom transform of the rendered layer only.*

2. **The `PageReference` value (semantics input, not glyphs).** Add an immutable value type ‚Äî `PageReference({required int pageNumber, required SurahAyahRange range, required int juz})` ‚Äî in `packages/quran` (or the existing structure-data module), derived **only** from the bundled QUL structure dataset, never from `GlyphLine.glyphCodes`. Its `///` doc states it is what the screen reader receives. The localized spoken string is composed at the feature layer through E08-T02's `Semantics` label + the ARB set (so it speaks fa/ckb/ar), with locale numerals via the E09 numeral path; this task only guarantees the *reference data* flows, never glyph codepoints. Do not give `PageReference` any field carrying a glyph code, an opaque address, or reconstructed text.

3. **The page stub.** Build a small deterministic `MushafLayout` stub fixture (2‚Äì3 lines, mixing a `LineType.ayah` line, a `surahName`/`basmala` line, with real-shaped placeholder `glyphCodes` strings and a fixed `PageGeometry`) under the test's fixtures ‚Äî **not** a network or asset load, and **not** Ahem (this is not a pixel golden; it is a data-identity check, so any stable opaque `glyphCodes` string is fine, but it must be the *same* string instance the assertion compares against). One stub, one expected `List<GlyphLine>`.

4. **The text-scale matrix.** Drive `assemblePage` (or the page-build provider, pumped under a `MediaQuery` with each `TextScaler`) across a matrix: `TextScaler.noScaling`, `TextScaler.linear(2.0)` (the 200% AA bar), and a large AX value (`TextScaler.linear(3.2)` or larger). Assert the assembled `List<GlyphLine>` ‚Äî line count, per-line `glyphCodes`, `LineType`, and the `PageGeometry` ‚Äî is **equal across the whole matrix and equal to the expected stub**. The point is precisely that the scaler is in scope and is *ignored* by the sacred path.

5. **TEST-FIRST ordering.** (a) Write `assemblePage`-identity-under-scale ‚Üí it should pass immediately if the path is already pure, and the test *documents* that purity as a guarded contract; (b) write reference-not-glyphs ‚Üí assert the `PageReference` carries surah/ayah/juz and contains no substring of the stub's `glyphCodes`; (c) only then add the `///` seam doc comments and, if E13 needs it, the `PageReference` type. If any assertion is hard to express, that difficulty is the signal a scale dependency has already leaked ‚Äî fix the path, never relax the test.

6. **Pitfalls to avoid.**
   - **Wrapping the mu·π£·∏•af in the chrome text-scale path.** A well-meant "let the OS setting make the page bigger" would feed `MediaQuery.textScalerOf` into glyph rendering and reflow the sacred layer ‚Äî the exact ¬ß5 anti-pattern and an R1 violation. The seam test exists to make that change fail loudly.
   - **Clamping instead of excluding.** Do **not** reach for `MediaQuery.withClampedTextScaling` around the mu·π£·∏•af ‚Äî that is T03's tool for *dense chrome rows*. The mu·π£·∏•af is excluded **entirely**, not capped; clamping still couples it to OS scale.
   - **Recomputing line breaks.** Any `softWrap`, width-wrapping, or `TextPainter` line measurement on the glyph layer recomputes breaks the dataset already fixed (R1, ¬ß11.2). The identity assertion catches a layout that changes shape; a why-comment names the rule.
   - **Leaking glyph codes to assistive tech.** Never pass `GlyphLine.glyphCodes` (opaque PUA addresses) as a `Semantics` label, `semanticsLabel`, or any reader-facing string. The reference is announced; the glyphs are `ExcludeSemantics`/decorative to the reader.
   - **Pixel-testing here.** This is a *data-identity* seam, not a fidelity golden. Do not add `matchesGoldenFile` ‚Äî the KFGQPC pixel goldens are owned by domain-mushaf-text-integrity and run in the golden CI lane; duplicating them here would be slow and miss the point.

Fixtures are inline literals ‚Äî no asset bundle, no clock. Required cases, written FIRST:

- **Layout identity under text scale** ‚Äî for the page stub, `assemblePage` produces a `List<GlyphLine>` that is equal (line count, each `glyphCodes`, each `LineType`) and a `PageGeometry` that is equal across `{noScaling, linear(2.0), linear(3.2)}`, and equal to the frozen expected stub. (If pumped through the provider, wrap each run in `MediaQuery(data: MediaQueryData(textScaler: ‚Ä¶))` to prove the scaler is in scope and ignored.)
- **No reflow / no runtime line-breaking** ‚Äî the assembled line count and per-line grouping are exactly the stub's; a variant stub whose words would *wrap* under a naive `softWrap` still produces the dataset's fixed line grouping (proving breaks come from data, not measurement).
- **Reference, not glyphs, to the reader** ‚Äî `PageReference` for the stub exposes surah/ayah-range/juz matching the bundled structure data, and a guard asserts no `PageReference` field contains any substring of the stub's `glyphCodes` (the reader can never receive an opaque glyph address as "text").
- **ExcludeSemantics on the glyph layer (if a widget stub is pumped)** ‚Äî a `SemanticsTester`/finder assertion that the glyph stub contributes no readable text node, only the `PageReference`-derived label is in the semantics tree.
- **Offline guard intact** ‚Äî the suite makes no network call; the throwing `HttpOverrides` is asserted installed (a stray connection is a loud, named failure, per eng-write-dart-test ¬ß8).

CI gates unchanged: no-network and banned-import gates green; no new `matchesGoldenFile` master; no coverage-percentage gate introduced.

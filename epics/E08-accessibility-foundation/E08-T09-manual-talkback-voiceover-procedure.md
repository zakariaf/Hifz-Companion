## Goal

A repeatable, dated **A9 manual assistive-technology procedure** exists in the repo: a TalkBack (Android) + VoiceOver (iOS) in-language checklist — run separately in **fa**, **ckb**, and **ar** — that a human executes on real devices over the **cold-start → first day → review → missed-day catch-up** path, the same four-flow journey the automated `integration_test` suite drives (J1–J4). It is the human pass that catches the RTL/multilingual 20% the automated gates (T07 `meetsGuideline`, T08 per-locale traversal goldens) structurally cannot: spoken intelligibility, reading-direction of announcements, and whether a screen reader can actually *operate* the daily flow in each voice. This task **authors and records** the procedure (a Markdown artifact, versioned and dated, with an explicit pass/fail record table per locale × platform) and adds a tiny CI guard that keeps the checklist honestly in sync with the real shell — it does **not** execute the human pass (that is E20's release gate) and builds no new screen.

## Implementation notes

This task is **documentation-first**: the deliverable is the recorded procedure. The accompanying sync-guard test is correctness-supporting (it keeps the doc from rotting), so write its assertions alongside the doc and let them fail until the doc's step ids match the journeys.

1. **Where the procedure lives.** Author `docs/engineering/manual-a9-screenreader-procedure.md` (sibling to `11-testing-strategy.md`, matching the gate-8 manual-review precedent). It is a versioned, dated artifact — **not** a `test/` file, **not** an `integration_test` — because A9 is explicitly "not automatable" ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md); [engineering 11 §8 gate 8](../../docs/engineering/11-testing-strategy.md)). Link it from the epic and from `12-localization-rtl-accessibility-impl.md` §8's gate row.

2. **Document header (provenance).** Top of the file: the gate id (**A9**), the conformance bar it serves (**WCAG 2.2 AA**, [design-system 09 §1](../../docs/design-system/09-accessibility-and-inclusivity.md)), the date authored, the app version/commit the steps were written against, and a one-line owner note: "Executed by E20 release readiness on real devices before each release; release-blocking." A revision-history line so each device run is dated.

3. **Preconditions block.** State plainly, as checkboxes the operator confirms before starting:
   - The automated accessibility gates are **green** on the build under test: A1 (`textContrastGuideline`), A6 (`androidTapTargetGuideline` / `iOSTapTargetGuideline`), A7 (`labeledTapTargetGuideline`) from **E08-T07**, and the per-locale RTL traversal goldens from **E08-T08**. (This pass verifies what those cannot, not what they already cover.)
   - **Real devices**, not just simulators: at least one physical Android (TalkBack) and one physical iOS (VoiceOver) device.
   - **Airplane mode ON** for the post-onboarding journeys (J2–J4) — the offline guarantee ([PRD C1, §17](../../docs/PRD.md)) is itself under test; the screen reader must operate the whole daily flow with the radio off.
   - The build's UI **language is set to the locale under test** (fa, then ckb, then ar) — the pass is run **three times per platform**, once per language, never English-only ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md)).

4. **The journey steps (mirror J1–J4, step-for-step).** Structure the checklist in four numbered sections whose ids are the **same** as the `integration_test` journeys so the sync-guard can match them:
   - **J1 — Cold start (onboarding).** Reader navigates: pick UI language → confirm muṣḥaf/riwāyah → core-pack download step → coverage (which juz held) taps → per-juz Solid/Shaky/Rusty rating → cycle preset. Check: every control is **announced with a localized label** (no English, no unlabeled icon); **no microphone prompt** appears ([PRD C2, R5](../../docs/PRD.md)); reading order is **RTL top-to-bottom**; the riwāyah name is spoken.
   - **J2 — First day (Today).** Reader reaches the Today list. Check: each **page-card is read as ONE merged phrase** ("Juz ۷ · page ۱۳۴ · weak"), not three fragments ([design-system 09 §7](../../docs/design-system/09-accessibility-and-inclusivity.md)); the **track chip and decay state are spoken as words** (never "color only" — the never-color-alone label from T06 is what the reader voices); section headers (Far/Near/New) are announced as headers; numerals are spoken in the **locale digit set**.
   - **J3 — Review (recite → grade → optional sign-off).** Reader opens the recite flow, reveals the page, grades. Check: the **muṣḥaf is announced by reference** (surah · ayah range · juz), never glyph codepoints ([design-system 09 §7](../../docs/design-system/09-accessibility-and-inclusivity.md)); the grade band (Again/Hard/Good/Easy) is operable, each button **labeled with its traditional verb**; "**page graded**" / "**sign-off recorded**" are **announced** and spoken **in reading order (RTL)** ([design-system 09 §7](../../docs/design-system/09-accessibility-and-inclusivity.md)); the recite confirmation is calm — **no celebratory announcement** ([PRD R3, C6](../../docs/PRD.md)).
   - **J4 — Missed-day catch-up.** With the gap already present (operator returns after a gap), reader reaches Today. Check: "**catch-up plan ready**" is **announced** in RTL; the banner is spoken as a **calm, supportive** plan, **never** a "you are behind" / overdue / shame phrasing ([PRD R3](../../docs/PRD.md); [design-system 09 §7](../../docs/design-system/09-accessibility-and-inclusivity.md)); the re-spread list is fully traversable and labeled.

5. **The intelligibility bar (A9 proper).** For each step, the operator records not just "labeled" but **"intelligible in this language's voice"** — Flutter's own testable rule: "the descriptions should be intelligible" ([engineering 12 §7](../../docs/engineering/12-localization-rtl-accessibility-impl.md)). Add an explicit line per locale: numerals voiced naturally (۴۵۶ in fa/ckb, ٤٥٦ in ar), each run spoken in the **right pronunciation voice** (Persian / Sorani / Arabic — `TextSpan.locale` is set), and announcements heard **in reading order**, not reversed.

6. **The pass/fail record table.** End the document with a result matrix the operator fills on each run: rows = the four journeys × the per-journey checks; columns = **fa·TalkBack, fa·VoiceOver, ckb·TalkBack, ckb·VoiceOver, ar·TalkBack, ar·VoiceOver**; each cell pass/fail with a free-text defect note. A footer line: "Date · device models · OS versions · app commit · operator," and the rule "**any single fail blocks the release**" ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md); [PRD §20](../../docs/PRD.md)).

7. **The sync-guard test (the only code).** Add a small `flutter_test` (it reads ARB labels and the journey id list — no widget pump needed beyond resolving localizations; `package:test` if it can read the lists directly) that:
   - parses the four journey-section ids out of the procedure doc and asserts they are exactly `{j1ColdStart, j2FirstDay, j3Review, j4Catchup}` — the same ids the `integration_test/` journeys use — so the manual checklist and the automated journeys can never silently diverge ([engineering 11 §6](../../docs/engineering/11-testing-strategy.md));
   - asserts every **announcement string** the procedure names ("catch-up plan ready," "page graded," "sign-off recorded") resolves to a **non-empty ARB key in all three locales** (fa/ckb/ar), so the human operator is never asked to listen for an announcement that has no localized text — reusing E08-T02's announce-label keys and the `nullable-getter: false` coverage.
   - This guard **does not** simulate a screen reader (impossible to automate per A9) — it only prevents documentation rot.

8. **Pitfalls to avoid.**
   - **Turning A9 into a CI job.** A9 is explicitly *not automatable* ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md); [engineering 11 §8 gate 8](../../docs/engineering/11-testing-strategy.md)). Do not write an `integration_test` that claims to "run the manual pass"; the value is the human ear. T07/T08 are the automated gates; this is the human one.
   - **English-only or one-locale shortcut.** The pass is fa **and** ckb **and** ar, each on TalkBack **and** VoiceOver — six runs. A checklist that bakes in English labels or a single locale defeats the gate's entire purpose ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md)).
   - **Simulator-only.** TalkBack/VoiceOver behave differently on real hardware (gestures, focus, announcement timing); the procedure requires physical devices ([design-system 09 §10](../../docs/design-system/09-accessibility-and-inclusivity.md)).
   - **Letting the muṣḥaf be read as glyphs.** A checklist step must explicitly confirm the reader speaks the page *reference*, never QPC codepoints — a regression here is a sacred-text and R1 failure ([design-system 09 §7](../../docs/design-system/09-accessibility-and-inclusivity.md); [PRD R1](../../docs/PRD.md)).
   - **A celebratory or shaming announcement.** The catch-up and grade announcements must be calm and supportive; a "Well done! 🎉"-style or "you're behind" announcement is an adab failure the human pass is specifically positioned to hear ([PRD R3, C6](../../docs/PRD.md); domain-adab-and-religious-integrity).
   - **Doc rot.** Without the sync-guard, the checklist drifts from the real journeys/labels; the guard ties step ids and announcement keys to the live ARB set and journey list.

## Tests

`test/manual_a9_procedure_sync_test.dart` (in the app/features package that can resolve `AppLocalizations`; `flutter_test`), REUSE SPDX header (`GPL-3.0-or-later`), throwing-`HttpOverrides` offline bootstrap installed, runs in CI's **fast** lane on every PR. The procedure itself is **not** a test — A9 is not automatable; this file only guards the recorded doc against rot. Fixtures are inline; no clock, no asset/network load. Required cases:

- **Journey ids in sync** — parse the four section ids from `manual-a9-screenreader-procedure.md` and assert they equal exactly `{j1ColdStart, j2FirstDay, j3Review, j4Catchup}`, the ids the `integration_test/` journeys use ([engineering 11 §6](../../docs/engineering/11-testing-strategy.md)); an added/removed/renamed journey fails the build until the checklist is updated.
- **Announcement keys localized in all three locales** — for each announcement the procedure tells the operator to listen for ("catch-up plan ready," "page graded," "sign-off recorded"), assert the corresponding E08-T02 ARB key resolves to a **non-empty** string under `Locale('fa')`, `Locale('ckb')`, and `Locale('ar')` — so the operator is never told to listen for text that does not exist localized.
- **No English label leak in the named-control set** — for the controls the J1–J4 steps name (grade verbs, track chip, catch-up banner, sign-off), assert their ARB labels under fa/ckb/ar are non-empty and differ from the `ar`-template English placeholder (reuses the T07/T08 label-coverage path; catches a control the manual pass would hear in English).
- **Offline guard intact** — the suite makes no network call; the throwing `HttpOverrides` is asserted installed (a stray connection is a loud, named failure, per eng-write-dart-test §8).

No new `integration_test` journey, no `matchesGoldenFile` master, and no coverage-percentage gate are introduced. The no-network and banned-import gates stay green.

# E20 — Release Readiness

No new feature work — this epic is the gate that decides whether the gift may ship. It assembles every release-blocking check from PRD §20 into one green release suite (no-network/banned-import, SHA-256 text & asset integrity, engine golden vectors + invariant properties, pinned-OS muṣḥaf and RTL/locale goldens, localization completeness, coverage published), drives the two human sign-offs the machine cannot supply (the scholarly muṣḥaf proof and the mutashābihāt dataset review) plus the always-on adab/religious-integrity checkpoint, publishes the OSS trust pack that turns "no data leaves the device" into a checkable claim, and ships the F-Droid + Google Play + Apple + signed-GitHub-Releases build from one signed, provenance-attested tag. Nothing here adds a screen; everything here proves the screens already built are safe to give away.

## Why this epic exists

For a *ṣadaqah jāriyah* app holding the words of Allah, "release" is not a launch event — it is the moment the non-negotiables are either honoured or quietly broken, and the religious-integrity requirements are *release-blocking by construction*: "any one, gotten wrong, ends the project" ([PRD §4](../../docs/PRD.md)). The whole-Quran retention problem this app exists to solve — a ḥāfiẓ losing 600+ pages *invisibly* ([PRD §2](../../docs/PRD.md)) — is only worth solving if the muṣḥaf it shows is byte-exact and the engine it trusts can only ever pull a page *forward*, never tell anyone a juz is "safe to drop" ([PRD R1, §7.6, §7.12](../../docs/PRD.md)). Those two covenants — *the sacred text is never put at risk* and *the engine may only make a page more frequent, never less* — are exactly the two the testing strategy makes un-tradeable, and this epic is where they stop being promises and become a build that either passes or fails ([docs/engineering/11-testing-strategy.md](../../docs/engineering/11-testing-strategy.md) §intro). The eight feature/foundation epics (E08–E19) each *write* the checks that prove their own surface; this epic *assembles them into the release contract*, re-runs the full suite from a signed tag, and refuses to publish on any red gate ([PRD §20](../../docs/PRD.md); [docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §4).

It exists for a second reason the category makes sharp: for an offline, no-AI, no-telemetry app *the repo is the audit*. The product's central promise — "no data leaves the device" ([PRD §17](../../docs/PRD.md)) — is only believable because the source is open, the single network code path is readable, and a stranger can rebuild the exact bytes from a signed tag ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §2, §8). So release-readiness is not just green CI; it is the trust pack (`LICENSE` + §7 App Store exception, `CONTRIBUTING.md` + DCO, `SECURITY.md`, `PRIVACY.md`, `docs/verify-yourself.md`, REUSE compliance), the strongest-truthful privacy declaration on every store, the F-Droid entry ticket the architecture already earns, and the multi-channel posture that keeps the gift reachable if one gatekeeper narrows access. And because two of the eight gates (the manual muṣḥaf proof and the mutashābihāt scholar sign-off) are human, not machine, this epic also owns the *checkpoint discipline* — the recorded adab/religious-integrity pass and the scholar's on-device proof — without which the green machine gates are necessary but not sufficient. The app is released when, and only when, every covenant is honoured: text fidelity, sect-neutrality, no gamification, servant-to-the-teacher, privacy-as-trust, and "nothing safe to drop."

## Scope

### In scope

- **The release-gate suite as one auditable contract.** Wire all eight PRD §20 gates into the gate→job mapping and a release workflow that re-runs them: gate 1 text & asset integrity (`restraint` SHA-256 re-hash + runtime verifier), gate 2 visual-diff (`golden` + on-device), gate 3 engine golden vectors, gate 4 invariant property tests, gate 5 localization & RTL completeness, gate 6 network restraint, gate 7 mutashābihāt dataset sign-off, gate 8 manual muṣḥaf review ([docs/engineering/11-testing-strategy.md](../../docs/engineering/11-testing-strategy.md) §8 mapping; [PRD §20](../../docs/PRD.md)).
- **The tag-triggered release workflow** (`release.yml`, `on: push: tags: ['v*.*.*']`): the pinned `subosito/flutter-action` toolchain, the full gate suite re-run, the reproducible Android build signed with *our* key, the SLSA build-provenance attestation, and the hard refusal-to-publish on any red gate; the binary ships lean (pinned tag + manifest hashes only), assets are never fetched at app-build time ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §4).
- **The airplane-mode acceptance run** that proves every post-onboarding surface — Today, Muṣḥaf, Mutashābihāt, Progress, Settings — works with zero network after the one-time core-pack download, the integration half of the offline covenant ([docs/engineering/11-testing-strategy.md](../../docs/engineering/11-testing-strategy.md) §7; [PRD §19.3](../../docs/PRD.md)).
- **The OSS trust pack** at the repo root: `LICENSE` (GPL-3.0-or-later **with** the §7 App Store additional permission, present before any outside copyright), `LICENSING.md`, `CONTRIBUTING.md` (DCO 1.1 verbatim + sign-off CI), `SECURITY.md`, `PRIVACY.md`, `docs/verify-yourself.md`, the `LICENSES/` texts + `.reuse/dep5`, and `reuse lint` green in CI ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §1, §2, §3, §8).
- **The per-store privacy posture and metadata**, gated: Apple "Data Not Collected" + `PrivacyInfo.xcprivacy` (`NSPrivacyTracking=false`, empty collected types, accurate required-reason codes) + mandatory privacy-policy link; Play "No data collected" Data-safety form; the `fastlane/metadata/{ar,fa,ckb}/` store listings as the single in-repo source of truth for all three stores; the App-Review business-model note (free, *ṣadaqah*, no IAP, no account, no ads, one-time static download) ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §2, §6, §7).
- **The F-Droid eligibility gate**: FLOSS-everything, the analytics/ads/backend/crash-SDK dependency allow-list, no NonFreeNet / NonFreeAssets / Tracking / Ads anti-feature, reproducible build shipping our key, and the multi-channel release (F-Droid primary, Play, App Store, signed GitHub Releases for Obtainium) ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §5, §7).
- **The human checkpoints, recorded.** The release-blocking adab/religious-integrity conscience pass over the *whole shipping app* (R1–R6: reverence, riwāyah named, zero tafsīr, sect-neutral, no gamification, no guilt/fear/loss, never "safe to drop", servant-to-the-teacher, no microphone, scholarly-review flags where pending); the qualified ḥāfiẓ/scholar's on-device muṣḥaf proof (sample pages, sajda marks, ayah numbering, basmala — gate 8); and the mutashābihāt dataset scholar sign-off (objective-wording scope — gate 7) ([PRD §20.7, §20.8](../../docs/PRD.md); [docs/design-system/13-islamic-identity-and-adab.md](../../docs/design-system/13-islamic-identity-and-adab.md) §1, §5, §6).

### Out of scope

- **Authoring the engine golden vectors / `glados` invariant properties** the gate runs → **E04 scheduling-engine** (this epic *runs* `dart test engine/` in the release job; it does not write the frozen `(state, grade, elapsed) → (D, S, due)` rows or the INV-1…INV-6 properties).
- **The SHA-256 asset manifest, the pinned-tag scheme, and the runtime fail-closed verifier** the hash gate re-checks → **E05 quran-data-and-rendering** (domain-asset-pack-integrity owns the manifest and device-side verifier; this epic re-hashes against it in CI).
- **The muṣḥaf-fidelity golden harness and the glyph-rendering rules** a fidelity golden compares against → **E05 quran-data-and-rendering** (domain-mushaf-text-integrity defines correct pixels; this epic pins the runner and consumes the goldens as a gate).
- **The localization-completeness gate machinery** (key coverage, the greps, the Arabic-plural check, per-locale RTL/numeral goldens) → **E09 localization-rtl-foundation** (this epic consumes gate 5 as a release-blocker, it does not build the pipeline).
- **The accessibility audit machinery** (TalkBack/VoiceOver, Dynamic Type, contrast, ≥48dp, non-color heat-map encoding) → **E08 accessibility-foundation** (this epic includes the recorded accessibility pass in the checklist; the gate definition is E08's).
- **The CLAIMS register, the science screen, and any user-facing factual number** → **E19 science-screen-and-claims** (this epic ships no new copy and originates no claim; every cited number traces to a claim already registered and sourced by its owning feature epic).
- **The feature copy / strings themselves** the adab checkpoint reviews → **E11–E19** (this epic runs the conscience pass over the assembled app and records sign-off; it does not write the strings).

## Dependencies

### Depends on

- **E08 accessibility-foundation** — the recorded accessibility pass (RTL focus/reading order, Dynamic Type, contrast ≥4.5:1, ≥48dp targets, non-color heat-map encoding, localized `Semantics`) that the release checklist confirms across the assembled app.
- **E09 localization-rtl-foundation** — gate 5 made machine-checkable: zero missing ARB keys, no hardcoded user-facing strings, the per-locale RTL + numeral goldens on real bundled fonts, and the banned-phrase (adab) lint the release suite re-runs.
- **E10–E19 (every feature epic)** — the assembled, shipping app whose covenants this epic verifies: the engine (E04 via E12), the muṣḥaf reader (E13), Today/recite-grade (E12), mutashābihāt (E14), progress/heat-map (E15), settings/profiles/teacher (E16), backup (E17), reminders (E18), and the science screen (E19) — each contributes the test the release suite runs and the surface the adab pass reviews.

### Enables

- **The release itself** — a signed, provenance-attested tag published to F-Droid (primary), Google Play, the Apple App Store, and signed GitHub Releases for Obtainium, with every release-blocking gate green and every human checkpoint recorded.
- **Auditable trust** — `docs/verify-yourself.md` lets any stranger rebuild the exact Android bytes, confirm "Data Not Collected" against the open source, observe airplane-mode operation, and re-derive the asset hashes; the OSS trust pack turns the privacy/offline/integrity promises into independently checkable claims.
- **Perpetual freedom in law** — the GPL-3.0-or-later + §7 App Store exception, the DCO-not-CLA contribution path, and the refusal to relicense the Quran assets keep the gift a gift for whoever maintains it next.

## Risks

- **A green gate that checks nothing.** A grep with a bad path glob, a golden rendered with Ahem instead of the real KFGQPC fonts, a floating Flutter version that silently re-renders goldens, or an auto-blessed vector exits green while the covenant it claims to protect goes unverified. *Mitigation:* every job pins an exact `flutter-version`; muṣḥaf goldens load the real fonts via `FontLoader`; CI never runs `--update-goldens`/`--update-vectors`; each gate is proven against a deliberate violation per eng-add-ci-check ([docs/engineering/11-testing-strategy.md](../../docs/engineering/11-testing-strategy.md) §5, §8).
- **Machine-green mistaken for release-ready.** Gates 7 and 8 are *human* — a passing CI run does not mean the muṣḥaf was proofed on a real device or the mutashābihāt dataset was scholar-reviewed. *Mitigation:* the release contract treats gates 7–8 as recorded sign-off artifacts; the release is blocked until both checkpoints and the adab conscience pass are captured, not merely until CI is green ([PRD §20.7, §20.8](../../docs/PRD.md); [docs/engineering/11-testing-strategy.md](../../docs/engineering/11-testing-strategy.md) §8).
- **The App Store exception retrofitted too late.** Once a second copyright holder exists, adding the §7 exception requires *their* consent — the literal VLC/GNU-Go failure that pulled GPL apps from the App Store. *Mitigation:* the exception ships in `LICENSE` before any outside contribution merges; the DCO flow makes every inbound commit GPL-3.0-or-later *with* the exception ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §1, §3).
- **A NonFreeAssets/NonFreeNet anti-feature forfeits F-Droid.** A font/audio license that forbids redistribution or derivatives earns NonFreeAssets; framing the one-time download as a runtime service dependency risks NonFreeNet. *Mitigation:* font/audio terms are verified before a pack ships; the airplane-mode acceptance run proves "offline forever" after the one-time static download; the dep allow-list catches a tracking/ads SDK at PR time ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §5).
- **A privacy declaration silently drifts.** A future plugin's required-reason API use, or an opt-in diagnostic that transmits, breaks "Data Not Collected"/"No data collected" while the form still reads zero. *Mitigation:* the privacy-posture release check fails on any code path that transmits user data; the manifest is audited against bundled plugins' own manifests; diagnostic logs stay local and are never transmitted ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §6; [PRD §17](../../docs/PRD.md)).
- **Single-channel dependence orphans users.** A hostile policy change at one gatekeeper (e.g. Google's 2025 developer-registration decree) could narrow access for a free app whose natural home is sideloading and F-Droid. *Mitigation:* the same reproducible, signed, attested build ships to four channels, and the signed-GitHub-Releases/Obtainium path is maintained as insurance ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §7).
- **Scope creep into feature work under the "release polish" banner.** A "while I'm here" copy tweak or a new screen smuggled into the release epic bypasses its owning epic's gates and review. *Mitigation:* this epic adds no screen, string, or claim; anything beyond verifying, packaging, and signing the already-built app is rejected in review and filed against the owning feature epic.
- **A red gate waved through to make the release date.** Pressure to ship can tempt a `continue-on-error` or a "just this once" merge on a skipped gate — exactly where a dropped diacritic or a clamp violation reaches a ḥāfiẓ. *Mitigation:* no gate is `continue-on-error`; the release job re-runs the full suite and refuses to publish on any red gate; the covenants outrank the calendar ([docs/engineering/13-oss-repo-and-release.md](../../docs/engineering/13-oss-repo-and-release.md) §4; domain-adab-and-religious-integrity).

## Goal

`.github/workflows/ci.yml` exists as the four-job pipeline from engineering 11 ¬ß8 ‚Äî **fast / golden / restraint / journeys** ‚Äî and `.github/workflows/release.yml` exists as the tag-triggered gate-re-run skeleton, so the gates the prior tasks built (T05's `tool/` scripts, T06's tests + throwing `HttpOverrides` bootstrap + golden/integration skeletons) become a check a PR either passes or fails. Every job pins `subosito/flutter-action@v2` to the exact README-baseline `flutter-version` with `channel: stable` and uses first-party actions only. CI runs the **same** files the prior tasks wrote ‚Äî no CI-only logic. The PRD ¬ß20 gate‚Üíjob mapping is documented in the workflow and kept in sync with the engineering 11 ¬ß8 table; no gate is `continue-on-error`; coverage is published, never gated on a percentage; signing keys are deferred to secrets/E20. On the epic's final PR all four jobs are green and the public logs show the gate results.

## Implementation notes

TEST-FIRST note: this task wires gates, it does not author the checks they run ‚Äî the "test-first" obligation is satisfied by E01-T08's deliberate-violation sweep (a CI gate that passes because it checks nothing is the epic's first risk). For *this* task, the verifiable artifact is the green four-job run on a real PR **plus** the proof that each job is wired to a check that can fail: do not declare a job done until you have seen it go red on a seeded violation (T05's seeded-violation matrix, re-run through CI) and green on revert. Author the workflow against the `eng-add-ci-check` `template.yml` and fill every `# TODO`.

1. **`.github/workflows/ci.yml`** ‚Äî copy the `eng-add-ci-check/template.yml` `ci` document. Triggers: `on: push: branches: [main]` and `on: pull_request`. Declare the pinned SDK **once** as `env: FLUTTER_VERSION: '<README baseline>'` (match `README.md` exactly ‚Äî e.g. `'3.38.0'`) and reference `${{ env.FLUTTER_VERSION }}` from every job's `subosito/flutter-action@v2` `with:` block, `channel: stable`. First-party actions only: `actions/checkout@v4`, `subosito/flutter-action@v2`, `fsfe/reuse-action@v6`, `reactivecircus/android-emulator-runner@v2` ‚Äî no third-party convenience wrappers. Every job begins `actions/checkout@v4` ‚Üí `subosito/flutter-action@v2` (pinned) ‚Üí `flutter pub get`.

2. **`fast` job** (`runs-on: ubuntu-latest`) ‚Äî `flutter analyze --fatal-infos` (this **includes** the path-scoped `avoid-banned-imports` / engine-purity lints from E01-T02's `analysis_options.yaml`: the import-path half of no-network gate 6 and the engine boundary); then `dart test engine/` (the placeholder vector test + the `glados` lane T06 scaffolds, ready for E04's frozen vectors + INV-1‚Ä¶INV-6 ‚Äî CI only **verifies**, never `--update-vectors`, fuzz stays OFF); then `flutter test --coverage --exclude-tags golden`; then a strip step `lcov -r coverage/lcov.info '*.g.dart' '*.drift.dart' '*.freezed.dart' '*/generated/*' -o coverage/lcov.info`. Leave a TODO + commented step to publish `lcov.info` to an OSS coverage uploader for auditability ‚Äî and **do not add a min-coverage gate** (a covered-but-unasserted line is worthless; the engine invariants are the bar ‚Äî 11 ¬ß10). Carry the commented ARB-key-check TODO (gate 5's `fast` half) ‚Äî the real `check_l10n_complete.sh`/`dart` ARB check is T05's; wire it in here only once T05's script is the canonical entrypoint.

3. **`golden` job** (`runs-on: ubuntu-latest` ‚Äî **pinned OS is load-bearing**, cross-OS golden drift is documented) ‚Äî `flutter test --tags golden`, which runs the real-`FontLoader` mu·π£·∏•af-fidelity goldens (gate 2) and the RTL `ar`/`fa`/`ckb` layout goldens (gate 5) T06 skeletons. In E01 there are no masters yet, so the tagged set is **dormant/empty** and the job is green by running zero golden tests ‚Äî E05 supplies the 604 reference images, the job's first real consumer. **Never** add `--update-goldens` to a CI step (it would make the gate assert nothing); never move this job to macOS/Windows.

4. **`restraint` job** (`runs-on: ubuntu-latest`) ‚Äî run T05's scripts verbatim: `bash tool/check_no_network.sh`, `bash tool/check_dep_allowlist.sh`, `bash tool/check_l10n_complete.sh` (the dep-allow-list is no-network gate 6's layer-b; the banned-SDK regex lives in the script, not the YAML); then `dart run tool/verify_asset_hashes.dart` (SHA-256 integrity gate 1 ‚Äî dormant/fail-closed on the present-but-empty manifest, lit by E05); then `uses: fsfe/reuse-action@v6` (`reuse lint` ‚Äî a missing/malformed per-file SPDX `GPL-3.0-or-later` header fails the build). Prefer invoking T05's `tool/` scripts over re-inlining the bash so there is exactly one source of truth (02 ¬ß7: CI runs the same files).

5. **`journeys` job** (`runs-on: ubuntu-latest`) ‚Äî `uses: reactivecircus/android-emulator-runner@v2` with `api-level: 24` (the min-Android baseline ‚Äî keep it == the README/`build.gradle` baseline) and `script: flutter test integration_test/`. This runs T06's one-journey launch skeleton now (green), and E04/E05/E07 grow it to the four PRD flows (cold-start ‚Üí first day ‚Üí review ‚Üí catch-up) ‚Äî **do not grow past four** without a decision-log amendment (11 ¬ß6). The airplane-mode acceptance run (the throwing-`HttpOverrides` third no-network layer, T06) lives in this job.

6. **`.github/workflows/release.yml` skeleton** ‚Äî copy the `template.yml` `release` document. `on: push: tags: ['v*.*.*']` (an exact, immutable tag ‚Äî never a moving ref). `permissions: contents: write, id-token: write, attestations: write` (Sigstore/SLSA, exercised in E20). One `release` job on `ubuntu-latest` (pinned Linux for golden + build stability), pinned `flutter-action` (keep `flutter-version` `==` `ci.yml`'s `FLUTTER_VERSION`), that **re-runs the full gate suite**: `flutter analyze --fatal-infos` (gate 6) ¬∑ `dart test engine/` (gates 3, 4) ¬∑ `flutter test --tags golden` (gates 2, 5) ¬∑ `dart run tool/verify_asset_hashes.dart` (gate 1) ¬∑ `fsfe/reuse-action@v6`. Leave the build/sign/attest steps as `# TODO (E20)` with the `secrets.KEYSTORE_BASE64`/`KEY_ALIAS`/`KEY_PASSWORD` env shape stubbed and a comment that the key lives **only** in encrypted secrets, never the repo. **No actual signing or store upload in this task.**

7. **The gate‚Üíjob mapping is documentation that must stay in sync.** Reproduce the engineering 11 ¬ß8 / `template.yml` PRD ¬ß20 gate‚Üíjob comment block verbatim at the top of `ci.yml` (gate 1‚Üírestraint+runtime ¬∑ 2‚Üígolden+journeys ¬∑ 3‚Üífast ¬∑ 4‚Üífast ¬∑ 5‚Üígolden+fast ¬∑ 6‚Üírestraint+fast ¬∑ 7‚Üírestraint+human ¬∑ 8‚Üíhuman). This block is the auditable release contract; a check added without a matching gate row, or a gate row with no job, is the bug `eng-add-ci-check` ¬ß1 forbids.

8. **Pitfalls to avoid.** Floating the SDK (`channel: stable` with no `flutter-version`) silently re-renders goldens and breaks reproducibility ‚Äî pin on **every** job, especially `golden` and `release`. `continue-on-error: true` on any gate step turns a release-blocker advisory ‚Äî never add it. Re-inlining T05's bash into the YAML creates two sources of truth that drift ‚Äî call the scripts. Adding `--update-goldens` or `--update-vectors` to a CI step makes the gate assert nothing. Adding a coverage-percentage threshold rewards asserting-nothing tests (11 ¬ß10) ‚Äî publish only. Letting CI `flutter build` *fetch the Quran assets at app-build time* couples the code/data release cycles (13 ¬ß4) ‚Äî the binary ships lean with the pinned tag + manifest hashes only; do not add an asset-download step. Third-party actions are an unaudited supply-chain surface ‚Äî first-party only. Putting the signing key (or any secret value) in the YAML instead of `secrets.*` is a release-blocking leak ‚Äî defer the whole signing path to E20.

## Tests

This task's verifiable artifact is a **green four-job CI run on a real PR** plus the proof that each job is wired to a check that *can* fail ‚Äî there are no new Dart/Flutter test cases here (the jobs run the tests T03/T04/T06 authored). Verification is the workflow executing on the real runner:

| Job | What it runs (authored elsewhere) | E01 expected outcome |
|---|---|---|
| `fast` | `flutter analyze --fatal-infos`; `dart test engine/` (T06 placeholder vector + glados lane); `flutter test --coverage --exclude-tags golden` (T06 per-package placeholder tests); `lcov -r` strip | green; coverage `lcov.info` published, generated files stripped, no % gate |
| `golden` | `flutter test --tags golden` (T06 golden skeleton, no masters) | green on the dormant/empty tagged set; lights up in E05 |
| `restraint` | `tool/check_no_network.sh`, `tool/check_dep_allowlist.sh`, `tool/check_l10n_complete.sh`, `dart run tool/verify_asset_hashes.dart` (T05), `fsfe/reuse-action@v6` | green on the clean tree; `verify_asset_hashes.dart` dormant/fail-closed on the empty manifest; `reuse lint` green |
| `journeys` | `flutter test integration_test/` on the min-Android emulator (T06 launch skeleton) | green on the one trivially-green launch journey |
| `release` (skeleton) | full gate re-run; build/sign/attest deferred to E20 | gate steps wired; no signing/upload exercised in E01 |

Gate-can-fail proof (the `eng-add-ci-check` mandate; formalized as the cross-gate sweep in **E01-T08**): for each wired check, re-run T05's seeded-violation through CI and confirm the owning job goes **red** with a named message, then green on revert ‚Äî a `package:http` in `engine` reddens `fast` (analyze) and `restraint` (`check_no_network.sh`); a banned SDK in a `pubspec.yaml` reddens `restraint` (`check_dep_allowlist.sh`); a missing SPDX header reddens `restraint` (`reuse lint`); (in E05) a flipped asset byte reddens `restraint` (`verify_asset_hashes.dart`); (in E04) a clamp-violating vector reddens `fast` (`dart test engine/`).

Offline / no-network guard: CI never fetches the Quran assets at app-build time (the binary ships lean ‚Äî pinned tag + manifest hashes only, 13 ¬ß4); the workflows add no asset-download step; the throwing-`HttpOverrides` airplane-mode acceptance run (T06) executes inside `journeys`, proving every post-onboarding screen works with zero network.

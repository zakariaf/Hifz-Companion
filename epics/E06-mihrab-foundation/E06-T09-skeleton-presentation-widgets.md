## Goal

The exact handful of dumb-View presentation primitives E07's walking-skeleton slice needs exist under `packages/features/lib/src/design_system/widgets/`, each taking display data only, RTL-native by construction, reading every value through tokens (`ColorScheme`/`TextTheme`/the extensions) and every string through `l10n.*` placeholders: a calm `MihrabCard` (a quiet `Card`/`ListTile` row), the restrained `FilledButton`/`FilledButton.tonal` styling and the `SegmentedButton` styling (component-theme tuning, no celebratory/badge/streak treatment), and the `AppearanceSwitcher` control (a `SegmentedButton<AppearanceSetting>` over `followSystem · Light · Sepia · Dark · Night` that emits a callback — it sets no theme itself). No page card, no grade band, no heat-map widget (those are E10). Each ships a `#Preview`/golden-stub specimen across the four appearances × three locales (fa/ckb/ar). This task styles and assembles; it composes no `ThemeData` (that is E06-T06) and wires no route or store (that is E07).

## Implementation notes

These are dumb Views (no engine math, no date logic, no persistence, no Quran glyph) — correctness here is *visual/structural* (token discipline, RTL geometry, touch size, the four-appearance recolor), so the goldens (Tests) are the primary contract and the per-widget specimens are written alongside the widgets.

1. **Location.** The design system lives inside the `features` umbrella package (E01). Create the widget tree under `packages/features/lib/src/design_system/widgets/`:
   - `mihrab_card.dart` — `MihrabCard`, the calm `Card`/`ListTile` row.
   - `mihrab_buttons.dart` — the `FilledButton`/`FilledButton.tonal`/`SegmentedButton` **styling** (component-theme builders + a thin styled wrapper where one aids reuse), not new button widgets.
   - `appearance_switcher.dart` — `AppearanceSwitcher`, the dumb-View control.
   Export the public surface through the `features` barrel under the `design_system` namespace; keep `lib/src/` internals private; REUSE SPDX header on every new file.

2. **`MihrabCard` — display-data-only row.** Immutable config: a leading widget slot (an `IconData`-backed leading affordance positioned at `start`), a `title`/`subtitle` taken as **`Widget`/`String` display data the caller already localized and numeral-formatted** (the card formats nothing itself — it shows no page number, juz index, or date of its own; those belong to E10's page card), and an optional `onTap`. Build on M3 `Card` → `surfaceContainerLow`/`surfaceContainer` (§3), `elevation` Level 0–1, **no `surfaceTint`**, inner padding `space.4`, row gap `space.2`, all via `EdgeInsetsDirectional`. Leading affordance at `start`, an optional trailing chevron at `end` using the auto-mirroring `Icons.arrow_forward_ios`/`chevron` family (mirrors in RTL). If `onTap` is non-null the whole row is one ≥48dp-tall hit target (§4). Text styles resolve from the `TextTheme` (E06-T04), never an ad-hoc `TextStyle`.

3. **Restrained button styling.** In `mihrab_buttons.dart` provide `FilledButtonThemeData`/`SegmentedButtonThemeData` builders (consumed by E06-T06's `ThemeData` composition) plus, where reuse warrants, a thin styled wrapper. Constraints from §5/§4: `FilledButton` minimum size ≥48dp wide and tall (the grade-band template wants ≥56dp tall — encode the tall variant as a documented `ButtonStyle` so E10's grade band reuses it), label style sourced from `buildMihrabTextTheme` (zero Arabic `letterSpacing`, E06-T04), color from `ColorScheme` roles only. `FilledButton.tonal` uses the tonal container role for the lower-emphasis secondary action. `SegmentedButton` styling: selected/unselected from `ColorScheme` roles, the check/selected indicator the plain M3 default (no custom celebratory glyph), segment min-height ≥48dp. **No** `Badge`, no streak/score chip, no success-green fill, no animated celebratory state anywhere (§5 anti-patterns; epic DoD).

4. **`AppearanceSwitcher` — a callback-emitting dumb View.** Render a `SegmentedButton<AppearanceSetting>` (M3 `Set`-based selection) over the five options `followSystem · light · sepia · dark · night` from E06-T03's `AppearanceSetting`. Props: `final AppearanceSetting selected;` and `final ValueChanged<AppearanceSetting> onChanged;` — the control **sets no theme and reads no setting store**; it reflects `selected` and emits `onChanged` (E06-T06 owns wiring it to the shell's appearance state). Each segment's label is an `l10n.*` string (`l10n.appearanceFollowSystem`, `l10n.appearanceLight`, `l10n.appearanceSepia`, `l10n.appearanceDark`, `l10n.appearanceNight` — **referenced** here; the transcreated fa/ckb/ar values are E09). If five segments overflow the compact width in ckb's longer transcreations, fall back to a labeled `ListTile` + radio group rather than truncating (reflow-not-truncate, §05/04 §7) — keep the same callback contract. Segments ≥48dp tall, `space.2` apart.

5. **The `#Preview` / golden specimen matrix (four appearances × three locales).** For each of the three widgets add a specimen widget (the CycleVault `#Preview` analog) that pumps the widget inside the composed `ThemeData` for each `MihrabAppearance` (Light · Sepia · Dark · Night) under `Directionality(textDirection: TextDirection.rtl)` for each of `fa`/`ckb`/`ar`, loading the **real bundled Vazirmatn/Estedad** via `FontLoader` (never `Ahem`). The specimens use `l10n` keys / placeholder display data, never hardcoded copy. These specimens are the seed E06-T11 promotes onto the pinned-OS golden lane (4 × 3 = 12 frames per widget).

6. **Pitfalls to avoid:**
   - **Scope creep into E10.** A real page card (page number + juz + track chip + decay indicator), a grade band (Again/Hard/Good/Easy), or any heat-map cell is E10 — reject in review and file against E10. This task ships only the *generic* `MihrabCard`, the button *styling*, and the switcher.
   - A `success`-green fill, a `Badge`, a streak/score chip, a checkmark "saved!" flash, or any celebratory/animated reward state on a button or card (epic DoD; §5) — there is no such token to reach for; saved/verified reads as icon + text in `color.accent.green`, not a celebration.
   - A raw `Color(0xFF…)`, an off-scale `EdgeInsets.all(13)`, a bare `Duration`, a `surfaceTint` veil, or a hardcoded `left`/`right` inset — every value is a token via `Theme.of(context).extension<T>()` / `ColorScheme` / `space.*`, every inset `EdgeInsetsDirectional` (the E06-T11 token-discipline + the `features/**` physical-side grep are gates).
   - A hardcoded user-facing string anywhere (including the switcher labels and any preview caption) — all copy via `l10n.*`; the previews reference keys, never literals.
   - The `AppearanceSwitcher` reaching into a settings store / calling `setState` to change the theme itself — it is a dumb View; it only reflects `selected` and emits `onChanged`.
   - Any `package:quran`/QPC-glyph reference, any `TextStyle` that re-typesets a Quran string — these are chrome widgets; the muṣḥaf never appears here (the wall is E06-T05).
   - A sub-48dp tap target (a small chevron-only hit, a thin segment) or two targets <8dp apart — §4 is a hard gate.

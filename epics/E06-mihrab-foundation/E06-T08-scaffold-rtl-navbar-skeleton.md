## Goal

Two presentation primitives live in the `features` theme folder and turn the screen-template and RTL-nav prose of design-system 05 §3/§4/§5 into reusable code. `MihrabScaffold` encodes the **bottom-action screen template**: full-bleed calm content scrolls in the upper/middle area, an optional primary action sits in a bottom band inside the one-handed thumb-reach zone (`space.4` padding above a `SafeArea` inset that clears the home indicator and the nav), and destructive/rare actions are reserved for the top corners. The five-tab `NavigationBar` chrome renders **Today · Muṣḥaf · Mutashābihāt · Progress · Settings in RTL order — Today (home) at the trailing/right edge** — wired to **no real routes** (selection is a plain in-widget index callback; `go_router` wiring and the real tab slices are E07). Every inset is `EdgeInsetsDirectional`/`AlignmentDirectional`, every interactive target ≥48×48dp with ≥8dp spacing, every distance a `space.*` token, every label an `l10n.*` key. Both are dumb Views taking only display data; no engine math, no date logic, no persistence, no Quran glyphs.

## Implementation notes

1. **Files** (one primary type per file, in the `features` umbrella package theme folder; `material`/`widgets` imports only — no engine/data/quran dependency):
   - `packages/features/lib/src/theme/mihrab_scaffold.dart` — the bottom-action scaffold chrome.
   - `packages/features/lib/src/theme/mihrab_navigation_bar.dart` — the RTL five-tab nav skeleton.
   Re-export both through the existing `packages/features/lib/src/theme/tokens.dart` barrel (and onward via `lib/features.dart`) so consumers import one symbol set, consistent with E06-T01/T07.
2. **`MihrabScaffold`.** A `StatelessWidget` wrapping M3 `Scaffold`, taking only display data: `Widget body` (the full-bleed scrolling content slot), an optional `Widget? bottomAction` (the thumb-band primary action), and an optional `Widget? bottomNavigationBar` (the caller passes `MihrabNavigationBar`; the scaffold does not own selection state). Background = `surface` full-bleed (no explicit color literal — let the `ThemeData` `scaffoldBackgroundColor` from T06 stand, or resolve `Theme.of(context).colorScheme.surface` by role; never a raw hex). When `bottomAction != null`, place it in a bottom band: `Padding(padding: EdgeInsetsDirectional.fromSTEB(space4, space4, space4, space4))` wrapped in `SafeArea(top: false)` so it clears the home indicator and sits above the nav, in the one-handed thumb zone (05 §5). Pull `space4` from `Theme.of(context).extension<SpacingTokens>()!.space4` — no numeric padding literal. Do **not** add a `FloatingActionButton`, celebratory affordance, or reward surface (05 §5 anti-pattern; EPIC DoD).
3. **`MihrabNavigationBar`.** A `StatelessWidget` over M3 `NavigationBar` with exactly **five** `NavigationDestination`s in **logical order Today · Muṣḥaf · Mutashābihāt · Progress · Settings** — declared in that single logical order; **RTL flips the geometry so Today renders at the trailing/right edge** automatically. Do not hand-reverse the list (that is the §3 anti-pattern; the flip is a `Directionality` consequence, not a manual reversal). Take `int selectedIndex` and `ValueChanged<int> onDestinationSelected` — a plain index callback, **wired to no real route** (the caller manages a local index in tests/previews; E07 replaces it with `go_router`). Each destination uses an auto-mirroring/neutral M3 `Icons.*` glyph and a **`l10n.*` label** (e.g. `l10n.navToday`, `l10n.navMushaf`, `l10n.navMutashabihat`, `l10n.navProgress`, `l10n.navSettings`) — never hardcoded text; the strings themselves are owned by E09's ARB pipeline, this task references the keys. Surface = `surfaceContainer`/`High` by role (02 §3) — the M3 default the T06 component theme sets; add no color literal.
4. **Touch targets (05 §4).** Each `NavigationDestination` already clears ≥48dp via M3; the bottom-band `FilledButton` (when supplied by the caller/T09) is sized ≥48dp tall and ≥48dp wide with `space.2` clearance — but this task only guarantees the *scaffold band* gives it ≥`space.4` padding and a SafeArea inset, not the button's own metrics (T09/T06 own the `FilledButton` theme). Assert the nav destinations meet the 48dp floor in the widget test.
5. **No real routes, no state ownership.** The scaffold and nav are **dumb Views**: selection is an `onDestinationSelected(int)` callback the parent drives; neither widget holds a `go_router`, a Riverpod provider, a `Navigator`, or any persistence. A preview/test host owns a local `int` index. This is the explicit E06↔E07 seam — the walking-skeleton wiring (composition root, routes, stores) is E07; this task ships only the presentation chrome E07 plugs into.
6. **RTL by construction, not by flag.** Do **not** wrap anything in a hardcoded `Directionality(TextDirection.rtl)` (the app-wide direction comes from the locale via `MaterialApp`, established in E01/T06). Tests/previews that need RTL pump an explicit `Directionality`/locale, but the production widgets read direction from context. All insets are `EdgeInsetsDirectional`/`AlignmentDirectional`; no `EdgeInsets.only(left:/right:)`, `Alignment.centerLeft/Right`, or `Positioned(left:/right:)` (the E06-T11 / `features/**` physical-side grep bans them).
7. **Token discipline.** Every distance resolves to a `space.*` token via `Theme.of(context).extension<SpacingTokens>()!`; the only colors are M3 roles (`colorScheme.surface`/`surfaceContainer*`); no raw hex, no off-scale dp, no bare `Duration`/`Curve` (the scaffold does not animate). E06-T11's no-raw-value check greps for violations.
8. **Pitfalls to avoid:** hand-reversing the destination list to "put Today on the right" (breaks the moment direction flips — declare logical order and let RTL place it); hardcoding `EdgeInsets.only(right: 16)` for the trailing edge (use `EdgeInsetsDirectional`); wrapping in a manual `Directionality.rtl` (hides physical-side bugs and breaks a Latin island); putting the primary action in a top app-bar/corner out of thumb reach, or a destructive action in the easy-reach bottom band (05 §5 anti-patterns); adding a `FloatingActionButton`/badge/streak/celebration surface (02 §5, 05 §5 anti-patterns; gamification forbidden); hardcoding an English nav label instead of an `l10n.*` key; the scaffold or nav owning route/selection state instead of taking a callback (defeats the E07 seam); injecting a numeric padding/`SizedBox(height: 16)` literal instead of `space.4`.

## Tests

`packages/features/test/theme/mihrab_scaffold_test.dart` and `packages/features/test/theme/mihrab_navigation_bar_test.dart` (file names mirror sources), `flutter_test` (`testWidgets`), REUSE `GPL-3.0-or-later` SPDX header, run under `flutter test` on the E01 fast CI lane with the throwing-`HttpOverrides` offline guard installed from the shared bootstrap. Each widget is pumped inside a `MaterialApp`/`Theme` carrying the T06-composed `ThemeData` (or a minimal `ThemeData` whose `extensions:` include `SpacingTokens.standard()`) and an explicit `Directionality`/locale — no real DB, no assets, no network, no wall clock. Required cases:

- **RTL destination order (the headline test):** pump `MihrabNavigationBar` under `Directionality(TextDirection.rtl)`; assert the five destinations resolve in logical order Today · Muṣḥaf · Mutashābihāt · Progress · Settings **and** that, by their on-screen x-positions, **Today sits at the trailing/right edge** and Settings at the leading/left — proving the geometry flipped without a hand-reversed list.
- **LTR mirrors:** the identical nav under `TextDirection.ltr` places Today at the left edge, proving the order is `Directionality`-derived, not a hardcoded physical placement.
- **Selection callback is a plain index, no route:** tapping a destination invokes `onDestinationSelected` with the tapped logical index; assert no `Navigator`/route push occurs (a `NavigatorObserver` records zero pushes) — the no-real-routes seam.
- **Touch-target floor:** assert each rendered `NavigationDestination` hit region is ≥48×48dp (05 §4); when a `bottomAction` is supplied, assert the scaffold's bottom band wraps it in a `SafeArea` with `space.4` directional padding.
- **Bottom-action placement:** pump `MihrabScaffold` with a sentinel `body` and a sentinel `bottomAction`; assert the action renders **below** the scrolling body and **above** the nav, within the SafeArea-insetted band (its top is greater than the body's bottom) — the thumb-zone template (05 §5).
- **ckb reflow:** pump the nav under the `ckb` locale with the longer transcreated labels; assert the bar lays out within the same insets without overflow/exception (the same template serving all three locales, 05 §3) — labels via `l10n.*`, real bundled UI font, never `Ahem`.
- **Token discipline (smoke):** the scaffold's bottom-band padding equals `SpacingTokens.standard().space4` (assert the resolved inset, not a literal), proving no raw dp leaked in.

CI lanes unchanged: the E01 fast lane runs these (no `Date()`, no networking symbols, throwing-`HttpOverrides` green; the `features/**` physical-side and ASCII-digit greps pass). The four-appearance × three-locale skeleton goldens that screenshot this scaffold + nav land in E06-T11.

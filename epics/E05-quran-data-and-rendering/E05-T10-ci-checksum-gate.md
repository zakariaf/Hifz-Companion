## Goal

The build-time half of R1 exists as a pure-Dart, no-rendering gate `tool/verify_asset_hashes.dart` and its tested core `verifyAssetIntegrity(MushafEdition, ReleaseManifest)`, wired into E01's `restraint` CI job and the tag-triggered release suite. The gate re-computes the SHA-256 of every file in the published GitHub Release core pack — the Uthmani text, the QUL layout dataset, and all **604** KFGQPC page fonts — and fails the build on any mismatch against (a) the manifest baked into the app binary and (b) the **authoritative Tanzil Uthmani hash** the text must equal, so the manifest itself cannot silently drift. It asserts exactly **604** fonts are present and unmodified and that the page count for the Madani 15-line edition is **604**. Verifying the triple `{text, layout, fonts}` together forecloses cross-edition layer mixing by construction. Written test-first: a single-byte-flipped release text, a missing/modified font, a wrong font count, and a manifest that drifted from the Tanzil anchor each fail the gate before the gate body is implemented.

## Implementation notes

**TEST-FIRST (correctness-critical):** write `verify_asset_integrity_test.dart` first; the single-byte-flipped text rejection, the missing-font and modified-font rejections, the wrong-font-count rejection, the layout-mismatch rejection, and the manifest-drifted-from-Tanzil-anchor rejection must all exist and **fail** before `verifyAssetIntegrity` is implemented. This is the build-time guard behind R1 — a wrong diacritic ends the project — so the rejections are the spec, and the implementation is whatever makes them go red→green.

1. **Pure core function** → `tool/src/verify_asset_integrity.dart` (importable from both the gate entrypoint and the test; plain Dart, `package:crypto` + `dart:io`/`dart:typed_data` only — **no `flutter_test`, no widget binding, no rendering, no `FontLoader`**). Signature per engineering 08 §6: `void verifyAssetIntegrity(MushafEdition ed, ReleaseManifest release)`. Model `ReleaseManifest` as a thin, injectable read-side over the *published release artifacts on disk in CI* — `Uint8List get textBytes`, `Uint8List get layoutBytes`, `Uint8List fontBytes(int page)`, `bool hasFont(int page)`, `int get fontCount` — so the test drives it with an in-memory fake (no real release download, no network). The function is **total and report-collecting**, not throw-on-first: accumulate every mismatch into a `List<IntegrityFailure>` (sealed: `TextDrift`, `TextMismatch`, `LayoutMismatch`, `FontMissing(page)`, `FontMismatch(page)`, `FontCountWrong(expected, actual)`, `PageCountWrong(expected, actual)`) and surface them all, so one CI run reports every drifted file rather than only the first — but a non-empty list is a hard failure.

2. **The five assertions, every one enumerated** (engineering 08 §6 + 11 §9 + PRD §11.3 steps 1–2):
   - `sha256(release.textBytes) == ed.textSha256` — the pinned text matches the published release (**TextMismatch** otherwise).
   - `ed.textSha256 == kAuthoritativeTanzilUthmaniSha256` — the manifest text hash equals the authoritative Tanzil Uthmani anchor, so the binary-baked manifest cannot itself silently drift from the upstream authority (**TextDrift** otherwise). This anchor is a `const String` from E05-T01's manifest constants (sourced from `tanzil.net/download/`), never recomputed inside the gate.
   - `sha256(release.layoutBytes) == ed.layoutSha256` — the QUL layout dataset matches (**LayoutMismatch** otherwise).
   - `ed.pageCount == 604` for the Madani 15-line edition, and `release.fontCount == ed.pageCount` — **PageCountWrong**/**FontCountWrong** otherwise. Drive the font loop with `ed.pageCount` (the triple's parameter — domain-mushaf-text-integrity step 1: never hardcode 604 in the loop bound), and *separately assert* `ed.pageCount == 604` so a wrong edition is caught, not silently iterated.
   - `for (var p = 1; p <= ed.pageCount; p++)`: `release.hasFont(p)` (**FontMissing(p)** otherwise) and `sha256(release.fontBytes(p)) == ed.fontSha256[p]` (**FontMismatch(p)** otherwise) — all 604 fonts present **and** unmodified.

3. **Cross-edition mixing is foreclosed by verifying the triple together.** All three layers are checked against the **same** `MushafEdition ed` (one `mushafId` binds one `{text, layout, fonts}`); there is no code path that accepts a layout from one edition with fonts from another — a QPC-V2 font paired with a different edition's layout fails its per-page hash. Make this structural: `verifyAssetIntegrity` takes exactly one `ed`, and the gate entrypoint resolves it from `EmbeddedManifest.core` (a single edition), never from two sources.

4. **Trust channel = the signed binary, never a sidecar.** The expected digests come **only** from `MushafEdition`/`EmbeddedManifest.core` (E05-T01's Dart constants compiled into the binary). The gate must never read, parse, or fall back to a `SHA256SUMS`/`.sha256` file shipped beside the release pack — an attacker who can swap the pack can swap its sidecar (engineering 09 §4; domain-asset-pack-integrity step 4). There is no file-read code path for "the expected hash."

5. **Hashing primitive.** Reuse E05-T03's `sha256OfFile` shape (chunked `startChunkedConversion`, lowercase hex) where the input is a file; for the in-memory `Uint8List` release-bytes read here, `sha256.convert(bytes).toString()` is acceptable (the release artifacts are bounded, and the gate runs on a CI runner, not low-end Android) — but **SHA-256 only**, never `md5`/`sha1`, and the comparison is exact lowercase-hex string equality (no prefix/substring match, no constant-time need — integrity, not a secret).

6. **The gate entrypoint** → `tool/verify_asset_hashes.dart` (the name engineering 11 §9 pins). REUSE SPDX header (`GPL-3.0-or-later`), `Future<void> main()`: resolve `ed` from `EmbeddedManifest.core`, build the `ReleaseManifest` over the published release artifacts present in the CI workspace, call `verifyAssetIntegrity`, then for each accumulated `IntegrityFailure` `stderr.writeln('::error::…')` (GitHub Actions annotation: `INTEGRITY:` for a per-file hash mismatch, `TEXT DRIFT:` for the Tanzil-anchor drift, `FONT MISSING/COUNT:` for the font-presence/count failures) and `exit(1)` if any failure was collected — release-blocking. No `print` of user data (there is none); the gate hashes bytes and is **locale/madhhab-blind** (eng-add-ci-check §10).

7. **Wire into E01's job shapes — author to plug in, don't reinvent.** Add the gate as a step in E01's existing `restraint` job (per the §8 gate→job mapping: gate 1 → `restraint` SHA-256 re-hash) and ensure the tag-triggered `release.yml` re-runs it; pin `subosito/flutter-action@v2` to the README-baseline `flutter-version` with `channel: stable` (eng-add-ci-check §2). The gate runs `dart run tool/verify_asset_hashes.dart` against the locally-present published release artifacts (downloaded in the workflow step *before* this gate, from the pinned immutable release tag) — it must **never** fetch the Quran assets at *app-build* time, and it adds **no** runtime network client (eng-add-ci-check §9; domain-asset-pack-integrity step 8). The job is `restraint`, not `golden` (no rendering); `continue-on-error` is forbidden.

8. **Necessary, not sufficient.** Document at the gate's head and in the epic risk that this CI check guards against *regression* of an already-correct reference — it does **not** prove the reference muṣḥaf is itself correct. The qualified ḥāfiẓ/scholar on-device proof (PRD §20.8) stays a release-blocking gate handled at **E20 release-readiness**; the pixel-fidelity proof is **E05-T11** (gate 2). This task makes a *drift* impossible to ship silently between reviews.

9. **Pitfalls to avoid:** hardcoding `604` as the font-loop bound instead of driving it with `ed.pageCount` while *separately* asserting `ed.pageCount == 604` (the triple's parameter is the source of truth; a hardcode hides a wrong-edition swap); throw-on-first-mismatch instead of accumulating (a CI run should report *every* drifted file, not just the first); reading or falling back to a sidecar `SHA256SUMS`/`.sha256` (the trust channel is the binary manifest); importing `md5`/`sha1` or comparing a truncated hex prefix; importing `flutter_test`/`FontLoader`/anything that *renders* (this is the pure byte-gate — pixels are E05-T11); recomputing the Tanzil anchor inside the gate instead of comparing against the pinned `const` (the anchor must be a fixed, reviewed constant, not a moving derivation); calling `gh attestation verify`/a Sigstore/minisign client in the gate's *required* path (attestation is audit-only — engineering 09 §4); floating the SDK on the job (re-renders nothing here, but breaks the pinned-toolchain rule and the reproducibility claim); `continue-on-error: true` on the gate, or skipping it on the release job (gate 1 is release-blocking); the gate "interpreting" bytes by locale/madhhab (it hashes, it does not read meaning).

## Acceptance criteria

- [ ] `verifyAssetIntegrity(MushafEdition ed, ReleaseManifest release)` exists in `tool/src/verify_asset_integrity.dart` as **pure Dart with no rendering** (no `flutter_test`, no widget binding, no `FontLoader`; imports `package:crypto` + `dart:io`/`dart:typed_data` only, verifiable by grep).
- [ ] The function is **total and report-collecting**: it accumulates a `List<IntegrityFailure>` (sealed `TextDrift`/`TextMismatch`/`LayoutMismatch`/`FontMissing`/`FontMismatch`/`FontCountWrong`/`PageCountWrong`) covering every drifted file in one pass; a non-empty list is a hard failure.
- [ ] It asserts `sha256(release.textBytes) == ed.textSha256` **and** `ed.textSha256 == kAuthoritativeTanzilUthmaniSha256` — both the release-match and the upstream-Tanzil anchor — and fails on either drift.
- [ ] It asserts `sha256(release.layoutBytes) == ed.layoutSha256`.
- [ ] It asserts exactly **604** fonts present and unmodified: `ed.pageCount == 604`, `release.fontCount == ed.pageCount`, and for every page `1..ed.pageCount` the font is present (`hasFont`) and `sha256(release.fontBytes(p)) == ed.fontSha256[p]`.
- [ ] The font loop bound is `ed.pageCount` (the triple's parameter), with `ed.pageCount == 604` asserted **separately** — no hardcoded `604` loop bound.
- [ ] The expected digests come only from `MushafEdition`/`EmbeddedManifest.core` (the binary-baked manifest); there is **no** code path that reads a sidecar `SHA256SUMS`/`.sha256`, and the Tanzil anchor is a pinned `const String`, never recomputed in the gate.
- [ ] Only one `MushafEdition` is verified per run; all three layers are checked against the same `ed`, so a layout-from-edition-A + fonts-from-edition-B combination fails (cross-edition mixing foreclosed).
- [ ] SHA-256 only (no `md5`/`sha1` import anywhere in the gate); comparison is exact lowercase-hex string equality.
- [ ] `tool/verify_asset_hashes.dart` carries the REUSE SPDX header (`GPL-3.0-or-later`), resolves `ed` from `EmbeddedManifest.core`, emits a `::error::` annotation per failure, `exit(1)`s on any failure, prints no user data, and is locale/madhhab-blind (hashes bytes, does not interpret them).
- [ ] The gate is wired into E01's **`restraint`** job (per the §8 gate→job mapping, PRD §20 gate 1) and re-run by the tag-triggered release workflow; the job pins `subosito/flutter-action@v2` to the README-baseline `flutter-version` with `channel: stable`; it is **not** `continue-on-error`; it fetches no Quran assets at app-build time and adds no runtime network client.
- [ ] A header comment (and the epic risk) records that the gate guards regression only — the scholar on-device proof (PRD §20.8, E20) and the pixel-fidelity visual-diff (E05-T11) are separate, non-substitutable gates.

## Tests

`tool/test/verify_asset_integrity_test.dart` — plain `package:test` (`dart test`, **no `flutter_test`**, no widget binding), run in CI's `fast`/`restraint` lane on every PR. Built over an in-memory `FakeReleaseManifest` and a fixture `MushafEdition` (synthetic tiny "text"/"layout" byte blobs + a small set of synthetic per-page font blobs with known digests, and a separate full-`pageCount` fixture for the count assertions) so the suite is **fully offline** — no real release download, no network, no real KFGQPC fonts (those are E05-T11's pixel concern). The throwing `HttpOverrides` from the test bootstrap stays green (this gate touches no socket). Required cases, written **FIRST**:

- **Happy path:** a manifest whose `textBytes`/`layoutBytes`/all `fontBytes(p)` hash exactly to `ed.textSha256`/`ed.layoutSha256`/`ed.fontSha256[p]`, with `ed.textSha256 == kAuthoritativeTanzilUthmaniSha256` and `ed.pageCount == release.fontCount == 604`, yields an **empty** failure list (the only green state).
- **Single-byte-flipped text → rejected:** flip one byte of `release.textBytes`; the digest differs by the avalanche property → exactly one `TextMismatch`, non-empty list, `main` would `exit(1)`.
- **Manifest drifted from Tanzil anchor → rejected:** `ed.textSha256` matches the release but `≠ kAuthoritativeTanzilUthmaniSha256` → `TextDrift` (proves the manifest cannot silently drift from upstream even when the release self-agrees).
- **Layout mismatch → rejected:** mutate `release.layoutBytes` → `LayoutMismatch`.
- **Missing font → rejected:** `hasFont(p)` false for some page → `FontMissing(p)` (a missing font is never silently skipped).
- **Modified font → rejected:** flip a byte of one `fontBytes(p)` → `FontMismatch(p)`.
- **Wrong font count → rejected:** `release.fontCount != ed.pageCount` (e.g. 603 or 605) → `FontCountWrong`; and a fixture with `ed.pageCount != 604` → `PageCountWrong` (the Madani 15-line invariant).
- **Cross-edition mixing → rejected:** fonts from a second fixture edition against the first edition's `fontSha256` → per-page `FontMismatch` (one `mushafId` binds one triple).
- **Anchors are constants:** assert `kAuthoritativeTanzilUthmaniSha256` is a 64-char lowercase-hex `const` (a regression guard that the anchor is fixed, not computed at runtime).
- **All failures reported in one pass:** a manifest with a flipped text byte **and** a missing font yields a list containing **both** `TextMismatch` and `FontMissing` (report-collecting, not throw-on-first).
- **Offline guard:** the suite imports no `package:dio`/`package:http`/`dart:io HttpClient` (grep-clean); it reads only in-memory fixtures.

CI integration: the gate runs as `dart run tool/verify_asset_hashes.dart` in E01's `restraint` job against the locally-present published release artifacts and in the tag-triggered `release.yml`; a deliberately-tampered fixture release in a CI smoke step proves the gate **exits non-zero** (a gate that cannot fail asserts nothing).

## Definition of Done

- [ ] All acceptance criteria met; `verify_asset_integrity_test.dart` written test-first (rejections red before the gate body), green locally and in CI on every PR; the gate runs in E01's `restraint` job and the release workflow and blocks publish on red.
- [ ] **Offline / no-network (non-negotiable):** the gate adds no network client and fetches no Quran assets at app-build time — it re-hashes release artifacts already present in the CI workspace from the pinned immutable tag; the suite imports no networking symbol and the throwing `HttpOverrides` bootstrap stays green.
- [ ] **No AI / no microphone:** nothing in the gate or its tests uses AI, ASR, or audio; the gate hashes bytes only.
- [ ] **Quran text fidelity (existential):** the gate fails the build on any text/layout/604-font SHA-256 drift vs the release **and** on any drift of the pinned text hash from the authoritative Tanzil Uthmani anchor; a single-byte-flipped text and a missing/modified font are gate-**rejected**; the trust root is the binary-baked manifest, never a sidecar; SHA-256 only (no MD5/SHA-1).
- [ ] **Cross-edition mixing foreclosed:** all three layers are verified against one `MushafEdition`; a layout/font combination from different editions fails; `ed.pageCount == 604` is asserted for the Madani 15-line edition.
- [ ] **RTL + fa/ckb/ar localization:** N/A by construction — a CI gate renders no user-facing string and stays locale/madhhab-blind (it hashes bytes, it does not interpret or display them); no ARB key is added.
- [ ] **Accessibility:** N/A by construction — no UI surface in this task.
- [ ] **Sect-neutral adab:** the gate interprets no byte by locale/madhhab/riwāyah — it only checks the pinned `riwayah`'s triple is intact; it renders no muṣḥaf and adds no decoration; it documents that it is necessary-not-sufficient (the scholar on-device proof, PRD §20.8/E20, and the E05-T11 pixel visual-diff remain separate gates).
- [ ] **Nothing safe to drop:** the gate is correctness-critical and release-blocking — never `continue-on-error`, never skipped on the release job, never weakened to make a build green.
- [ ] **Deterministic tests:** the suite is pure `package:test` over in-memory fixtures with fixed synthetic digests — no real release download, no network, no clock, no real fonts; the same inputs always produce the same failure list; the gate script carries the REUSE SPDX header, typed errors, is `dart format`/analyzer clean, and prints no user data.

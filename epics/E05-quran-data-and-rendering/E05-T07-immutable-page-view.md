## Goal

`MushafPageView` exists in `packages/quran` as the **dumb, immutable** renderer of one mu·π£·∏•af page: a `StatelessWidget` that draws each `GlyphLine` with `Text(line.glyphCodes, textDirection: TextDirection.rtl, style: TextStyle(fontFamily: qpcFontFamily(page), fontFamilyFallback: const []))` ‚Äî the font selection **is** the shaping; the OS shaper is never asked to lay out Quran text. Glyph codes stay opaque (no normalize/split/search/log-as-verse). A pure `assemblePage(pageNumber, layout)` groups words **strictly** by `page-{p}-line-{l}` from the QUL geometry, **never** by verse, with no `softWrap` / width-wrap / `TextPainter` line-breaking. The page lives in its own pipeline ‚Äî **no** `type.*` token, **no** shared `TextStyle`, **no** `MediaQuery.textScalerOf` (OS text-scale). A missing glyph surfaces as **visible tofu** (caught by E05-T11's visual-diff); the view **refuses to render any unverified asset**, deferring to the integrity layer rather than drawing an unverified page. This task ships the glyph layer + page assembly only; the overlay painter is E05-T08, and zoom/theme/RTL-paging/riwƒÅyah chrome is E05-T09.

## Implementation notes

TEST-FIRST (correctness-critical): the line builder and `assemblePage` are the sacred-text seam this task exists to make safe. Write, **before** the implementation body, the widget test asserting every mu·π£·∏•af line carries `fontFamilyFallback: const []` (no fallback ever reaches the sacred path) and the `assemblePage` unit asserting a line containing words from two different ayƒÅt stays one line grouped by `page-{p}-line-{l}` (never split by verse). Both must exist and fail first.

1. **Files & package** ‚Üí `packages/quran/lib/src/render/mushaf_page_view.dart` (the `MushafPageView` widget + the private `_GlyphLayer`/`_buildGlyphLine`) and `packages/quran/lib/src/render/assemble_page.dart` (the pure `assemblePage`), both barrel-exported from `lib/quran.dart`. Rendering, layout geometry, and glyph-font handling live **only** in `packages/quran` (enforced by `tool/check_quran_isolation.sh`, engineering 02 ¬ß). `qpcFontFamily` / `GlyphLine` / `LineType` / `ImmutableGlyphPage` are the `quran`-module types defined alongside font registration (E05-T06 / engineering 08 ¬ß2/¬ß3) ‚Äî import them, define exactly one copy of the `QPC_P${page.padLeft(3,'0')}` convention.

2. **`assemblePage`** ‚Üí `List<GlyphLine> assemblePage(int pageNumber, MushafLayout layout)`, exactly per engineering 08 ¬ß3: read `layout.wordsOnPage(pageNumber)` (each word already carries `lineNumber`, `position`, the opaque glyph ref), bucket into `byLine[w.lineNumber]`, then for each `lineNo` in **sorted** order emit `GlyphLine(pageNumber: pageNumber, lineNumber: lineNo, type: layout.lineType(pageNumber, lineNo), glyphCodes: words.map((w) => w.glyphCode).join())`. Grouping is **strictly** by `page-{p}-line-{l}` ‚Äî **never** by verse boundary (one mu·π£·∏•af line mixes words from multiple ayƒÅt). The join is plain `String` concatenation of opaque glyph codes ‚Äî **not** shaping, **not** a `TextSpan` split. The function is pure (no `BuildContext`, no IO, no clock); it consumes the geometry E05-T05 loaded and is the seam E05-T11's golden harness drives. `pageCount`/`lineCount` are read from the edition/QUL data ‚Äî never a literal `604`/`15`.

3. **The line builder** (`_buildGlyphLine(GlyphLine line)`), exactly per engineering 08 ¬ß2 / domain-mushaf-text-integrity step 3:
   ```dart
   Text(
     line.glyphCodes,                          // opaque QPC codes ‚Äî NEVER normalize/split/search/log as text
     textDirection: TextDirection.rtl,
     softWrap: false,                          // ¬ß3 no runtime line-breaking on Quran text
     maxLines: 1,
     style: TextStyle(
       fontFamily: qpcFontFamily(line.pageNumber), // the font IS the typeset page
       fontFamilyFallback: const <String>[],  // ¬ß2 no fallback on the sacred path ‚Äî missing glyph => visible tofu
       // page-font metric from the reader's own size, NOT a type.* token and NOT MediaQuery.textScalerOf
     ),
   )
   ```
   - `fontFamilyFallback: const <String>[]` is non-negotiable and on **every** line ‚Äî a single fallback entry re-shapes the sacred string the moment a glyph is missing (engineering 08 ¬ß2 pitfall; the epic's first risk). A missing glyph **must** surface as visible tofu, not be papered over.
   - The glyph string is passed straight to `Text` ‚Äî **no** `String` munging, `split`, normalize, `toLowerCase`, regex, search, comparison, or `debugPrint`-as-verse anywhere near `glyphCodes`.
   - `softWrap: false` + `maxLines: 1` ensure Flutter never width-wraps the line; the line's extent is the printed line, justified by the bundled word geometry (T09 sizing), never by the shaper splitting on spaces.

4. **The glyph layer** ‚Üí a private `_GlyphLayer(StatelessWidget)` that lays the page's `GlyphLine`s as a `Column(mainAxisSize: MainAxisSize.min, crossAxisAlignment: CrossAxisAlignment.stretch, children: [for (final line in glyphPage.lines) _buildGlyphLine(line)])`. The order of lines is the dataset order (`lineNumber` ascending, fixed by `assemblePage`); the view **never** reorders, mirrors, or re-cuts lines.

5. **`MushafPageView`** ‚Üí a dumb `StatelessWidget` taking an already-resolved, **verified** `ImmutableGlyphPage` (its `lines` from `assemblePage`) and rendering the glyph `_GlyphLayer` under an explicit `Directionality(textDirection: TextDirection.rtl)`. **No** transform is applied here (zoom/`ColorFilter` is E05-T09); the overlay `CustomPaint` sibling is left as a documented seam for E05-T08 (this task paints no marker, holds no `OverlayMarker`). The view holds **no** `type.*` token, **no** shared/inherited `TextStyle`, and **never** reads `MediaQuery.textScalerOf(context)` ‚Äî the mu·π£·∏•af is its own pipeline (engineering 08 ¬ß5; typography ¬ß1/¬ß7).

6. **Refuse-unverified is the integrity layer's verdict, surfaced here as absence.** This view renders only an `ImmutableGlyphPage` it was handed; if the asset for a page is unverified, E05-T06's font registration never produced its `QPC_P###` family and/or the resolve path (E05-T09) hands no verified page ‚Äî so the view shows the calm "refuse to render Quran text" state (T09 chrome), never an unverified page and never a substitute font. A missing `QPC_P###` family makes `fontFamilyFallback: const []` surface as tofu, which is the intended fail-loud behaviour. This task does **not** invent its own retry/refusal copy (that is E05-T04/T09 `l10n`); it makes the absence visible, never papered over.

7. **No network, no IO, no clock, no AI.** `MushafPageView` and `assemblePage` open no socket, read no file at render time, call no `DateTime.now()`, and touch no AI/ASR/audio. The verified glyph page and geometry are handed in from the `/quran` providers (authored with eng-create-riverpod-store at the reader composition root ‚Äî E05-T09), not constructed in this view.

8. **Pitfalls to avoid:** any `fontFamilyFallback` entry (even one) on a mu·π£·∏•af line; styling the page with a `type.*` token / `Theme.of(context).textTheme` / a shared `TextStyle`; letting `MediaQuery.textScalerOf` reach the page (it must not reflow with OS text-scale); `softWrap: true` / default soft-wrap / a `TextPainter`/`RichText`-with-wrapping on Quran text; grouping `assemblePage` by verse/ayah instead of `page-{p}-line-{l}`; splitting `glyphCodes` into `TextSpan`s or per-word `Text`s (a fragmented Arabic-script run triggers diacritic-clipping); normalizing/searching/logging `glyphCodes`; **mirroring/flipping/rotating** the glyph layer for RTL (the line's `TextDirection.rtl` is the only direction handling; the page is never reflected ‚Äî eng-rtl-and-bidi-layout step 9); applying any `intl` `NumberFormat` to a glyph (ayah numbers on the page are part of the immutable layer); hardcoding `604`/`15` instead of reading `pageCount`/`lineCount`/the dataset; importing `dio`/`http`/`HttpClient`/`drift` (this file is `packages/quran` ‚Äî outside the networking-allowed `packages/assets` and the Drift-only `packages/data`); painting an overlay/marker here (that is E05-T08); applying zoom/`ColorFilter`/chrome here (that is E05-T09).

## Tests

`packages/quran/test/render/mushaf_page_view_test.dart` and `packages/quran/test/render/assemble_page_test.dart` (mirror the source paths), `flutter_test` (the bootstrap installs E01's throwing `HttpOverrides`; this suite must never trip it ‚Äî rendering is offline). The widget test pumps `MushafPageView` under an RTL `Directionality` over a small synthetic `ImmutableGlyphPage` (a few `GlyphLine`s with placeholder opaque codes); `assemblePage` is unit-tested over a synthetic `MushafLayout` fake. Required cases, the first two written FIRST:

- **Empty fallback on every line (TEST-FIRST, central case)**: pump a page of several `GlyphLine`s; walk every realized `Text`/`RichText` widget under `MushafPageView` and assert each one's `style.fontFamilyFallback` is `const <String>[]` (empty) and its `style.fontFamily == qpcFontFamily(line.pageNumber)` ‚Äî proving no fallback ever reaches the sacred path and the font selection is the shaping.
- **Group by page-line, never by verse (TEST-FIRST)**: an `assemblePage` over a layout whose line 3 holds the last words of ayah 5 **and** the first words of ayah 6 yields exactly one `GlyphLine` for `lineNumber: 3` (its `glyphCodes` = the concatenation of all those words' codes, in `position` order); no verse boundary splits the line; lines come back in ascending `lineNumber`.
- **Opaque glyph codes untouched**: the `GlyphLine.glyphCodes` reaching the rendered `Text` is byte-identical to the assembled string (no normalize/re-encode/trim); a code containing the QPC special-Unicode range passes through verbatim.
- **No runtime line-breaking**: each line's `Text` has `softWrap: false` and `maxLines: 1`; `assemblePage` invokes no width/`TextPainter` computation (the function takes no width/constraints argument); changing the available width in a widget test never changes the number or content of lines.
- **Own pipeline / no text-scale reflow**: pumping the same page under `MediaQuery(textScaler: TextScaler.linear(2.0))` produces an **identical** widget tree of lines (no `type.*` token resolved, `MediaQuery.textScalerOf` not read) ‚Äî the mu·π£·∏•af does not reflow with OS chrome text-scale.
- **`pageCount`/`lineCount` from data**: `assemblePage` over a 5-line synthetic page emits 5 lines; over a 16-line synthetic edition emits 16 ‚Äî proving no `604`/`15` is baked in (swappability, R2).
- **Not mirrored**: the line order and per-line `TextDirection.rtl` are identical whether ambient `Directionality` is rtl or ltr (the mu·π£·∏•af direction is explicit per line, never derived/flipped from chrome); the glyph layer is never wrapped in a `Transform`/mirror in this task.
- **Offline guard**: the suite runs with the throwing `HttpOverrides` installed and never trips it; no real network, no `rootBundle` load, no clock.

No 604-page real-font golden in this task ‚Äî that is the E05-T11 visual-diff gate (which renders this exact `MushafPageView` with the real KFGQPC fonts on min-OS iOS+Android). Here the glyph page/geometry are synthetic fakes and no real font is loaded, so the unit/widget suite is fast and deterministic.

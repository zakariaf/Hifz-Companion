## Goal

The app's "today" becomes an **injected `CalendarDate`**, read from the device's local civil day exactly once at the app edge and threaded through the session ‚Äî never `DateTime.now()` in a scheduling path. `todayFor(DateTime now)` lives at the boundary and uses `.toLocal()` once (the same single conversion `civilDayOf` performs in E02-T03) to name the user's local day; a Riverpod `todayProvider` exposes the resulting `CalendarDate`, and every engine/repository call site receives `ref.read(todayProvider)`. The un-overridden provider reads the real clock once at session start; a test or golden run swaps it for a fixed date via `overrideWithValue` (or a `FixedClock`-style literal), so the schedule is reproducible on any runner in any zone. "Today" is captured at session start and never re-derived mid-session ‚Äî a midnight rollover cannot re-shuffle an open Today screen. No `timezone`/IANA dependency is added.

## Implementation notes

TEST-FIRST is not required for this thin boundary (it is plumbing, not the off-by-one-day arithmetic of T01/T02), but the **override behaviour must be proven by test**: a fixed-date `overrideWithValue` makes `ref.read(todayProvider)` return that literal date with zero clock read ‚Äî write that assertion alongside the implementation.

1. **`todayFor` file**: `packages/data/lib/src/dates/today_for.dart` (the layer-2 app-edge module that owns the clock side effect ‚Äî **not** `engine/`, which reads no clock). Mirror the placement of `civilDayOf` from E02-T03 (same `dates/` subfolder, same package); if T03 placed `civilDayOf` in a shared edge file, co-locate `todayFor` beside it. `CalendarDate todayFor(DateTime now) { final local = now.toLocal(); return CalendarDate.ymd(local.year, local.month, local.day); }` ‚Äî verbatim 07 ¬ß5. It depends on `engine`'s `CalendarDate` (downward only); it imports **no** `package:flutter`, no `dart:io`, no `intl`/`hijri`/`shamsi_date`.
2. **The single `.toLocal()`**: `todayFor` reads the local zone **once** via `now.toLocal()` ‚Äî the same boundary discipline as `civilDayOf` (¬ß3). It does **not** use `DateTime.utc`-derived `(y,m,d)` for "today" (that would name the UTC day; a 23:00-local review in Tehran is *today*, not tomorrow). A why-comment cites 07 ¬ß5 and names the refusal of a UTC `(y,m,d)` "today".
3. **`todayProvider`**: `final todayProvider = Provider<CalendarDate>((ref) => todayFor(DateTime.now()));` ‚Äî verbatim 07 ¬ß5, in the shell's date-provider wiring (`packages/app/lib/src/providers/today_provider.dart` or the composition-root providers file the shell owns, per `eng-define-service-boundary` ¬ß4). The provider body is a **thin wire only** ‚Äî no business logic, no clock arithmetic beyond the `todayFor` call. It is a `Provider` (synchronous DI value), **not** a `Notifier`/`AsyncNotifier`; there is no mutation, no persistence, no republish here.
4. **`DateTime.now()` appears exactly once.** The single `DateTime.now()` in the whole app's scheduling surface is inside this provider's default body (the one sanctioned clock read, the analogue of `civilDayOf`'s `.toLocal()` edge). Engine call sites, repositories, controllers, and the day-plan receive `ref.read(todayProvider)` / take `today: CalendarDate` as a parameter ‚Äî never `DateTime.now()`. The E01 CI banned-import grep flags any `DateTime.now()` outside this `todayFor`/provider edge.
5. **Capture-once / no mid-session rollover**: "today" is resolved once when the session reads it (or once at session start and threaded through). A long Today session does **not** re-read the provider on a timer; the next app session recomputes it. If the shell needs an explicit session anchor, read `todayProvider` once into the session/read-model at construction ‚Äî do not re-`watch` it on a clock tick. A why-comment cites 07 ¬ß5 (no midnight re-shuffle).
6. **The fixed-date double**: the test/preview override is a plain literal ‚Äî `todayProvider.overrideWithValue(CalendarDate.ymd(2026, 6, 16))` ‚Äî installed in a `ProviderContainer(overrides: [...])` or `ProviderScope(overrides: [...])`. No mock framework, no codegen (`eng-define-service-boundary` ¬ß5). If the broader app models the clock as the `Clock` interface from `eng-define-service-boundary` (`abstract interface class Clock { CalendarDate today(); }`), the override is a `FixedClock(CalendarDate.ymd(...))`; otherwise the `Provider<CalendarDate>` override is sufficient and is what 07 ¬ß5 specifies. Match whichever shape E02-T03 / E07 established for the clock boundary ‚Äî do **not** introduce a second clock abstraction.
7. **Pitfalls to avoid**: deriving "today" from a UTC `(y,m,d)` instead of `.toLocal()` (names the wrong civil day across midnight ‚Äî the ¬ß3/¬ß5 refusal); calling `DateTime.now()` anywhere reachable from the engine, a repository, a controller, or a view (the determinism break ‚Äî only the provider default may); making `todayProvider` a `Notifier`/`AsyncNotifier` or putting business logic in it (it is a thin DI wire); re-`watch`ing `todayProvider` so a midnight tick re-shuffles an open session; adding the `timezone`/IANA package (dead weight ‚Äî we never convert across zones); placing `todayFor` in `engine/` (the engine reads no clock ‚Äî this is the data/app-edge layer); reaching `today` through a global singleton instead of the provider.

## Acceptance criteria

## Tests

Two thin suites ‚Äî the pure day function (`package:test`) and the provider override (`flutter_test`/`ProviderContainer`). The shared throwing-`HttpOverrides` offline bootstrap (E01-T06) stays installed; neither suite touches the network by construction. REUSE SPDX header on each file. Fixed dates are constructed `CalendarDate.ymd(...)` / explicit `DateTime` literals ‚Äî no host wall clock is read in an assertion.

`packages/data/test/dates/today_for_test.dart` (`package:test`, mirrors the source name):
- **Local-day naming**: `todayFor(DateTime(2026, 6, 16, 9, 0))` (a local-zone literal) returns `CalendarDate.ymd(2026, 6, 16)`.
- **23:00-local lands on the local day (the ¬ß5/T10 boundary intent)**: a `DateTime` constructed at `23:00` local on a given day returns *that* local day's `CalendarDate`, never the next UTC day ‚Äî assert `todayFor` does not roll forward. (The hostile-timezone variant ‚Äî same instant named differently under `TZ=Asia/Tehran`/`Pacific/Kiritimati`/`UTC` ‚Äî is **E02-T08/T10**; this suite pins the local-naming contract.)
- **`.toLocal()` is applied once**: a UTC instant passed in is named by its *local* civil day (`instant.toLocal()`), matching the `civilDayOf` boundary of E02-T03 for the same moment ‚Äî `todayFor(utcInstant) == civilDayOf(utcInstant, localNow: utcInstant)`.

`packages/app/test/providers/today_provider_test.dart` (`flutter_test` / `ProviderContainer`, mirrors the source name):
- **Fixed-date override**: `ProviderContainer(overrides: [todayProvider.overrideWithValue(CalendarDate.ymd(2026, 6, 16))])`; `container.read(todayProvider) == CalendarDate.ymd(2026, 6, 16)` with no clock read ‚Äî this is the deterministic double the engine goldens (E04) and the DST matrix (E02-T08) rely on.
- **Stable within a session**: two reads of the overridden `todayProvider` return the identical `CalendarDate` (the provider is a synchronous DI value, not recomputed per read); demonstrates the capture-once contract a session anchor depends on.
- **Default reads the clock once (smoke)**: an un-overridden container's `todayProvider` returns a `CalendarDate` equal to `todayFor(DateTime.now())` evaluated at read time ‚Äî a single, sanctioned clock read; no assertion depends on the *value* (which is host-dependent), only that it is a `CalendarDate` produced via the `todayFor` edge.

The full schedule-zone-independence matrix (T4/T5: a fixed `today` drives a byte-identical schedule under `TZ=Asia/Tehran`/`Pacific/Kiritimati`/`UTC` and across a DST-change date) belongs to **E02-T08** and consumes this provider ‚Äî it is not this task. The 23:00-local cross-zone vector (T10) is **E02-T10**.

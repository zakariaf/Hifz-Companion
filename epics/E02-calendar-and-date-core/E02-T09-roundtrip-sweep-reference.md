## Goal

Pin the conversion-fidelity matrix over the `CalendarPresenter` (E02-T05) and its hardened Hijri path (E02-T07) so the round-trip is identity at scale and the published reference values match. Three vectors land as a committed suite: **T6** ‚Äî `CalendarDate ‚Üí Jalali ‚Üí CalendarDate` is identity for **every** date in a multi-decade sweep; **T7** ‚Äî `CalendarDate ‚Üí Hijri(Umm al-QurƒÅ) ‚Üí CalendarDate` is identity for every date **in** the supported AH range and falls back to a **Gregorian label** out of range (never throws); **T9** ‚Äî a documented Gregorian‚ÜîHijri (and Gregorian‚ÜîJalƒÅlƒ´) **reference pair** matches the published Umm al-QurƒÅ / JalƒÅlƒ´ value. Every reference value comes from an **independently published** conversion, never the `hijri`/`shamsi_date` library asserting against its own forward+inverse, and the Hijri sighting caveat (from E02-T07) is carried in copy beside the T9 vector. This task is the sweep harness over conversions T05/T07 already built; it adds no new conversion path and no new copy.

## Implementation notes

TEST-FIRST (correctness-critical): this is the conversion-fidelity gate the whole epic's calendar layer rests on ‚Äî a silent round-trip drift corrupts every dated surface in E10‚ÄìE19, and "the library asserting against itself" is the exact T9 trap. This task is *almost entirely* test code; write the T6 sweep, the T7 in-range/out-of-range sweep, and the T9 independently-sourced reference case **first** and confirm they exercise the real `CalendarPresenter`/conversion helpers (red if a conversion regressed) before considering the task done. No production conversion code is authored here unless the sweep surfaces a gap that belongs to T05/T07 (then fix it there, with its own test).

1. **File**: a new test file `packages/l10n/test/calendar/calendar_roundtrip_sweep_test.dart`, mirroring the source tree under the `l10n` package `test/` (per 11 ¬ß2 / E01-T03), alongside the E02-T05 `calendar_presenter_test.dart` and E02-T07 `calendar_presenter_hijri_test.dart`. **No new production file** ‚Äî the conversions under test already exist in `packages/l10n/lib/src/calendar/calendar_presenter.dart`. REUSE SPDX header (`GPL-3.0-or-later`).
2. **Runner**: `flutter_test` (the `l10n` layer depends on `intl`/`shamsi_date`/`hijri` and Flutter localizations, so the engine's `package:test`-only rule does not apply here ‚Äî per `eng-write-dart-test` ¬ß2 the *engine* uses `package:test`, the *display layer* uses `flutter_test`). The round-trip helpers themselves are pure, so the cases need no widget binding beyond what `flutter_test` provides; no `pumpWidget` for a pure conversion.
3. **Round-trip helpers (test-local, over the presenter's conversion path)**: assert identity on the **`CalendarDate` serial day**, not on a localized label. For T6: `CalendarDate ‚Üí DateTime.utc(y,m,d) ‚Üí Jalali.fromDateTime(g) ‚Üí toGregorian() ‚Üí CalendarDate.ymd(...)`; for T7: the same shape via `HijriCalendar.fromDate(g)` ‚Üí `hijriToGregorian(...)` (Umm al-QurƒÅ). Use the **same `DateTime.utc(d.year, d.month, d.day)` pure bridge** the presenter uses (07 ¬ß1/¬ß4) ‚Äî never a local `DateTime(...)`. Keep these helpers thin and local to the test, calling the same library entry points the presenter does, so a regression in the presenter's conversion is caught; do not re-implement an epoch or a leap rule.
4. **T6 ‚Äî multi-decade JalƒÅlƒ´ sweep**: iterate `epochDay` from a fixed start `CalendarDate.ymd(2000, 1, 1)` to a fixed end `CalendarDate.ymd(2050, 12, 31)` (a >50-year span; widen only if cheap) via `addDays(1)`, and assert `roundTripJalali(d) == d` for every date ‚Äî including Nowruz boundaries, every leap configuration, and year edges. The span is **named constants** (`_sweepStart`/`_sweepEnd`) with a why-comment citing 07 ¬ß7 T6; no magic literal mid-loop. Keep the loop deterministic and fast (a half-century of integer round-trips is cheap); no wall clock, no randomness in the spine (a `glados` identity property over generated `(y,m,d)` triples is an allowed *addition*, not the replacement for the deterministic sweep).
5. **T7 ‚Äî Hijri in-range identity + out-of-range fallback sweep**: iterate the dates **inside** `[_hijriMinSupported, _hijriMaxSupported]` (the E02-T07 named bounds ‚Äî import/reference them, do **not** re-declare a second copy) and assert `roundTripHijri(d) == d`, covering each Hijri month boundary and a leap-year configuration. Then assert that a date **below** `_hijriMinSupported` and one **above** `_hijriMaxSupported` make `CalendarPresenter(..hijriUmmAlQura..).format(d)` return the **Gregorian** label and `returnsNormally` (never throw) ‚Äî the ¬ß6 totality contract, swept at the boundary rather than at a single point. Reuse E02-T07's range bounds as the single source of truth.
6. **T9 ‚Äî independently published reference pairs (the anti-self-assertion case)**: freeze a small set of `(Gregorian, Umm al-QurƒÅ Hijri)` and `(Gregorian, JalƒÅlƒ´)` reference tuples as in-file `const` records, **each with its published-source URL in a why-comment** (e.g. an academic/government Umm al-QurƒÅ converter and an Iranian Solar-Hijri reference ‚Äî never `HijriCalendar`/`Jalali`'s own output). Assert the presenter's converted `(year, month, day)` for each Gregorian input equals the published Hijri/JalƒÅlƒ´ triple. The Hijri case carries the E02-T07 sighting caveat in a `// 07 ¬ß6` comment beside the vector (the civil approximation may differ by a day by sighting); the assertion is on the **civil Umm al-QurƒÅ value**, asserting no observance/sighting ruling.
7. **Pitfalls to avoid**: asserting T9 against `HijriCalendar.fromDate(...).hijriToGregorian(...)` (proves only self-consistency ‚Äî the ¬ß7 refusal; use an **independent** published pair); building the sweep with `Duration`/`DateTime.add` instead of `addDays` (reintroduces the DST surface the epoch-day integer removes); round-tripping through a local `DateTime(y,m,d)` instead of `DateTime.utc` (bakes in the device zone); re-declaring the Hijri range bounds instead of consuming E02-T07's constants (two drifting copies); making the Hijri out-of-range case expect a throw (it must **fall back**, per ¬ß6 ‚Äî `returnsNormally`); inventing a `C-###` CLAIMS id or a citation token for the carried caveat (E19 grades it; E02-T07 owns the wording); a `glados` property with a fixed lucky seed that hides a regression; touching the network in any conversion path (both packages are bundled ‚Äî the throwing-`HttpOverrides` guard must stay green); pushing this sweep into a slow `integration_test` journey (it is a fast pure-conversion unit, not one of the four PRD journeys).

## Tests

This task **is** the test deliverable. `packages/l10n/test/calendar/calendar_roundtrip_sweep_test.dart`, `flutter_test`, the shared throwing-`HttpOverrides` offline bootstrap (E01-T06) installed, REUSE SPDX header, all dates as literals or `addDays` walks (no wall clock). Required cases, written **first** and confirmed to exercise the real presenter:

- **T6 ‚Äî JalƒÅlƒ´ round-trip is identity (multi-decade sweep)**: loop `epochDay` from `_sweepStart` (`CalendarDate.ymd(2000, 1, 1)`) to `_sweepEnd` (`CalendarDate.ymd(2050, 12, 31)`) via `addDays(1)`; for each `d`, `expect(roundTripJalali(d), d)`. Explicitly include Nowruz/year-boundary and JalƒÅlƒ´ leap-year dates in the swept range; assert the loop actually iterated the expected count (no off-by-one that silently skips the span).
- **T7 ‚Äî Hijri round-trip is identity in range**: loop every date in `[_hijriMinSupported, _hijriMaxSupported]` (E02-T07 bounds); `expect(roundTripHijri(d), d)`; cover each Umm al-QurƒÅ month boundary and a Hijri leap configuration.
- **T7 ‚Äî Hijri out-of-range falls back to Gregorian (never throws)**: for a date below `_hijriMinSupported` and one above `_hijriMaxSupported`, assert `CalendarPresenter(CalendarSystem.hijriUmmAlQura, locale).format(d)` equals the Gregorian label for that date and `expect(() => ...format(d), returnsNormally)`; a boundary/library-failure case also falls back, never propagates.
- **T9 ‚Äî independently published Umm al-QurƒÅ reference pairs**: for each frozen `(Gregorian, expected Hijri)` tuple (source URL in a why-comment, **not** library-generated), assert the presenter's converted `(hYear, hMonth, hDay)` equals the published triple; a comment beside the table carries the ¬ß6 sighting caveat and names each source.
- **T9 ‚Äî independently published JalƒÅlƒ´ reference pairs**: the same shape for `(Gregorian, expected JalƒÅlƒ´)` tuples from a published Solar-Hijri reference (not `shamsi_date`'s own output), asserting the converted `(jYear, jMonth, jDay)`.
- **Optional `glados` identity property**: an *addition* (not a replacement) ‚Äî over generated valid `(y,m,d)` triples, `roundTripJalali`/`roundTripHijri` (when in range) is identity, relying on shrinking; no fixed lucky seed.

The DST/timezone schedule matrix (T1‚ÄìT5) is **E02-T08** (the engine half); the numeral (T8) and local-day (T10) vectors and the date-gate CI wiring are **E02-T10**; the single-pair T9 smoke and the Hijri range-guard implementation are **E02-T05/T07**. This task is the multi-decade T6/T7 sweep and the independent-oracle T9 reference set only.

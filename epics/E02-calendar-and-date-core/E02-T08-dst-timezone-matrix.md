## Goal

The first half of the date-correctness release gate exists as a pinned, hostile-timezone test suite: T1–T5 from engineering/07 §7. `addDays` is proven DST-immune (`2026-03-07.addDays(1) == 2026-03-08`, exactly +1 `epochDay` across the US spring-forward); `daysUntil` is proven exact across both a spring-forward and a fall-back boundary (no 23h/25h `inDays` artifact); the trust-clamp shape (`due.epochDay ≤ ceiling.epochDay`) is proven over randomized `(S, R, cycle, today)` inputs via `glados`; and the *schedule* is proven byte-identical when the same injected `today` + state are built under `TZ=Asia/Tehran`, `TZ=Pacific/Kiritimati`, and `TZ=UTC`, and on a DST-change-date `today` against the no-DST control. The suite is wired into CI so it runs **under each DST zone explicitly** — a green run under `TZ=UTC` alone is never accepted as proof. A red vector here blocks release (PRD §20 gate 5).

## Implementation notes

TEST-FIRST (correctness-critical): this is a pure test/CI task — the artifacts it produces *are* the assertions. Every vector below is written and must be runnable before any helper it needs is finalized; T1/T2's spring-forward/fall-back vectors and T4/T5's cross-zone identity must exist and be capable of going red before the `TZ`-matrix runner is wired. No production engine code changes here; if T3 reveals a clamp that can exceed the ceiling, the fix lands in E04, not in this suite.

1. **DST/timezone vector file**: `engine/test/dst_timezone_matrix_test.dart` (mirrors the gate's name), **`package:test` only** — no `flutter_test`, no widget binding (eng-write-dart-test §2). All dates are literal `CalendarDate.ymd(...)`; nothing reads a wall clock (§3). The shared throwing-`HttpOverrides` bootstrap (E01-T06) is installed so any stray socket is a loud named failure (§8). REUSE `GPL-3.0-or-later` SPDX header.

2. **T1 — `addDays` DST-immune (spring-forward)**: assert `CalendarDate.ymd(2026, 3, 7).addDays(1) == CalendarDate.ymd(2026, 3, 8)` **and** `CalendarDate.ymd(2026,3,7).daysUntil(CalendarDate.ymd(2026,3,8)) == 1` — i.e. exactly +1 `epochDay` across the US spring-forward night (2026-03-08, when a naive 24h `Duration` add would land on the wrong civil day). This is the same vector E02-T02 ships focused on `nextDue`; here it is asserted on the value type directly and re-run under the zone matrix (notes 6–7).

3. **T2 — `daysUntil` exact across DST (both directions)**: pick a date pair straddling the spring-forward boundary and a pair straddling the fall-back boundary (e.g. across 2026-11-01, US fall-back), and assert `daysUntil` returns the exact integer calendar-day count — never the 23h (`inDays == 0`) spring-forward artifact nor the 25h fall-back artifact a `Duration`/`difference` would produce. Include a multi-day span that *contains* a transition (e.g. a 30-day span spanning the change) and assert it equals the exact day count, a same-day pair (`== 0`), and a reversed pair (negative). Each assertion names the transition it straddles so a red test points at the boundary.

4. **T3 — trust-clamp shape over randomized inputs (`glados`)**: a `glados` property over generated `(S, R, cycle, today)` — the `scheduleCase` generator shape from eng-write-engine-golden-vector — asserting `due.epochDay <= ceiling.epochDay` for every generated input, with shrinking relied on for the minimal counterexample and **no fixed lucky seed**. This is the by-construction, date-side half of PRD §7.12 INV-1 (the property over real engine `Card` state lives in E04); restate the covenant in a comment (`// PRD §7.6: SR may only make a page MORE frequent — never later than the ceiling`). Where E04's clamp is not yet importable, drive the property through the `dueWithCeiling` primitive (E02-T02) over generated `(idealDays, ceilingDays, today)` — the same `min`-over-`epochDay` shape the real clamp resolves to — and note in the file that the full-engine INV-1 property is wired in E04. Reference `cycleCeilingDays`/`targetR` by name, never as literals.

5. **The `TZ`-matrix runner (T4/T5 core)**: a small test helper, `withProcessTimeZone(String tz, void Function() body)`, that sets the process zone for the duration of `body` and restores it after. On the Dart VM, `Platform.environment['TZ']` is read at process start, so the in-process override is not portable — therefore **the authoritative T4/T5 mechanism is the CI job running the *same* suite once per `TZ` env value** (note 9), and the in-suite helper is a best-effort local convenience that the CI matrix backstops. The suite must be *correct under whatever zone the process is in*: it asserts identity *between* zone runs by comparing each run's output to a single committed golden, so three CI invocations under three `TZ` values each compare to the same golden and any zone-dependence shows as one invocation diverging. Place the helper in `engine/test/support/timezone_matrix.dart`.

6. **T4 — schedule byte-identical across zones**: build the schedule with a fixed injected `today = CalendarDate.ymd(2026, 3, 8)` (a US DST-change date) and a pinned fixture state, then assert the result equals a single committed golden — `expect(buildToday(state, today), equals(goldenSchedule))`. Because the engine reads no clock and `today` is the same `CalendarDate` literal under every zone, the assertion passing under all three CI `TZ` invocations *is* the byte-identity proof (07 §7's illustrative test). Until E04's real `buildToday` is importable, drive a **date-pipeline stand-in** — the E02-T02 day-math primitives over the pinned fixture (e.g. mapping each fixture card's `(lastReviewDay, idealDays, ceilingDays)` through `dueWithCeiling`/`nextDue`) — and assert *its* output is zone-identical; leave a TODO + a doc-comment that E04 swaps in the real `buildToday` against the same golden, the runner unchanged. The golden is committed as a plain Dart fixture (a list of `(pageId, dueEpochDay)` rows), human-readable in the diff, never the engine asserting against its own live output.

7. **T5 — schedule DST-transition-independent**: assert the schedule built with `today` on the DST-change date (the 2026-03-08 spring-forward in a DST zone) equals the schedule built with the *same* fixed `today` + state as a no-DST control (e.g. the `TZ=UTC` / `Pacific/Kiritimati`-no-DST invocation). Same golden, same fixture; the only varying input across the compared runs is the process zone, which must not change the result. This is T4's sibling specialized to the transition date — it catches a regression where a transition date specifically perturbs the pipeline even when steady-state zones agree.

8. **Drive `today` through the injected provider, not a clock**: T4/T5 obtain `today` as a fixed `CalendarDate.ymd(...)` literal (the deterministic double E02-T04 ships via `todayProvider.overrideWithValue(...)`); the schedule build receives it as a parameter. No part of this suite calls `DateTime.now()` — that would make the cross-zone comparison depend on when CI runs. The point of setting `TZ` is solely to prove the *pipeline* ignores it, given a fixed `today`.

9. **CI wiring (the run-under-DST-zones requirement)**: extend the engine test CI job (or add a `date-gate` matrix leg) so the DST/timezone suite runs **once per zone** — `TZ=Asia/Tehran`, `TZ=Pacific/Kiritimati`, `TZ=UTC` — as separate invocations (a GitHub Actions `strategy.matrix.tz` over `dart test engine/ -P date-matrix`, or three explicit `env: { TZ: … }` steps). All three must pass against the same committed golden; the gate is **green only when every zone leg is green**, and a `TZ=UTC`-only run is explicitly not sufficient (07 §7 refusal). E02-T10 owns folding this into the full T1–T10 date-gate job and the dependency/allow-list checks; this task contributes the T1–T5 legs and the zone-matrix shape. Follow the CI-shape conventions in `docs/engineering/11-testing-strategy.md` §8 (the fast / golden / gate jobs and the PRD-gate→job mapping); do not invent a parallel runner.

10. **Pitfalls to avoid**: asserting only under one zone (the whole bug class hides in a no-DST zone — the suite must compare each zone run to a shared golden, and CI must run all three legs); using `DateTime.now()` for `today` (makes the cross-zone comparison time-dependent and non-reproducible); hand-typing the expected schedule golden from intuition rather than committing a generated, reviewed fixture (07 §7 refusal — and for T3/T4 the reference must not be the engine asserting against its own live output); letting a `Duration` or `DateTime.add` creep into a test helper "to build a vector" (reintroduces the exact off-by-one the gate exists to catch — build vectors with `CalendarDate.ymd`/`addDays` only); pinning one lucky `glados` seed in T3 that hides a clamp regression; introducing a second clock/timezone abstraction instead of the `todayProvider` double from E02-T04; adding the `timezone`/IANA package (the suite needs only the process `TZ`, not arbitrary-zone conversion).

## Tests

This task *is* the test suite; the artifacts and their cases:

`engine/test/dst_timezone_matrix_test.dart` (`package:test` + `package:glados`, mirrors the gate name) — the throwing `HttpOverrides` bootstrap installed; all dates literal `CalendarDate.ymd(...)`; no wall clock; runs in CI **once per `TZ` zone**:

- **T1 — addDays DST-immune (spring-forward)**: `CalendarDate.ymd(2026,3,7).addDays(1) == CalendarDate.ymd(2026,3,8)`; `ymd(2026,3,7).daysUntil(ymd(2026,3,8)) == 1` — exactly +1 `epochDay` across 2026-03-08.
- **T2 — daysUntil exact across DST (both directions)**: a spring-forward pair and a fall-back pair (around 2026-11-01) each yield the exact day count; a 30-day span containing a transition yields exactly 30; a same-day pair yields `0`; a reversed pair yields the negated count — each case naming the boundary it straddles.
- **T3 — trust-clamp shape (`glados`)**: `Glados<ScheduleCase>(any.scheduleCase).test(...)` (or over `(idealDays,ceilingDays,today)` against `dueWithCeiling`) asserting `due.epochDay <= ceiling.epochDay` for every generated input; shrinking relied on; covenant named in a comment; no fixed seed.
- **T4 — schedule byte-identical across zones**: `expect(buildToday(pinnedFixture, CalendarDate.ymd(2026,3,8)), equals(goldenSchedule))` (date-pipeline stand-in until E04), re-run under each `TZ` leg against the same committed golden — divergence in any zone leg fails the gate.
- **T5 — schedule DST-transition-independent**: the DST-change-date schedule equals the no-DST control schedule for the same fixed `today` + fixture; same golden.

`engine/test/support/timezone_matrix.dart` — the `withProcessTimeZone(tz, body)` local-convenience helper and the committed `goldenSchedule` / `pinnedFixture` fixtures (plain Dart `(pageId, dueEpochDay)` rows), REUSE SPDX header, no clock, no network.

CI: a `date-matrix` leg (or `strategy.matrix.tz` over the engine test job) runs the suite under `TZ=Asia/Tehran`, `TZ=Pacific/Kiritimati`, `TZ=UTC`; the gate passes only when every leg passes. No widget/golden/integration tests in this task (pure arithmetic + schedule identity, no UI, no DB). Offline is asserted by construction via the throwing `HttpOverrides`; nothing here touches the network.

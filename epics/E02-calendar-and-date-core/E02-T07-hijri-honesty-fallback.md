## Goal

Harden the Hijri path of the `CalendarPresenter` (built in E02-T05, in `packages/l10n/lib/src/calendar/calendar_presenter.dart`) so a Hijri date is rendered **only** as **Umm al-Qurā**, labelled as such with a localized "(Umm al-Qurā)" qualifier, and **every** `CalendarDate → Hijri` conversion is range-guarded: an out-of-range date returns the Gregorian label instead of throwing, so a date label can never crash a screen. Add the standing one-line **civil-approximation caveat** as an ARB string slot (fa/ckb/ar) that a calendar surface can render near a Hijri date or the Settings calendar picker — wording: an observance's start may differ by a day by sighting. Bake in **no** single-region sighting offset as a default, key **no** deadline/reminder/guarantee off a Hijri date being exact, and issue **no** fiqh or sighting ruling — sect/madhhab-neutral. The caveat copy is **handed to E19** for grading as a CLAIMS row; **no CLAIMS id is invented in this task**.

## Implementation notes

TEST-FIRST (correctness-critical): the range-guard is the exact "a date label crashes a screen" failure mode this task exists to remove, and T9 is the "library asserting against itself" trap. Write the **T7 in-range identity + out-of-range fallback** case and the **T9 independently sourced reference-pair** case **before** wiring `_hijriLabel`, and watch them fail (red) first.

1. **File**: edit `packages/l10n/lib/src/calendar/calendar_presenter.dart` (the presenter from E02-T05) — add/finish the private `_hijriLabel(DateTime g)` and the supported-range guard; **do not** create a new presenter. The Hijri conversion uses `package:hijri` (`HijriCalendar.fromDate(g)`, Umm al-Qurā) — already a `packages/l10n` dependency (E02-T05); **not** importable from `packages/engine` (the banned-import gate proves it).
2. **Supported-range constants**: declare `_hijriMinSupported` / `_hijriMaxSupported` as named `DateTime` bounds for the Umm al-Qurā tabulated range (per 07 §4 / §6 — the `hijri` family is tabulated ~1937–2076 CE; use the documented bound, **never** an inlined magic literal without a `// 07 §6` why-comment naming the source). Every real `due_at` falls comfortably inside; the guard exists for backup dates, cold-start "when memorized" inputs, and defensive totality.
3. **Range-guard + fallback** (the core of this task): `_hijriLabel` checks `g.isBefore(_hijriMinSupported) || g.isAfter(_hijriMaxSupported)` (and wraps the `HijriCalendar.fromDate` call in a typed `catch` for any library failure) and **returns the Gregorian label** (`_gregorianLabel(g, locale)`, E02-T05) on either path — never an exception, never an empty/garbled string, never a logged-off-device error. The presenter is **total**: `format(...)` always returns a valid localized label.
4. **The "(Umm al-Qurā)" qualifier**: the in-range Hijri label carries a localized "(Umm al-Qurā)" qualifier (an ARB string, fa/ckb/ar) so a Hijri date is never shown as "the Hijri date" in the absolute. Compose `day monthName year` from the `hijri` package's localized month tables (not `intl`'s Gregorian-only `DateFormat`), then the qualifier; numerals are remapped **downstream** by E02-T06.
5. **The standing civil-approximation caveat slot**: add **one** ARB key (e.g. `hijriCivilApproximationCaveat`) to `packages/l10n/lib/src/arb/app_ar.arb` (the template/base, `ar`) plus transcreated `fa` and `ckb` values — wording in plain, reverent, honest register: *the Hijri date is a civil (Umm al-Qurā) approximation; an observance's start may differ by a day by sighting*. It is a **caveat slot**, not a placement decision: a calendar surface (Settings picker, a date chip) renders it near the Hijri date — the where/when is **E16/E19**; this task ships the string and exposes it. Carry an `@hijriCivilApproximationCaveat` description noting `needs-scholarly-review (E19)` and "register as a CLAIMS row — no id here".
6. **No deadline/offset/ruling** (adab guardrails): the daily reminder and every schedule key off the **local civil day** (E02-T04), never a Hijri date being exact — assert nothing here re-routes them. Do **not** wire the `hijri` package's per-region `adjustments` map as a default; any such offset is an explicit later user/community act, never an app default (07 §6 pitfalls). The copy issues no fiqh ruling and names no sect/madhhab.
7. **Pitfalls to avoid**: a Hijri conversion that can throw to a widget (the §6 refusal — must fall back, never propagate); presenting the civil Umm al-Qurā date as the authoritative date of Ramaḍān/ʿĪd; baking one region's sighting offset in as "the" Hijri date; asserting T9 against `HijriCalendar`'s own forward+inverse (proves only self-consistency — use an **independent** published pair); concatenating raw ASCII digits into the label (numerals are E02-T06's downstream remap); inventing a `C-###` CLAIMS id or a citation in the caveat copy (E19 grades it); placing the caveat as mandatory/guilt-laden chrome (it is an honest informational note, dismissible context, never a gate).

- [ ] One **caveat ARB key** (e.g. `hijriCivilApproximationCaveat`) exists in `app_ar.arb` + `fa` + `ckb`, plain/reverent/honest, stating the date is a civil Umm al-Qurā approximation that may differ by a day by sighting; its `@`-description carries the `needs-scholarly-review (E19)` + "register as CLAIMS row, no id here" marker.

## Tests

`packages/l10n/test/calendar/calendar_presenter_hijri_test.dart` (mirrors the source under the `l10n` package `test/` tree), `package:test`/`flutter_test` as appropriate to the presenter's API, the shared throwing-`HttpOverrides` offline bootstrap installed (no date path opens a socket — both calendar packages are bundled), REUSE SPDX header. Required cases, the first two written **first**:

- **T7 — in-range Hijri round-trip is identity**: for a sweep of in-range dates (including each Hijri month boundary and a leap configuration), `CalendarDate → Hijri(Umm al-Qurā) → CalendarDate` is identity; the rendered label contains the "(Umm al-Qurā)" qualifier and the correct Umm al-Qurā `day/month/year`.
- **T7 — out-of-range falls back to Gregorian (never throws)**: a date below `_hijriMinSupported` and one above `_hijriMaxSupported` each return the **Gregorian** label; `format(...)` is asserted **not** to throw (`returnsNormally`) for both; a simulated `hijri`-library failure (or the boundary case) also falls back, never propagates.
- **T9 — independently published reference pair**: a frozen, **independently sourced** Gregorian↔Umm al-Qurā reference pair (constant documented in-file with its published source URL, not generated by `HijriCalendar`) matches the presenter's output exactly; a why-comment names the source and the sighting caveat (07 §6 / §7).
- **Caveat copy renders per locale (no ASCII digits, RTL)**: pump/format the caveat string under `Directionality.rtl` for `fa`/`ckb`/`ar`; assert the key resolves in all three (no missing-key fallback to `ar`), the register reads calm/honest, and any numerals in the string are locale digits with **zero** ASCII `[0-9]`.
- **Adab / neutrality guard (string-level)**: assert the caveat and qualifier copy contain **no** fiqh-ruling/sect/madhhab terms, no "must/should/don't", no guilt/loss phrasing, and no `C-###`/citation token — a banned-pattern grep over the new ARB values is green.
- **No deadline keyed off Hijri**: a unit assertion that the reminder/schedule path consumes only the injected local civil day (E02-T04), unaffected by `CalendarSystem.hijriUmmAlQura` being selected (the schedule is byte-identical across calendar systems — day-distance is calendar-invariant).

The **multi-decade** Jalālī+Hijri round-trip **sweep** (T6/T7 at scale) and the reference-**pair** harness are wired in **E02-T09**, which consumes this task's in-range/out-of-range behaviour and the documented reference pair; this task ships the Hijri-path behaviour and its two pinning vectors.

## Goal

The last two rows of the date-correctness matrix are pinned and the whole T1–T10 suite is wired into the release gate. **T8** formats a fixed `CalendarDate` in `fa`/`ckb`/`ar` and asserts Extended Arabic-Indic digits for `fa`/`ckb`, Arabic-Indic for `ar`, and zero ASCII digits in any rendered date. **T10** drives a 23:00-local review instant in a positive-UTC-offset zone through `civilDayOf` and asserts the resulting civil day is the **local** calendar date, never UTC's next date. Both vectors join T1–T7 and T9 in a single date-correctness suite that CI runs as part of [PRD §20 gate 5](../../docs/PRD.md): the engine half (`dart test engine/`) and the display half (`flutter test`) both run under the throwing-`HttpOverrides` offline guard that asserts no date or calendar path opens a socket, and a single red date vector blocks the release exactly as a missing ARB key does.

## Implementation notes

TEST-FIRST (correctness-critical): both T8 and T10 fail silently in the dangerous direction — a Latin digit leaking into a localized date, or a 23:00 review counted as tomorrow, each passes a test that only checks the output is "non-empty"/"a valid date". Write the `ckb`/`ar` no-ASCII assertions and the 23:00-local `civilDayOf` assertion **first**, watch them fail against any pre-existing leak, then confirm green against the T06/T03 implementations. This task adds no new production code beyond the test vectors and the CI wiring; if a vector goes red because production is wrong, the fix lands in the owning sibling (T06 for numerals, T03 for `civilDayOf`), not here.

1. **T10 — local-day boundary, `engine/` boundary helper.** The civil-day conversion `civilDayOf` lives at the app edge (E02-T03), built on the pure `engine/` `CalendarDate`. Construct a UTC instant for 23:00 local in a fixed **positive**-offset zone (Tehran, UTC+03:30) — e.g. the UTC moment whose `.toLocal()` is `2026-06-16 23:00` local — feed it through `civilDayOf`, and assert the result equals `CalendarDate.ymd(2026, 6, 16)` (the **local** day), never `CalendarDate.ymd(2026, 6, 17)` (UTC's next date). Drive the local zone deterministically the same way T4/T5 do (process `TZ` / host-clock override, never the real machine zone), so the vector is reproducible on any runner; do **not** read `DateTime.now()` (eng-write-dart-test §2/§3). Add a fall-back/control row (a 01:00-local review in the same zone, plus a negative-offset zone) so the assertion proves "uses the local day," not "happens to subtract a day."
2. **T8 — numeral block, display layer.** Re-use E02-T06's `CalendarPresenter` + `toLocaleNumerals` exactly — do **not** author a second formatter. Format one fixed `CalendarDate` (e.g. `CalendarDate.ymd(2026, 6, 16)`) through the presenter for each `{fa, ckb, ar}` and assert every numeric character is in the locale's block (`fa`/`ckb` → U+06F0–U+06F9; `ar` → U+0660–U+0669), zero ASCII digits `0x30–0x39`, and **no cross-block bleed** (`fa`/`ckb` output free of U+0660–U+0669; `ar` output free of U+06F0–U+06F9). This is the gate-level restatement of T06's unit assertion; keep them consistent (same fixed date, same locales) so the gate and the unit test cannot disagree.
3. **One date-correctness suite, two runners.** Group the engine-side vectors (T1–T7 from T08/T09, plus T10's `civilDayOf` boundary which builds only on `CalendarDate`) under `engine/test/` so they run in `dart test engine/`; keep the display-side vectors (T8 numerals, T9 presenter reference pair) under the app/display package so they run in `flutter test`. Do not move a `package:test` engine vector into a `flutter_test` file or vice-versa (eng-write-dart-test §2). Tag nothing `@Tags(['golden'])` here — these are arithmetic/string assertions, not pixel goldens (the per-locale RTL *screenshot* golden is E09/E10's, shared by reference, not duplicated).
4. **CI wiring, no new job, no new gate.** The suite rides the **existing** `fast` job in `.github/workflows/ci.yaml` (eng-add-ci-check §1, §5; testing-strategy §8): `dart test engine/` already runs the engine vectors, `flutter test --coverage --exclude-tags golden` already runs the display vectors. Confirm both are reached; if a path is missing, add the run step on the pinned `subosito/flutter-action@v2` (`flutter-version: '3.38.0'`, `channel: stable`) — never float the SDK (eng-add-ci-check §2). Update the gate→job mapping table comment so **gate 5** explicitly lists the date vectors alongside the RTL goldens and the ARB-key check. Do **not** mark the step `continue-on-error`, do **not** add a fifth job, and do **not** invent a new gate number — date correctness is a sub-check of the existing gate 5 (eng-add-ci-check §1, §9).
5. **Offline guard is the gate's other job.** The shared throwing-`HttpOverrides` bootstrap (eng-write-dart-test §8) is installed for every test in this suite, engine and display alike, so the assertion "no date or calendar path opens a socket" is structural: a stray connection is a named failure, not a silent 400. The engine suite is offline by construction (no Flutter, no `intl`, no `dart:io HttpClient`); the display suite installs the throwing override in `setUpAll`. Neither suite is the asset-downloader, so neither opts out.
6. **DST-zone discipline carries through.** The gate is **never** considered green from a `TZ=UTC`-only run (07 §7 refusal; domain-calendars Rule 12). T10's positive-offset zone and T08's `TZ=Asia/Tehran`/`Pacific/Kiritimati` rows must actually execute under their simulated zones in CI; document in the suite header that a UTC-only pass proves nothing about a Tehran user crossing midnight.
7. **Pitfalls to avoid**: asserting T10 with a *zero*- or *negative*-offset zone (the bug — UTC's next date and the local date coincide, so the test can't fail); reading the machine's real zone instead of pinning `TZ` (non-reproducible, green on the author's laptop, red in CI); duplicating T06's numeral helper into the gate test instead of re-running the real presenter (a second formatter drifts from production); using the **wrong** digit block per locale in the expectation (`٤` for `fa`, `۴` for `ar` — the cross-bleed defect T8 exists to catch); adding a fifth CI job or a new gate number for "the date gate" (it is gate 5); marking the date step `continue-on-error: true` or letting a UTC-only run count as green; tagging the arithmetic vectors `golden` (they are not pixel comparisons and must run in the fast job, not the pinned golden job); auto-running `--update-vectors`/`--update-goldens` in CI (CI only *verifies* — eng-add-ci-check §5).

## Acceptance criteria

- [ ] **T10 (local-day boundary)** is pinned: a 23:00-local review in a fixed positive-UTC-offset zone (e.g. Tehran, UTC+03:30), driven through `civilDayOf` with a deterministic zone override, yields `CalendarDate.ymd(2026, 6, 16)` (the local day) and **not** `CalendarDate.ymd(2026, 6, 17)` (UTC's next date); a control row (early-hours / negative-offset zone) proves the assertion is "uses the local day," not an incidental −1.
- [ ] **T8 (numeral block)** is pinned at the gate: the same fixed date rendered through E02-T06's `CalendarPresenter` is Extended Arabic-Indic (U+06F0–U+06F9) for `fa` and `ckb`, Arabic-Indic (U+0660–U+0669) for `ar`, with **zero** ASCII digits and **no** cross-block bleed in any of the three.
- [ ] The full **T1–T10** matrix exists as one date-correctness suite split correctly across `dart test engine/` (T1–T7, T10 boundary) and `flutter test` (T8, T9), with no engine vector in a `flutter_test` file and no display vector in a `package:test` file.
- [ ] The suite is wired into the **existing `fast` CI job** on the pinned `subosito/flutter-action@v2` (`flutter-version: '3.38.0'`, `channel: stable`); no new job is added, no new gate number is invented, and the gate→job mapping comment lists the date vectors under **PRD §20 gate 5**.
- [ ] A throwing `HttpOverrides` is installed for every test in the suite (engine and display); a deliberately-introduced socket open in a date/calendar path fails the suite loudly (verified once, then removed).
- [ ] No step in the date path is `continue-on-error`; flipping any single T1–T10 vector red (locally demonstrated) fails the `fast` job and would block the release.
- [ ] T4/T5 (from T08) and T10 actually run under their simulated non-UTC zones in the suite; the suite header documents that a `TZ=UTC`-only pass is not considered green.
- [ ] No new CLAIMS id is introduced (T8/T10 are locale-rendering and date-boundary correctness, not graded claims); both test files carry the REUSE `GPL-3.0-or-later` SPDX header, full-word names, and pass `dart format` and the analyzer.

## Tests

The vectors **are** the deliverable; there is no production code to cover beyond the suite and the CI wiring.

`engine/test/calendar_date_local_day_test.dart` — **`package:test`** (pure engine; no `flutter_test`, no widget binding; eng-write-dart-test §2). The shared throwing-`HttpOverrides` bootstrap is installed; all dates are literal `CalendarDate`/instant fixtures, no wall clock. Required cases, written FIRST:

- **T10 — 23:00-local lands on the local day**: build the UTC instant whose `.toLocal()` is `2026-06-16 23:00` in a pinned UTC+03:30 zone; `expect(civilDayOf(instant), CalendarDate.ymd(2026, 6, 16))`; assert it is **not** `CalendarDate.ymd(2026, 6, 17)`.
- **T10 — control, early hours**: a `01:00-local` review the same day in the same zone still maps to `2026-06-16` (the conversion uses the local day, not a fixed offset subtraction).
- **T10 — control, negative offset**: a `23:00-local` review in a fixed negative-offset zone maps to its own local day (so the rule is "local day," symmetric, not "Tehran-specific").
- **T1–T7 + T10 assembled**: re-export / group the T08 (T1–T5) and T09 (T6/T7) engine vectors so `dart test engine/` runs the engine half of the matrix as one reported suite alongside T10.

`test/calendar_date_gate_test.dart` (display package) — **`flutter_test`** (presenter + `intl` are display-layer; eng-write-dart-test §2). Throwing `HttpOverrides` installed in `setUpAll`; fixed `CalendarDate` and locale, no wall clock. Required cases, written FIRST:

- **T8 — `fa` block**: `CalendarPresenter(jalali, Locale('fa'))` on `CalendarDate.ymd(2026, 6, 16)` → every numeric char in U+06F0–U+06F9; zero `0x30–0x39`; zero U+0660–U+0669.
- **T8 — `ckb` block**: same date, `Locale('ckb')` → Extended Arabic-Indic only, no ASCII, no Arabic-Indic bleed.
- **T8 — `ar` block**: same date, `Locale('ar')` through `hijriUmmAlQura` and `gregorian` → every numeric char in U+0660–U+0669; zero ASCII; zero Extended Arabic-Indic.
- **T8 — no-ASCII matrix**: `{gregorian, jalali, hijriUmmAlQura} × {fa, ckb, ar}` on the fixed date asserts no rendered string contains any `0x30–0x39` — proving the remap is the unconditional last step, consistent with the T06 unit test.
- **T9 re-run**: the published Umm al-Qurā reference pair from T09 runs in this same gate suite (re-export), so the display half of the matrix is one reported suite.
- **Offline guard self-proof** (run once, then delete): a temporary date-path socket open is caught by the throwing `HttpOverrides` and fails the suite — proving the guard is live, not decorative.

No widget/golden/integration tests in this task: the per-locale RTL **screenshot** golden (real `fa`/`ckb`/`ar` fonts, FSI/PDI, logical layout) is E09/E10's, shared into gate 5 by reference; this task asserts arithmetic and string-level correctness only. CI wiring change: `.github/workflows/ci.yaml` `fast` job confirmed to run both `dart test engine/` and `flutter test` over the suite, with the gate→job comment updated for gate 5; no new job, no `continue-on-error`.

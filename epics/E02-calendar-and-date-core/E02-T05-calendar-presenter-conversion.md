## Goal

`CalendarSystem { jalali, hijriUmmAlQura, gregorian }` and `CalendarPresenter(system, locale)` exist in the display-layer `l10n` package (`packages/l10n/lib/src/calendar/calendar_presenter.dart`) as the **single** place a pure-engine `CalendarDate` becomes localized human text. `format(CalendarDate)` converts via pure-Dart, offline, BSD-licensed packages — `shamsi_date` (Jalālī, the `fa` default), `hijri` (Umm al-Qurā), `intl`'s `DateFormat` (Gregorian only) — taking the month name and era from the **calendar package's** localized tables (never `intl`'s Gregorian-only `DateFormat` for a non-Gregorian system). The calendar is an explicit `CalendarSystem` parameter sourced from a Settings value, **never** inferred from `Locale.current`, and Hijri and Gregorian are offered to every locale. These three display dependencies are declared at the display layer only and are provably absent from `engine/` (the E01 banned-import gate). This task ships the conversion boundary and the month-name/era composition; the downstream locale-numeral remap is E02-T06 and the Hijri range-guard/fallback + honesty caveat is E02-T07 — this task wires the seams they fill.

## Implementation notes

This task is correctness-sensitive at the display boundary (a wrong calendar parameter or an `intl`-rendered Hijri label corrupts every dated surface in E10–E19), but it is not float arithmetic — pin it with per-system label tests, not a frozen FSRS vector. Write the per-system / per-locale label assertions (below) alongside the presenter; do not let a Hijri or Jalālī label silently fall through to `intl`'s Gregorian `DateFormat`.

1. **File**: `packages/l10n/lib/src/calendar/calendar_presenter.dart` in the **display-layer** `l10n` package — never `engine/` (07 §4: calendars are a display concern; the engine is calendar-agnostic). One primary boundary per file: `CalendarSystem` (the enum) and `CalendarPresenter` (the converter). Re-export from the `l10n` barrel `packages/l10n/lib/l10n.dart` as `export 'src/calendar/calendar_presenter.dart' show CalendarSystem, CalendarPresenter;` so Settings (E16) can name the contract.
2. **Imports**: `package:shamsi_date`, `package:hijri`, `package:intl`, and the `engine` barrel (for `CalendarDate`) — all permitted **here**, in `l10n`. These three packages are pinned in `packages/l10n/pubspec.yaml` and the workspace allow-list (E01); they are **banned in `engine/`** by the E01 banned-import / no-network grep gate, which is the audit this task must keep green. Add no `dart:io`, no `HttpClient`, no network of any kind — both calendar packages are bundled pure Dart (07 §4 / PRD C1).
3. **`CalendarSystem` enum**: verbatim `enum CalendarSystem { jalali, hijriUmmAlQura, gregorian }` (07 §4). This is the conversion contract a Settings value selects; do **not** add a `fromLocale` factory or any locale-inference path — the default (`jalali` for `fa`) is chosen at the Settings layer (E16), never here, and never from `Locale.current` (07 §4 Pitfalls; PRD §13.3).
4. **`CalendarPresenter`**: `final CalendarSystem system; final Locale locale; const CalendarPresenter(this.system, this.locale);` exactly per 07 §4. The single public method `String format(CalendarDate d)`: build the pure bridge `final g = DateTime.utc(d.year, d.month, d.day);` (a `(y,m,d)` calculator, **not** a clock — `DateTime.utc`, never local `DateTime(...)`), then `switch (system)` to `_gregorianLabel` / `_jalaliLabel` / `_hijriLabel`. Return the converted label; the **downstream** numeral remap (`toLocaleNumerals`) is the E02-T06 seam — leave the call site marked, do not implement the digit map here.
5. **Per-system composition** — month name + era from the **calendar package**, never `intl`'s Gregorian-only `DateFormat`:
   - `_jalaliLabel`: `Jalali.fromDateTime(g).formatter` → `'${f.d} ${f.mN} ${f.yyyy}'` (day · monthName · year), the localized Jalālī month name (`f.mN`) coming from `shamsi_date`'s tables.
   - `_hijriLabel`: `HijriCalendar.fromDate(g)` → format with the **Umm al-Qurā** month name from `hijri`'s tables, labelled as Umm al-Qurā; the range-guard Gregorian fallback and caveat are the **E02-T07** seam (here it is a labelled call, never an `intl` Gregorian render of a Hijri date).
   - `_gregorianLabel`: the one path that uses `intl`'s `DateFormat(pattern, locale)` directly (Gregorian is `intl`'s only calendar).
6. **The seam contract** (do not over-build): this task converts and composes the label string; it does **not** (a) remap numerals (E02-T06), (b) implement the Hijri out-of-range fallback or the caveat copy (E02-T07), (c) run the round-trip/reference sweep (E02-T09), or (d) isolate the run for RTL or place it in a widget (E09). Mark each downstream seam with a `// TODO(E02-T06/T07)` pointer; keep the presenter a pure, total label function.
7. **Docs & standards**: `///` on `CalendarSystem`, each enum value, `CalendarPresenter`, and `format`; the class doc states "the ONLY place a `CalendarDate` becomes localized text" and that the calendar is an explicit parameter, never inferred from locale. The `format` body carries why-comments citing 07 §4 at the `DateTime.utc` bridge and at the month-name-from-calendar-package decision. Effective Dart casing, `dart format` / `dart analyze --fatal-infos` clean, REUSE SPDX header (`GPL-3.0-or-later`).
8. **Pitfalls to avoid**: rendering a Jalālī or Hijri date through `intl`'s `DateFormat` (it is Gregorian-only — the wrong month name/era, the §4 refusal); a local `DateTime(d.year, d.month, d.day)` instead of `DateTime.utc` in the bridge (bakes in the device zone); inferring the calendar from `Locale.current` or adding a `CalendarSystem.fromLocale` (07 §4 refusal); putting `CalendarSystem`/`CalendarPresenter` in `engine/` (breaks engine purity — the E01 gate fails); leaking the calendar packages' Latin digits to a caller as final UI text without the E02-T06 remap seam marked; presenting the Hijri label as "the Hijri date" in the absolute rather than Umm al-Qurā (07 §6); making `format` throw on an out-of-range Hijri date (the engine/display layer is total — the fallback is E02-T07, never an exception); a network call in any date path.

## Tests

`packages/l10n/test/calendar/calendar_presenter_test.dart` (mirrors the source name under the `l10n` `test/` tree), `flutter_test` (the `l10n` layer depends on `intl` and Flutter localizations, so the engine's `package:test`-only rule does **not** apply here — per `eng-write-dart-test` §2 the *engine* uses `package:test`; the display layer uses `flutter_test`). Every date is a constructed `CalendarDate.ymd(...)` literal; no wall clock is read. The shared throwing-`HttpOverrides` offline bootstrap (E01-T06) stays installed; this suite touches no network by construction. REUSE SPDX header on the file. Required cases:

- **Per-system label shape (T8 month-name/era half)**: for a fixed date (e.g. `CalendarDate.ymd(2026, 6, 16)`), assert `CalendarPresenter(CalendarSystem.jalali, …).format(d)` yields the correct Jalālī day · month-name · year from `shamsi_date`; `CalendarSystem.hijriUmmAlQura` yields the correct Umm al-Qurā month-name (labelled Umm al-Qurā); `CalendarSystem.gregorian` yields the `intl`-formatted Gregorian label. (Numeral-block assertions are **E02-T06**; this case asserts the calendar and month name are right.)
- **Each locale renders each calendar (explicit-parameter proof)**: a 3×3 matrix over `{fa, ckb, ar} × {jalali, hijriUmmAlQura, gregorian}` — each cell produces a non-empty, system-appropriate label; in particular `fa`+`gregorian` and `ar`+`hijriUmmAlQura` render (Hijri/Gregorian are offered to every locale), proving the calendar is the `system` parameter, never inferred from `Locale.current`.
- **Wrong-renderer guard**: assert the Jalālī and Hijri labels are **not** equal to `intl`'s Gregorian `DateFormat` output for the same date (the month name/era differ), proving the non-Gregorian month name comes from the calendar package, not from `intl`.
- **Pure-bridge / no-clock**: `format` is deterministic for a fixed `CalendarDate` regardless of the host's timezone or wall clock (two calls are equal; no `DateTime.now()` reachable); the presenter is `const`-constructible.
- **T9 reference pair (single-pair smoke)**: one documented, independently published Gregorian↔Umm-al-Qurā reference pair (from a published source, **not** the library asserting against itself) formats to the expected Umm al-Qurā month/year. (The multi-decade Jalālī + in-range Hijri identity round-trip sweep and the broader reference set are **E02-T09**.)

The full T6/T7 round-trip identity sweep, the T9 reference set, the T8 numeral-block assertions, and the Hijri out-of-range Gregorian-fallback test belong to **E02-T06/T07/T09** and consume this presenter — they are not this task.

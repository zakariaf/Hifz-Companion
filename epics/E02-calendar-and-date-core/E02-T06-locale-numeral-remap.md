# E02-T06 — Downstream locale-numeral remap on dates (Extended/Arabic-Indic, no ASCII)

| | |
|---|---|

| **Size** | S (≈0.5-1 day) |

## Goal

The `CalendarPresenter` from E02-T05 maps its converted date's numeric fields (day, year, and any numeric month) to the active locale's own digit block **strictly downstream of the calendar conversion**: convert the `CalendarDate` first, then pass every numeric field through `intl` `NumberFormat` with a pinned numbering system — Extended Arabic-Indic (`۰۱۲۳۴۵۶۷۸۹`, U+06F0–U+06F9) for `fa`/`ckb`, Arabic-Indic (`٠١٢٣٤٥٦٧٨٩`, U+0660–U+0669) for `ar` — so a `hijri`/`shamsi_date`/`intl`-Gregorian formatter's Latin digits never reach the UI. The remap lives in a single `toLocaleNumerals(latin, locale)` boundary called as the last step of `CalendarPresenter.format`, never as raw ASCII string concatenation. A test formats a known date in `fa`/`ckb`/`ar` and asserts each one renders the correct digit block with **zero** ASCII digits (the T8 vector).

## Implementation notes

TEST-FIRST (correctness-critical): a Latin digit leaking into a localized date is a release-gate-blocking localization defect ([PRD §20 gate 5](../../docs/PRD.md)), and the failure is silent in any test that only checks the date is "non-empty". Write the T8 `fa`/`ckb`/`ar` digit-block assertions and the no-ASCII guard **before** wiring `toLocaleNumerals` — the three locale expectations must exist and fail (presenter still emits Latin digits) before the remap is added.

1. **File**: `toLocaleNumerals` lives beside the `CalendarPresenter` from E02-T05 (the display layer — `features/` or `l10n/`, **not** `engine/`; `intl` is a display-only dependency and the E01 banned-import grep keeps it out of `engine/`). Reuse the project's existing `numberFormatFor(locale)` from the RTL/numeral layer (eng-rtl-and-bidi-layout `template.dart`) — do **not** author a second numeral formatter; this task wires the date path into the one that already exists.
2. **`numberFormatFor(locale)` numbering system, pinned**: `fa` and `ckb` use a `NumberFormat` whose locale carries `-u-nu-arabext` (Extended Arabic-Indic, U+06F0–U+06F9); `ar` uses `-u-nu-arab` (Arabic-Indic, U+0660–U+0669). Pin the numbering-system subtag explicitly — never rely on `intl`'s bare-locale digit default, which is inconsistent between date and number formatting (eng-rtl-and-bidi-layout Rule 7). These are two **distinct** Unicode blocks: `ar` must never show `۴` (Persian four) and `fa`/`ckb` must never show `٤` (Arabic four).
3. **`toLocaleNumerals(String latin, Locale locale)`**: the remap operates on the already-converted, Latin-digit label string the per-system formatter produced. Format each numeric field (day, year, and a numeric month if the system emits one) through `numberFormatFor(locale)` and substitute it for its ASCII run — do **not** hand-map ASCII code points `0x30–0x39` to a target block with string replacement as the *primary* mechanism (`intl` `NumberFormat` is the source of truth so grouping/sign/zero behaviour stays locale-correct). Prefer feeding the presenter the numeric *fields* (`f.d`, `f.yyyy`) through `NumberFormat` **before** assembling the label, so there is no ASCII run to find; if E02-T05's per-system label already concatenated Latin digits, remap field-by-field, never with a blanket `replaceAll` over the whole string (which would also corrupt any Latin technical token — there is none in a date, but the discipline is the rule).
4. **The month NAME and era are NOT numerals** — they come from the calendar package's localized tables (`shamsi_date`'s `mN`, `hijri`'s month names, `intl`'s `DateFormat` for Gregorian) and pass through `toLocaleNumerals` unchanged (they contain no ASCII digits). This task only touches the numeric fields; it never re-localizes, re-orders, or re-cases a month name (07 §4: the calendar package and `intl` each own a different half of the string).
5. **Call site, last step, once**: `toLocaleNumerals` is the **final** line of `CalendarPresenter.format` (07 §4 listing), applied identically to all three `CalendarSystem` branches — Gregorian, Jalālī, and Hijri (and, via E02-T07, the Hijri→Gregorian fallback label). There is exactly one numeral-remap boundary, mirroring the presenter being the one conversion boundary; a view never remaps digits itself.
6. **Bidi isolation is out of scope here** (eng-rtl-and-bidi-layout Rules 5–6): wrapping the finished date run in FSI/PDI and placing it with logical insets happens where the widget renders it (E09/E10). This task guarantees the *digits inside the string* are the locale block; it does not isolate or position the run. Do not add a `Directionality`, an `isolate()`, or any `EdgeInsetsDirectional` here.
7. **Pitfalls to avoid**: a blanket `replaceAll('0','۰')…` chain across the whole label (corrupts non-numeral text and silently desyncs from `NumberFormat`'s rules); using the **wrong block** for a locale (`-u-nu-arab` for `fa`, or `-u-nu-arabext` for `ar` — the `٤`/`۴` swap defect); relying on `intl`'s bare-locale default numbering instead of pinning the subtag; remapping the month *name* or accidentally stripping its diacritics; running the remap **before** conversion (so the calendar formatter re-introduces Latin digits afterward — the remap must be strictly downstream); letting a percentage/page-count elsewhere reuse this *date-only* helper (those have their own `numberFormatFor` call at their own widget — this helper is the date path); concatenating a raw ASCII year into the string "just for the year" (the exact leak the no-ASCII grep and T8 catch).

## Acceptance criteria

- [ ] `toLocaleNumerals(latin, locale)` exists beside `CalendarPresenter` in the display layer (not `engine/`); the E01 banned-import gate stays green (no `intl`/`hijri`/`shamsi_date` reachable from `engine/`).
- [ ] It is the **final** transform in `CalendarPresenter.format`, applied identically to the Gregorian, Jalālī, and Hijri branches (and the E02-T07 Hijri→Gregorian fallback), so no `CalendarSystem` path can emit a Latin-digit date.
- [ ] Numeric fields format through `numberFormatFor(locale)` with the numbering system pinned per locale — `-u-nu-arabext` for `fa`/`ckb`, `-u-nu-arab` for `ar`; the helper reuses the existing RTL/numeral `numberFormatFor`, not a second formatter.
- [ ] A known date formatted in `fa` and `ckb` contains **only** Extended Arabic-Indic digits (U+06F0–U+06F9) for every numeric field; the same date in `ar` contains **only** Arabic-Indic digits (U+0660–U+0669).
- [ ] No rendered date in any of the three locales contains an ASCII digit `0x30–0x39`; the `fa`/`ckb` output contains no `ar` (`٤`) digit and the `ar` output contains no `fa`/`ckb` (`۴`) digit — the blocks never cross.
- [ ] The month name and era pass through unchanged (no re-localization, no diacritic loss); only numeric fields are remapped.
- [ ] A grep over the presenter/helper finds no blanket whole-string ASCII-digit `replaceAll` chain and no raw ASCII-digit concatenation into a localized date string; the file passes `dart format` and the analyzer with the REUSE `GPL-3.0-or-later` SPDX header present.

## Tests

`test/calendar_presenter_numerals_test.dart` beside the E02-T05 presenter suite, **`flutter_test`** (the presenter and `intl` formatting are display-layer, so this is not an `engine/` `package:test` file; eng-write-dart-test §2). `today`/the formatted date are literal `CalendarDate.ymd(...)` values; nothing reads a wall clock (§3). The shared throwing-`HttpOverrides` bootstrap is installed so any stray socket is a loud named failure (§8). Runs under `flutter test`, and is re-run by E02-T10 as the T8 vector under the date gate and shared with the E09 per-locale numeral golden. Required cases, written FIRST:

- **T8 — `fa` digit block**: format a fixed `CalendarDate` (e.g. `CalendarDate.ymd(2026,6,16)`) through `CalendarPresenter(jalali, Locale('fa'))`; assert every numeric character in the result is in U+06F0–U+06F9 (Extended Arabic-Indic) and the string contains **zero** characters in `0x30–0x39` and **zero** in U+0660–U+0669 (no Arabic-Indic bleed).
- **T8 — `ckb` digit block**: same fixed date through `CalendarPresenter(jalali, Locale('ckb'))` (Jalālī offered for Kurdish, PRD §13.3); assert Extended Arabic-Indic only, no ASCII, no Arabic-Indic — `ckb` shares `fa`'s `-u-nu-arabext` block, distinct from `ar`.
- **T8 — `ar` digit block**: same fixed date through `CalendarPresenter(hijriUmmAlQura, Locale('ar'))` and through `CalendarPresenter(gregorian, Locale('ar'))`; assert every numeric character is in U+0660–U+0669 (Arabic-Indic) and there are **zero** ASCII digits and **zero** Extended Arabic-Indic (no `fa` bleed) — covering that the remap is system-agnostic.
- **No-ASCII guard across all systems × all locales**: a matrix over `{gregorian, jalali, hijriUmmAlQura} × {fa, ckb, ar}` on a fixed date asserting no rendered string contains any `0x30–0x39` — proving the remap is the unconditional last step of `format`, not a per-branch afterthought.
- **Month name preserved**: assert the localized month name from the calendar package is present and byte-identical before and after `toLocaleNumerals` (it carries no ASCII digit to remap), so the numeral pass does not touch or corrupt non-numeric text.
- **Downstream ordering**: a guard that `toLocaleNumerals` applied to an already-localized-numeral string is idempotent (re-running finds no ASCII to convert) and that the per-system Latin label fed *into* it is the only source of digits — pinning that conversion happens before the remap, never after.

No widget/golden/integration tests in this task — the per-locale RTL **golden** screenshot (real bundled `fa`/`ckb`/`ar` UI fonts, FSI/PDI isolation, logical layout) belongs to E09/E10 and the date gate in E02-T10; this task asserts the **string-level** digit blocks. The offline guard is asserted by construction via the throwing `HttpOverrides`; nothing here touches the network.

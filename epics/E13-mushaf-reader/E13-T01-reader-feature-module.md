## Goal

The `/features/mushaf` feature module exists as the real reader scaffold: a folder `packages/features/lib/src/mushaf/` with the canonical anatomy — a dumb `MushafReaderScreen` `ConsumerWidget` View, its 1:1 `MushafReaderViewModel` `AsyncNotifier`, a scoped `mushaf_providers.dart`, and a `widgets/` dir — wired into E07's single `ShellRoute` bottom nav **in place of the inert Muṣḥaf placeholder tab**. The `GoRoute` carries typed, optional deep-link params that resolve to a starting page (`?page=`/`?juz=`/`?hizb=`/`?surah=`), so a heat-map or drill hand-off can open the reader at a target. The View renders the current page through E05's `MushafPageView` behind the redirect guard, names the active riwāyah in chrome, and re-shapes/re-typesets/re-derives nothing. This task lands the navigable skeleton and its downward-only dependency set only — the paged navigator (T03), jump-to (T04), overlays (T05), and controls (T06) fill it in.

## Implementation notes

TEST-FIRST (correctness-critical): the deep-link param parse/clamp and the ViewModel's UI-state mapping are the two places an off-by-one on a sacred page boundary or a wrong-page landing can enter. Write the route-param parse unit and the ViewModel wiring test (below) **before** their bodies; the legal-range and clamp cases must exist and fail first.

1. **Folder anatomy** (`docs/engineering/02-project-structure.md` §3.4, per `eng-add-feature-module`). Create, inside the existing `features` umbrella package, **not** a new package:
   ```
   packages/features/lib/src/mushaf/
   ├── mushaf_screen.dart        # MushafReaderScreen — dumb ConsumerWidget entry View
   ├── mushaf_view_model.dart    # MushafReaderViewModel — 1:1 AsyncNotifier + immutable MushafReaderState
   ├── widgets/                  # leaf views (one primary type per file; filled by T03/T06/T07)
   └── mushaf_providers.dart     # scoped Riverpod providers (NOT global)
   ```
   One primary public type per file, `lower_snake_case` filenames named for the type.

2. **`MushafReaderViewModel` (1:1 `AsyncNotifier`)** in `mushaf_view_model.dart`. `class MushafReaderViewModel extends AsyncNotifier<MushafReaderState>` exposing one immutable `MushafReaderState` value. `MushafReaderState` is an immutable value type (`const` ctor + `copyWith`) carrying at minimum the active `MushafEdition` (for the riwāyah chrome) and the `initialPage` resolved from the deep-link param; the live current-page/zoom/theme/overlay-toggle state is owned by **T02's reader-state store** and read by this ViewModel — keep the seam. `build()` reads the active `MushafEdition` via an injected provider (E16 owns the swap; here read whichever is active) and the verified-readiness gate; it reaches **no** DAO, calls **no** engine math, and contains **no** `DateTime.now()`. Provider: `final mushafReaderViewModelProvider = AsyncNotifierProvider.autoDispose.family<MushafReaderViewModel, MushafReaderState, ProfileId>(MushafReaderViewModel.new);` — `family`-keyed by `ProfileId` (weak-line refs key on the active profile in T05), `autoDispose`d so the heavy reader does not survive its screen on low-end Android.

3. **`MushafReaderScreen` (dumb `ConsumerWidget`)** in `mushaf_screen.dart`. `class MushafReaderScreen extends ConsumerWidget` taking the typed deep-link params via its constructor (`final int? initialPage; final int? initialJuz; final int? initialHizb; final int? initialSurah;`). `build()` reads exactly one controller (`ref.watch(mushafReaderViewModelProvider(profile))`), `.when(loading/error/data)` with a **calm** retry on `error` (never a guilt/fear message), and in `data` embeds E05's `MushafPageView` for the current page, wrapped in `Directionality(textDirection: TextDirection.rtl)`. No repository call, no engine call, no `try/catch` business logic, no `softWrap`/`TextPainter` on the page — only show/hide if-statements, layout, simple routing. The riwāyah/`displayName` chrome label and the page renderer are the only content this task wires; the navigator (T03), jump-to surface (T04), overlay toggles (T05), and controls (T06) are seams the screen reserves slots for.

4. **`GoRoute` under the single `ShellRoute`** in `app/composition/router.dart` (the composition root — touched here, not a feature file). Add `GoRoute(path: '/mushaf', builder: (context, state) => MushafReaderScreen(initialPage: _intParam(state, 'page'), initialJuz: _intParam(state, 'juz'), initialHizb: _intParam(state, 'hizb'), initialSurah: _intParam(state, 'surah')))`. Params are **optional query params** (`state.uri.queryParameters['page']`), typed via `int.tryParse`, so `/mushaf` opens at the user's last/first page and `/mushaf?juz=3` deep-links a target. Parse + legal-range validation (`page∈1..604`, `juz∈1..30`, `hizb∈1..60`, `surah∈1..114`) lives in a small pure helper `mushafReaderRouteFromUri` so it is unit-testable without a router pump; an out-of-range or unparseable value is **clamped/dropped to a safe default page** (never a crash, never a silent wrong-boundary landing). Resolving a juz/ḥizb/sūrah param to its **page** is read from the fixed bundled structure (PRD §6.1) and is **owned by T04** — here the param is carried through to the screen; T04 wires the resolver and seeks the controller.

5. **Replace the inert placeholder; preserve RTL nav order.** Swap E07's placeholder Muṣḥaf destination for `MushafReaderScreen` at the existing nav index; the bottom-nav order stays Today · Muṣḥaf · Mutashābihāt · Progress · Settings (RTL, rightmost = Today/home), mirroring automatically under the app-wide `Directionality`; directional icons mirror. Do **not** add an imperative `Navigator.push`; do **not** bypass the router's redirect guard — the new route inherits it, so no Quran screen renders before the core pack is verified and a profile exists (E05/E07).

6. **Downward-only dependencies (the audit map).** `features` may import `engine`, `data`, `quran` (the `MushafPageView`/`MushafEdition` types), `l10n`, `profiles`, plus `flutter_riverpod` + `go_router`. The module imports **no** `package:http`/`dio` (it fetches nothing), **no** `drift`/DAO (no DB from a feature), and **no** other feature's `src/`. The engine is never reached from a widget; the View reads its controller, the controller reads providers. Verify by grep over `lib/src/mushaf/`.

7. **Strings & tokens (stubs only here; T09 fills fa/ckb/ar).** Any user-facing string already introduced (the tab label, the riwāyah chrome, the calm retry copy, the `Semantics` label on the tab) comes from `AppLocalizations` in the `l10n` package (`ar` template + `fa`/`ckb`), never a hardcoded literal; colours/type/spacing referenced by name (`color.*`/`type.*`/`space.*`) — no inline hex/pt/ms. T09 completes the transcreations and Semantics; this task wires the keys and leaves none hardcoded.

8. **Pitfalls to avoid:** putting two primary types in one file, or naming the View `mushaf_view.dart` (it is `mushaf_screen.dart`); an un-keyed "current mushaf state" provider or `autoDispose` on an app-scope singleton (key by `ProfileId`, `autoDispose` the reader); calling `DateTime.now()` in the ViewModel (the reader needs no clock — defer any date to the injected `clockProvider` if ever needed); reaching a DAO/the engine from the View; an optimistic republish or any `review_log` append from reader state (display-only — no engine mutation); setting `softWrap`/adding a `fontFamilyFallback`/swapping fonts per theme on the embedded page (E05 owns the glyph layer — relax nothing); resolving a juz/ḥizb/sūrah deep-link by **calculation** instead of reading the bundled structure (an off-by-one on a sacred boundary — defer the resolver to T04); navigating from inside the controller; opening a socket or fetching anything at runtime; any badge/streak/score/confetti/ornament/page-flip fanfare on the page (forbidden, not discouraged).

- [ ] A `GoRoute('/mushaf')` is registered under the single `ShellRoute` with optional typed deep-link params `page`/`juz`/`hizb`/`surah`; `/mushaf`, `/mushaf?page=255`, and `/mushaf?juz=3` all resolve to a `MushafReaderScreen` opened at a valid page.
- [ ] `mushafReaderRouteFromUri` (pure helper) parses each param via `int.tryParse`, validates the legal ranges (`page∈1..604`, `juz∈1..30`, `hizb∈1..60`, `surah∈1..114`), and clamps/drops an out-of-range or unparseable value to a safe default page — never crashes, never lands on a wrong boundary; the juz/ḥizb/sūrah→page **resolution** is carried through to T04, not computed here.
- [ ] `MushafReaderViewModel` exposes one immutable `MushafReaderState`, is registered as `AsyncNotifierProvider.autoDispose.family<…, ProfileId>`, reaches no DAO/engine, contains no `DateTime.now()`, and mutates no persisted state (no `review_log` append, no `due_at` re-derivation).
- [ ] `MushafReaderScreen.build` reads exactly one controller, `.when` surfaces a **calm** retry on error, and embeds E05's `MushafPageView` under an RTL `Directionality`; it sets no `softWrap`, adds no `fontFamilyFallback`, and swaps no font per theme on the page.
- [ ] Dependency grep over `lib/src/mushaf/` shows no `package:http`/`dio`, no `drift`/DAO import, and no import of another feature's `src/`; the engine is not imported by any widget.
- [ ] Every user-facing string in the module comes from `AppLocalizations` (no hardcoded UI literal); tokens referenced by name (no inline hex/pt/ms).
- [ ] The screen renders only behind the redirect guard (no Quran screen before the core pack is verified and a profile exists); no imperative `Navigator.push`, no controller-initiated navigation.

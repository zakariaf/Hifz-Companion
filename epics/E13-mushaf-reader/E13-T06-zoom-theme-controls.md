# E13-T06 — Reader controls: zoom (uniform scale, independent of OS text-scale) + light/sepia/dark theme toggle (ColorFilter) as layer transforms

## Goal

The reader's zoom and light/sepia/dark theme controls exist as **layer transforms over E05's render frame** and nothing more. `packages/features/lib/src/mushaf/widgets/reader_zoom_control.dart` and `.../reader_theme_control.dart` bind the E13-T02 reader-state store's `zoom` (a uniform scale factor) and `theme` (`ReaderTheme.light`/`.sepia`/`.dark`) commands to two calm chrome controls; the values flow into E05's `mushafPageView(pageNumber, theme, zoom)` frame, where `zoom` becomes a single `Transform.scale` (RTL `Alignment.topRight` origin) and `theme` selects a single `ColorFilter`. There is exactly **one font per page**: dark mode is a colour filter, never a "dark font"; zoom is the muṣḥaf's *own* scale, read from the store and **independent of `MediaQuery.textScalerOf`** (the OS chrome text-scale), so a printed line break never moves at any zoom step. The controls write only display-only reader state (no engine mutation, no review, no persisted-instant change); they re-typeset, re-flow, and font-swap nothing.

## Implementation notes

TEST-FIRST: the "zoom is a uniform layer transform independent of OS text-scale" and "dark/sepia is a `ColorFilter`, not a font swap, and no line break moves" behaviours are correctness-critical for the sacred surface. Write the no-reflow / one-font-per-page widget assertions and the theme×zoom golden (Tests section) **before** the control bodies.

1. **File / anatomy** (eng-add-feature-module, established by E13-T01): two leaf widgets under the reader feature — `packages/features/lib/src/mushaf/widgets/reader_zoom_control.dart` (one primary public type `ReaderZoomControl`) and `packages/features/lib/src/mushaf/widgets/reader_theme_control.dart` (one primary public type `ReaderThemeControl`), plus a tiny shared `reader_zoom_steps.dart` constant if a discrete zoom band is used. Both are `ConsumerWidget`s read into `MushafReaderScreen` (E13-T01); they import the E13-T02 reader-state store, E05's `mushafPageView`/`ReaderTheme` types, `flutter`/`riverpod`, and the generated `l10n` — **no** `drift`/DAO, no `package:http`/`dio`/`HttpClient`, no engine-mutation symbol, no `DateTime.now()`.

2. **The transform lives in E05's frame, not in the control** (ui-mushaf-page-view rule 5; eng-08 §5): the controls do **not** build the `Transform.scale`/`ColorFiltered` themselves — they only set `zoom`/`theme` in the store, and the page slot (E13-T03) passes those into `mushafPageView(pageNumber: p, theme: state.theme, zoom: state.zoom)`. This task's job is the *control chrome + the binding*, so the single sanctioned transform frame stays in E05 and is never duplicated. Confirm the page slot wraps exactly one `ColorFiltered` over one `Transform.scale(alignment: Alignment.topRight)` over the glyph+overlay `Stack`, and that overlays sit *inside* the scale (they scale with the page; they are never re-laid-out per zoom).

3. **Zoom control** (eng-08 §5; type-04 §7):
   - The control reads `state.zoom` and calls `ref.read(mushafReaderStateProvider.notifier).setZoom(z)`. Use a **discrete zoom band** (e.g. a small ordered `List<double>` such as `[1.0, 1.25, 1.5, 2.0, 2.5]`) driven by `−` / `+` icon buttons (or a `SegmentedButton` of named steps) rather than a continuous slider, so each step is a named, repeatable choice and there is no fine-grained "fit to width" temptation. The min/max band is a constant in `reader_zoom_steps.dart`; `setZoom` already `assert`s finiteness/bounds (E13-T02), so the control simply clamps to the band.
   - **Zoom is the muṣḥaf's own scale, not OS text-scale**: the control **never** reads or writes `MediaQuery.textScalerOf(context)` / `TextScaler`, and the value it sets flows only into E05's `Transform.scale` — not into any `TextStyle.fontSize`, not into the page's text-scaler. The control's *own* label/buttons are ordinary chrome and *do* scale with the OS `TextScaler` (type-04 §7); the **page** does not.
   - The origin is E05's concern (`Alignment.topRight`, RTL), so the page grows from the top-right (the natural reading start in RTL) as zoom increases — the control adds no origin/anchor logic of its own.

4. **Theme control** (ui-settings-picker rules 1/2/3; ds-03 §3/§7):
   - A single-choice control over `ReaderTheme.light`/`.sepia`/`.dark` (the three values E13-T02's enum declares; the design system's fourth appearance, *Night*, is out of scope for this task's three-value contract — do not add it here). Render as a `SegmentedButton<ReaderTheme>` (short, mutually-exclusive set) or `RadioListTile`s; selected state is the segment fill/check **and** the localized text label, **never colour alone** (SC 1.4.1). It reads `state.theme` and calls `setTheme(t)`.
   - The value selects E05's `ReaderTheme.glyphColorFilter` (eng-08 §5) — a single `ColorFilter` over the whole glyph+overlay layer. The control performs **no per-theme font swap**, sets no `fontFamily`, and touches no glyph: dark/sepia is a colour transform of the rendered layer only.
   - The toggle is a **display transform** (ui-settings-picker rule 3): it mutates no `due_at`, no stored instant, no engine state; it only re-tones the rendered page. It offers the choice and defaults to the OS/light-sepia preference (ds-03 §3, "never force dark mode as a look"); it makes no sleep claim and no "recommended" framing.

5. **Composition with the page slot is reflow-free** (domain-mushaf-text-integrity rule 7; eng-08 §5): threading `zoom`/`theme` into `mushafPageView` must not introduce any `softWrap`, `TextPainter`, width-wrap, or per-zoom re-layout of the glyph lines — the layout is the bundled QUL geometry regardless of scale; `Transform.scale` enlarges the *rendered* layer, it does not re-break lines. Assert (test) that the same page at zoom `1.0` and zoom `2.0` has identical relative line positions (the higher zoom is a pure pixel scale of the lower).

6. **No-dashboard restraint on the controls** (domain-mushaf-text-integrity rule 10; adab): the controls add **no** confetti/streak/badge on a zoom or theme change, no "you zoomed in" toast, no celebration, no glow/ornament on the page; a theme/zoom change is a quiet M3 state-layer change. Hide/show choreography (the edge-receding auto-hiding chrome) is **E13-T08's** job — this task makes the controls calm and self-contained, not their reveal animation.

7. **Pitfalls to avoid**: building a second `Transform.scale`/`ColorFiltered` *inside* the control instead of letting E05's frame own the single sanctioned transform (two transforms = a chance to drift from the RTL `topRight` origin and the one-filter rule); reading `MediaQuery.textScalerOf`/`TextScaler` into `zoom` (couples the sacred zoom to OS chrome text-scale — forbidden by type-04 §1/§7 and E13-T02's contract); using a continuous `Slider` for zoom (a "fit to width" temptation and a non-repeatable step — prefer a discrete band; ui-settings-picker forbids a slider for a 2+-value choice); swapping `fontFamily` for dark mode or adding a "dark font" (one font per page, dark = filter — eng-08 §5); signalling the selected theme by colour alone (SC 1.4.1 — pair shape + text); letting a zoom or theme change introduce `softWrap`/`TextPainter`/width-wrap or move a printed line break; adding *Night* as a fourth value (out of this task's three-value `ReaderTheme` contract); persisting a `review_log`/mutating a `Card`/re-deriving a `due_at` from a control (display-only); hardcoding a user-facing label in the widget instead of an `l10n.*` key (strings are E13-T09); celebrating the change.

## Acceptance criteria

- [ ] `packages/features/lib/src/mushaf/widgets/reader_zoom_control.dart` and `.../reader_theme_control.dart` exist (each one primary public type, mounted inside `MushafReaderScreen`); they import the E13-T02 store, E05's `mushafPageView`/`ReaderTheme`, `flutter`/`riverpod`, and `l10n` only — no `drift`/DAO, no `package:http`/`dio`/`HttpClient`, no engine-mutation symbol, no `DateTime.now()` (verifiable by grep).
- [ ] The zoom control sets `state.zoom` via the store's `setZoom` over a **discrete zoom band** (a constant ordered `List<double>`, min/max bounded), driven by `−`/`+` (or named steps) — **no continuous `Slider`** in the tree (assert no `Slider` present).
- [ ] The zoom control reads/writes **no** `MediaQuery.textScalerOf`/`TextScaler`; the zoom value flows only into E05's `Transform.scale` and never into any `TextStyle.fontSize` or the page's text-scaler — the muṣḥaf's zoom is independent of OS chrome text-scale.
- [ ] The theme control is a single-choice control over exactly `ReaderTheme.light`/`.sepia`/`.dark`; selected state is shape **and** localized text (never colour alone, SC 1.4.1); it sets `state.theme` via `setTheme`; *Night* is not added.
- [ ] `theme` selects E05's `ColorFilter` only — the control performs **no per-theme font swap**, sets no `fontFamily`, and touches no glyph; dark/sepia is a colour transform of the rendered layer (one font per page).
- [ ] The page slot composes exactly one `ColorFiltered` over one `Transform.scale(alignment: Alignment.topRight)` over the glyph+overlay `Stack` (E05's frame, not duplicated in the control); overlays sit inside the scale (they scale with the page).
- [ ] Neither control mutates engine state — no `review_log` write, no `Card` mutation, no `due_at`/instant re-derivation; both are display-only reader-state writes; a zoom/theme change re-tones/re-scales the rendered layer and introduces no `softWrap`/`TextPainter`/width-wrap and moves no printed line break.
- [ ] Neither control opens a socket or fetches anything at runtime; neither adds a confetti/streak/badge/toast/glow on a zoom or theme change; each interactive target is ≥48dp `touch.min`, carries an `l10n.*`-keyed `Semantics` label (values land in E13-T09), and references design tokens by name only; every new Dart file carries the REUSE SPDX header (`GPL-3.0-or-later`) and a `///` on each public declaration.

## Tests

Dart tests run in CI on every PR (`flutter test`); muṣḥaf goldens load the **real KFGQPC page fonts** via `FontLoader` (never Ahem, which draws squares) under an RTL `Directionality`. A throwing `HttpOverrides` is installed via the shared test bootstrap for every test below (the controls touch no socket — C-048). **Write the no-reflow / one-font-per-font widget assertions and the theme×zoom golden first.**

- `features/test/mushaf/reader_zoom_control_test.dart` — **zoom binding + OS-text-scale independence** (widget, `ProviderContainer`/`overrideWith` over a fake store): (a) tapping `+`/`−` calls `setZoom` with the next/previous band value exactly once and through no other path; (b) the band is bounded — at the max step `+` is a no-op, at the min step `−` is a no-op; (c) **no `Slider`** exists in the widget tree; (d) wrapping the control in a `MediaQuery` with a large `TextScaler` (e.g. `TextScaler.linear(2.0)`) does **not** change the value passed to `setZoom` and does **not** scale the page layer — the zoom value is the store's, never the OS text-scale; (e) the control reads/writes no `MediaQuery.textScalerOf` for the page (a grep over the source asserts the page-zoom path uses no `textScaler`).
- `features/test/mushaf/reader_theme_control_test.dart` — **theme binding + single-choice integrity** (widget): (a) selecting each of `light`/`sepia`/`dark` calls `setTheme` with that value exactly once; (b) exactly one option reads selected at a time, and the selected option exposes a `Semantics` selected value (radiogroup) and a text label, not colour alone; (c) the control sets no `fontFamily` and swaps no font per theme (assert the page's `MushafPageView` receives the same `pageNumber`/font across all three themes — only the `ColorFilter` differs); (d) *Night* is not selectable (only the three contracted values render).
- `features/test/mushaf/reader_no_reflow_test.dart` — **no-reflow / one-font-per-page** (widget, real or stub geometry): render the page slot at zoom `1.0` and at zoom `2.0` and assert the glyph lines' **relative** positions are identical (the page at `2.0` is a pure pixel scale of `1.0` — no line break moved, no re-wrap); assert no `softWrap`/`TextPainter`/width-wrap was introduced by the zoom/theme wrapper, and that the same dedicated `qpcFontFamily(page)` is used across `light`/`sepia`/`dark` and across zoom steps (one font per page, dark = filter not a "dark font").
- `features/test/mushaf/reader_zoom_theme_golden_test.dart` — **theme × zoom golden** over the **real KFGQPC fonts** (`FontLoader`, never Ahem) under `Directionality(TextDirection.rtl)`: render a representative page (page 1 with the basmala/surah banner) through E05's `mushafPageView` for the matrix `{light, sepia, dark} × {zoom 1.0, zoom ~2.0}`, `matchesGoldenFile` each; assert the `light`/`sepia`/`dark` goldens of the *same zoom* differ only by colour transform (identical glyph outlines/positions — the `ColorFilter` recolours, it does not move a glyph), and the `zoom 2.0` golden differs from `zoom 1.0` only by a uniform `topRight`-anchored scale (no line break moved). (The full per-locale/overlay golden matrix is E13-T10; this task supplies the control-level zoom×theme goldens E13-T10 composes with overlays/locales.)
- **Offline / no-network + banned-import guard** (eng-write-dart-test): the throwing `HttpOverrides` is active for every test above; a banned-import grep over both control source files finds no `http`/`dio`/`HttpClient`/`drift`/DAO symbol, no engine-mutation symbol, no `DateTime.now()`/`Calendar.current`, and (for the page-zoom path) no `MediaQuery.textScalerOf`/`TextScaler` driving the page.

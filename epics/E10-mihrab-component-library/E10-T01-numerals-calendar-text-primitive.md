## Goal

The shared leaf rendering primitive every other E10 component leans on exists in the `l10n` package: a single `numberFormatFor(locale)` numeral path (Extended Arabic-Indic ۰۱۲ for `fa`/`ckb` via `-u-nu-arabext`, Arabic-Indic ٠١٢ for `ar` via `-u-nu-arab`, pinned on the tag — never a sublocale default), a `localeDigits(int, locale)` shaper that is the only way an integer becomes text, and a `CalendarPresenter` that maps an E02 `CalendarDate` + the user's `CalendarSystem` (Jalālī · Hijri Umm al-Qurā · Gregorian) + locale into a localized date label with numerals remapped **downstream** of the calendar conversion and the run FSI/PDI-isolated. No surface in the library ever splices a raw ASCII digit (`"Page " + n.toString()`) or hardcodes a Gregorian grid again — they call this path. Backed by a test-first per-locale numeral golden, a Jalālī/Hijri format check, a "switch never mutates the stored day" assertion, and an offline guard.

## Implementation notes

TEST-FIRST: this primitive is correctness-critical pure logic (a wrong numeral block or a non-isolated run is a defect a native reader sees instantly). Write the unit cases in `numerals_test.dart` and the format cases in `calendar_presenter_test.dart` (below) **before** the implementation — the "fa/ckb render U+06F0-range, ar renders U+0660-range, never ASCII" assertion and the "Hijri never throws, falls back out of range" assertion must exist and fail before the bodies are written.

1. **Location (the `l10n` package, never the engine).** Create:
   - `packages/l10n/lib/src/numerals.dart` — `numberFormatFor(Locale)` (the one pinned-numbering-system `NumberFormat` factory) and `localeDigits(int, Locale)` (the only int→text shaper). Also expose a `localePercent(...)` for the heat-map cell's percentage and an integer formatter with grouping suppressed where a thousands separator would be wrong on a 3-digit page number.
   - `packages/l10n/lib/src/calendar_presenter.dart` — the `CalendarSystem { jalali, hijriUmmAlQura, gregorian }` enum and the `CalendarPresenter` value type (`const CalendarPresenter(this.system, this.locale)`) with `String format(CalendarDate d)`.
   - `packages/l10n/lib/src/localized_runs.dart` — `localizedPageJuz(...)` and `localizedPagesDue(...)`: the canonical mixed-run callers that format each token, isolate it, and inject it as an `AppLocalizations` ARB placeholder.
   Export the public surface (`numberFormatFor`, `localeDigits`, `localePercent`, `CalendarSystem`, `CalendarPresenter`, `localizedPageJuz`, `localizedPagesDue`) through the `l10n` package barrel; keep `lib/src/` internals private. REUSE SPDX header (`GPL-3.0-or-later`) on every new file.

2. **`numberFormatFor(locale)` — pin the numbering system on the tag.** Switch on `locale.languageCode`: `fa` → `NumberFormat.decimalPattern('fa-u-nu-arabext')`, `ckb` → `'ckb-u-nu-arabext'`, `ar` → `'ar-u-nu-arab'`. Never trust the bare-sublocale default — `intl`'s Eastern-digit emission disagrees between dates and numbers (12-impl §5). `localeDigits(value, locale)` is `numberFormatFor(locale).format(value)` and is the **only** path from an `int` to a chrome string; a raw `n.toString()` or an ASCII-digit string-replace is the exact bug this prevents (and is grep-banned in `features/**`). Suppress grouping for index-style numbers (a `pageNumber: 253` must render `۲۵۳`, never `۲۵۳` with a separator artifact).

3. **`CalendarPresenter.format(CalendarDate)` — convert first, remap numerals last (07 §4).** Read the pure `(year, month, day)` off the `CalendarDate` (it carries no zone, no clock — never widen it to a `DateTime` instant for a clock read; `DateTime.utc(y, m, d)` is permitted only as the pure calculator the package needs). Then:
   - **Gregorian** → `DateFormat.yMMMMd(locale.toString()).format(...)` (month names from `intl`).
   - **Jalālī** → `Jalali.fromGregorian(...)` (`shamsi_date`, BSD-3, pure Dart, offline); take the day / localized month name / year from the package's formatter.
   - **Hijri Umm al-Qurā** → `HijriCalendar.fromDate(...)` (`hijri`, BSD-2, pure Dart, offline); take `hDay`/`longMonthName`/`hYear`.
   The package output may carry Latin digits — that is fine; **remap downstream**: build the label from numeric fields each passed through `localeDigits(..., locale)` (preferred) or re-map the rendered string's ASCII digits through the numeral path, so no Latin digit ever reaches a widget on a Jalālī/Hijri date.

4. **Hijri is a labelled, caveated, range-guarded civil approximation (07 §6).** The Hijri branch returns the **Umm al-Qurā civil** label; the surface (the picker preview / the date row) carries the standing one-line honesty caveat ("Umm al-Qurā civil date — not an observance ruling") as an `AppLocalizations` key, never an assertion of the date of Ramaḍān/ʿĪd. Guard the supported AH/Gregorian range: on out-of-range **or** any conversion failure, return the Gregorian label silently and locally (no throw, no off-device log) — a date row must never crash. The presenter is sect/madhhab-neutral and issues no fiqh ruling; the caveat copy goes past domain-adab-and-religious-integrity.

5. **The calendar/numeral switch is a pure re-render (07 §2; 12-design §5).** `CalendarPresenter` is a `const`-constructible value object over `(system, locale)`; it reads a `CalendarDate` and mutates nothing — no stored instant, no `due_at`, no engine state. The same `CalendarDate.epochDay` formatted under jalali/hijri/gregorian and fa/ar yields different labels over a byte-identical underlying day. A "due in 7 days" interval is seven calendar days in every calendar — the calendar choice changes only the label (assert this; §3.5 below).

6. **Mixed runs are formatted then FSI/PDI-isolated into ARB placeholders.** `localizedPageJuz({page, juz, locale, l10n})` formats each number via `localeDigits` (`۲۵۳`, `۱۳`), wraps each in `isolateLtr(...)` (the known-direction digit run — prefer it over first-strong FSI per 12-impl §4; the helper is **imported** from eng-rtl-and-bidi-layout, never re-implemented here), then injects the isolated tokens as `AppLocalizations` placeholders (`l10n.pageJuz(pageToken, juzToken)` over `"صفحة {page} · جزء {juz}"`) — never `"Page " + ...`. `localizedPagesDue(count, ...)` shapes the count to locale digits and routes it through an ICU `plural` message (`l10n.pagesDue(count)`), so Arabic's six CLDR categories and digit shaping are both correct — never `"$count pages"`.

7. **Pitfalls to avoid:**
   - **A raw `toString()` / ASCII-digit splice.** `"Page " + n.toString()`, a hand-rolled ASCII→Eastern map, or a raw `int` passed into `Text`/an ARB message — all banned; every number flows through `localeDigits`/`numberFormatFor`.
   - **The wrong numeral block.** Shipping Arabic-Indic `٤٥٦` to a Persian reader (or the Extended set to `ar`) — the two blocks are distinct, non-interchangeable codepoints; the per-locale numeral golden (§ Tests) catches a swap.
   - **Remapping before converting.** Re-shaping digits before the calendar conversion, or letting the calendar package's Latin digits reach a widget — numerals are remapped *after* the conversion, never inside it.
   - **A view constructing a calendar object.** A `Jalali`/`HijriCalendar` built in a widget instead of asking the presenter — the conversion lives only in `CalendarPresenter`.
   - **A throwing Hijri conversion.** An out-of-range AH date that throws instead of falling back to Gregorian — the branch is range-guarded and total.
   - **Mutating from a render path.** Any write to a stored instant / `due_at` / engine state from formatting — rendering is a pure display transform.
   - **Reaching the muṣḥaf.** Routing the Quran page's printed ayah numbers / juz markers through `numberFormatFor` — the `intl` toolkit stops at the chrome; the glyph layer (E13) is never re-shaped (domain-mushaf-text-integrity).
   - **A network calendar/font source.** `shamsi_date` and `hijri` are bundled pure Dart; no CDN, no network calendar, no `google_fonts` runtime call.

## Acceptance criteria

- [ ] `packages/l10n/lib/src/numerals.dart` exposes `numberFormatFor(Locale)` whose numbering system is **pinned on the tag** — `fa-u-nu-arabext` / `ckb-u-nu-arabext` (Extended Arabic-Indic) / `ar-u-nu-arab` (Arabic-Indic) — never a sublocale default, and `localeDigits(int, Locale)` as the only int→text shaper; index numbers render with no spurious grouping separator.
- [ ] `localeDigits(456, Locale('fa'))` and `Locale('ckb')` contain only U+06F4 U+06F5 U+06F6 and no ASCII `4`/`5`/`6`; `localeDigits(456, Locale('ar'))` contains only U+0664 U+0665 U+0666 and no ASCII — proven by unit assertion.
- [ ] `packages/l10n/lib/src/calendar_presenter.dart` exposes `CalendarSystem { jalali, hijriUmmAlQura, gregorian }` and a `const`-constructible `CalendarPresenter(system, locale)` whose `format(CalendarDate)` converts via the calendar package (Gregorian `intl` / Jalālī `shamsi_date` / Hijri `hijri`) and **remaps numerals downstream** so no Latin digit reaches the output on a Jalālī/Hijri date; no view constructs a `Jalali`/`HijriCalendar`.
- [ ] The Hijri branch returns a labelled **Umm al-Qurā** civil date, carries the standing caveat copy (an `AppLocalizations` key), asserts no observance date, and **falls back to a Gregorian label out of range or on conversion failure rather than throwing** (total function).
- [ ] Formatting the same `CalendarDate` under different `CalendarSystem`/locale leaves the underlying `CalendarDate.epochDay` (and any caller's `due_at`) byte-identical — the presenter mutates no instant, `due_at`, or engine state.
- [ ] `localizedPageJuz(...)` formats each token via `localeDigits`, wraps each in the **imported** `isolateLtr` (not re-implemented), and injects them as `AppLocalizations` placeholders — no `"Page " + ...` concatenation; `localizedPagesDue(...)` routes a locale-shaped count through an ICU `plural` message (six `ar` categories).
- [ ] `CalendarDate` is imported read-only from the `engine` package and never widened to a `DateTime` instant for a clock read; the `l10n` package adds no Flutter import to the engine and the `engine` imports nothing from `l10n`.
- [ ] The public surface is exported through the `l10n` barrel; `lib/src/` internals stay private; every new file carries the REUSE SPDX header; `dart format` and `dart analyze --fatal-infos` are clean; every public member has a `///` doc comment.

## Tests

Unit logic in `packages/l10n/test/` with **`package:test`** (pure formatting, no widget binding); per-locale render goldens in `packages/l10n/test/` (or `packages/features/test/design_system/` when run through T02's `pumpComponentMatrix`) as Flutter golden tests on the **real bundled UI fonts** via `FontLoader` (never `Ahem`), `@Tags(['golden'])` for the pinned-OS lane, all suites under the throwing-`HttpOverrides` offline bootstrap. Fixture dates are explicit `CalendarDate.ymd(...)` values — no wall clock.

- `numerals_test.dart` (unit, **test-first**):
  - **Per-locale numeral block:** `localeDigits(456, …)` for `fa`/`ckb` contains only U+06F0-range digits and **no** ASCII; for `ar` only U+0660-range and **no** ASCII; a swapped block (Arabic-Indic to a Persian locale) fails.
  - **No grouping on an index:** a 3-digit page number renders three locale digits with no thousands separator; a percentage renders through `localePercent`.
  - **Pinned numbering system:** the formatter's locale carries the `-u-nu-` extension; a deliberately bare-locale formatter that emits ASCII is rejected (guards against trusting the sublocale default).
- `calendar_presenter_test.dart` (unit, **test-first**):
  - **Jalālī format:** `CalendarPresenter(CalendarSystem.jalali, Locale('fa')).format(CalendarDate.ymd(2026, 3, 21))` equals the documented Nowrūz-day Jalālī label with Extended Arabic-Indic digits (numerals downstream of the conversion).
  - **Hijri is Umm al-Qurā, range-guarded, total:** an in-range date yields an Umm al-Qurā-labelled Hijri label in the locale digits; an out-of-range date yields a **graceful Gregorian fallback, not an exception**; the standing caveat key accompanies the Hijri surface.
  - **Switch never mutates the stored day:** the same `CalendarDate` formatted under jalali/hijri/gregorian × fa/ar yields different labels over a byte-identical `epochDay`; a caller's `due_at` is unchanged across renders (07 §2; 12-design §5).
  - **Calendar-invariant interval:** "due in 7 days" is seven calendar days in all three systems (day-distance is calendar-invariant).
- `mixed_run_test.dart` (unit + a small bidi golden): `localizedPageJuz(page: 253, juz: 13, …)` wraps each token in LRI…PDI and the visual order stays page-then-juz under RTL (no "30 of 7" flip); `localizedPagesDue` places the locale-shaped count in the correct `ar` plural category.
- `numeral_calendar_golden_test.dart` (`@Tags(['golden'])`): for `fa`/`ckb`/`ar`, golden-capture a `localeDigits(456)` render, a `localizedPageJuz(253, 13)` run, and a `CalendarPresenter` date under each calendar — on the **real bundled UI fonts** (Persian digits and Sorani letters actually exercised), pinned DPR/size/theme, animations off; `--update-goldens` is local-and-reviewed (CI never blesses).
- **Offline / no-AI guard:** every suite runs under the throwing-`HttpOverrides` (`useOfflineTestPolicy()`); a negative test asserts a deliberate `HttpClient()` attempt from a formatting path throws the named `StateError` — `shamsi_date`/`hijri` rendering touches no socket and no model. The banned-import gate stays green over the new code.

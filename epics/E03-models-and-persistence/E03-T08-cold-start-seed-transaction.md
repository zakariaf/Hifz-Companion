## Goal

`ColdStartRepository.seedColdStart(Profile profile, List<CardSeed> seeds)` exists in `packages/data/lib/src/repositories/cold_start_repository.dart` as the cold-start provisioning write path: it opens **exactly one outer** `db.transaction` that inserts the `profile` row, **batch**-inserts the 600+ `card` seed rows (`batch.insertAll`), and inserts the profile's `cycle_config` row — all-or-nothing, `await`ing every query, so a failure mid-seed (a constraint trip, an FK violation, a disk error) leaves **zero** rows and no partially-provisioned profile. The returned `Future` resolves only after the durable WAL commit (persist-before-republish). The conservative-prior seed values themselves (Solid `D=3,S=60`; Shaky `D=5,S=14`; Rusty `D=7,S=4`; coverage capture; stale-time decay — PRD §7.10) are computed **upstream** and owned by **E11**; this method only persists the `CardSeed`s it is handed, recomputing nothing. A test asserting a mid-seed failure leaves the store at zero rows is written **first** (red before the body).

## Implementation notes

**TEST-FIRST (correctness-critical).** Write `cold_start_repository_test.dart`'s mid-seed-failure case (the zero-rows assertion) **before** the `seedColdStart` body. The test must compile and **fail** (red) against a stub/unimplemented `seedColdStart` first; only then implement the body to turn it green. A partially-provisioned profile (a `profile` row with half its cards, or cards with no `cycle_config`) is exactly the silent-corruption this guarantee forbids — proving all-or-nothing with a failing test before the code exists is the point.

1. **File**: `packages/data/lib/src/repositories/cold_start_repository.dart`, one primary type per file — `ColdStartRepository`. It wraps the injected Drift handle from E03-T05 (the `data`-package interface over `HifzDatabase`, reached as a constructor dependency / `ref.watch`ed `Provider`, never a global singleton). It depends on `models` and the E03-T06 DAOs; `package:drift`/`package:sqlite3` stay confined to `data` — no Drift symbol crosses into `engine`/`features`/`quran`. Re-export from the package barrel `packages/data/lib/data.dart`.

2. **`Profile` / `CardSeed` / `CycleConfig` are upstream outputs, consumed not recomputed.** `seedColdStart(Profile profile, List<CardSeed> seeds)` takes the immutable `Profile` value type, the list of 600+ `CardSeed`s, and the `CycleConfig` (carried on the `Profile` or passed alongside, mirroring engineering 05 §3's `p.cycleConfig`). A `CardSeed` is a `models`-level value type carrying the seed `track`, `d`, `s`, `CalendarDate dueAt`/`lastReviewedDay` (or `null` for an `UNMEMORIZED` un-held page), and `page_id`. `seedColdStart` does **no** D/S/`due_at` arithmetic and applies **no** stale-time decay — those are E11's engine math; this method persists what it is handed. If `CardSeed`/`CycleConfig` value types do not yet exist in `models`, resolve them as a typed dependency (E03-T01's record family / E11's cold-start I/O DTOs); do not re-derive a prior here.

3. **The transaction body — one outer `db.transaction`, every query `await`-ed, in §3 order**:
   ```dart
   // packages/data/lib/src/repositories/cold_start_repository.dart — data sees drift; features await this.
   Future<void> seedColdStart(Profile profile, List<CardSeed> seeds) async {
     try {
       await _db.transaction(() async {
         // 1. INSERT the profile FIRST — the card FK (profile_id) requires it to exist.
         await _profileDao.insert(profile.row);                  // await — the footgun
         // 2. BATCH-insert the 600+ card seeds in one batch (not 600 awaited inserts).
         await _cardDao.insertAllSeeds(seeds);                   // await — batch.insertAll
         // 3. INSERT the cycle_config (the cycle ceiling, named-preset config).
         await _cycleConfigDao.insert(profile.cycleConfig.row);  // await
       });
     } on CouldNotRollBackException catch (error) {
       // Even the ROLLBACK failed: surface as recovery-needed, log locally only, never transmit.
       throw const ColdStartWriteError.rollbackFailed();         // map, do not swallow
     } on DriftRemoteException catch (error) {
       throw const ColdStartWriteError.seedFailed();             // typed, no bare catch
     }
   }
   // When this Future resolves, every row is durably on disk (synchronous=FULL).
   // The onboarding controller (E11) republishes ONLY AFTER this returns — persist-before-publish.
   ```
   Every query inside `transaction(() async` is `await`-ed (the §3 footgun: an un-awaited query can run after the transaction closes → data loss). **`profile` is inserted before the card batch** because `card.profile_id` is a foreign key into `profile` (FKs are `ON`, E03-T04) — inserting cards first would trip the FK. The 600+ cards go through **one** `batch.insertAll`, not 600 individually-awaited inserts (the §3 batched-insert pattern). No nested transaction/savepoint is needed for the three steps here (it is available per §3 but unused). The method does not `print`/`debugPrint` the `Profile`, `seeds`, or any row (no logging of user data, §17).

4. **Un-held pages seed as `UNMEMORIZED`.** Per PRD §7.10 step 1, pages the user did not mark as held are seeded `track = 'UNMEMORIZED'` with `due_at = NULL` — the `card` `CHECK (track='UNMEMORIZED' OR due_at IS NOT NULL)` permits exactly this and rejects a held card with no `due_at`. Whether a given page is held vs un-held is E11's decision; this method persists the `CardSeed.track`/`dueAt` it is handed and relies on the `CHECK` to reject a malformed combination (which the all-or-nothing rollback then surfaces, leaving zero rows).

5. **Sealed persistence error type — one type, typed `catch`, never swallowed.** Define one sealed `ColdStartWriteError` (a sealed class / `freezed` union with `seedFailed`, `rollbackFailed`, `constraintViolated` cases) in `packages/data/lib/src/repositories/`, surfaced to the onboarding feature to handle exhaustively. Map Drift's `CouldNotRollBackException` and constraint/`DriftRemoteException` failures to it; every `catch` is typed (`on … catch (e)`, never bare `catch (_)`); no write error is swallowed. User-facing copy is **not** authored here — the onboarding layer renders a calm "couldn't finish setting up — try again" in fa/ckb/ar (E03-T05 boundary rule). The engine stays total and Drift-free; throwing is confined to this I/O boundary. (If E03-T07's `ReviewWriteError` was generalized to a shared `PersistenceWriteError`, reuse it rather than duplicating; otherwise a sibling sealed type is fine — do not gold-plate a premature abstraction.)

6. **Persist-before-republish is structural.** This method returns `Future<void>` and resolves only after the durable commit; the onboarding controller (E11) navigates to the first day / republishes state strictly after the awaited `seedColdStart` completes — never an optimistic "you're all set" before the 600+ cards are durably on disk. This task ships the repository method and its units only — the onboarding controller/`Notifier` and the cold-start placement UI that *call* it are E11. Do not republish, emit a stream value, navigate, or flip a UI flag inside `seedColdStart`.

7. **Determinism / clock.** `seedColdStart` reads no wall clock — the `CalendarDate` seed days and the `created_at` UTC instant arrive inside the `Profile`/`CardSeed` value types (the injected `Clock`/today entered upstream at the onboarding layer, E03-T05). The method never calls `DateTime.now()`. Tests inject the days as literal `CalendarDate`s and the instant as an explicit UTC `DateTime`.

8. **Pitfalls to avoid**: inserting cards before the `profile` row exists (trips the `card.profile_id` FK — `profile` first); 600 individually-awaited inserts instead of one `batch.insertAll` (slow, and not the §3 pattern); a missing `await` inside `transaction(() async` (the release-blocking footgun — the test and a grep audit catch it); splitting the seed across two transactions (loses all-or-nothing — a `profile` could commit without its cards); republishing / navigating / "all set" before the commit resolves (a window where onboarding claims success the disk does not hold); **recomputing** a seed (D, S) or applying stale-time decay in this method (E11's engine math is the only sink — 01 §4); writing a reference table (`page`/`mushaf`) — they are read-only, only referenced; a bare `catch (_)` or swallowed write error; `print`/`debugPrint`ing the profile/seeds (no user-data logging, §17); a Drift `Companion`/`TableInfo` escaping `data`; persisting derived health/streak/score; `synchronous=NORMAL` assumptions (the write connection is `FULL`, E03-T04).

## Tests

All tests are pure `dart test` under `packages/data/test/repositories/`, run against an in-memory `NativeDatabase.memory()` store (E03-T05's double, with `foreign_keys=ON` so the FK ordering is exercised) using a `FixedClock` and literal `CalendarDate`s — no `flutter_test`, no widget binding, no wall clock. The reference tables the seed references (`page`, `mushaf`) are pre-populated in `setUp` from a small fixture so FKs resolve. Each file carries the REUSE SPDX `GPL-3.0-or-later` header, uses full-word/unit-bearing names, typed `catch`, and asserts behaviour (row counts / commit state) not lines. The shared throwing-`HttpOverrides` offline bootstrap is installed so any stray network call fails loudly. Required file and cases (the mid-seed-failure case written **first**, red before the body):

- `cold_start_repository_test.dart`
  - **Mid-seed failure leaves zero rows (written first)**: a seed where one `CardSeed` is malformed (e.g. a held card with `due_at = NULL` that trips the `card` `CHECK`, or an injected `CardDao` that throws partway through the batch). Assert: after the call the store holds **zero** `profile` rows, **zero** `card` rows, and **zero** `cycle_config` rows for that profile — all-or-nothing, no partially-provisioned profile. Assert the method throws the sealed `ColdStartWriteError`, not a raw Drift exception, and republishes nothing.
  - **Happy path is exactly one transaction, all rows present**: a valid `Profile` + 604 `CardSeed`s (a mix of held FAR/NEAR/active and un-held `UNMEMORIZED`) + a `CycleConfig` commits; after the awaited call the store holds the one `profile`, all 604 `card` rows (counts match; held cards carry the seeded `(track, d, s, due_at)` verbatim — no recomputation; un-held cards are `UNMEMORIZED` with `due_at = NULL`), and the one `cycle_config`. A read-back card's `due_at` equals the `CardSeed.dueAt` exactly (CalendarDate-serial identity), proving consume-not-recompute.
  - **FK ordering holds**: the `profile` is committed before the `card` batch within the transaction (cards reference it) — proven by the happy path committing without an FK error, and by a variant where the injected `ProfileDao` throws asserting no `card` row landed.
  - **Persist-before-republish ordering**: a recording DAO/handle whose `insert` closures assert, at call time, that no observable in-memory republish has occurred yet — the method itself emits nothing mid-transaction (the controller's republish is out of scope here).
  - **Memory never newer than disk**: after a failed `seedColdStart` followed by a successful one (a retry), the store holds exactly the successful seed's rows — the failed attempt left nothing behind.

No golden, widget, or `integration_test` is in scope — this is a `data`-tier write-path unit. The end-to-end cold-start placement → first-day device journey is E11/E12; the onboarding/RTL goldens belong to those feature epics.

## Goal

The `onboarding` feature exists under `packages/features/lib/src/onboarding/` as the canonical four-file module ŌĆö a dumb `onboarding_screen.dart` `ConsumerWidget`, its 1:1 `onboarding_view_model.dart` `Notifier`, a `widgets/` leaf folder, and a scoped `onboarding_providers.dart` ŌĆö reached through the `/onboarding` `GoRoute` that already sits behind E07's readiness redirect guard. The ViewModel holds an immutable, finite, capped, back/resume-safe **captured state** for the whole flow ŌĆö `(locale, mushaf choice, download state, coverage set, JuzConfidence map, memorizedOn map, cycle preset, daily budget)` ŌĆö plus the step cursor and the commands to advance/go back. It is a pure capture surface: it writes **no** persisted rows (the placement commit is E11-T09), invents **no** `(D, S)`, reads "today" only from the injected `CalendarDate` clock, and never `DateTime.now()`. This task ships the scaffold + controller + step plumbing; the sibling tasks fill each step's View.

## Implementation notes

TEST-FIRST (correctness-critical): the captured-state model and the controller's cursor/back/resume behaviour are pinned by a `ProviderContainer.test()` suite written **before** the controller body ŌĆö the resume-rebuild, the back-without-loss, and the "no write path reachable from the controller" assertions must exist and fail first. The View widgets are not test-first (they arrive in the sibling step tasks).

1. **Folder anatomy** (`packages/features/lib/src/onboarding/`, per eng-add-feature-module ┬¦2): `onboarding_screen.dart` (the dumb `ConsumerWidget` host that reads one controller and shows the current step's View), `onboarding_view_model.dart` (the 1:1 `Notifier<OnboardingState>` ŌĆö the cursor + captured state + commands), `widgets/` (the shared step-chrome leaves this task introduces: a back/next bar, a step-progress affordance; the *step content* widgets are added by T02ŌĆōT08), and `onboarding_providers.dart` (scoped providers ŌĆö the `autoDispose` controller and any small per-step view-local state). One primary public type per file, `lower_snake_case`.

2. **The captured-state value type.** An immutable `OnboardingState` (`@immutable`, `copyWith`, value-equality) holding: `Locale? locale`; `MushafChoice? mushaf` (the named riw─üyah/edition value, not "the Quran"); `CorePackPhase downloadPhase` (`awaitingFirstDownload` / `downloading` / `downloadInterrupted` / `ready` ŌĆö the enum E05/E11-T04 owns; T01 references it, does not define the download mechanics); `Set<int> coverage` (held juz numbers 1ŌĆō30; absence = `UNMEMORIZED`, never a stored "0%"); `Map<int, JuzConfidence> confidence` (per held juz, from the engine's enum); `Map<int, CalendarDate> memorizedOn` (optional, sparse ŌĆö a juz absent here simply has no stale-time date); `CyclePreset? cyclePreset`; `int? dailyBudgetMinutes`; and `OnboardingStep cursor`. No `DateTime`, no `R`/`D`/`S` field, no "readiness %", no streak/score field ŌĆö the state is inputs only. Live the value type in the feature (it is UI-state, not a persisted model); the persisted `CardSeed`/`Profile`/`cycle_config` rows are E03/E11-T09's, never assembled here.

3. **The step cursor is finite and capped.** A sealed `enum OnboardingStep` enumerating exactly the ┬¦12.1 sequence (`welcomePrivacy`, `language`, `riwayahConfirm`, `download`, `coverage`, `confidence`, `whenMemorized`, `cyclePreset`, `done`) ŌĆö no open feed, no dynamic step injection. The controller exposes `next()` and `back()` commands that move the cursor with **guards**: `next()` only advances when the current step's minimal precondition is met (a locale chosen before leaving `language`; `downloadPhase == ready` before leaving `download` ŌĆö the fail-closed gate, so no coverage/confidence step is reachable before the muß╣ŻßĖźaf is verified); `back()` never loses already-captured values. The cursor is a pure function of the captured state where possible, so a rebuild lands on the right step.

4. **Back/resume-safe.** Going `back()` then `next()` must round-trip the captured value unchanged (e.g. re-opening `confidence` shows the juz the user already rated). Because no rows are written before E11-T09's commit, an app kill mid-flow simply restarts onboarding with an empty `OnboardingState` (the redirect guard re-routes here, E07) ŌĆö there is **no** half-captured persisted state to reconcile and **no** half-seeded `Card`. Document this "abandon = clean restart, commit-once-at-end" contract in the controller's `///` doc comment so a reviewer can see the *sanad*-safe property by construction.

5. **Provider shape** (eng-create-riverpod-store ┬¦2, ┬¦9): the controller is an `autoDispose`, `family`-keyed `NotifierProvider` keyed by the active `ProfileId` (so re-running placement for a second profile in teacher mode is isolated). It is **not** an app-scope singleton. Commands (`setLocale`, `setMushaf`, `toggleJuz`, `rate(juz, JuzConfidence)`, `setMemorizedOn(juz, CalendarDate)`, `clearMemorizedOn(juz)`, `setCyclePreset`, `setDailyBudget`, `next`, `back`) each return a new `OnboardingState` via `copyWith` and republish ŌĆö none of them touches a repository, a DAO, or `db.transaction`.

6. **"Today" is injected.** Any step that needs the current day (the `whenMemorized` "today" bound, or computing the `SerialDay` later handed to `coldStartCard`) reads `ref.watch(clockProvider).today()` returning a `CalendarDate`; the controller passes it down. **No `DateTime.now()` / `Calendar.current` / `TimeZone.current` anywhere** in the module ŌĆö a grep over the file is part of the gate.

7. **Route wiring** (eng-add-feature-module ┬¦9): confirm the `/onboarding` `GoRoute` exists under the app shell's `ShellRoute`/router and points at `OnboardingScreen`; E07 already installed the redirect guard that routes an un-ready device (no verified core pack / no profile) here before any Quran screen renders ŌĆö **do not** bypass it with an imperative `Navigator.push`, and **do not** add onboarding to the bottom-nav tab order (it is a pre-shell gate, not a tab). Inter-step movement is internal (the screen swaps the step View on cursor change), not a stack of `GoRoute`s.

8. **The View is dumb; chrome is RTL by construction.** `OnboardingScreen` reads the one controller, renders the step-chrome (back/next bar, progress affordance) and the current step's child View, and holds only show/hide + which-step-View routing ŌĆö no business `try/catch`, no repository call, no engine call in `build()`. Layout uses logical `start`/`end` (`EdgeInsetsDirectional`, `AlignmentDirectional`); back/next mirror via `Icons.arrow_*`; no hardcoded `Directionality` (RTL comes from the locale, E09). Design comes by token name (`color.*`/`type.*`/`space.*`/`motion.*`) ŌĆö no inlined hex/pt/ms. No gamified chrome: the step-progress affordance is a calm, non-celebratory indicator, never a streak/score/confetti.

9. **Strings deferred but keyed.** This task introduces no copy of its own beyond placeholder chrome labels; every user-facing string (including any "step N of M" counter) is an `AppLocalizations` key consumed through E09's pipeline and authored/transcreated in **E11-T10** (ar template + fa/ckb). Do not inline a literal; if a label is needed to compile the chrome, add the key in the ARB now and let T10 own the transcreated values.

10. **Pitfalls to avoid:** writing a `Card`/`cycle_config` row anywhere in T01 (that is E11-T09's single write path ŌĆö T01 must have **no** reachable `data` write); inventing `(D, S)` or storing a seeded `D`/`S`/`R`/readiness-% on the state; storing an un-held juz as a "0%"/"missing" value rather than mere absence from `coverage`; calling `DateTime.now()`; navigating from inside the controller; making onboarding a bottom-nav tab or bypassing the redirect guard; an app-scope (non-`autoDispose`, non-`family`) controller that leaks one profile's captured state into another; a `useState`/`setState` mutation of the captured state outside the controller; hardcoded left/right or a hardcoded `Directionality`; a second state/DI library (`get_it`/Bloc/legacy providers).

## Tests

Test-first; `package:flutter_test` + Riverpod test harness; runs in CI under both TZ pins. Controller tests use `ProviderContainer.test()` with `overrideWith` faking `clockProvider` (a fixed `CalendarDate`) and the active-profile provider ŌĆö no widget pump, no DB, no clock read.

- **`packages/features/test/onboarding/onboarding_view_model_test.dart`** (written first):
  - **Resume-rebuild:** after a sequence of capture commands (`setLocale`, `toggleJuz` ├ŚN, `rate`, `setMemorizedOn`, `setCyclePreset`, `setDailyBudget`), reading the state back yields exactly the captured values; a fresh container with an empty state lands on `welcomePrivacy` (clean restart).
  - **Back-without-loss:** `next()` to `confidence`, capture ratings, `back()` to `coverage`, `next()` again ŌåÆ the previously captured `confidence` map is byte-equal (round-trip preserves data).
  - **Cursor guards:** `next()` from `language` with no locale does not advance; `next()` from `download` with `downloadPhase != ready` does not advance (the fail-closed gate ŌĆö coverage/confidence are unreachable before the muß╣ŻßĖźaf is verified).
  - **Coverage absence semantics:** un-toggling a juz removes it from `coverage` (it becomes `UNMEMORIZED` by absence) and is **not** stored as a `0`/"missing" value; toggling does not touch `confidence`/`memorizedOn` for other juz.
  - **No write path / no seeds:** an assertion (by faking a repository provider and asserting zero interactions, plus a static grep test in CI) that no controller command reaches a `data` write; a check that the state exposes no `D`/`S`/`R`/readiness field.
  - **Injected clock:** `setMemorizedOn` and any `SerialDay` derivation use the overridden fixed `CalendarDate`; no real clock is read.
  - **`family` isolation:** the controller keyed on profile A and profile B hold independent captured states.
- **`packages/features/test/onboarding/onboarding_screen_test.dart`** (widget): pumping `OnboardingScreen` over a fake controller renders the cursor's step host and the back/next chrome; tapping next/back issues the controller command (not a `Navigator.push`); the screen makes no repository/engine call.

# E07-T02 — DI Provider set + ProviderScope composition root with throwing placeholders (engine, repository, clock, active-profile gate)

## Goal

`app/lib/composition/providers.dart` exists as the app's dependency-injection surface: the DI `Provider` set — `schedulerProvider` (the pure engine), `appDatabaseProvider`, `cardRepositoryProvider`, `clockProvider`, `activeProfileProvider` (`NotifierProvider<ActiveProfileController, ProfileId>`), and `appReadyProvider` — declared so that the IO-bearing placeholders (`appDatabaseProvider`, `appReadyProvider`) **throw when read un-overridden**, and `main()` wires the live services exactly once by opening the WAL Drift database and supplying it through `ProviderScope(overrides:)`. After this task the shell has one auditable composition root: the only file that sees the Drift database, the engine, the clock, and the asset gate at once; no global singleton, no `get_it`, no second DI container, and the active-profile `Notifier` is the sole profile gate every downstream `family` provider keys off. The router, the shell scaffold, the repository transaction body, the cold-start seed, and the Today queue are the *consumers* wired in by T03–T08 against this set; this task is the wiring, not the bodies.

## Implementation notes

This task is wiring, not arithmetic — but it is the structural enforcement point for "no global singleton" and "live services wired once," so the `ProviderContainer.test()` cases under **Tests** are written FIRST and prove the throwing-placeholder and single-wiring behaviour before `main()` is filled in.

1. **File**: `app/lib/composition/providers.dart`, the one DI surface (02 §2). Imports `package:flutter_riverpod/flutter_riverpod.dart` (**modern API only — never `package:flutter_riverpod/legacy.dart`**), the engine package (`Scheduler`), the data package (`AppDatabase`, `CardRepository`), the models package (`ProfileId`), and the `Clock`/`appReady` interfaces from T01. Every Dart file carries the REUSE SPDX header (`SPDX-License-Identifier: GPL-3.0-or-later`).

2. **`schedulerProvider`** — `final schedulerProvider = Provider<Scheduler>((ref) => const Scheduler());`. The engine is pure and `const`-constructible; it imports nothing from Riverpod or Flutter (04 §1.2; E04). It takes **no** injection — `today`/`card`/`grade` are plain parameters at the call site inside the repository. Never put interval/trust-clamp math behind this provider; it only hands back the engine.

3. **`clockProvider`** — `final clockProvider = Provider<Clock>((ref) => const LiveClock());`. The clock defaults to the live impl because it opens no IO at construction (eng-define-service-boundary piece 3). "Today" enters the shell **only** here, as a `CalendarDate` (T01 owns `Clock`/`CalendarDate` semantics); no `DateTime.now()` anywhere in this file or any consumer.

4. **`appDatabaseProvider`** — a **throwing placeholder**: `final appDatabaseProvider = Provider<AppDatabase>((ref) => throw StateError('appDatabaseProvider must be overridden in main()'));`. Reading it un-overridden is a loud startup failure, not silent null data (04 §1.2; eng-define-service-boundary piece 3). The live handle is constructed in `main()` only.

5. **`cardRepositoryProvider`** — `final cardRepositoryProvider = Provider<CardRepository>((ref) => CardRepository(db: ref.watch(appDatabaseProvider), scheduler: ref.watch(schedulerProvider), clock: ref.watch(clockProvider)));` exactly per 04 §1.2. The repository is the single write path and the data layer's single source of truth; features reach the engine/DB **only** through it (T05 writes its `recordReview` body — not here). No DAO is exposed as a provider; DAOs are reachable only from inside `data/` repositories.

6. **`activeProfileProvider`** — `final activeProfileProvider = NotifierProvider<ActiveProfileController, ProfileId>(ActiveProfileController.new);` where `ActiveProfileController extends Notifier<ProfileId>` exposes `build()` returning the current id and a `switchTo(ProfileId)` command that sets `state`. This is the **sole** profile gate (04 §1.2, §5): switching it recomputes every `family`-keyed downstream provider. No global mutable "current user", no second slot, no static field. `ProfileId` is a stable, equatable value from the models package. The controller does **not** persist a "last active profile" in this task (that is Settings/profiles, E16) — `build()` returns the seeded/first profile id; keep it minimal.

7. **`appReadyProvider`** — declared here as the readiness gate the router reads (04 §6): `final appReadyProvider = Provider<bool>((ref) => throw StateError('appReadyProvider must be overridden in main()'));`, a **throwing placeholder** overridden in `main()` from the T01 `appReady`/asset-gate boundary (core pack verified + a profile exists). It is a thin wire over the T01 boundary — no verification logic here (the SHA-256 fail-closed verifier lives in the `assets` package, E05). The router that consumes it is E07-T03; this task only exposes it.

8. **`main()`** in `app/lib/main.dart` — the composition root and the **only** place live services are constructed (04 §1.2): `WidgetsFlutterBinding.ensureInitialized();` then open the WAL Drift database once (the T01/E03 live handle, `synchronous=FULL`), resolve the live `appReady` boundary, then `runApp(ProviderScope(overrides: [appDatabaseProvider.overrideWithValue(db), appReadyProvider.overrideWith(...)], child: const HifzApp()));`. `clockProvider` needs **no** override (its `LiveClock` default opens no IO). `main.dart` contains **no business logic** — any logic in `app/lib/` is a review-blocking bug (02 §2 Pitfalls). The one-time core-pack download/verify trigger lives in `app/lib/bootstrap/first_run.dart` (E11/E05), not in `main()`'s body for this epic.

9. **App-scope lifetime**: none of `schedulerProvider`, `appDatabaseProvider`, `cardRepositoryProvider`, `clockProvider`, `appReadyProvider`, `activeProfileProvider` is `autoDispose`d — they live for the app's lifetime (04 §5). `autoDispose` + `family` belongs to the per-screen consumers (the Today queue, E07-T07), not the DI set.

10. **Pitfalls to avoid**: importing `flutter_riverpod/legacy.dart` (CI-failing grep); adding `get_it`/Bloc/`provider` as a second DI mechanism; constructing the live `AppDatabase` inside a provider body or a widget instead of `main()` (re-creation footgun); defaulting `appDatabaseProvider`/`appReadyProvider` to a live service that silently opens IO at import; putting verification/scheduling logic inside any provider (a provider is a thin wire); a `DateTime.now()` call anywhere reachable from this file; a global mutable "current profile" field shadowing `activeProfileProvider`; `autoDispose`-ing an app-scope singleton; opening any network socket from the composition root.

## Acceptance criteria

- [ ] `app/lib/composition/providers.dart` exists declaring exactly `schedulerProvider`, `appDatabaseProvider`, `cardRepositoryProvider`, `clockProvider`, `activeProfileProvider`, and `appReadyProvider`; it imports `flutter_riverpod` (modern API) and no `legacy.dart`, no `get_it`, no `provider`, no Bloc (verifiable by grep over the file).
- [ ] `appDatabaseProvider` and `appReadyProvider` are throwing placeholders: reading either from a `ProviderContainer` with no override throws (a `StateError`), proving a forgotten wiring fails loudly at startup rather than returning null.
- [ ] `schedulerProvider` returns the pure `const Scheduler()`; `clockProvider` returns the live `Clock` default (no override needed); neither performs any scheduling/date math in the provider body.
- [ ] `cardRepositoryProvider` constructs `CardRepository` from `appDatabaseProvider` + `schedulerProvider` + `clockProvider` and nothing else; no DAO is exposed as a provider.
- [ ] `activeProfileProvider` is a `NotifierProvider<ActiveProfileController, ProfileId>`; `switchTo(id)` changes the published `ProfileId`; there is no other profile state anywhere (no static field, no global "current user").
- [ ] `main()` (in `app/lib/main.dart`) opens the WAL Drift database once and supplies `appDatabaseProvider.overrideWithValue(db)` and the live `appReadyProvider` override in a single root `ProviderScope`; it constructs no other live service and contains no business logic; `clockProvider` is left un-overridden.
- [ ] No app-scope DI provider in the set is `autoDispose`d; no `DateTime.now()`/`Calendar.current`/`TimeZone.current` appears in `providers.dart` or `main.dart`; no networking symbol is imported by either file.
- [ ] Every public declaration carries a `///` doc comment; the file passes `dart format` and the analyzer/lint config; the REUSE SPDX header is present.

## Tests

`app/test/composition/providers_test.dart` (mirrors the source path), `package:flutter_test` + `ProviderContainer.test()`, no widget tree, run under `flutter test` in CI on every PR. A throwing `HttpOverrides` is installed in `setUpAll` so any stray network call from a wired provider fails loudly (offline guard, eng-define-service-boundary). Fixtures use the T01 deterministic fakes — `FixedClock(CalendarDate(2026, 6, 16))`, `AppDatabase(NativeDatabase.memory())`, and the fake-ready `appReady` double. Cases, written FIRST where they pin correctness-critical behaviour:

- **Throwing placeholders (test-first)**: `ProviderContainer.test()` with **no overrides**; `expect(() => container.read(appDatabaseProvider), throwsStateError)` and the same for `appReadyProvider` — proving an un-wired IO boundary is a loud failure.
- **Engine + clock defaults resolve**: `container.read(schedulerProvider)` returns a `Scheduler`; `container.read(clockProvider)` returns the live `Clock` with no override; reading either throws nothing.
- **Repository wiring**: with `appDatabaseProvider.overrideWithValue(AppDatabase(NativeDatabase.memory()))` and `clockProvider.overrideWithValue(FixedClock(...))`, `container.read(cardRepositoryProvider)` constructs a `CardRepository` whose injected `db`, `scheduler`, and `clock` are exactly the overridden instances (single-wiring identity).
- **Composition-root single wiring**: a helper that builds the overrides list (as `main()` does) yields a container where `cardRepositoryProvider` reads without throwing and the database instance is the one passed to `overrideWithValue` — i.e. the live handle is wired once, not re-created per read.
- **Active-profile gate**: `container.read(activeProfileProvider)` returns the seeded `ProfileId`; after `container.read(activeProfileProvider.notifier).switchTo(otherId)`, the published value is `otherId`; a downstream `family`-keyed test provider observes the new id (proving the gate recomputes dependents, no stale "current user").
- **Offline guard**: the throwing `HttpOverrides` from `setUpAll` is asserted in place; reading the full provider set against the in-memory fakes performs no network call.
- **No-legacy / no-second-DI grep** (CI gate, not a Dart test, referenced here): the E01 banned-import grep over `app/lib/` finds no `flutter_riverpod/legacy.dart`, `get_it`, `package:provider`, or Bloc import, and no `DateTime.now()` in `composition/` or `main.dart`.

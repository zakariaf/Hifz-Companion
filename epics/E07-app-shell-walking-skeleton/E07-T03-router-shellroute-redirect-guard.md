## Goal

`routerProvider` exists in `app/lib/src/composition/router.dart` as a `Provider<GoRouter>` reading the E07-T02 DI set — one `ShellRoute` hosting the persistent RTL bottom nav (Today · Muṣḥaf · Mutashābihāt · Progress · Settings, rightmost = home), typed deep-link route params (`/mushaf/page/:pageId` → `int.parse`), and the **redirect guard** that routes an un-ready device — core pack unverified *or* no profile, read through `appReadyProvider` — to `/onboarding` before any Quran-rendering route can resolve. A local-notification / deep-link tap is a typed location the router resolves only *after* the guard, so it can never bypass onboarding or surface an unverified muṣḥaf. The guard is built **test-first**: the redirect-guard suite — an un-ready device cannot reach a Quran route; a deep link to `/mushaf/page/3` resolves only after `appReady` flips true — exists and fails before the router body is written. The `HomeShell` chrome itself, the screen widgets, and live wiring in `main` are E07-T04; this task owns the route table and the guard, with the destination screens stubbed.

## Implementation notes

TEST-FIRST: write the redirect-guard suite (`router_redirect_test.dart`, cases below) **before** the router body. The "un-ready ⇒ cannot reach a Quran route" and "deep link resolves only after `appReady`" cases must exist and fail before `routerProvider` is implemented — the guard is correctness-critical (it is R1 in code).

1. **File**: `app/lib/src/composition/router.dart`, one primary declaration — `final routerProvider = Provider<GoRouter>((ref) { … });`. It lives in the `app/` composition root (the only target that may depend on every package), **not** in a feature; a feature never declares the router (`eng-add-feature-module` "Do NOT use … for the thin `app/` shell").
2. **Redirect guard** — copy the §6 shape exactly: read `final ready = ref.read(appReadyProvider);` (the gate = core pack verified **and** a profile exists, composed in T01/T02), `final onboarding = state.matchedLocation.startsWith('/onboarding');` then `if (!ready && !onboarding) return '/onboarding'; if (ready && onboarding) return '/today'; return null;`. Use `ref.read`, not `ref.watch`, inside the closure (the guard is consulted per navigation, not a build dependency); when `appReady` flips, drive a `GoRouter` refresh via a `refreshListenable` bound to the `appReadyProvider` so an in-flight deep link re-resolves — wire the listenable from the provider, do not poll.
3. **Route table** — one top-level `GoRoute(path: '/onboarding', …)` (builder → the minimal onboarding stub for this task; the real `OnboardingFlow` is E07-T06) **outside** the shell, then one `ShellRoute(builder: (_, __, child) => HomeShell(child: child), routes: [...])` containing the five tab `GoRoute`s in RTL order: `/today`, `/mushaf` (with nested `GoRoute(path: 'page/:pageId', …)`), `/mutashabihat`, `/progress`, `/settings`. `initialLocation: '/today'`. Destination builders are minimal `const` stub widgets here (each a keyed `Semantics`-labelled `SizedBox`/`Placeholder`-style sentinel with a stable a11y identifier) so the route table and guard are testable before T04 supplies real screens; `HomeShell` is referenced by type from T04 (a thin pass-through stub if T04 has not landed).
4. **Typed deep-link param**: the nested route parses `int.parse(state.pathParameters['pageId']!)` and passes the `int pageId` to the (stub) reader builder — the typed boundary §5 requires; never pass the raw `String` downstream. A malformed `pageId` (non-int) must fail closed to the guard/onboarding or a calm not-found route, never throw into the render tree.
5. **`appReady` semantics**: this task consumes the boolean gate; it does **not** compute verification or profile existence (T01 owns the asset/`appReady` boundary, T02 wires the providers). The router only asks `ref.read(appReadyProvider)`. Do not read `clockProvider`/the DB here — the router computes nothing.
6. **No navigation from logic; no imperative push.** All primary navigation is typed `go_router` locations under the one `ShellRoute`; the bottom nav switches branches via `context.go('/<tab>')` (or the shell's branch navigator), never `Navigator.push(MaterialPageRoute(...))`. A notification/deep-link tap (E18) enters as a location string the router resolves *after* the guard — no controller calls the router.
7. **RTL by construction**: nav order and any directional chrome use logical start/end only — never hard-coded left/right; the bar mirrors under the app-wide RTL `Directionality` (all three locales are RTL). This task ships no nav *labels* itself (T04 owns the `HomeShell` chrome + localized labels); it only fixes the **route order** Today · Muṣḥaf · Mutashābihāt · Progress · Settings so T04 inherits it.
8. **Pitfalls to avoid**: `ref.watch` inside `redirect` (rebuild storm); reading the gate once at provider-construction instead of per-navigation (a device that becomes ready mid-session stays stuck on `/onboarding` — the missing `refreshListenable`); an imperative `Navigator.push` that bypasses the guard and renders an unverified muṣḥaf (R1 breach); a router that observes connectivity or fetches a route (there is no server — C1/C-048); leaving the `/mushaf` and `/mushaf/page/:pageId` routes reachable while `!ready`; `autoDispose` on `routerProvider` (it is an app-scope singleton); passing the raw `String` `pageId` downstream; constructing a live service inside the provider (placeholders are overridden in `main` — T04).

## Acceptance criteria

- [ ] `app/lib/src/composition/router.dart` declares `final routerProvider = Provider<GoRouter>(...)`; it reads `appReadyProvider` (and nothing it should not — no DB, no clock, no network import); it is app-scope, not `autoDispose`d.
- [ ] The route table is exactly: top-level `/onboarding` outside the shell; one `ShellRoute` containing `/today`, `/mushaf` (+ nested `page/:pageId`), `/mutashabihat`, `/progress`, `/settings` in that RTL order; `initialLocation: '/today'`.
- [ ] The redirect guard returns `/onboarding` for an un-ready device on any non-onboarding location, `/today` for a ready device on `/onboarding`, and `null` otherwise — matching §6 verbatim.
- [ ] **No Quran-rendering route resolves while `appReady` is false**: a request to `/mushaf` or `/mushaf/page/:pageId` on an un-ready device lands on `/onboarding`, never on the reader stub.
- [ ] A deep link (e.g. `/mushaf/page/3`) issued while un-ready is held at `/onboarding`, then **re-resolves to the reader route once `appReady` flips true** via the `refreshListenable` — proving the notification/deep-link tap is resolved only after the guard.
- [ ] The `page/:pageId` param is parsed to `int` via `int.parse` and passed typed; a non-int `pageId` fails closed (guard/onboarding or a calm not-found), never throwing into the render tree.
- [ ] No imperative `Navigator.push`/`MaterialPageRoute`, no connectivity observation, no networking import anywhere in `router.dart` (grep-verifiable); no `ref.watch` inside the `redirect` closure.
- [ ] Every `public` declaration carries a `///` doc comment; the file carries the REUSE SPDX header (`GPL-3.0-or-later`) and passes the analyzer/lint config and `dart format`.

## Tests

`app/test/composition/router_redirect_test.dart` (mirrors the source name), `flutter_test` + `ProviderContainer.test()` / `MaterialApp.router` pump, run with `flutter test` in the fast CI job. Written **FIRST**. `appReadyProvider` and `activeProfileProvider` are faked via `overrideWith` (the T01 fake-ready / not-ready doubles); no live DB, no asset IO, no real `OnboardingFlow`/reader — the destination stubs carry stable a11y identifiers the test finds. The shared bootstrap installs the throwing `HttpOverrides` (the router must touch no socket). Required cases:

- **Un-ready ⇒ onboarding (the guard).** Override `appReadyProvider` → `false`; pump the router at `initialLocation: '/today'`; assert the resolved location is `/onboarding` and the onboarding stub identifier is present.
- **Un-ready ⇒ Quran route unreachable.** With `appReady = false`, navigate to `/mushaf` and to `/mushaf/page/5`; assert the resolved location is `/onboarding` and **no** reader stub renders — a ḥāfiẓ-facing R1 proof.
- **Ready ⇒ leaves onboarding.** Override `appReadyProvider` → `true` starting at `/onboarding`; assert redirect to `/today`.
- **Deep link resolves only after appReady.** Start un-ready, issue `context.go('/mushaf/page/3')` → held at `/onboarding`; flip the faked gate to ready (notify the `refreshListenable`) → assert the router now resolves to `/mushaf/page/3` and the parsed `pageId == 3` reaches the reader stub. This is the test-first acceptance for "a deep link resolves only after `appReady`."
- **Typed param parse.** `/mushaf/page/12` yields `int pageId == 12` at the stub; a non-int (`/mushaf/page/x`) fails closed (guard/onboarding or not-found), no exception escapes into the tree.
- **RTL route order.** Assert the `ShellRoute` child routes are declared in the order Today · Muṣḥaf · Mutashābihāt · Progress · Settings (the contract T04's chrome inherits).
- **Offline guard.** The throwing `HttpOverrides` is installed; the whole suite passes with zero network attempts (a stray connection is a named failure).

The end-to-end notification-tap-after-guard path on the real onboarding/reader is exercised by the E07-T10 `integration_test` spine journey; this task's unit/widget suite is the deterministic, fakes-only proof. CI no-network / banned-import / `DateTime.now()` gates stay green over the new file.

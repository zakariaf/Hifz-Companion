# E09-T08 — Calendar-display layer: render E02's converted (y,m,d) into locale numerals, isolate the run, CLDR week-start; no conversion here

## Goal

A render-only calendar-display chrome layer exists in the `l10n` package that takes a date **E02 already converted and already digit-remapped** — `CalendarPresenter(system, locale).format(CalendarDate)` (Hijri Umm al-Qurā / Jalālī / Gregorian, with its numeric fields already in the locale block via E02-T06's `toLocaleNumerals`) — and places it correctly inside an RTL line: it runs that label through E09-T05's `bidi.dart` isolation helper before display, and it exposes the locale's CLDR-driven week-start (Saturday for `fa`/`ar`) for any week-grouping chrome, never a hardcoded Monday/Sunday. The Settings calendar picker selects an explicit `CalendarSystem` (default Jalālī for `fa`) that is threaded *into* the presenter — it never infers the calendar from `Locale.current`. This task performs **no** calendar conversion and **no** `CalendarDate` math (both stay in E02): it owns only the bidi isolation of the converted run, the week-start lookup, and the picker→`CalendarSystem` wiring.

## Implementation notes

This task is correctness-sensitive at the bidi boundary — an un-isolated date run reorders the line ("30 of 7") and scrambles the screen-reader order — but the *digits* and the *conversion* are already correct when this layer receives the string (E02). Write the isolation and week-start assertions alongside the code; do not let a date reach a widget un-isolated, and do not re-introduce conversion or numeral math here.

1. **File & package boundary.** New source lands in `packages/l10n/lib/src/calendar/calendar_display.dart` (the `l10n` package, the same one holding `bidi.dart`/`numerals.dart`/`icon_mirror_policy.dart`), exported from the package barrel `packages/l10n/lib/l10n.dart`. `engine/` is untouched (no strings, no direction, no Flutter import); the E01 banned-import gate must stay green — this file may import `package:flutter`, `package:intl`, the `l10n` siblings (`bidi.dart`), and E02's `CalendarPresenter`/`CalendarSystem` (re-exported from the `l10n` barrel by E02-T05), but it must **not** import `package:hijri`, `package:shamsi_date`, or any `CalendarDate`-math symbol beyond the value type it passes through.
2. **`CalendarDisplay` — the render-only seam.** Add a small, total helper (a plain function or a `const`-constructible value type, not a widget that owns state) whose single job is: given a `CalendarPresenter` and a `CalendarDate`, produce the **isolated** label string ready to drop into an ICU `{date}` placeholder. Sketch:
   ```dart
   /// Render a converted, locale-numeralled date for placement in an RTL line.
   /// Conversion + numeral remap are E02's; this isolates the run for bidi safety.
   String isolatedDateLabel(CalendarPresenter presenter, CalendarDate date) {
     final converted = presenter.format(date); // E02: converted + digit-remapped
     return isolateRtl(converted);             // E09-T05 helper — known RTL direction
   }
   ```
   The label is **known-direction** (a localized month name + locale-block digits → RTL content), so use `isolateRtl` (RLI…PDI), not `isolate` (FSI), per impl-12 §4 / the skill Rule 6 — FSI's first-strong would mis-guess if the string opened with a digit. Do not split the string; isolate it as one run.
3. **The ICU placeholder, not concatenation.** A date-bearing chrome label (e.g. Today's "next due / last reviewed") is one ICU message with a `{date}` placeholder authored in `app_ar.arb` (E09-T01 pipeline, `eng-add-localized-string`), and the isolated string from step 2 is injected as that placeholder — never `'…' + dateString` and never a hard-spliced substring (impl-12 §4; ds-12 §3). The localized label stays in a single `Text`/`TextSpan` (ds-12 §3). The date value reaching the placeholder is **already** the isolated, digit-correct run; the ARB value itself carries no physical-side or hardcoded-RTL assumption (the surrounding chrome direction is locale-derived, E09-T04).
4. **Week-start from CLDR, not a constant.** Expose the locale's first-day-of-week from CLDR `weekData` for any week-grouping chrome (a Progress week roll-up, a date-picker grid header) — Saturday for `fa`/`ar`. Read it from `intl`/`MaterialLocalizations` locale data (e.g. `MaterialLocalizations.of(context).firstDayOfWeekIndex`, which is CLDR-sourced), or a thin `firstDayOfWeekFor(locale)` that defers to that data — never a literal `DateTime.monday`/`DateTime.sunday`. The Hijri/Jalālī calendar choice does **not** change the week-start (week-start is locale data, calendar is a separate display transform); both pass through unchanged. This task ships the lookup; the actual week-grid widget is a later feature epic's consumer.
5. **The Settings calendar picker selects an explicit `CalendarSystem`.** The picker (built fully in E16 settings) sets a `CalendarSystem { jalali, hijriUmmAlQura, gregorian }` value that is threaded into the `CalendarPresenter` this layer renders with. This task wires the *selection→presenter* path and the **default** (Jalālī for `fa`, Hijri/Gregorian offered to every locale), making explicit that the calendar is a chosen value, never read off `Locale.current` (07 §4 refusal; ds-12 §5). No `CalendarSystem.fromLocale` factory is added here or anywhere. The picker option *labels* ("Jalālī", "Hijri (Umm al-Qurā)", "Gregorian") are ARB strings (E09-T01); selecting one is a pure display transform that mutates no engine state and no due-date (ds-12 §5).
6. **TEST-FIRST for the isolation guard.** A date reaching a widget un-isolated is a release-gate-blocking RTL/accessibility defect that is silent in any test only checking the date is "non-empty". Write the "the rendered date run is wrapped in RLI…PDI" assertion and the "no raw concatenation of the date into chrome copy" guard **before** wiring `isolatedDateLabel` into a label — the isolation expectation must exist and fail (the bare presenter output carries no isolate controls) before the helper is added.
7. **Pitfalls to avoid.** Converting the calendar here (constructing a `Jalali`/`HijriCalendar`, importing `hijri`/`shamsi_date`) — that is E02's, and it breaks this layer's purity (the §5 boundary); remapping numerals here (re-running `numberFormatFor` over an already-remapped string, or worse a blanket `replaceAll` — E02-T06 already did it, the digits are correct on arrival); using `isolate` (FSI) where the run's RTL direction is known (first-strong mis-detects on a leading digit — use `isolateRtl`); hard-splicing the date into chrome copy with `+` instead of an ICU `{date}` placeholder (RTL word order will not survive); hardcoding a Monday/Sunday week-start instead of reading CLDR `firstDayOfWeekIndex` (Saturday for fa/ar); inferring the calendar from `Locale.current` or adding a `fromLocale` factory (07 §4 / ds-12 §5 refusal); letting this date path touch a muṣḥaf ayah number (the printed page numbers are the immutable glyph layer, never re-rendered by `intl` — ds-12 §8); reading `DateTime.now()` for "today" or the week (today is injected — 07 §5); `EdgeInsets.only(left:/right:)` or a hardcoded `Directionality` in any surface this touches (direction is locale-derived — E09-T04).

## Acceptance criteria

- [ ] `packages/l10n/lib/src/calendar/calendar_display.dart` exists and is re-exported from `packages/l10n/lib/l10n.dart`; it lives in the **display-layer** `l10n` package, never `engine/`; the E01 banned-import gate stays green (no `hijri`/`shamsi_date`/`CalendarDate`-math symbol imported here, none reachable from `engine/`).
- [ ] The layer takes E02's `CalendarPresenter.format(CalendarDate)` output — already converted **and** already locale-numeral-remapped — and performs **no** conversion and **no** numeral remap of its own (no `numberFormatFor`/`replaceAll`/`Jalali`/`HijriCalendar` in this file).
- [ ] The converted date run is isolated via E09-T05's `bidi.dart` helper before display, using `isolateRtl` (known RTL direction), not `isolate`/FSI; the isolated value is injected into an ICU `{date}` placeholder, never concatenated or hard-spliced, and stays in a single `Text`/`TextSpan`.

- [ ] No `DateTime.now()` in this file; "today" and any date arrive as injected `CalendarDate` values (E02-T04 `todayProvider`); the file reads no wall clock.

- **Digits pass through untouched (boundary proof)**: the isolated label's numeric characters are exactly the locale block E02-T06 produced — Extended Arabic-Indic (U+06F0–U+06F9) for `fa`/`ckb`, Arabic-Indic (U+0660–U+0669) for `ar` — and contain zero ASCII digits, confirming this layer neither re-maps nor corrupts the digits (it isolates only).
- **Week-start is CLDR Saturday for fa/ar**: assert the resolved first-day-of-week index for `fa` and `ar` is Saturday (and not Monday/Sunday); assert the value comes from locale data, not a constant; assert it is identical across `CalendarSystem` choices for the same locale (calendar does not change week-start).
- **Calendar is the explicit parameter (no locale inference)**: render the same `CalendarDate` for `fa`+`gregorian` and `ar`+`hijriUmmAlQura` (calendars offered to every locale) and confirm each produces the system-appropriate isolated label — proving the calendar is the threaded `CalendarSystem`, never read off `Locale.current`; assert no `fromLocale` path exists.
- **Default for `fa` is Jalālī**: the picker's default `CalendarSystem` for `fa` is `jalali`; selecting `hijriUmmAlQura` then re-rendering changes only the label, not any engine/due-date state (display-transform proof).
- **No conversion/numeral math here (purity)**: a grep/import test that `calendar_display.dart` imports no `package:hijri`/`package:shamsi_date`, defines no `numberFormatFor`, and contains no ASCII-digit `replaceAll` chain — the conversion and numeral remap are provably E02's.

No engine golden vectors here (this layer originates no schedule and no number). The per-locale RTL+numeral **golden screenshot** of the isolated, digit-correct date on the **real** bundled `fa`/`ckb`/`ar` UI fonts belongs to **E09-T10** (and the E02-T10 date gate shares the T8 numeral vector); this task asserts the string-level isolation, week-start, and picker wiring.

## Goal

In the pure-Dart `engine/`, a logged wrong-branch swap raises `Difficulty` on **every** member of the confusable group — not only the page that was just recited — by adding a named confusion bump to `D` (clamped `[1,10]`) inside the **one** `onReview`/group-resolution path. Because the FSRS stability equation carries the `(11−D)` factor, a higher `D` automatically yields a smaller stability gain and therefore a **shorter interval** on each bumped member, so the unpracticed twin is pulled forward and never left to be suppressed — with **no bespoke frequency override** and **no second scheduler**. The behaviour is deterministic (no `DateTime.now()`, no `Random`; `today` injected) and pinned test-first: a frozen golden vector proving the per-member `D` rise → shorter `due`, and a `glados` invariant property proving *every* group member is bumped (never just the recited node), both exist and **fail** before the bump is wired.

## Implementation notes

**TEST-FIRST:** write the engine golden vector + the `glados` invariant property in `## Tests` below **before** wiring the bump. The "per-member D rise → shorter `due`" frozen row and the "every group member is bumped, not just the recited node" property must exist and **fail** before the bump code is written.

1. **Where the bump lives — the single engine path, in `engine/`.** Extend the existing review path (`packages/engine/lib/src/review.dart` or wherever E04 placed `onReview`) so a confused page's `D` is bumped through the **same `D` channel** as the weak-line factor, *not* a new function that recomputes intervals. The bump enters exactly where §4 adds the weak-line term:
   ```dart
   // Confusion channel: a logged swap bumps D on EVERY group member; the
   // (11−D) factor in stabilityOnSuccess turns the higher D into a shorter
   // interval automatically — no bespoke frequency override. PRD §9.2; science 05 §4.
   d = (d + kConfusionDifficultyBump * confusionWeightOf(card)).clamp(1.0, 10.0);
   ```
   `kConfusionDifficultyBump` is a **named constant** in the engine constants block (§8), beside `kWeakLineFactor`/`kLapseDifficultyBump` — never an inline literal. The bump is additive into `D`; `trustClamp` still runs **last** (so the bump can only pull a page *forward*, never past the cycle ceiling — INV-1 holds unchanged).
2. **"Every group member" is the whole point — model the bump as a group operation, not a per-card field read.** A swap is a property of the **edge/group**, so the bump must reach the *unpracticed* sibling that was **not** recited today. Two clean ways, pick the one that fits E04's seam (and keep it pure):
   - a pure `applyConfusionBump(List<Card> groupMembers, ConfusionContext ctx) → List<Card>` that maps the bump over **all** members of the confusable group (resolved from the T01 dataset + T03 graph passed in as data), each member re-run through the `D` clamp; or
   - `onReview` accepts the recited card plus the **group context** (members + the relevant `confusion_edge` weight) so the same update emits the bumped `D` for the recited card *and* a list of bumped siblings.
   Either way the engine receives group membership and edge weight **as injected data** (the dataset/graph are loaded by T01/T03 and handed in by the shell) — the engine never reads Drift, never opens the dataset file, never derives "which verses are similar" at runtime (no AI/inference; PRD C2).
3. **Full strength regardless of source — the bump is not throttled by self vs teacher.** Per §4, the confusion signal applies at full strength whether the swap was self-reported or teacher-flagged; `sourceConfidence` (`kSelfConfidence`/teacher `1.0`) scales **only** the stability *gain* in `onReview`, which is already there and **must stay untouched**. So `applyConfusionBump` takes **no** source/confidence parameter that scales the `D` bump — a self-reported swap bumps `D` identically to a teacher-flagged one (the mutashābihāt skill's Do/Don't: "Don't drop a self-reported swap because it wasn't a teacher sign-off").
4. **`confusionWeightOf` / the bump magnitude is a small pure function — bounded, monotonic, no ML.** The bump scales with the pair's logged `confusion_edge.weight` (more swaps → larger `D` bump → shorter interval), but it is a plain, bounded, deterministic function of the user's own logged history — never an inferred or trained value, never unbounded. Keep `kConfusionDifficultyBump * weight` clamped by the `[1,10]` `D` clamp so a heavily-confused pair cannot push `D` out of range. The `weight` itself is grown by T03's `logSwap`; this task only **consumes** it.
5. **No second scheduler, no bespoke interval math.** The shorter interval comes **only** from the existing `(11−D)` factor in `stabilityOnSuccess` — this task must **not** add a `nextDue = …` override, must **not** divide an interval, must **not** introduce a "confusion penalty days" term, and must **not** touch `buildToday`/`expandMutashabihat` (sibling-massing is E14-T05). A grep over the change should find no new interval/`due_at` arithmetic — only a `D` addition feeding the unchanged FSRS equation.
6. **Determinism + purity preserved.** No `DateTime.now()`, no `Random`, no Flutter, no Drift import anywhere in the change; `today` stays the injected `SerialDay`; identical `(card, group, edge, grade, today)` inputs produce a byte-identical bumped result. The `onReview is pure` and the determinism (INV-4) properties must still hold after the change.
7. **Pitfalls to avoid:**
   - **Bumping only the recited page** and leaving the unpracticed twin to rot — the exact retrieval-induced-forgetting failure the invariant property must catch; the bump maps over **every** group member.
   - **A bespoke frequency/interval override** beside `onReview` — the shorter interval must fall out of `(11−D)`; no `nextDue`/penalty-days math.
   - **Throttling the `D` bump by source** — full strength always; `sourceConfidence` scales only the existing S move.
   - **An inline bump literal** instead of the named `kConfusionDifficultyBump` constant — breaks the §8 "constants by name" rule and the FSRS-6 re-freeze story.
   - **Reading the dataset/graph inside `engine/`** (a Drift call, a file read, a runtime "similar verses" derivation) — group membership + edge weight arrive as **injected data**; the engine stays pure (no AI/inference, no I/O).
   - **Letting the bump escape the `[1,10]` clamp** or bypass `trustClamp` — a bumped page is still clamped to its cycle ceiling (INV-1); the bump may only pull it *forward*.
   - **Generating the golden vector from the engine under test** — compute the expected `(D, S, due)` once from `dart-fsrs` (fuzz OFF) so a fixture cannot agree with a bug.

## Tests

Engine vectors + properties live in `packages/engine/test/` (mirroring E04's layout — e.g. `packages/engine/test/vectors/confusion_bump_vectors_test.dart` and `packages/engine/test/invariants/confusion_bump_invariant_test.dart`), `package:test` (+ `package:glados` for properties), run under `dart test packages/engine/`. **No** `flutter_test`, **no** widget binding, **no** font, **no** `HttpOverrides` (the engine is pure Dart). `today` is a constructed `SerialDay` literal (`day(130)`); elapsed days is integer subtraction. Required cases, written **FIRST**:

- **Per-member D rise → shorter due (golden vector):** a frozen `FsrsVector`-style row — a card with a baseline `(D, S)` reviewed `Good` with **no** swap vs the **same** card with a logged swap of weight `w` — asserts the swapped run's `D` is exactly `baselineD + kConfusionDifficultyBump * f(w)` (clamped) and its resulting `due` offset is **strictly smaller** than the no-swap run's, both to `closeTo(_, 1e-6)`. The expected `(D, S, due)` is computed once from `dart-fsrs` (`enableFuzzing: false`, `kDefaultWeights45`), never from the engine under test.
- **Every group member is bumped (invariant property):** a `glados` property over a generated `(group of N≥2 members, swap edge, grade-sequence, today)` history asserts that after a logged swap **all** members of the group have `D' ≥ D` and `due' ≤ due` (each pulled forward), and that **no** member is left at its pre-swap `D` — i.e. the unpracticed twin (the member *not* recited today) is bumped too. Comment restates C-029 / science 05 §4 at the assertion. Rely on shrinking for the minimal counterexample; no fixed lucky seed.
- **Full strength regardless of source:** two runs differing **only** in `Source` (self vs teacher) produce an **identical** `D` bump (the bump is not source-scaled); the stability *gain* still differs by `kSelfConfidence` exactly as before (proving only the S move is confidence-scaled, the `D` bump is not).
- **No bespoke override / `(11−D)` does the work:** assert the swapped run's interval equals what `interval(s', targetR)` yields for the bump-shortened `S'` — i.e. the shorter `due` is fully explained by the `(11−D)`-driven smaller stability gain, with no extra subtraction; a property that the change introduces no interval term outside `stabilityOnSuccess`.
- **Clamp + trust-clamp respected:** a pathologically heavy `confusion_edge.weight` cannot push `D` above `10` (clamp holds) and a bumped page's `due` never exceeds `cycleCeilingDays(card, config)` (INV-1 still holds — the bump only pulls forward).
- **Determinism unchanged:** the existing `onReview is pure` / INV-4 (two runs fingerprint-equal) properties still pass with the bump wired; no `DateTime.now()`/`Random` is reachable.

All cases run under `dart test packages/engine/` in CI on every PR (gate 3 golden vectors + gate 4 invariant properties of the PRD §20 release contract); vectors are regenerated only via the human-reviewed `--update-vectors` flag, never auto-blessed.

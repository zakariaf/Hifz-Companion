# Local Persistence in Flutter with Drift over SQLite — Research Note

**Topic:** Drift on SQLite for Hifz Companion's local-first store — schema, migrations, transactional crash-safety, and optional at-rest encryption (the 2026 `sqlite3` v3 build-hook path with SQLite3MultipleCiphers or re-instated SQLCipher, contrasted with the deprecated `*_flutter_libs` packages and the `sqflite_sqlcipher` fallback).
**Compiled:** 2026-06-16 · all package versions, APIs, and pragmas web-verified against primary sources (Drift docs, pub.dev, SQLite.org, SQLite3MultipleCiphers docs) as of June 2026 — current at the time of writing: drift 2.34.x, `package:sqlite3` 3.3.3 (SQLite 3.53.1, sqlite3mc 2.3.3), flutter_secure_storage 10.3.1.

This note is the evidence dossier behind the engineering synthesis doc `05-persistence-and-encryption.md` and the decision-log entry on the local store. It frames everything against the PRD's locked constraints: the store is **per-device, never synced, never transmitted** (PRD §17, C1); the `review_log` is an **append-only audit trail** that a teacher/sanad-respecting design must never silently mutate (PRD §10.3); a missed write or a corrupted card row is a *trust* failure in an app whose whole promise is "nothing decays silently" (PRD §7.12); and the bundled Quran reference tables are read-only and checksum-governed (PRD §11.3, R1). Crash-safety is therefore a non-negotiable value, encryption-at-rest is a *defense-in-depth* option (the data is religious-practice data, not medical/PII, so the bar differs from CycleVault), and dependency minimalism matters because this is an open-source *ṣadaqah* project the community will audit.

---

## What the evidence says

### 1. Drift is the right local-first layer for a pure-Dart, golden-tested design

- **Drift** (formerly Moor) is "a reactive library to store relational data in Dart and Flutter applications," built on SQLite, MIT-licensed, by Simon Binder; latest **2.34.0** (June 2026), supporting Android, iOS, Linux, macOS, Windows, and web ([Drift on pub.dev](https://pub.dev/packages/drift)). It gives compile-time-checked queries (in SQL or Dart), generated DAOs per table, auto-updating result streams, transactions, and a first-class migration system ([Drift docs](https://drift.simonbinder.eu/)).
- Type-safe, compile-time-verified queries matter directly for R1-adjacent correctness: the `card`/`review_log`/reference schema (PRD §10) is large and relational, and a mistyped column in a hand-rolled `sqflite` string query would only fail at runtime. Drift's code generator catches schema/query mismatches at build time ([Drift docs](https://drift.simonbinder.eu/)).
- Drift separates the **database definition** from the **execution backend** (`NativeDatabase`, `WasmDatabase`, `SqfliteDatabase`). The engine package the PRD mandates (`/engine`, pure Dart, zero I/O — PRD §19.1) never touches Drift at all; Drift lives only in `/data`, which feeds the engine plain state and persists results. This keeps the determinism/golden-test guarantee (PRD §7.12) intact: the math is testable without a database.

### 2. The native SQLite backend gives WAL crash-safety, which is the load-bearing durability property

- Drift's `NativeDatabase` runs on the `sqlite3` Dart package (FFI to a real SQLite). SQLite's **Write-Ahead Logging (WAL)** is the mode that makes "persist on every grade" crash-safe: instead of writing original pages to a rollback journal, "the original content is preserved in the database file and the changes are appended into a separate WAL file," and a commit is a single appended commit record — the main database file is untouched until checkpoint ([SQLite: WAL](https://sqlite.org/wal.html)).
- WAL's safety properties are exactly what a daily review log needs: commits are atomic, "writers do nothing that would interfere with the actions of readers" so "writers and readers can run at the same time," and the scheme is robust against crashes and power loss with far fewer `fsync`s than a rollback journal ([SQLite: WAL](https://sqlite.org/wal.html)). For Hifz Companion the writer is the grade/sign-off path and the readers are the reactive UI (heat-map, Today list) — concurrent, non-blocking, consistent snapshots.
- One durability nuance to encode as a deliberate decision: `PRAGMA synchronous=NORMAL` in WAL mode skips `fsync` on each commit, so "transactions are no longer durable and might rollback following a power failure or hard reset," with sync deferred to checkpoint ([SQLite: WAL](https://sqlite.org/wal.html)). For a teacher sign-off (a *sanad* act we must not lose), the safe posture is `synchronous=FULL` on the write connection, accepting the extra `fsync` cost — the write volume is tiny (a handful of rows per review).
- WAL means the live database is a **family of files** — the main file plus `-wal` and `-shm` ([SQLite: WAL](https://sqlite.org/wal.html)). This is a backup gotcha (see §8): copying just the `.sqlite` file mid-session loses uncheckpointed commits.

### 3. Drift's transaction API makes every multi-row review write atomic, with automatic rollback

- A review is not one write: `onReview` (PRD §7.7) appends to `review_log`, updates the `card` row's D/S/`due_at`, may create `line_block` rows, and may bump `confusion_edge` weights. These must be **all-or-nothing** — a half-applied review corrupts the audit trail. Drift's `db.transaction(() async {...})` runs them atomically; "none of their changes is visible to the main database until the transaction is finished," and the block **rolls back automatically when it throws** ([Drift: Transactions](https://drift.simonbinder.eu/dart_api/transactions/)).
- Two correctness rules the docs stress: (a) **every query inside the transaction must be `await`-ed**, because "without `await`, some queries might be operating on the transaction after it has been closed," causing data loss or crashes; and (b) if even the `ROLLBACK` fails, Drift surfaces both errors via `CouldNotRollBackException` ([Drift: Transactions](https://drift.simonbinder.eu/dart_api/transactions/); [CouldNotRollBackException](https://pub.dev/documentation/drift/latest/drift/CouldNotRollBackException-class.html)). Drift adds runtime checks that throw if a transaction is used after close ([Drift: Transactions](https://drift.simonbinder.eu/dart_api/transactions/)).
- Nested transactions (savepoints) have been supported since Drift 2.0 on `NativeDatabase`/`WasmDatabase`/`SqfliteDatabase`: an inner transaction sees the outer state and commits atomically into it, or reverts on throw ([Drift: Transactions](https://drift.simonbinder.eu/dart_api/transactions/)). Useful for cold-start (PRD §7.10), where seeding 600+ cards should be one outer transaction so a failure mid-seed leaves no partial profile.
- Drift also exposes efficient **batched** statements for bulk inserts ([Drift docs](https://drift.simonbinder.eu/)) — the right tool for the cold-start calibration seed and for importing a backup.

### 4. Drift's migration system is explicit, versioned, transactional, and testable — the correct posture when a bad migration equals lost hifz history

- Migrations are driven by an integer `schemaVersion` and a `MigrationStrategy` with three callbacks: `onCreate` (first run — typically `m.createAll()`), `onUpgrade` (version bumped — incremental schema changes via the `Migrator`), and `beforeOpen` (after migration, before queries — e.g. enabling pragmas, seeding) ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)). The `Migrator` API is direct SQL-level: `createAll()`, `createTable(...)`, `addColumn(table, column)` ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)).
- **Each migration runs in a transaction** — atomic, so a failed `onUpgrade` does not leave a half-migrated database ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)). Combined with the WAL/transaction guarantees above, an interrupted upgrade rolls back cleanly.
- Drift ships **guided migrations and schema testing**: `dart run drift_dev make-migrations` exports a JSON schema snapshot per version and generates `.steps.dart` step-by-step migration files plus test scaffolds, and the `stepByStep()` helper handles incremental version bumps (`from1To2: (m, schema) async {...}`) ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)). The docs are blunt that manual migrations are "error-prone and can lead to data loss" and recommend the guided, tested path ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)).
- Schema-snapshot export exists precisely because "drift's code generator can only see the current state of your schema"; storing older-version snapshots lets the tooling **validate** a written migration against the real old→new diff, and verify upgrades from *every* prior version ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)). For an append-only `review_log` that must survive years of app updates, this verify-every-path capability is the safeguard.
- Two operational notes the docs flag: write queries only against the *latest* schema (don't `select`/`update` a column before the migration that adds it has run), and enable `PRAGMA foreign_keys = ON` inside `beforeOpen` if FK enforcement is wanted — SQLite leaves FKs off by default ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)).

### 5. The 2026 encryption story changed: `sqlite3` v3 build hooks + SQLite3MultipleCiphers, *not* the old `*_flutter_libs` packages

- This area was rewritten in late 2025 / early 2026; ignore older blog posts. **`package:sqlite3` v3** is "a major update relying on build hooks and code assets to load SQLite" — it bundles SQLite with your app via Dart build hooks (instead of a `DynamicLibrary`), and its 3.0.0 notes are explicit: "You should drop your dependencies on `sqlite3_flutter_libs` and `sqlcipher_flutter_libs` when upgrading" ([sqlite3 changelog](https://pub.dev/packages/sqlite3/changelog); [sqlite3 hooks topic](https://pub.dev/documentation/sqlite3/latest/topics/hook-topic.html)).
- The bundled engine moves with the package: **3.3.1** shipped "SQLite 3.53.0 and SQLite3 Multiple Ciphers version 2.3.3," **3.3.2** upgraded to **SQLite 3.53.1**, and **3.3.3** (the current release as of June 2026) explicitly **"Restore[d] support for SQLCipher"** ([sqlite3 changelog](https://pub.dev/packages/sqlite3/changelog)). So as of mid-2026 there are genuinely **two** encryption back-ends reachable from the v3 build-hook model — the sqlite3mc bundle (the recommended default below) and a re-instated SQLCipher build — but **both go through the hook/`user_defines` mechanism**, not the dead `*_flutter_libs` packages.
- Encryption is now selected declaratively in `pubspec.yaml` by pointing the hook at the SQLite3MultipleCiphers source — no separate crypto package:
  ```yaml
  hooks:
    user_defines:
      sqlite3:
        source: sqlite3mc
  ```
  This bundles the "SQLite3MultipleCiphers build providing encryption support" instead of stock SQLite. The hook's `source` accepts `sqlite3` (default upstream), `sqlite3mc`, the re-instated `sqlcipher`, and the non-bundling `system` / `process` / `executable` options — so the single hook switch is the *one* place encryption is turned on, with no Dart-side change ([sqlite3 hooks topic](https://pub.dev/documentation/sqlite3/latest/topics/hook-topic.html); [Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)).
- Because SQLite3MultipleCiphers implements the SQLite C API, **no Dart code changes** are needed — `package:drift/native.dart` and `package:sqlite3` work unchanged; you simply set a key before first use ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). Drift's recommended pattern sets the key in `NativeDatabase`'s `setup` callback (runs before drift touches the DB):
  ```dart
  NativeDatabase.createInBackground(
    file,
    setup: (rawDb) {
      rawDb.execute("PRAGMA key = 'passphrase';");
    },
  );
  ```
  ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)).
- The old Flutter-libs packages are **deprecated and now no-ops**: `sqlcipher_flutter_libs` is at `0.7.0+eol` with metadata "Not used anymore, update to version 3.x of package:sqlite3 instead," "relates to version 2.x of `package:sqlite3`, and is obsolete after upgrading," and "Starting from version `0.7.0`, this package no longer does anything"; `sqlite3_flutter_libs` similarly "no longer does anything" from 0.6.0 ([sqlcipher_flutter_libs on pub.dev](https://pub.dev/packages/sqlcipher_flutter_libs); [sqlite3_flutter_libs changelog](https://pub.dev/packages/sqlite3_flutter_libs/changelog)). Drift's encryption docs say `sqlcipher_flutter_libs` "is no longer necessary after upgrading to drift 2.32.0" ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). The lesson is *which* SQLCipher you reach for: the FFI route is now SQLCipher (or sqlite3mc) **bundled inside `package:sqlite3` v3 via build hooks**, never the standalone `*_flutter_libs` packages.
- This also retires a notorious footgun: the historic conflict where depending on **both** `sqlcipher_flutter_libs` and `sqlite3_flutter_libs` (the latter pulled in transitively, e.g. via `drift_flutter`) caused the *regular* sqlite3 to win the link and SQLCipher to be silently unavailable — the cause of "This database needs to run with SQLCipher, but that library is not available!" errors ([drift #3162](https://github.com/simolus3/drift/issues/3162); [drift #3702](https://github.com/simolus3/drift/issues/3702)). The single-source build-hook model removes the dual-link ambiguity entirely.

### 6. SQLite3MultipleCiphers crypto design: a strong default, with a selectable SQLCipher-compatible scheme

- **SQLite3MultipleCiphers (sqlite3mc)** is "an extension to the public domain version of SQLite that allows applications to read and write encrypted database files," MIT-licensed, maintained by Ulrich Telle (2020–2026) ([SQLite3MultipleCiphers overview](https://utelle.github.io/SQLite3MultipleCiphers/)). It supports several cipher schemes: AES-128-CBC and AES-256-CBC (wxSQLite3), **ChaCha20-Poly1305** (sqleet), **AES-256-CBC + HMAC (SQLCipher-compatible)**, RC4, Ascon-128, and the AEGIS family ([SQLite3MultipleCiphers overview](https://utelle.github.io/SQLite3MultipleCiphers/)).
- The **recommended default** is ChaCha20-Poly1305 HMAC ([SQLite3MultipleCiphers overview](https://utelle.github.io/SQLite3MultipleCiphers/)) — a modern AEAD, fine for a greenfield store with no legacy file to read.
- If SQLCipher-format compatibility is wanted (e.g. to interoperate with files other tools produce), the **SQLCipher scheme** is selectable: AES-256-CBC, PBKDF2 key derivation with a random 16-byte salt in the file header (SHA1/SHA256/SHA512), per-page random 16-byte IV, and per-page HMAC. v4 uses HMAC-SHA512 (64-byte tag) with **256,000 PBKDF2 iterations**; legacy versions 1–3 use weaker parameters (4,000 / 4,000 / 64,000 iterations) and should not be used for new databases ([SQLite3MultipleCiphers: SQLCipher cipher](https://utelle.github.io/SQLite3MultipleCiphers/docs/ciphers/cipher_sqlcipher/)). In sqlite3mc, AES-256-CBC encrypts each 4096-byte page individually with its own random IV, and the HMAC of ciphertext+IV is appended and verified on read ([sqlite3mc AES-256 cipher](https://utelle.github.io/SQLite3MultipleCiphers/docs/ciphers/cipher_aes256cbc/); web-verified design summary).
- A **raw 32-byte key** can be supplied instead of a passphrase (skipping PBKDF2) — the right choice when the key is a random value held in the OS keystore rather than a typed human password, removing the per-open key-derivation delay.
- The bundled SQLite is part of the correctness surface: `package:sqlite3` **3.3.3** (current, June 2026) ships **SQLite 3.53.1** with **SQLite3MultipleCiphers 2.3.3** (added in 3.3.1) ([sqlite3 changelog](https://pub.dev/packages/sqlite3/changelog)). Track point releases — the bundled SQLite version is part of your durability surface (the same lesson the broader SQLite ecosystem learned from WAL-reset corruption bugs).

### 7. Encryption pitfalls that bite specifically: silent-plaintext builds, wrong-key behavior, and WAL re-keying

- **Silent plaintext if the build is misconfigured.** If `source: sqlite3mc` is not actually in effect (wrong workspace root, hook not applied), `PRAGMA key` is a no-op against stock SQLite and you ship a *plaintext* database that *looks* encrypted. Drift's docs give the guard — assert the cipher is live at open:
  ```dart
  bool _debugCheckHasCipher(Database db) =>
      db.select('PRAGMA cipher;').isNotEmpty;
  // assert(_debugCheckHasCipher(rawDb));
  ```
  ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). Promote this to a real (non-debug) startup check if encryption ships.
- **Wrong key fails late and opaquely.** "If you specify an incorrect key, you will not get an error message right away," but the first access throws `SQLITE_NOTADB` — "file is encrypted or is not a database" ([SQLite3MultipleCiphers FAQ](https://utelle.github.io/SQLite3MultipleCiphers/docs/faq/faq_overview/)). The app must treat `SQLITE_NOTADB` on open as "wrong/missing key," not "corrupt DB."
- **WAL + re-keying don't mix in sqlite3mc.** The FAQ states a key/page-size change requires that "the database must not be in WAL journal mode" ([SQLite3MultipleCiphers FAQ](https://utelle.github.io/SQLite3MultipleCiphers/docs/faq/faq_overview/)); community reports add that in-place WAL rekeying has caused corruption and only re-keys one pooled connection (others then throw `SQLITE_NOTADB`). Practical rule: never `PRAGMA rekey` an open WAL database in place — to change the key, switch off WAL or export to a freshly-keyed database (mirroring the SQLCipher `cipher_migrate`/export pattern). Pinning a current bundled sqlite3mc version is also prudent, as WAL handling has had regressions in past point releases.
- **Legacy/compat migration pragmas** exist if a pre-existing SQLCipher file must be read: run `PRAGMA cipher = 'sqlcipher'` and `PRAGMA legacy = 4` *before* `PRAGMA key` ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). Hifz Companion starts greenfield, so it can simply use the modern default and avoid the legacy axis entirely.

### 8. Key storage and the WAL-aware backup contract

- The encryption key must live in the OS-native secure store, never in the DB or code. **`flutter_secure_storage`** (10.3.1, BSD-3-Clause) uses the **iOS/macOS Keychain** and, on **Android**, RSA-OAEP + AES-GCM via the platform KeyStore ([flutter_secure_storage on pub.dev](https://pub.dev/packages/flutter_secure_storage)). Its iOS `accessibility` option `first_unlock_this_device` ("access only after the device is unlocked for the first time since installation on this device") is the sensible default for a key that must be readable by background notification scheduling but never leave the device or migrate to backups ([flutter_secure_storage on pub.dev](https://pub.dev/packages/flutter_secure_storage)). Generate a random 32-byte key at first launch, store it there, and feed it raw to sqlite3mc.
- **The live database must not be backed up by file copy.** WAL means three files (`.sqlite`, `-wal`, `-shm`); copying only the main file loses uncheckpointed commits ([SQLite: WAL](https://sqlite.org/wal.html)). The PRD's backup feature (§16 — versioned JSON or SQLite dump, user-initiated, no network) should export a *consistent snapshot*: either serialize the relational data to the documented versioned JSON, or use a checkpoint-then-copy / `VACUUM INTO` to produce a single consistent file. Never hand the user a half-written WAL set.
- The append-only `review_log` (PRD §10.3) interacts with this: backup/restore is the only sanctioned path that touches it in bulk, so the backup format must carry a **schema/app version stamp** (consistent with Drift's `schemaVersion`) so an older app refuses or migrates a newer file deterministically.

### 9. Two SQLCipher-compatible paths exist — prefer the FFI hook, treat `sqflite_sqlcipher` as the fallback

- If a SQLCipher-format file (not sqlite3mc's modern default) is actually required, there are now two routes. (a) The **FFI route**: `package:sqlite3` 3.3.3 re-instated a SQLCipher build reachable through the same `user_defines` hook, and sqlite3mc itself can read/write SQLCipher-format files via `PRAGMA cipher = 'sqlcipher'; PRAGMA legacy = 4;` before `PRAGMA key` ([sqlite3 changelog](https://pub.dev/packages/sqlite3/changelog); [Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). (b) The **platform-channel route**: Drift documents the **`encrypted_drift`** package's `EncryptedExecutor`, which runs on **`sqflite_sqlcipher`** ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)). `sqflite_sqlcipher` is a fork of `sqflite` that links native SQLCipher on iOS and Android and adds an optional `password` parameter to `openDatabase`; it bundles SQLCipher ~4.x and auto-runs `PRAGMA cipher_migrate` to upgrade older files ([sqflite_sqlcipher on pub.dev](https://pub.dev/packages/sqflite_sqlcipher)).
- Route (b) is viable but weaker for our purposes: it pulls a second native database dependency (a larger audit/supply-chain surface for an OSS project), uses the platform-channel `sqflite` backend rather than the FFI `NativeDatabase` (losing some of Drift's native concurrency model), and reintroduces exactly the kind of native-link coupling the `sqlite3` v3 hook model was created to retire. For Hifz Companion — which is greenfield and needs *no* SQLCipher-format interop — the right answer is the sqlite3mc default; `sqflite_sqlcipher` is a documented fallback, not the recommendation.

### 10. Encryption is *optional* here — and that is a deliberate, honest scoping call

- Unlike CycleVault (reproductive-health data, where app-level encryption and decoy/duress are headline features), Hifz Companion stores **religious-practice telemetry**: which muṣḥaf pages a profile has reviewed and how fluently. There is no PII (a profile is a typed display name — PRD §17), no account, and nothing leaves the device. The honest threat model is *device theft / shared-device curiosity*, not network exfiltration or forensic extraction of medical history.
- Therefore the right posture is: **WAL crash-safety and transactional integrity are mandatory; at-rest encryption is an opt-in setting** (e.g. a teacher/halaqa shared-device that holds several students' progress is a reasonable trigger). The architecture must make encryption a **build-hook + key-setup toggle** that costs nothing when off and is a single declarative change when on — which the `sqlite3mc` hook model delivers. OS-level file protection (iOS Data Protection, Android FBE) already covers the data at rest while the device is locked, as defense-in-depth, for free.

## Implications for Hifz Companion

1. **Adopt Drift 2.34.x on `NativeDatabase` (FFI `sqlite3` v3) as the local store** in `/data` only; the `/engine` package stays pure Dart with zero Drift/IO so determinism and golden tests (PRD §7.12, §19.1) hold. Reference tables (`page`, `line`, `ayah`, `surah`, `mushaf`, mutashābihāt) are read-only and checksum-governed (PRD §11.3); user tables (`card`, `review_log`, `line_block`, `confusion_edge`, `cycle_config`, `profile`, `app_meta`) are read-write per profile (PRD §10).
2. **Run WAL with `synchronous=FULL` on the write connection.** The review/sign-off write volume is tiny; pay the extra `fsync` to guarantee a teacher sign-off (a *sanad* act) is durable across a power loss, rather than accepting the `synchronous=NORMAL` "may roll back after power failure" window ([SQLite: WAL](https://sqlite.org/wal.html)).
3. **Wrap every review in one `db.transaction`.** `onReview` (PRD §7.7) — append `review_log`, update `card`, maybe create `line_block`, maybe bump `confusion_edge` — is atomic; the block auto-rolls-back on throw. **Audit every query for a missing `await` inside transactions** (a lint/review checklist item) since the docs flag un-awaited queries as a data-loss cause ([Drift: Transactions](https://drift.simonbinder.eu/dart_api/transactions/)). Seed cold-start (PRD §7.10) inside one outer transaction with batched inserts so a partial profile can never persist.
4. **Use Drift guided migrations from day one.** Commit the exported JSON schema snapshot per `schemaVersion`, generate `.steps.dart`, and keep the generated migration tests green in CI — because a botched migration on an append-only `review_log` is irreversible hifz-history loss ([Drift: Migrations](https://drift.simonbinder.eu/Migrations/)). Enable `PRAGMA foreign_keys = ON` in `beforeOpen` to enforce the profile/page FKs.
5. **If/when encryption ships, use the build-hook path, not the deprecated SQLCipher packages.** Add `hooks: user_defines: sqlite3: source: sqlite3mc`, set the key in `NativeDatabase`'s `setup` via `PRAGMA key`, and use the **sqlite3mc default (ChaCha20-Poly1305)** since the store is greenfield ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/); [sqlite3mc overview](https://utelle.github.io/SQLite3MultipleCiphers/)). Do **not** depend on `sqlcipher_flutter_libs`/`sqlite3_flutter_libs` (both EOL no-ops) and do not link two native sqlite sources.
6. **Add a hard "is encryption actually on?" assertion at startup.** Check `PRAGMA cipher;` returns non-empty before any write, promoted from debug-only to a real guard, so a misconfigured build can never silently ship a plaintext database that looks encrypted ([Drift: Encryption](https://drift.simonbinder.eu/platforms/encryption/)).
7. **Treat `SQLITE_NOTADB` on open as "wrong/missing key," never "corrupt."** Surface a key-recovery flow, not a "your data is corrupted" message ([SQLite3MultipleCiphers FAQ](https://utelle.github.io/SQLite3MultipleCiphers/docs/faq/faq_overview/)).
8. **Never rekey a live WAL database in place.** To rotate the key, export to a freshly-keyed database rather than `PRAGMA rekey` — sqlite3mc disabled WAL rekeying after corruption reports ([SQLite3MultipleCiphers FAQ](https://utelle.github.io/SQLite3MultipleCiphers/docs/faq/faq_overview/)).
9. **Store the DB key in `flutter_secure_storage`** (Keychain / Android KeyStore), as a random 32-byte raw key with `first_unlock_this_device` accessibility, non-syncing — so the key never migrates to cloud backups and the encrypted backup file + key remain the only off-device path ([flutter_secure_storage on pub.dev](https://pub.dev/packages/flutter_secure_storage)).
10. **Make backup WAL-aware and version-stamped.** Never copy the live `.sqlite`/`-wal`/`-shm` set; export a consistent snapshot (versioned JSON per PRD §16, or `VACUUM INTO` / checkpoint-then-copy), stamped with the Drift `schemaVersion`, so import is deterministic across app versions and the append-only `review_log` is never silently merged wrong ([SQLite: WAL](https://sqlite.org/wal.html); [Drift: Migrations](https://drift.simonbinder.eu/Migrations/)).
11. **Keep encryption optional and honestly scoped.** Mandatory: WAL crash-safety + transactional integrity. Optional (build-hook toggle): at-rest encryption, with the natural trigger being a shared teacher/halaqa device holding multiple students' progress (PRD §15.3). Rely on OS file protection as free defense-in-depth regardless.
12. **Pin and track the bundled SQLite + sqlite3mc versions** (currently SQLite 3.53.1 / sqlite3mc 2.3.3 via `package:sqlite3` 3.3.3) as part of the durability surface, updating promptly on point releases that fix WAL/corruption issues ([sqlite3 changelog](https://pub.dev/packages/sqlite3/changelog)). Note that `package:sqlite3` 3.3.3 re-instated a SQLCipher build alongside sqlite3mc — Hifz Companion deliberately chooses the **sqlite3mc default** and does not opt into SQLCipher-format compatibility, since there is no legacy file to interoperate with.

# Asset Integrity & Distribution — Research Note

**Topic:** Distributing and verifying Quran asset packs offline — hosting versioned static packs on GitHub Releases, runtime SHA-256 integrity pinning, supply-chain integrity (signing, attestation, immutability), and how an account-free, telemetry-free app *proves* it operates fully offline after a single trusted download.

**Compiled:** 2026-06-16

This note is the evidence dossier behind the engineering synthesis docs *08-quran-data-and-immutable-rendering* and *09-asset-packs-and-offline-integrity*, and behind PRD §11.1, §11.1.1, §17, §19.3, and release gates §20.1/§20.6. Our hard constraints make this the load-bearing trust mechanism of the whole product: the app downloads Quran assets **once** from a public GitHub repo, must guarantee the bytes are byte-for-byte the authoritative muṣḥaf (R1 — a single altered diacritic ends the project), and must then run in airplane mode forever, sending no per-user data, ever (C1). Everything below is in service of that contract.

---

## What the evidence says

### 1. GitHub Releases gives stable, CDN-backed, predictable download URLs — but is "fine for a tarball," not a hardened distribution platform

GitHub Releases serves release assets from clean, predictable URLs of the form `https://github.com/<owner>/<repo>/releases/download/<tag>/<filename>`, fetchable with a plain HTTPS `GET` and no authentication for public repos ([w3tutorials: GitHub latest-release download links](https://www.w3tutorials.net/blog/is-there-a-link-to-github-for-downloading-a-file-in-the-latest-release-of-a-repository/)). Under the hood the request is **HTTP-redirected to a signed, time-limited URL** on GitHub's asset CDN (`release-assets.githubusercontent.com`, formerly `objects.githubusercontent.com`), so any client must follow redirects (`curl -L`, or an HTTP library configured to follow 302s) — but the *asset bytes behind a published release are stable* and can be made permanently so (Finding 2) ([GitHub Community: Identifiers in release-assets URL](https://github.com/orgs/community/discussions/167206); [URLert: githubassets.com CDN](https://www.urlert.com/domains/githubassets.com)). On reliability, GitHub's own status page reports strong rolling-90-day availability — Pages ~99.96%, API Requests ~99.94%, Git Operations ~99.86% at compilation — broken out per component ([GitHub Status](https://www.githubstatus.com/)). The honest caveat is contractual, not technical: the **99.9% quarterly SLA applies only to paid Enterprise Cloud customers; free and Team plans have no uptime guarantee** ([GitHub Enterprise SLA](https://github.com/github/docs/blob/main/content/site-policy/site-policy-deprecated/github-enterprise-service-level-agreement.md)). Practitioners also note "GitHub Releases is fine for attaching a zip or tarball, but it was never designed to be a proper release hosting platform" — no download analytics or platform breakdown ([dev.to: Why GitHub Releases suck for binary distribution](https://dev.to/gaborboros/why-github-releases-suck-for-binary-distribution-289i)). For us the missing analytics is a *feature* (we want no download telemetry) and the absent SLA is acceptable **precisely because the network is needed exactly once** — a transient outage delays first-run setup but never touches an already-provisioned offline device. The residual risk is "latest" URL brittleness, which we design around by pinning an exact tag, not `latest`; free CDN mirrors (jsDelivr, Statically) exist if a fallback host is ever needed ([jsDelivr](https://github.com/jsdelivr/jsdelivr); [Statically](https://github.com/staticallyio/statically)).

### 2. GitHub now offers *immutable releases* (GA 28 Oct 2025): once published, assets and the Git tag cannot be changed, deleted, or moved

This is the single most important supply-chain primitive available to us, and it shipped recently enough to be easy to miss. As of **28 October 2025**, immutable releases are generally available: "Once you publish a release as immutable, its assets can't be added, modified, or deleted," and "Tags for new immutable releases are protected and can't be deleted or moved" ([GitHub Changelog, 2025-10-28](https://github.blog/changelog/2025-10-28-immutable-releases-are-now-generally-available/)). The docs are precise: the tag "is locked to a specific commit, cannot be changed, and cannot be deleted while the release exists," and all attached files "are protected from modification or deletion" ([GitHub Docs: Immutable releases](https://docs.github.com/en/code-security/concepts/supply-chain-security/immutable-releases)). Crucially for a long-lived *ṣadaqah* project, immutable releases include **repository-resurrection protection**: "Even if you delete a repository and create a new one with the same name, you cannot reuse tags that were associated with immutable releases in the original repository" — closing the attack where someone re-creates the repo to re-bind a trusted tag to malicious assets. One residual gap to design around: if the maintainer *deletes* an immutable release, the tag name cannot be reused, but the assets at the old URL stop resolving — so the app must treat a 404 as "keep using the verified local copy," never "re-fetch anything that now matches."

### 3. Immutable releases auto-generate a signed *release attestation* binding tag + commit SHA + asset digests, verifiable with Sigstore

Publishing an immutable release "automatically generates a release attestation, which is a cryptographically verifiable record of a release containing the release tag, commit SHA, and release assets" ([GitHub Docs: Immutable releases](https://docs.github.com/en/code-security/concepts/supply-chain-security/immutable-releases)). These attestations use the **Sigstore bundle format**, so "you can easily verify the authenticity and integrity of assets, both on GitHub and in external environments" ([GitHub Changelog, 2025-10-28](https://github.blog/changelog/2025-10-28-immutable-releases-are-now-generally-available/)). This sits on top of GitHub's broader **artifact attestations** system: an attestation "binds some subject (a named artifact along with its digest) to a SLSA build provenance predicate using the in-toto format," signed with a short-lived Sigstore-issued certificate from GitHub's (or the public-good) Fulcio CA, with the bundle recorded in an immutable, publicly readable Rekor transparency log ([Ian Lewis: Understanding GitHub Artifact Attestations](https://www.ianlewis.org/en/understanding-github-artifact-attestations); [Sigstore Blog: cosign verify bundles](https://blog.sigstore.dev/cosign-verify-bundles/)). The attestation is *evidence about provenance*, not a safety guarantee — GitHub is explicit that "artifact attestations are not a guarantee that an artifact is secure," they are claims to evaluate against a policy ([GitHub Docs: Artifact attestations](https://docs.github.com/en/actions/concepts/security/artifact-attestations)). For our purposes the attestation is a **build/release-time CI control**, not a runtime mobile control: our CI (and any auditor) can run `gh attestation verify` to confirm the published pack was built from our source on our workflow; the mobile app itself relies on the simpler pinned-digest check (Finding 4).

### 4. SHA-256 is the right runtime integrity primitive: NIST-standard (FIPS 180-4), 128-bit collision resistance, and a single flipped byte changes the whole digest

SHA-256 is specified in NIST **FIPS 180-4**, the Secure Hash Standard, whose digests are explicitly intended "to detect whether messages have been changed since the digests were generated" ([NIST FIPS 180-4](https://csrc.nist.gov/pubs/fips/180-4/upd1/final)). It produces a 256-bit digest with an expected **collision resistance of ~128 bits**, and despite sustained effort no collisions have been found ([Wikipedia: SHA-2](https://en.wikipedia.org/wiki/SHA-2); [Wikipedia: Collision resistance](https://en.wikipedia.org/wiki/Collision_resistance)). The integrity property we rely on is the avalanche effect: "If even a single byte changes — whether from corruption, a failed download, or deliberate tampering — the resulting hash will be completely different" ([bitkode: Verify file integrity with SHA-256](https://bitkode.com/how-to-verify-file-integrity-with-sha-256-windows-mac-linux/)). This is exactly the guarantee R1 needs — a one-character change to a single diacritic in the Uthmani text yields a totally different digest, so a tampered or truncated pack can never silently pass. Web SRI (the browser equivalent) underscores the algorithm choice: the SRI spec accepts **only `sha256`, `sha384`, `sha512`** and forbids MD5/SHA-1 as cryptographically broken for integrity ([MDN: Subresource Integrity](https://developer.mozilla.org/en-US/docs/Web/Security/Defenses/Subresource_Integrity)).

### 5. Pin the *exact* expected hash to a *specific version*, never to "latest" — and refuse to load on mismatch (the SRI model)

The established best practice, drawn from Subresource Integrity, is to ship the *expected* hash inside the trusted client and pin it to an exact, versioned URL, never a moving `latest` pointer: "Always pin to a specific version … not a 'latest' tag, as if the file changes, the hash won't match" ([andrewlock: Avoiding CDN supply-chain attacks with SRI](https://andrewlock.net/avoiding-cdn-supply-chain-attacks-with-subresource-integrity/)). The verification semantics are equally important: the browser "will then calculate the hash of the resource contents … and compare the result with all the specified values: if the actual value matches … then the browser will load the resource, **otherwise it will refuse to load the resource**, and return a network error" ([MDN: Subresource Integrity](https://developer.mozilla.org/en-US/docs/Web/Security/Defenses/Subresource_Integrity)). SRI is positioned as "a defense against supply chain attacks by ensuring that the files your web application fetches have exactly the contents that you expect them to have" ([MDN](https://developer.mozilla.org/en-US/docs/Web/Security/Defenses/Subresource_Integrity)). Our PRD §11.1.1 is precisely this model translated to a mobile app: the binary "ships with the pinned SHA-256 checksums and pinned release version of every asset pack," a downloaded pack is "rejected and re-fetched unless its hash matches exactly," and the app "refuses to render Quran text from any unverified asset" — fail-closed, exactly like an SRI mismatch.

### 6. The threat model demands defense-in-depth: download over HTTPS, verify after download, and confirm the expected hash through a second trusted channel

General guidance on verified downloads converges on three layers. First, transport: "Always download files and checksums over HTTPS and use a secure hash algorithm like SHA-256" ([transloadit: Downloading and verifying file hashes](https://transloadit.com/devtips/hashing-files-with-curl-a-developer-s-guide/)). Second, post-download verification: "Build the habit of 'verify after download' … Verifying after download ensures you have the official build, not a backdoored version" ([bitkode](https://bitkode.com/how-to-verify-file-integrity-with-sha-256-windows-mac-linux/)). Third — the subtle one — the expected hash itself must come from a channel *independent* of the download: if verification fails, "do not run the file until you've confirmed the official hash from a second trusted channel — for example, the project's GitHub releases page … or a signed SHASUMS.gpg" ([bitkode](https://bitkode.com/how-to-verify-file-integrity-with-sha-256-windows-mac-linux/)). A checksum hosted *next to* the file it describes provides only corruption detection, not tamper resistance — an attacker who can replace the pack can replace its sidecar checksum too. The defense is to bake the expected digest into the **signed app binary** (which the user got from a trusted app store), making the app itself the independent channel.

### 7. Optional hardening: signed releases (Sigstore/cosign, or minisign) and SLSA build provenance give an auditor a cryptographic source-to-artifact chain

Beyond a pinned digest, the asset *repository* can publish signatures so anyone — not just our CI — can independently verify provenance. **Sigstore** (Linux Foundation/OpenSSF) provides keyless signing: `cosign` signs artifacts, **Fulcio** issues short-lived certificates tied to an OIDC identity, and **Rekor** is a public transparency log of every signing event ([oneuptime: Implement supply chain security with Sigstore](https://oneuptime.com/blog/post/2026-01-25-sigstore-supply-chain-security/view); [GitGuardian: Sigstore and Cosign](https://blog.gitguardian.com/supply-chain-security-sigstore-and-cosign-part-ii/)). For a tiny, fully-offline-verifiable alternative, **minisign** (and OpenBSD `signify`) sign files with Ed25519; "the signatures and keys are tiny and stored in base64, meaning their distribution is very easy" and need no PKI ([jedisct1/minisign README](https://github.com/jedisct1/minisign/blob/master/README.md)). The framing concept is **SLSA build provenance** — "a signed document that makes claims about how an artifact is built, which … consumers can use … to determine whether software belongs within their software supply chain" ([Legit Security: SLSA provenance deep dive](https://www.legitsecurity.com/blog/slsa-provenance-blog-series-part-2-deeper-dive-into-slsa-provenance)). With GitHub immutable releases (Finding 3) this comes nearly for free via `actions/attest-build-provenance` ([actions/attest-build-provenance](https://github.com/actions/attest-build-provenance)).

### 8. Reproducible (deterministic) builds let the *community* independently re-derive our pinned hashes from source — the strongest "auditable" claim

Because this is an open, *waqf*-style data repo (PRD §11.1), reproducibility multiplies trust. "A build is reproducible if given the same source code, build environment and build instructions, any party can recreate bit-by-bit identical copies of all specified artifacts," giving "an independently-verifiable path from source to binary code" ([Reproducible Builds project](https://reproducible-builds.org/)). The security payoff is direct: reproducible builds "verify that the binaries that you download match the original, untampered source code" and act as "a strong countermeasure against attacks where binaries do not match their source code" ([Reproducible Builds project](https://reproducible-builds.org/); [Wikipedia: Reproducible builds](https://en.wikipedia.org/wiki/Reproducible_builds)). If the pack-build pipeline (Tanzil text → normalized DB, QUL layout → tables, font manifest) is deterministic, anyone can re-run it and confirm our published SHA-256 — turning the pinned digest from "trust us" into "verify yourself."

### 9. Flutter has everything needed natively: the Dart-team `crypto` package for SHA-256 (chunked for large files) and standard offline storage paths

The official **`crypto`** package, published by the Dart team, implements SHA-1/224/256/384/512, MD5, and HMAC ([pub.dev: crypto](https://pub.dev/packages/crypto)). Small inputs use `sha256.convert(bytes)`; large packs (multi-megabyte fonts, audio) should be hashed incrementally via `sha256.startChunkedConversion(...)` fed from a chunked file stream so the whole file is never held in memory ([djangocas: Calculate file crypto hash in Flutter](https://djangocas.dev/blog/flutter/calculate-file-crypto-hash-sha1-sha256-sha512/)). On the storage side, downloaded packs belong in app-private storage: `getApplicationDocumentsDirectory()` maps to iOS `NSDocumentDirectory` and Android's app data dir and is the right home for data the app needs offline ([dhiwise: getApplicationDocumentsDirectory deep dive](https://www.dhiwise.com/post/a-deep-dive-into-getapplicationdocumentsdirectory)). The conventional split — durable, hard-to-re-download data in documents; cheap re-downloadable caches in the temporary directory — argues for placing the *verified core pack* in documents (it is the irreplaceable, verified muṣḥaf) and treating in-flight downloads as temporary until verification passes ([dhiwise](https://www.dhiwise.com/post/a-deep-dive-into-getapplicationdocumentsdirectory)).

### 10. TLS certificate pinning is *not* warranted here — and OWASP increasingly discourages it for third-party hosts

It is tempting to pin GitHub's TLS certificate, but OWASP guidance cuts the other way: pinning "should only be done when the client and server sides are both controlled by the same party," and "major platform maintainers and industry experts now discourage the use of SSL pinning except in very special cases" ([OWASP MASTG: Testing certificate pinning](https://mas.owasp.org/MASTG/tests/android/MASVS-NETWORK/MASTG-TEST-0022/); [OWASP: Certificate and Public Key Pinning](https://owasp.org/www-community/controls/Certificate_and_Public_Key_Pinning)). Industry vendors agree it is **"not recommended for Android applications due to the high risk of future server configuration changes, such as changing to another Certificate Authority, rendering the application unable to connect ... without a client software update,"** with current best practice being to "not pin unless you have a niche regulatory or extreme threat model" and instead rely on the platform PKI trust store, Certificate Transparency, and HSTS ([Approov: Certificate Pinning on Android](https://approov.io/blog/securing-https-with-certificate-pinning-on-android); [caverav: The Obsolescence of SSL Pinning](https://caverav.cl/posts/ssl-pinning/ssl-pinning/)). We do **not** control GitHub's certificates, and they rotate; pinning them would be a self-inflicted outage waiting to happen and would gain us little, because our real integrity guarantee is the content hash, not the transport. The correct posture is enforce modern TLS (1.2+/1.3) for the one-time download, then rely on the pinned SHA-256 of the *content* — transport security protects confidentiality and tampering-in-flight, the content digest protects integrity end to end regardless of which CDN edge served the bytes.

### 11. "Provable offline operation" is a build-time *and* audit-time property: the only network client is the asset downloader, asserted in CI

PRD C1/§19.3 promises the app is offline after setup and never sends per-user data. This is provable, not merely asserted, by three converging facts in the evidence above. (a) The download is a plain unauthenticated HTTPS `GET` to a public asset URL — the request carries only the public URL, no body, no identifiers ([w3tutorials](https://www.w3tutorials.net/blog/is-there-a-link-to-github-for-downloading-a-file-in-the-latest-release-of-a-repository/)) — so even the one network call leaks nothing about the user. (b) Because assets live on a CDN, no first-party server of ours exists to receive data; there is literally nothing for the app to phone home to. (c) Open source + reproducible builds (Finding 8) means an auditor can read the source and confirm a single network code path. The standard mobile technique to make this checkable is to assert, in CI, that the release binary links **no** analytics/ads/backend SDKs and that the only outbound client is the pack downloader — the build-time analogue of "verify after download" applied to the app's own network surface.

### 12. Source/licence integrity is part of asset integrity: the canonical sources are Tanzil (text) and QUL/KFGQPC (layout + glyph fonts), each with attribution terms

Integrity is not only "are the bytes intact" but "are they the *right, lawfully-sourced* bytes." The Uthmani text source is **Tanzil**, whose terms of use state: "Permission is granted to copy and distribute verbatim copies of the Quran text provided here, but changing the text is not allowed," and require clearly indicating Tanzil as the source with a link to tanzil.net ([Tanzil: Download / Terms of Use](https://tanzil.net/download/)). (Some Tanzil distributions are additionally described as CC BY 3.0; the operative, conservative requirement is *verbatim copies + attribution* — which aligns perfectly with R1's byte-for-byte rule.) Page layout and per-page glyph fonts come from the **Quranic Universal Library (QUL)** and KFGQPC: the QPC font set is "604 separate fonts … each corresponding to one page of the Madinah Mushaf," available as TTF/WOFF/WOFF2 ([QUL: QPC V2 Font](https://qul.tarteel.ai/resources/font/249); [QUL home](https://qul.tarteel.ai/)). The integrity manifest must therefore pin not just one text blob but a per-asset hash list (one entry per font, dataset, layout file) and record each asset's source + licence, so a CI run proves *both* that every byte matches *and* that every asset is attributable (PRD §11.3, §20.7).

## Implications for Hifz Companion

1. **Host packs on GitHub Releases as immutable releases, pinned by exact tag.** Publish each versioned pack (core, optional reciter, future alt-muṣḥaf) as a GitHub *immutable* release so assets and tags are tamper-locked and resurrection-protected (Findings 2–3). The app references an **exact tag** baked into the binary, never `latest` (Finding 5). Treat a 404 on a pinned URL as "use the verified local copy," never as a trigger to fetch something else.

2. **Make the pinned SHA-256 manifest the runtime gate, fail-closed.** Ship, inside the signed app binary, a manifest of the expected SHA-256 for **every** asset (each of the 604 fonts, the text DB, the layout dataset, the mutashābihāt dataset). After download, hash each file with the Dart-team `crypto` package — chunked for large files (Finding 9) — and compare. Any mismatch ⇒ reject, re-fetch once, and if still mismatched, **refuse to render Quran text** and surface a calm, honest error (PRD §11.1.1; Findings 4–5). This is SRI's "refuse to load on mismatch," applied to the muṣḥaf.

3. **The app binary is the independent trust channel — never the sidecar checksum.** Do not trust a checksum file sitting next to the pack on the same release (Finding 6). The expected digests live in the app the user installed from a trusted store; that store's app signing is our "second trusted channel." This is what upgrades the check from corruption-detection to tamper-resistance.

4. **Add signing + provenance as a CI/audit control, not a mobile runtime dependency.** Use `actions/attest-build-provenance` (free with immutable releases) and optionally publish a tiny **minisign** signature + public key in the repo (Findings 3, 7). The *mobile app* still verifies only the pinned digest (simple, offline, no PKI in the app); the *signatures* let maintainers, F-Droid reviewers, and the community independently verify source-to-artifact provenance via `gh attestation verify` / `cosign` / `minisign`.

5. **Build the packs reproducibly.** Make the pack-build pipeline deterministic (sorted inputs, fixed timestamps, pinned toolchain) so any third party can re-derive — and thus re-confirm — our published hashes from the open source data (Finding 8). Document the exact command. This is the concrete meaning of the engineering pillar "open-source & auditable."

6. **Store verified packs in app-documents; verify in temp first.** Download to the temporary directory, verify, and only then move the pack into `getApplicationDocumentsDirectory()` as the canonical offline copy (Finding 9). The reference Drift DB is built from the *verified* assets; a verified-at timestamp goes in `app_meta` (PRD §10.2). Never render from an unverified file, even transiently.

7. **Do not pin TLS certificates; do enforce modern TLS.** Skip cert pinning against GitHub (Finding 10) — it buys little and risks outages on rotation. Require TLS 1.2+/1.3 for the one-time fetch and let the content hash carry end-to-end integrity. The download client must send no auth, cookies, or identifiers — only the public asset URL.

8. **Make "offline" a release gate, not a promise.** Add CI checks (PRD §20.6) that assert the release binary links no analytics/ads/backend SDKs and that the single outbound network path is the pack downloader (Finding 11). Pair with a manual airplane-mode acceptance test: install, complete the one-time core download, enable airplane mode, and confirm every Quran feature works indefinitely. This is how C1 becomes *demonstrable*.

9. **Pin source + licence per asset, and verify both in CI.** The integrity manifest records, per asset, its SHA-256 **and** its source/licence (Tanzil verbatim-copy + attribution for text; QUL/KFGQPC for layout/fonts — Finding 12). CI fails if a hash drifts *or* if attribution metadata is missing, so the build proves the muṣḥaf is simultaneously intact, authentic, and lawfully redistributed (PRD §11.3, §20.7). Surface the riwāyah + source attribution in-app (R2).

10. **Design the one-time download as the only moment trust is extended — then never again.** Onboarding clearly explains the single required download, retries gracefully if offline at first run (PRD §12.1), and from that point forward the app extends *zero* further trust to the network: no remote config, no silent re-fetch, no "latest" check. The verified local muṣḥaf is the root of trust for the lifetime of the install.

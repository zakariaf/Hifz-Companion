# Flutter i18n & RTL for Hifz Companion — research note

**Topic:** Flutter internationalization and right-to-left correctness for a fully-RTL, three-locale app (Persian `fa`, Kurdish Sorani `ckb`, Arabic `ar`) — the `intl` package + `gen_l10n`/ARB pipeline, the `Directionality`/`TextDirection` model, Unicode bidi isolation for mixed-script strings, locale-specific Eastern Arabic-Indic numerals, and the layout discipline that makes mirroring correct. Survey of the official Flutter/Dart docs, the `intl` and Flutter `foundation` API references, the Unicode/W3C bidi standards, and the `dart-lang/i18n` issue tracker.
**Compiled:** 2026-06-16 · research agent · Status: primary-source survey, every URL web-verified live.

Context from the PRD that constrains every choice below: all three shipping languages are **RTL** and **fully translated**, with **localizable religious terminology** (the sabaq/sabqi/manzil vocabulary differs by region), **locale-appropriate numerals and calendars**, and a release gate that demands **zero missing ARB keys, no hardcoded user-facing strings, RTL golden screenshots per locale, and verified numeral/calendar rendering** ([PRD §13](../../PRD.md), [PRD §20](../../PRD.md)). One subtlety dominates the whole note: the Quran text is **never** localized — it is always the Uthmani muṣḥaf rendered through dedicated glyph fonts, so only the **UI chrome** is the subject of i18n, while the muṣḥaf is a separate immutable-rendering concern ([PRD §11](../../PRD.md), [PRD §13.1](../../PRD.md)). This note covers the chrome. Arabic-script rendering of the Quran itself is covered in the sibling note `arabic-script-rendering-fonts.md`.

## What the evidence says

### 1. The official pipeline is `flutter_localizations` + `intl` + `gen_l10n` over ARB files, with code generation on

Flutter's first-party internationalization path is prescriptive and code-generated. You add the framework localizations and `intl` (`flutter pub add flutter_localizations --sdk=flutter` and `flutter pub add intl:any`), then **enable the `generate` flag** under the `flutter:` section of `pubspec.yaml` ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)). A root `l10n.yaml` configures the pipeline, with documented defaults: `arb-dir` ("The directory where the template and translated arb files are located. The default is `lib/l10n`."), `template-arb-file` (default `app_en.arb`), `output-localization-file` (default `app_localizations.dart`), and `output-class` (default `AppLocalizations`) ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).

- ARB ("Application Resource Bundle") is a JSON format. Each translatable string is a key→value entry; **"the key of each entry is used as the method name of the getter, while the value of that entry contains the localized message"** — and metadata keys prefixed with `@` carry `description`, `placeholders`, and translator context ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).
- Generation is automatic: **"Now, run `flutter pub get` or `flutter run` and codegen takes place automatically"**, or invoke `flutter gen-l10n` manually. The output is a **type-safe** Dart class — every key becomes a getter/method, so a typo or a missing key is a compile error rather than a runtime miss ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).

This directly serves the PRD's CI gate ("zero missing ARB keys; no hardcoded user-facing strings"): code generation makes a referenced-but-undefined key fail the build, and a lint/grep over widget source can ban string literals in the UI ([PRD §13.6](../../PRD.md), [PRD §20](../../PRD.md)).

### 2. Direction is derived from the locale automatically — `GlobalWidgetsLocalizations` sets the ambient text direction

You do not, in general, hardcode RTL. The `localizationsDelegates` list wires in the framework's global localizations, and Flutter's own docs are explicit about the division of labor: **"`GlobalMaterialLocalizations.delegate` provides localized strings and other values for the Material Components library. `GlobalWidgetsLocalizations.delegate` defines the default text direction, either left-to-right or right-to-left, for the widgets library"** ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)). So when the active locale is `ar`/`fa`/`ckb`, the widgets library's ambient direction becomes RTL with no per-widget code.

The minimal `MaterialApp` wiring is:

```dart
MaterialApp(
  localizationsDelegates: AppLocalizations.localizationsDelegates, // includes the three Global* delegates
  supportedLocales: AppLocalizations.supportedLocales,             // generated from the ARB set
  // ...
)
```

The generated `AppLocalizations.localizationsDelegates` / `.supportedLocales` lists keep the delegate set and the locale list in sync with the ARB files ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).

### 3. `Directionality` / `TextDirection` is the mechanism behind all mirroring — and the rule is "use direction-relative APIs, never physical sides"

The ambient direction is carried by a single inherited widget. `Directionality` **"determines the ambient directionality of text and text-direction-sensitive render objects"**; its `textDirection` argument is required, and direction-sensitive widgets read it via `Directionality.of(context)` (**"the text direction from the closest instance of this class that encloses the given context"**) or the null-safe `Directionality.maybeOf(context)` ([Flutter: Directionality class](https://api.flutter.dev/flutter/widgets/Directionality-class.html)). `MaterialApp`/`WidgetsApp` install this ambient `Directionality` from the resolved locale (via the global delegate, §2), so app-wide RTL is a consequence of locale selection, not a manual `Directionality` wrap.

The underlying enum lives in `dart:ui`: `TextDirection.rtl` ("The text flows from right to left (e.g. Arabic, Hebrew).") and `TextDirection.ltr` ("The text flows from left to right (e.g., English, French.)"). It governs the order of runs of bidirectional text, default alignment, and how render objects lay out boxes horizontally. Critically, **"the lowest levels of the Flutter framework do not have a default text reading direction"** — at the `dart:ui` level direction must be explicit — whereas the widgets library introduces the ambient `Directionality` that supplies a default ([Flutter: TextDirection enum](https://api.flutter.dev/flutter/dart-ui/TextDirection.html)).

The single highest-leverage discipline follows from this: **resolve sides logically, never physically.** Padding, alignment, and insets have direction-relative forms — `EdgeInsetsDirectional` (start/end), `AlignmentDirectional`, `Directionality`-aware `Positioned.directional` — and the framework resolves them to absolute values per the ambient `TextDirection` (e.g. `Padding` "depend[s] on `Directionality` to resolve `EdgeInsetsDirectional` objects into absolute `EdgeInsets` objects") ([Flutter: Directionality class](https://api.flutter.dev/flutter/widgets/Directionality-class.html)). Any hardcoded `EdgeInsets.only(left: …)`, `Alignment.centerLeft`, or fixed positive `x`-offset bypasses mirroring and becomes an RTL bug. For an app that is RTL in **every** shipping locale, this is not a future-proofing nicety — it is the primary layout contract.

### 4. ARB carries ICU plural/select, which the three languages genuinely need

ARB messages use **ICU MessageFormat**, parsed and code-generated by `gen_l10n`. Named placeholders are `{name}` and are typed in the `@`-metadata; plurals and gender/selects are first-class ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).

- **Plural** — `"{count, plural, =0{…} =1{…} other{…}}"`; the docs note a pluralized message **"must include a `num` parameter,"** and `intl` applies the correct CLDR plural category per target language. Arabic is the canonical hard case: CLDR defines **six** plural categories for Arabic (`zero`, `one`, `two`, `few`, `many`, `other`), so "3 pages due" vs "11 pages due" vs "100 pages due" inflect differently — a correctness requirement the ARB/`intl` stack handles only if the translator supplies all needed categories ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).
- **Select** — `"{gender, select, male{…} female{…} other{…}}"`; the docs warn **"comparison between the parameter and the actual value is case-sensitive."** This is the right tool for the **localizable terminology** requirement: the regional sabaq/sabqi/manzil term-set can be modeled as a `select` over a region key, or simply as parallel keys with a region override file ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization), [PRD §13.4](../../PRD.md)).
- **Escaping** — literal braces and apostrophes (common in fa/ar copy) require `use-escaping: true` in `l10n.yaml`; then a string of characters wrapped in single quotes is ignored by the parser and a doubled `''` yields one literal apostrophe ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).

### 5. Kurdish Sorani (`ckb`) is a custom locale — Flutter's *Material* localizations don't ship it, so a delegate is required

The app's own strings come from the ARB set regardless of locale, but `GlobalMaterialLocalizations` (the built-in widget tooltips, "Cancel"/"OK", date-picker chrome, etc.) ships a fixed list of supported locales, and **Central Kurdish `ckb` is not among them** — a real-world report (resolved as "use a custom delegate") confirms the `ckb_IQ` "MaterialLocalizations delegate … not found" warning that results ([flutter/flutter #35103 — Localization support for Central Kurdish, locale 'ckb'](https://github.com/flutter/flutter/issues/35103)). The documented remedy is to provide a **custom `GlobalMaterialLocalizations` subclass and `LocalizationsDelegate`** for the unsupported language, overriding the required string getters and registering it in `localizationsDelegates`; the Flutter docs walk through exactly this pattern for a non-shipped locale ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)). Community packages (e.g. `ckb_localizations` / `flutter_kurdish_localization` on pub.dev) implement that delegate and can be used or vendored ([pub.dev: ckb_localizations](https://pub.dev/packages/ckb_localizations)).

Two practical consequences:
- `ckb` must be declared with `Locale.fromSubtags(languageCode: 'ckb')` and a delegate supplied; the docs prefer `Locale.fromSubtags` because it supports `scriptCode` (relevant if a script tag like `ckb_Arab` is ever needed) ([Flutter: Internationalizing Flutter apps](https://docs.flutter.dev/ui/accessibility-and-internationalization/internationalization)).
- Sorani uses the Arabic script, so once `ckb` resolves, the global widgets delegate yields **RTL** automatically (§2); the custom Material delegate only fills in chrome strings, not direction ([flutter/flutter #35103](https://github.com/flutter/flutter/issues/35103)).

### 6. Mixed-direction strings need Unicode bidi **isolation**, and the recommended primitive is FSI…PDI

Hifz UI constantly mixes scripts inside one line: a Latin/ASCII page number or a surah name beside RTL Persian, "Juz 7", a date, a percentage. Without isolation, the Unicode Bidirectional Algorithm can reorder neighbouring runs and produce visually scrambled output (the classic "number jumps to the wrong end" bug). The governing standard is **Unicode UAX #9 (the Bidirectional Algorithm)**, which now explicitly recommends *isolates* over the older embedding/override codes: **"the use of the directional isolates instead of embeddings is encouraged in new documents,"** because an isolate **"functions like a neutral character on the ordering of the surrounding characters,"** whereas the deprecated `LRE`/`RLE`/`LRO`/`RLO`+`PDF` could perturb text outside their intended scope. UAX #9 also confirms that **digits within a number keep their logical order** regardless of surrounding direction (rules W4–W5) — a "specific number" is never reversed ([Unicode UAX #9 — Unicode Bidirectional Algorithm](https://www.unicode.org/reports/tr9/)).

Flutter exposes these control characters directly on the `Unicode` class in `foundation`, with the isolate set:

| Constant | Codepoint | Meaning |
|---|---|---|
| `Unicode.FSI` | U+2068 | **First Strong Isolate** — "Treat the following text as isolated and in the direction of its first strong directional character that is not inside a nested isolate." |
| `Unicode.LRI` | U+2066 | Left-to-Right Isolate |
| `Unicode.RLI` | U+2067 | Right-to-Left Isolate |
| `Unicode.PDI` | U+2069 | Pop Directional Isolate (ends an isolate scope) |
| `Unicode.LRM` / `Unicode.RLM` / `Unicode.ALM` | U+200E / U+200F / U+061C | direction marks (incl. Arabic Letter Mark) |

([Flutter: Unicode class](https://api.flutter.dev/flutter/foundation/Unicode-class.html), [Flutter: Unicode.FSI](https://api.flutter.dev/flutter/foundation/Unicode/FSI-constant.html)). The pattern for an embedded run of unknown direction (e.g. a user-typed profile name, or a Latin technical token in RTL copy) is `FSI + text + PDI`.

The `intl` package provides higher-level helpers so you rarely hand-concatenate control characters:
- **`BidiFormatter`** — "formatting display text in a potentially opposite-directionality context without garbling layout issues." Constructed as `BidiFormatter.RTL`/`.LTR`/`.UNKNOWN`; `wrapWithUnicode(text)` wraps a string with the appropriate Unicode controls for plain-text output (the right choice for Flutter `Text`, which is not HTML), estimating direction when not given ([Dart intl: BidiFormatter class](https://pub.dev/documentation/intl/latest/intl/BidiFormatter-class.html), [Dart intl: BidiFormatter.wrapWithUnicode](https://api.flutter.dev/flutter/package-intl_intl/BidiFormatter/wrapWithUnicode.html)).
- **`Bidi`** — static utilities: `estimateDirectionOfText` ("Estimates the directionality of text using the best known general-purpose method (using relative word counts)"), `detectRtlDirectionality`, `isRtlLanguage` ("Check if a BCP 47 / III languageString indicates an RTL language"), `hasAnyRtl`/`hasAnyLtr`, and the embedding/mark constants ([Dart intl: Bidi class](https://pub.dev/documentation/intl/latest/intl/Bidi-class.html)).

(Note FSI's caveat, per Flutter's own constant doc: first-strong detection can mis-guess when the first character is the "wrong" script — e.g. an Arabic string opening with an ASCII quote may detect as LTR — so for a run whose direction you actually *know*, prefer the explicit `RLI`/`LRI` (or `BidiFormatter.RTL`/`.LTR`) over `FSI` ([Flutter: Unicode.FSI](https://api.flutter.dev/flutter/foundation/Unicode/FSI-constant.html)).)

### 7. Locale-specific numerals: Persian/Kurdish use **Extended** Arabic-Indic, Arabic uses Arabic-Indic — two different Unicode blocks

The PRD's numeral rule maps onto two distinct, non-interchangeable Unicode ranges:

| Set | Codepoints | Glyphs | Used by |
|---|---|---|---|
| **Arabic-Indic** | U+0660–U+0669 | ٠١٢٣٤٥٦٧٨٩ | Arabic (`ar`) |
| **Extended Arabic-Indic** (a.k.a. Persian) | U+06F0–U+06F9 | ۰۱۲۳۴۵۶۷۸۹ | Persian (`fa`), Kurdish Sorani (`ckb`), Urdu |

Wikipedia's Unicode table is explicit that these are **separate codepoints even where they look alike**: "Each numeral in the Persian variant has a different Unicode point even if it looks identical to the Eastern Arabic numeral counterpart," and the Persian set has **visually distinct glyphs for 4, 5, and 6** ([Wikipedia: Eastern Arabic numerals](https://en.wikipedia.org/wiki/Eastern_Arabic_numerals)). This is why a `fa` UI must use `۴۵۶`, not `٤٥٦` — using the Arabic block for Persian is a real localization defect a Persian reader will notice.

The correct production path is `intl`'s `NumberFormat`, which takes a locale and renders the locale's digits: it exposes `localeZero` ("the code point for the locale's zero digit") and applies locale symbols/digits in `format()` ([Dart intl: NumberFormat class](https://pub.dev/documentation/intl/latest/intl/NumberFormat-class.html)). **The known sharp edge** (which the engineering note must encode): `intl`'s digit handling across Arabic *sub*locales is inconsistent — bare `ar` and some country variants do not all emit Eastern digits, and the behavior **"is implemented in dates, but not in NumberFormat,"** per the maintainer-tracked issue ([dart-lang/i18n #197 — odd handling of Arabic locales and their digits (ar_DZ vs ar_EG vs ar)](https://github.com/dart-lang/i18n/issues/197)). The robust workarounds, drawn from the same issue thread and `intl`'s locale machinery:
- Pin the **numbering system via a Unicode `-u-nu-` extension** on the locale tag — `-u-nu-arab` forces Arabic-Indic, `-u-nu-latn` forces Western — so digit choice is explicit and not left to sublocale defaults.
- For dates, `DateFormat.useNativeDigits` ("Should we use native digits for printing DateTime, or ASCII") and its default-setter `useNativeDigitsByDefaultFor` control whether native (locale) digits are emitted, giving an explicit per-locale toggle that parallels the number path ([Dart intl: DateFormat.useNativeDigits](https://pub.dev/documentation/intl/latest/intl/DateFormat/useNativeDigits.html), [dart-lang/i18n #197](https://github.com/dart-lang/i18n/issues/197)).

The non-negotiable companion rule from the PRD: **never concatenate raw ASCII digits into localized strings** — always route a number through `NumberFormat` for the active locale (or an explicit numbering-system extension), and pass the result *into* the ICU placeholder, so the localized sentence and its digits agree ([PRD §13.3](../../PRD.md)).

### 8. Testing RTL is structural, not optional — and does not require a finished translation

Because direction follows the locale and the app is RTL in all three locales, RTL correctness is a *layout* property that can regress on any screen. The cheapest guards are: (a) override `Directionality`/locale in widget tests to assert mirrored geometry; (b) golden-screenshot each screen per locale (the PRD's "RTL golden screenshots per locale" gate); and (c) a static grep gate banning physical-side APIs (`EdgeInsets.only(left:`, `Alignment.centerLeft/Right`, `Positioned(left:`) in feature code, allowing only the `*Directional` variants and start/end insets ([Flutter: Directionality class](https://api.flutter.dev/flutter/widgets/Directionality-class.html), [PRD §20](../../PRD.md)). The W3C i18n guidance frames *why* this is layout-wide and not merely text alignment — base direction reorders the whole UI, not just paragraphs ([W3C i18n: Structural markup and right-to-left text in HTML](https://www.w3.org/International/questions/qa-html-dir)).

## Implications for Hifz Companion

1. **Adopt the first-party `gen_l10n` pipeline verbatim; make codegen a build gate.** `flutter_localizations` + `intl`, `generate: true`, an `l10n.yaml` (`arb-dir: lib/l10n`, `template-arb-file: app_ar.arb` since Arabic is our base content language, `output-class: AppLocalizations`), and the three Global* delegates. The generated type-safe class turns "missing key" into a compile error — directly satisfying the "zero missing ARB keys / no hardcoded user-facing strings" release gate. Add a grep/lint gate that bans string literals inside `features/` widgets so every user-facing string is forced through `AppLocalizations` (§1, [PRD §13.6](../../PRD.md), [PRD §20](../../PRD.md)).

2. **Let locale drive direction; never wrap the app in a hardcoded RTL `Directionality`.** Wiring `GlobalWidgetsLocalizations.delegate` and supporting only `ar`/`fa`/`ckb` means the ambient direction is RTL by construction. Reserve manual `Directionality(TextDirection.rtl/ltr)` for two narrow cases: forcing **LTR** around a Latin-only technical island, and any in-app language preview. Do **not** assume RTL in code where the locale already guarantees it (§2–3).

3. **Ban physical-side layout APIs in feature code — this is the top RTL rule.** Mandate `EdgeInsetsDirectional`/`AlignmentDirectional`/`*Directional` and `start`/`end` everywhere; forbid `EdgeInsets.only(left:/right:)`, `Alignment.centerLeft/Right`, and fixed positive `x`-offsets via a CI grep gate. Mirror the bottom nav (Today is rightmost), back/next chevrons, and any progress/decay direction. Because we are RTL in *every* locale, a single physical-side slip is a guaranteed visible bug, not a latent one (§3, [PRD §13.2](../../PRD.md)).

4. **Model the regional sabaq/sabqi/manzil term-sets with ICU `select` or a region-override ARB, not by editing base strings.** The track labels, grade verbs, and cycle names are regional ([PRD §13.4](../../PRD.md)); represent the term-set choice as a `select` over a region key (or a per-region override file the build merges) so swapping a vocabulary set is one file, never a code change. Keep the Kurdish (Sorani) defaults flagged "needs native + scholar review" in the ARB `@description` metadata until confirmed (§4, [PRD §21](../../PRD.md)).

5. **Treat `ckb` as a custom locale with a vendored Material-localizations delegate.** Declare `Locale.fromSubtags(languageCode: 'ckb')`, supply a custom `GlobalMaterialLocalizations` subclass (or vendor `ckb_localizations`) so built-in widget chrome is translated, and verify the delegate is registered before `AppLocalizations`. Our own strings already come from the ARB set; the delegate only covers framework chrome. Direction comes free once `ckb` resolves (Arabic script ⇒ RTL) (§5).

6. **Provide one bidi-isolation helper and route every mixed-script string through it.** A small `Bidi`-backed utility that wraps embedded runs (page numbers, "Juz N", surah names, dates, Latin tokens, user-typed profile names) in `FSI…PDI` (or `BidiFormatter.RTL/.LTR.wrapWithUnicode` when direction is known) prevents the "number jumps to the wrong end" class of bug. Prefer explicit `RLI`/`LRI` over `FSI` whenever the run's direction is known, because first-strong detection mis-guesses on leading punctuation. Add this to the localization checklist so no raw concatenation of opposite-direction runs ships (§6).

7. **Numerals: per-locale `NumberFormat`, with an explicit numbering-system extension to dodge the `intl` Arabic-digit inconsistency.** Persian and Kurdish must render Extended Arabic-Indic (U+06F0–U+06F9, `۴۵۶`); Arabic must render Arabic-Indic (U+0660–U+0669, `٤٥٦`) — different Unicode blocks, never substitutable. Because bare `ar` does not reliably emit Eastern digits in `NumberFormat` ([dart-lang/i18n #197](https://github.com/dart-lang/i18n/issues/197)), pin the numbering system explicitly (e.g. `ar-u-nu-arab`, `fa-u-nu-arabext` semantics, or `-u-nu-latn` where Western is wanted) and **golden-test the rendered digit glyphs per locale**. Never concatenate ASCII digits into a localized sentence — format the number, then inject it into the ICU placeholder (§7, [PRD §13.3](../../PRD.md), [PRD §20](../../PRD.md)).

8. **Keep Quran numerals and UI numerals as separate concerns.** Ayah numbers, juz/ḥizb markers, and sajda marks rendered *on the muṣḥaf page* come from the immutable glyph layer ([PRD §11.2](../../PRD.md)) — they are not `NumberFormat` output and must never be re-typeset by the i18n layer. The `NumberFormat`/numeral rules in §7 apply only to **chrome** (Today list, Progress stats, dates, percentages), keeping the sacred-text-fidelity boundary intact (§7, [PRD §11](../../PRD.md), [PRD §13.1](../../PRD.md)).

9. **Make Arabic's six plural categories a translation contract.** ARB/`intl` will produce correct Arabic plurals only if the translator supplies the needed CLDR categories (`zero/one/two/few/many/other`). Any count-bearing string ("N pages due", "N days in your catch-up plan") must use an ICU `plural` message, and the ARB review checklist must verify Arabic supplies every category — a missing `few`/`many` is a silent grammatical error a native reader will catch (§4).

10. **Bake RTL into the test matrix as a first-class, always-on gate, not a future locale.** Per-locale RTL golden screenshots of Today, Muṣḥaf chrome, Mutashābihāt, Progress, and Settings; widget tests that assert mirrored geometry under each `Directionality`; numeral-glyph goldens per locale; and the physical-side grep gate. This is the PRD's §20 localization gate made concrete, and it is cheap because no finished translation is needed to exercise layout direction (§8, [PRD §20](../../PRD.md)).

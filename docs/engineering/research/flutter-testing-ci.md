# Flutter & Dart Testing and CI — Research Note

**Topic:** Testing the Hifz Companion codebase — the Flutter/Dart test pyramid (unit, widget/golden, `integration_test`); making the pure-Dart scheduling engine deterministic and golden-testable; and a GitHub Actions CI that runs all of it and asserts that **no networking, analytics, or backend SDK ships in a release build** (PRD §19.3, §20 gate 6).
**Compiled:** 2026-06-16

This note is the evidence dossier behind the engineering synthesis doc `11-testing-strategy.md`. It assumes the locked constraints: a pure-Dart engine with **no I/O and an injected "today"** (PRD §7.12, §19.1), **immutable QPC glyph rendering** that must be visually proofed against reference muṣḥaf images (PRD R1, §11.3), **RTL fa/ckb/ar** golden screenshots (PRD §20 gate 5), and **fully offline** operation whose network-restraint must be machine-checked (PRD §20 gate 6). Everything below is sourced to official Flutter/Dart docs, the relevant flutter/flutter issues, or the maintained tooling packages.

---

## What the evidence says

### 1. The Flutter test pyramid: many cheap unit/widget tests, few expensive integration tests

Flutter's own testing guidance defines three layers with an explicit cost/confidence tradeoff. **Unit tests** (the `test` package) verify "a single function, method, or class," mock external dependencies, do no disk I/O or rendering, and are quick and low-maintenance but low-confidence. **Widget tests** (`flutter_test`) verify a single widget's layout, rendering, and interaction in a headless environment; quicker than integration, higher confidence than unit. **Integration tests** (`integration_test`) verify "the complete app or a large part of the app," run on a real device or emulator, and are the slowest, highest-maintenance, highest-confidence tier ([Flutter: Testing Flutter apps](https://docs.flutter.dev/testing/overview)). The official recommendation is blunt: "a well-tested app has many unit and widget tests, tracked by code coverage, plus enough integration tests to cover all the important use cases" ([Flutter: Testing Flutter apps](https://docs.flutter.dev/testing/overview)). For Hifz Companion this maps directly: the engine and DAOs get a broad base of unit tests, the recite/grade flow and heat-map get widget + golden tests, and a small set of `integration_test` cases cover the make-or-break journeys (cold-start → first day → review → catch-up, per PRD §20).

### 2. Unit tests are the natural home of the deterministic scheduling engine

Unit tests use the standalone `test` package: `test()` defines a case, `expect(actual, matcher)` asserts, and `group()` organizes; files end in `_test.dart`, live in `test/`, and run with `flutter test` ([Flutter: An introduction to unit testing](https://docs.flutter.dev/cookbook/testing/unit/introduction)) (the same package's `setUp()` runs shared fixtures before each case). Because the engine is specified as **pure Dart with zero I/O and `today` injected** (PRD §7.12, §19.1), it can be tested with the plain `test` package and **needs no Flutter binding at all** — the fastest, most stable tier. The PRD's invariant list (a memorized page's `due_at` is never later than its cycle ceiling; FAR/manzil items are never dropped; the engine never implies "safe to stop"; "identical inputs → identical schedule") is a ready-made unit/property test suite (PRD §7.12).

### 3. Determinism requires injecting time — never call `DateTime.now()` inside the engine

Test flakiness in time-dependent code comes from reading the wall clock. Two complementary patterns remove it:

- **Inject the date into the engine (the primary rule).** The PRD already mandates that the scheduler take "today" as a parameter and contain no wall-clock call (PRD §7.12, §19.3). This is what makes "identical inputs → identical schedule" a *testable* property: a golden test pins `today`, feeds a fixed `(Card, grade)` history, and asserts the exact `due_at`/track output. No `clock` or `fake_async` is needed *inside* the engine precisely because time is already an argument — the strongest, simplest form of determinism.
- **`clock` for the app layer around the engine.** The `clock` package is "a fakeable wrapper for dart:core clock APIs": library code calls `clock.now()` / `clock.stopwatch()` instead of `DateTime.now()`, and a test wraps execution in `withClock(Clock.fixed(...))` to pin time deterministically ([Dart: `clock` package](https://pub.dev/packages/clock)). Use it for the non-engine code that *does* read the clock — "days since last review" displays, the daily-notification scheduler, catch-up-gap detection.
- **`fake_async` for time-driven async.** `fake_async` "makes it easy to deterministically test code that uses asynchronous features like Futures, Streams, Timers, and microtasks," letting a test advance time explicitly with `FakeAsync().run((async) => async.elapse(...))`; critically, if you create your `DateTime`s/`Stopwatch`es "using the clock package's `clock.now()` … FakeAsync will automatically override them to use the same notion of time" ([Dart: `fake_async` package](https://pub.dev/packages/fake_async)). This lets a test fast-forward three days and assert the catch-up banner appears, with zero real waiting and zero flakiness.

### 4. Golden tests pin pixels — the right tool for the immutable muṣḥaf and RTL chrome, with sharp caveats

`matchesGoldenFile` is "an asynchronous matcher" that compares a `Finder`/`ui.Image` against a master reference image identified by a key; callers must use `expectLater` and `await` it. Golden files are created/updated with `flutter test --update-goldens` ([Flutter: `matchesGoldenFile`](https://api.flutter.dev/flutter/flutter_test/matchesGoldenFile.html)). This is the mechanism behind PRD's R1 visual-diff gate (render every page, diff against reference muṣḥaf images). But the same API doc carries the load-bearing warnings: "Golden file tests… can be more sensitive to underlying changes than other types of tests… A change to the version of Flutter or to the operating system on which the test is run can result in a slightly different rendering," and custom fonts "may render differently on different platforms" — a golden generated on one OS "is unlikely to match a golden generated on a different operating system" ([Flutter: `matchesGoldenFile`](https://api.flutter.dev/flutter/flutter_test/matchesGoldenFile.html)). The default test font is **Ahem**, which renders every glyph as a solid square unless real fonts are explicitly loaded via a `FontLoader` ([Flutter: `matchesGoldenFile`](https://api.flutter.dev/flutter/flutter_test/matchesGoldenFile.html)).

### 5. Golden drift is real and version/engine-sensitive — pin everything or goldens lie

The caveats above are not theoretical. Maintainers report goldens changing purely from a host OS upgrade ("Goldens images changed after upgrade to macOS 26.4," [flutter/flutter #184182](https://github.com/flutter/flutter/issues/184182)), font weight rendering differently under the **Impeller** engine versus **Skia** ([flutter/flutter #119317](https://github.com/flutter/flutter/issues/119317)), and — most tellingly — golden tests painting *different pixels from the same Ubuntu Docker image* depending on the host OS, so even a pinned container does not fully isolate text rendering ([flutter/flutter #131559](https://github.com/flutter/flutter/issues/131559)). The community consensus for stable goldens is therefore a fully controlled environment: **run goldens on a single OS (Linux x64) in CI**, ensure **consistent font loading**, and **lock themes, surface sizes, and animation state** (settled or disabled) so nothing varies between runs ([LeanCode: Golden tests — common mistakes & best practices](https://leancode.co/glossary/golden-tests-in-flutter)) — plus **pinning the Flutter SDK version**, since the framework itself warns that "a change to the version of Flutter… can result in a slightly different rendering" ([Flutter: `matchesGoldenFile`](https://api.flutter.dev/flutter/flutter_test/matchesGoldenFile.html)). For Hifz this is doubly important: the QPC per-page glyph fonts and the fa/ckb/ar UI fonts must be the bundled assets, loaded into the test harness, so the golden compares *the real typeset page* and catches a dropped diacritic (PRD R1).

### 6. Two golden modes: real-text "platform" goldens vs. font-independent "CI" goldens

The `alchemist` package formalizes the platform-drift problem with two modes. **Platform tests** render real, readable text, but their output "is dependent on the platform the test is running on" because "individual platforms are known to render text differently than others" — so they are for local development. **CI tests** "circumvent this" by rendering with the **Ahem** font (which "solely renders square characters"), replacing text blocks "with colored squares" and shadows with opaque colors, so their output is "always consistent regardless of the host platform" ([Alchemist package](https://pub.dev/packages/alchemist)). This is a deliberate tradeoff: CI goldens are stable across machines but no longer assert *exact glyph shapes*. For Hifz the muṣḥaf-fidelity goldens (R1) cannot use the Ahem trick — they must assert real KFGQPC glyphs — so those run pinned on one OS; the *layout/chrome* goldens (RTL mirroring, heat-map structure) can use the font-independent style to stay green across contributor machines. An alternative for layout-only assertions is a textual snapshot via `toStringDeep()` of the widget tree — a first-party Flutter diagnostics API that emits an indented tree dump — which is diff-friendly and font-agnostic, so it catches structural/RTL-mirroring regressions without depending on pixel rendering at all.

### 7. Widget tests drive the daily flow headlessly

Widget tests use `testWidgets((tester) async {...})`, building UI with `tester.pumpWidget(...)`, advancing frames with `pump([duration])`, settling animations with `pumpAndSettle()`, locating widgets with `find.byType` / `find.text` / `find.byKey`, and asserting with matchers like `findsOneWidget` / `findsNothing` ([Flutter: An introduction to widget testing](https://docs.flutter.dev/cookbook/testing/widget/introduction)). This is the tier for Hifz's reveal-on-tap recite flow, grade buttons, catch-up banner, and the heat-map — all testable without a device. RTL correctness (PRD §13.2, §20 gate 5) is asserted by pumping each screen under `Directionality(textDirection: TextDirection.rtl)` per locale and golden-capturing the result.

### 8. `integration_test` covers the end-to-end journeys on a real device

The `integration_test` package ships with the Flutter SDK; add it with `flutter pub add "dev:integration_test:{sdk: flutter}"`, place tests in a top-level `integration_test/` directory, call `IntegrationTestWidgetsFlutterBinding.ensureInitialized()` at the top of `main()`, then reuse the same `flutter_test` APIs (`testWidgets`, finders, matchers). Tests run with `flutter test integration_test/app_test.dart` against a connected device/emulator or desktop ([Flutter: Check app functionality with an integration test](https://docs.flutter.dev/testing/integration-tests)). The binding "executes tests on a physical device or emulator" so it exercises the real SQLite/Drift stack, real asset loading, and real rendering — the right place for Hifz's headline journeys: cold-start onboarding → first generated day → review → missed-day catch-up (PRD §7.10, §7.9, §20).

### 9. The test harness blocks the network by default — a feature we should weaponize

A subtle but decisive fact: "In non-browser tests, [`TestWidgetsFlutterBinding`] overrides `HttpClient` creation with a fake client that always returns a status code of 400. This is to prevent tests from making network calls, which could introduce flakiness" ([Flutter: `TestWidgetsFlutterBinding`](https://api.flutter.dev/flutter/flutter_test/TestWidgetsFlutterBinding-class.html)). This is widely (and confusingly) observed as "all HTTP requests return 400" in widget tests ([flutter/flutter #77245](https://github.com/flutter/flutter/issues/77245), [#35318](https://github.com/flutter/flutter/issues/35318)). For most apps it is an annoyance; for an **offline-by-design** app it is a gift. Hifz tests should deliberately **leave the default in place** for every test except the single asset-downloader test, and can additionally install a custom `HttpOverrides.global` that throws on any connection attempt, turning "code tried to use the network" into an immediate, explicit test failure rather than a silent 400.

### 10. Asserting "no networking ships" needs a build-time check, not just runtime tests

Runtime tests prove the running code does not call out; they cannot prove a release binary is *free of* an analytics/ads/backend SDK that might phone home (PRD §20 gate 6, §19.3). Runtime restraint and static restraint are different guarantees, and the PRD's "release binary contains no analytics/ads/backend SDKs" gate is a *static* claim. Two complementary build-time checks close the gap:

- **Allow-list dependency audit.** The committed `pubspec.lock` is the source of truth for what the app actually links — it "becomes the best source of truth … because it shows the versions your app actually installs" ([Vulert: Dart & Flutter dependency security](https://vulert.com/blog/dart-flutter-dependency-security/)) — and `flutter pub deps` prints the full transitive graph. A small CI step parses the resolved package set and **fails if any package outside an explicit allow-list appears** (no `firebase_*`, no `google_analytics`, no ad SDK, no `sentry`/crash-reporter, no `google_fonts` runtime fetch). Because the audit runs against the *lock file*, it catches a forbidden package pulled in transitively by an innocent-looking direct dependency, which is exactly where an analytics shim would hide.
- **Forbidden-import lint rules scoped per layer.** Even an allow-listed package (`package:http`, needed by the downloader) must not be importable from the engine or domain layers. The `import_rules` analyzer plugin enforces this in `analysis_options.yaml` — e.g. `target: lib/engine/** / disallow: package:http/**` — so a networking import in the pure engine fails `dart analyze` ([import_rules](https://github.com/fujidaiti/import_rules)); DCM's `avoid-banned-imports` does the same with per-path `paths`/`deny`/`message`/`severity` entries ([DCM: avoid-banned-imports](https://dcm.dev/docs/rules/common/avoid-banned-imports/)). The **one** sanctioned networking path — the asset-pack downloader (PRD §11.1.1, §19.3) — is confined to a single module (`lib/assets/**`); every other call site is rejected at build time.

Together these turn "fully offline" from a promise into a build invariant: the release cannot link a phone-home SDK (audit) and the engine cannot even *import* a socket (lint), independent of whether any test happens to exercise that path.

### 11. Property-based testing hardens the engine invariants

The PRD's invariants are universal claims ("for *every* card and *every* grade sequence, `due_at` never exceeds the cycle ceiling," PRD §7.12), which is exactly what property-based testing checks. `glados` is the established Dart property-based framework: `Glados<int>().test('…', (a) { expect(...); })` first "generates increasingly complex, random inputs until one breaks the property," then in a shrinking phase "gradually simplifies the input and returns the smallest input that's still breaking the property," reporting e.g. "Tested 1 input, shrunk 25 times. Failing for input: [0]"; custom generators are written as `Any` extensions with `combine2` (for data classes) / `choose` (for enums) ([glados](https://pub.dev/packages/glados)). It is mature rather than fast-moving (latest **1.1.7**, MIT; no recent releases as of mid-2026), but its API is stable and its only role here is in the test target — a defensible test-only dependency. A generator for `(Card, List<Grade>, today)` sequences lets a handful of property tests assert the trust-clamp and never-dropped-manzil invariants across thousands of random schedules — far stronger than hand-picked fixtures, and matching PRD §20 gate 4 ("invariant tests as property-based checks"). (If `glados` ever stalls, the zero-dependency fallback is a seeded `Random` driving `package:test` parameterized cases — you lose automatic shrinking but keep deterministic, reproducible generation by logging the seed.)

### 12. CI: `subosito/flutter-action` is the standard runner; pin it for golden stability

The de-facto GitHub Actions setup is `subosito/flutter-action@v2` (latest release tag `v2.23.0`), which downloads and configures the SDK on Linux, Windows, and macOS and accepts `channel`, `flutter-version`, `cache`, and `pub-cache` inputs ([subosito/flutter-action](https://github.com/subosito/flutter-action)). Pin the action to a full commit SHA (not a floating major) on a public, audited repo so the workflow's own supply chain is reproducible. A minimal job is `actions/checkout` → `subosito/flutter-action@v2` (with `channel: stable`) → `flutter pub get` → `flutter analyze` → `flutter test`. Because goldens are version- and OS-sensitive (Finding 5), the golden job must **pin a specific `flutter-version` and run on a fixed runner** (e.g. `ubuntu-latest`, Linux x64) so reference images stay stable; tagging golden tests (`@Tags(['golden'])` + `--tags=golden`) lets CI isolate and run them in that controlled job. A concrete job split that maps to the PRD gates:

| Job | Runner | Runs | PRD gate |
|---|---|---|---|
| `verify` (every push) | `ubuntu-latest` | `flutter analyze`; engine unit + `glados` property tests; widget tests; `--coverage` | §20.3, §20.4 |
| `golden` (pinned) | `ubuntu-latest`, pinned `flutter-version` | `flutter test --tags=golden` with real fonts loaded | §20.2, §20.5 |
| `restraint` | `ubuntu-latest` | dependency allow-list audit + `dart analyze` import bans | §20.6 |
| `assets` | `ubuntu-latest` | SHA-256 manifest check vs published GitHub Release | §20.1 |
| `integration` | macOS/Linux + emulator | `flutter test integration_test/` headline journeys | §20 (other) |

Standard GitHub-hosted runners — including macOS — are **free for public repositories** ([GitHub Docs: Actions billing](https://docs.github.com/billing/managing-billing-for-github-actions/about-billing-for-github-actions)), so the integration job costs nothing for an open-source *ṣadaqah* repo and the public CI logs themselves become part of the audit/trust story.

### 13. Coverage and reporting are first-class in CI

`flutter test --coverage` emits `coverage/lcov.info`, and `genhtml` turns it into a browsable HTML report ([Code With Andrea: Flutter test coverage](https://codewithandrea.com/articles/flutter-test-coverage/)). The standard refinement is to strip generated files (`*.g.dart`, `*.drift.dart`, `*.freezed.dart`) from the lcov before reporting — via `lcov --remove` or an `--coverage-package`/exclusion glob — so the number reflects hand-written code rather than codegen. For an open-source *ṣadaqah* project, the lcov can be uploaded to a free-for-OSS service (Codecov/Coveralls) via their GitHub Action to make the engine's coverage publicly auditable — consistent with the "open-source & auditable" engineering value and PRD §21's open-source intent. (No coverage *number* is promised here; the engine's invariants matter more than a percentage.)

## Implications for Hifz Companion

1. **Engine = plain `test` package, no Flutter binding.** Keep `/engine` a pure-Dart package; test it with `package:test`, inject `today`, and never import `flutter_test`. This is the fastest, most stable tier and enforces the "pure, deterministic, no I/O" rule structurally (PRD §7.12, §19.1).
2. **Codify every PRD §7.12 invariant as a property test in `glados`.** Generate random `(Card, grade-sequence, today)` inputs and assert: `due_at ≤ cycle ceiling`, FAR/manzil never dropped, lapse demotes, no "safe to stop" output, identical inputs → identical schedule. Shrinking will hand you the minimal failing schedule when an invariant breaks (PRD §20 gate 4).
3. **Pin golden goldens.** Run muṣḥaf-fidelity goldens (R1 visual-diff) on **one OS** with a **pinned Flutter version**, with the **real KFGQPC and UI fonts loaded** via `FontLoader` (never Ahem), fixed `devicePixelRatio`/size/theme, animations disabled. A one-pixel diacritic change must fail the build (PRD R1, §11.3).
4. **Two golden tiers.** Use real-glyph goldens only where fidelity is the point (the page render); use font-independent / `toStringDeep` snapshots for layout-and-RTL-mirroring goldens so they stay green across contributor machines and OSes (Findings 5–6, PRD §20 gate 5).
5. **RTL goldens per locale.** Capture each key screen under `Directionality.rtl` for fa, ckb, and ar, with locale numerals/calendars rendered, as the concrete artifact for PRD §20 gate 5 ("RTL golden screenshots per locale").
6. **Reserve `integration_test` for the journeys that define the product.** Cold-start → first day → review → missed-day catch-up, exercising real Drift/SQLite and asset loading (PRD §7.9–§7.10, §20). Keep this set small; lean on unit/widget tests for everything else (Finding 1).
7. **Make the network the enemy in tests.** Keep the default 400-blocking `HttpClient` in place for all tests, and add a global `HttpOverrides` that *throws* on connect, so any accidental network call fails loudly. Only the asset-downloader test opts out (Finding 9, PRD §17, §19.3).
8. **Two-layer "no networking ships" gate (PRD §20 gate 6).** (a) A CI dependency-allow-list step that fails if any analytics/ads/backend/crash-SDK appears in the resolved graph; (b) `import_rules` / DCM `avoid-banned-imports` forbidding networking imports everywhere except the single whitelisted downloader module. Together these make "fully offline" a build invariant, not a promise.
9. **Asset-integrity checks belong in CI, not just runtime.** Re-verify the pinned SHA-256 of every asset pack (text, 604 page fonts, mutashābihāt) against the published GitHub Release in CI, so a tampered or drifted asset fails the build before it can ship (PRD §11.3, §20 gate 1) — detailed in the sibling note `asset-integrity-and-distribution.md`.
10. **CI shape.** One fast job (`analyze` + unit + widget + property tests + coverage) on every push; one pinned, Linux-only golden job (`--tags=golden`); a dependency/import-restraint job; and the integration-test journeys on an emulator. Use `subosito/flutter-action@v2` with `cache: true` and a pinned `flutter-version` (Finding 12).
11. **Publish coverage.** Emit `lcov.info`, strip generated files, and upload to a free-for-OSS coverage service so the engine's test coverage is publicly auditable — fitting a project built openly as *ṣadaqah jāriyah* (Finding 13, PRD §21).

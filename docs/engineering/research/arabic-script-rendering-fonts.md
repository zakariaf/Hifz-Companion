# Arabic-Script & Quran Text Rendering in Flutter — Research Note

**Topic:** Rendering Arabic-script and Quran text faithfully in Flutter — the complex-text-shaping pipeline (HarfBuzz/Skia/Impeller) and where it drops or misplaces diacritics; Persian/Arabic/Kurdish-Sorani UI-font coverage; and the KFGQPC per-page glyph-font (QCF/QPC) "immutable-muṣḥaf" approach that lets us draw an exact printed page without trusting the OS shaper. Scoped to a fully-offline, no-AI Flutter app whose first non-negotiable value is that *one wrong diacritic ends the project* (PRD R1; Blueprint §1 Engineering value 1).

**Compiled:** 2026-06-16 · Research-agent topic: Quran text-rendering fidelity · Scope: Flutter's text stack and open Arabic/Quran rendering issues; the Quran Foundation / QUL (Tarteel) glyph-font documentation; KFGQPC font licence terms; Tanzil Quranic-font catalogue; W3C ALReQ and Kurdish-typography requirements for the fa/ckb/ar UI chrome. This is the evidence dossier behind `engineering/08-quran-data-and-immutable-rendering.md` and feeds `engineering/12-localization-rtl-accessibility-impl.md`.

> Framing note. There are **two distinct rendering problems** in this app and conflating them is the classic mistake. (1) **The Quran muṣḥaf itself** — sacred, fixed, must be byte-perfect and pixel-faithful; we solve it by *refusing to shape it at all* and drawing pre-baked per-page glyph fonts (PRD §11.2; R1). (2) **The UI chrome** — buttons, labels, the heat-map legend, in fa/ckb/ar — which *is* ordinary complex text and *does* go through Flutter's shaper, so it must use a UI font with full Persian and Kurdish-Sorani coverage. This note keeps the two strictly separate, because the engineering obligations are opposite: for the Quran we distrust the shaper; for the chrome we depend on it.

## What the evidence says

### 1. Flutter's text stack shapes complex scripts with HarfBuzz — capable, but a layer we cannot afford to trust for the muṣḥaf

Flutter does not ship its own Arabic shaper. Its text engine (historically a fork of Android's Minikin called **libtxt**, now folding into Skia's text layout) divides the work: **libtxt** handles font selection, bidi and line-breaking; **HarfBuzz** performs glyph selection and shaping; **Skia** (or **Impeller**) rasterises ([Flutter: Engine architecture](https://chromium.googlesource.com/external/github.com/flutter/flutter/+/refs/tags/3.36.0-0.5.pre/docs/about/The-Engine-architecture.md); [Esfahbod, *State of Text Rendering 2024*](https://behdad.org/text2024/)). HarfBuzz is the same industry-standard shaper used by Chrome, Android, and Adobe's World-Ready Composer; its job is precisely "glyph selection, positioning, and mark attachment" for complex scripts ([Esfahbod, *State of Text Rendering 2024*](https://behdad.org/text2024/)).

The point for us is **not** that HarfBuzz is bad — it is excellent — but that **shaping is a transformation with platform-specific inputs** (the FreeType/CoreGraphics rasteriser differs between Android and iOS; the engine differs between Skia and Impeller). Quran text is uniquely unforgiving: it carries dense stacked vocalisation marks (e.g. *shadda + kasra*, *small high* marks, *waṣla*, *maddah*) whose **mark attachment** is exactly the step most likely to vary. A guarantee that *every glyph on every page is identical on every device and OS version* (PRD §20 gate 2 — visual-diff on min-OS iOS/Android) cannot be founded on a shaping step whose output we do not control. That is the whole motivation for §3–§5.

### 2. Flutter has a documented, recurring history of *exactly* the failure mode the muṣḥaf cannot tolerate: misplaced/broken Arabic diacritics and ligatures

This is not a theoretical worry. The Flutter tracker contains long-lived, reproducible bugs in Arabic vocalisation and ligature shaping:

- **Quran diacritics land on the wrong side of the letter.** In flutter/flutter#16886, the test word `يَسۡ‍َٔلُونَ` rendered with the **KFGQPC Uthmanic Script HAFS** font shows "the diacritics moved to the beginning of the letter not the end" in Flutter, while *the same font renders correctly in native iOS (Xcode)* — the reporter notes this was "a known issue with Android" ([flutter#16886, *Arabic Quran diacritics problem*](https://github.com/flutter/flutter/issues/16886)). A misplaced ḥaraka changes what is read; for a muṣḥaf this is catastrophic.
- **Specific letters/ligatures shatter into disconnected parts depending on the font.** flutter#143975 reports the combination `الإ` rendering "incorrectly with disconnected parts" under Google's **Lateef** and **Scheherazade New**, while **Amiri** renders it correctly — same text, same Flutter, different font; still **open**, triaged P2 to the Engine team, found in 3.19/3.20 ([flutter#143975](https://github.com/flutter/flutter/issues/143975)).
- **The rendering backend itself can corrupt Arabic.** flutter#119805 was a **P0** Impeller-on-iOS bug where Arabic text rendered incorrectly (and font weight failed to apply) — engine-level, in the rasteriser, not the app ([flutter#119805, *[Impeller] Incorrect Arabic Text Rendering*](https://github.com/flutter/flutter/issues/119805)). iOS-specific diacritic corruption was also reported in flutter#54529 ([flutter#54529](https://github.com/flutter/flutter/issues/54529)).
- **The same backend can break cursive joins.** flutter#147577 reports that **Impeller** inserts "unwanted gaps between Arabic characters" (and elsewhere overlaps glyph ends) for IBM Plex Sans Arabic and other Arabic families, "whereas Skia renders everything fine" except at small sizes — a join failure that depends purely on which rasteriser is active ([flutter#147577, *[Impeller] Unwanted gaps between Arabic characters*](https://github.com/flutter/flutter/issues/147577)).
- **The two mobile platforms can disagree with each other on the same text.** flutter#32734 documents an Arabic character "rendered differently between iOS and Android" — the most direct possible proof that shaping output is not a single deterministic function of the input, but varies by platform rasteriser ([flutter#32734](https://github.com/flutter/flutter/issues/32734)). This is exactly why a sacred-text guarantee cannot rest on shaping, and why the PRD's visual-diff gate must run on *both* min-OS targets (PRD §20.2).

The throughline: Arabic-shaping correctness in Flutter is **font-dependent, platform-dependent, and backend-dependent** — the *same string* can join, break, or stack marks differently across fonts (#143975), across iOS vs Android (#32734), and across Skia vs Impeller (#147577, #119805) — and the marks most at risk are the vocalisation marks the Quran is saturated with. For UI chrome (§6–§7) these bugs are survivable annoyances; for the muṣḥaf they are project-ending. Hence: do not let the shaper near the muṣḥaf.

### 3. The fidelity-preserving technique used across the serious Quran-tech ecosystem: per-page **glyph fonts** where each glyph is a whole pre-shaped word

The open Quran-rendering ecosystem (Quran.com / Quran Foundation, Tarteel's QUL, and downstream apps) converged on a technique that sidesteps shaping entirely. The KFGQPC **QCF/QPC** fonts are *glyph-based*: rather than mapping characters to letter-forms, the font maps codepoints to **"each Quranic word as a unique glyph, which is a visual representation of an entire word"** ([QUL/Tarteel, *Glyph-Based Fonts*](https://qul.tarteel.ai/docs/glyph-based)). The shaping, ligature-joining, and mark-stacking were done **once, by the calligrapher (Uthman Taha) and the King Fahd Complex**, and frozen into the glyph outlines. At runtime the engine is not asked to *shape* anything — it just draws glyph N from font P. The OS shaper's discretion is removed from the sacred path.

Because the glyphs are pre-baked per the *printed* page, the system needs **one font per muṣḥaf page**: "each font corresponding to a specific page of the Quran. Most Muṣḥaf has 604 pages" ([QUL/Tarteel, *Glyph-Based Fonts*](https://qul.tarteel.ai/docs/glyph-based)). Versions: **QPC V1** (Taha's 1405H print), **QPC V2** (1423H edition, the common modern Madani look), **V4** (adds tajwīd colour data in the font) ([QUL/Tarteel, *Glyph-Based Fonts*](https://qul.tarteel.ai/docs/glyph-based); [QuranPortal, *Rendering the Quran Muṣḥaf Digitally*](https://quranportal.io/blog/rendering-the-quran-mushaf-digitally)). A newer **QCF V4** reduces the file count from 604 to ~47 by reusing glyph tables across pages ([MohamadHajjRabee, *quran-qcf4*](https://github.com/MohamadHajjRabee/quran-qcf4)).

### 4. The QCF glyph codes are **not human-readable Unicode text** — and treating them as text silently corrupts the page

This is the single most important operational detail, and it has a precise consequence in the Quran Foundation tutorial. Each word ships with two representations: **`code_v2`** (special glyph codepoints that the per-page font interprets visually) and **`text_qpc_hafs`** (standard Arabic Unicode, a fallback). The warning is explicit: **"QCF glyph codes contain special Unicode characters that must be rendered using `innerHTML` … not `textContent`. Using `textContent` will display incorrect characters"** ([Quran Foundation, *Integrating Quran Font Rendering*](https://api-docs.quran.foundation/docs/tutorials/fonts/font-rendering/)).

The matching prohibition: **"If you attempt to use `text_qpc_hafs` (standard Unicode) with a QCF font file, the characters won't match the glyph definitions … producing garbled output or placeholder squares"** — and conversely the per-page font *must* match the page: `p{pageNumber}-v2` loads `…/v2/woff2/p{pageNumber}.woff2` ([Quran Foundation, *Integrating Quran Font Rendering*](https://api-docs.quran.foundation/docs/tutorials/fonts/font-rendering/)). Two things follow for us: (a) the glyph-code string and its page-font are an **inseparable pair**; storing one without pinning the other is a corruption waiting to happen; (b) the glyph codes are **opaque** — they are addresses into a glyph table, *not* the Quranic text, so they must never be confused with, parsed as, or substituted for the actual Uthmani text (which we store separately and checksum — PRD R1).

There is a clean Unicode alternative — **QPC Hafs / IndoPak are ordinary Unicode fonts** that take normal Arabic text and render acceptably with "standard text rendering" ([Quran Foundation, *Integrating Quran Font Rendering*](https://api-docs.quran.foundation/docs/tutorials/fonts/font-rendering/)). The trade-off is exactly the §1–§2 trade-off: a Unicode font is simple but hands shaping back to the OS; the per-page glyph font is heavier but guarantees pixel-faithful, shaper-independent pages. For a sacred-text-fidelity app the glyph font is the correct, and the ecosystem's standard, choice.

### 5. Faithful page layout is **data, not runtime line-breaking** — and the per-page font only works if the layout matches it

The glyph font renders correct *glyphs*; it does not by itself place lines. Faithful muṣḥaf layout comes from a separate **page-layout dataset**: each word carries `page_number`, `line_number`, and `position`, and you **"group words by `page-{page_number}-line-{line_number}` … not by verse boundaries"** because "a single Muṣḥaf line often contains words from multiple verses" — so naive verse-based wrapping "would incorrectly fragment verses across lines, destroying the authentic physical layout" ([Quran Foundation, *Page Layout API Guide*](https://api-docs.quran.foundation/docs/tutorials/fonts/page-layout/)). Different muṣḥaf editions physically differ (IndoPak 15-line = 610 pages, 16-line = 548), so page boundaries **must** be taken from data, never recomputed ([Quran Foundation, *Page Layout API Guide*](https://api-docs.quran.foundation/docs/tutorials/fonts/page-layout/)). This is precisely PRD §11.2's rule ("line breaks and page breaks come only from the bundled layout data") and §6.1's "the app never recomputes" the structure.

The QuranPortal write-up adds a layout subtlety worth recording: because each word is a separate inline element, justified mushaf lines need a whitespace-collapse trick — **"setting `font-size: 0` on the container eliminates whitespace, then individual characters restore their proper size"** — to stop the renderer injecting spurious gaps between word-glyphs ([QuranPortal, *Rendering the Quran Muṣḥaf Digitally*](https://quranportal.io/blog/rendering-the-quran-mushaf-digitally)). It also notes line typography is tuned **per edition** so each word "fits perfectly on its designated line" ([QuranPortal](https://quranportal.io/blog/rendering-the-quran-mushaf-digitally)) — i.e. the layout and the font are co-designed and cannot be mixed across editions.

### 6. There is a ready Flutter precedent that bundles the 604 per-page fonts offline — proving the approach fits our constraints

The pure-Flutter `qcf_quran` package bundles **604 page-specific QCF fonts (QCF_P001…QCF_P604)** "to ensure every page looks exactly like the printed Madani Muṣḥaf, with correct ligatures, verse endings, and scaling," explicitly accepting that "~600 font files … increases the package size but guarantees 100% accurate rendering **without internet dependency**," and resolving the correct `fontFamily` automatically from the page number (with a separate `QCF_BSML` family for verse-number/symbol glyphs) ([qcf_quran (pub.dev)](https://pub.dev/packages/qcf_quran)). This is strong existence-proof that the immutable-muṣḥaf approach is (a) implementable in plain Flutter, (b) fully offline — matching PRD C1 — and (c) compatible with Flutter's standard font bundling: fonts are declared per-`family` under `flutter: fonts:` in `pubspec.yaml` and selected via `TextStyle(fontFamily: …)`; Flutter "does not simulate" missing weights/styles and "you cannot override a font's weight or style," so each variant must be a real file ([Flutter: *Use a custom font*](https://docs.flutter.dev/cookbook/design/fonts)). (Our packs are *downloaded once and checksum-verified* rather than app-bundled — PRD §11.1.1 — but the rendering mechanics are identical.)

### 7. The UI chrome (fa/ckb/ar) is a *different* problem: pick a UI font with verified Persian **and** Kurdish-Sorani coverage

Everything above concerns the Quran. The app's buttons, labels, dates and numerals are ordinary complex text and *do* pass through HarfBuzz (§1), so here the obligation flips to **glyph coverage**:

- **Persian/Arabic UI font.** **Vazirmatn** (Rastikerdar) is a modern Persian/Arabic family, 9 weights, and from v27 states coverage for "Persian, Arabic, … Kurdish, Pashto, Urdu …" ([Rastikerdar, *Vazirmatn*](https://github.com/rastikerdar/vazirmatn)). **Noto Naskh Arabic** is a classical Naskh face that "supports … Persian, Kurdish, and Central Kurdish" with ~1,598 glyphs ([Google Fonts, *Noto Naskh Arabic*](https://fonts.google.com/noto/specimen/Noto+Naskh+Arabic)). Both are open-licensed and bundleable offline (PRD §13.5 — no font-CDN fetch).
- **Kurdish-Sorani is the real coverage risk.** Sorani is written with an Arabic-based alphabet that adds letters absent from core Arabic — consonants **پ چ ژ گ ڤ ڕ** and the mandatory vowels **ۆ ۇ ۊ ە ێ** — and, "unlike the regular Arabic script, which is an abjad, Kurdish Arabic is an alphabet in which **vowels are mandatory**" ([Wikipedia, *Kurdish typography*](https://en.wikipedia.org/wiki/Kurdish_typography)). A font missing any of these shows tofu (▯) or, worse, *silently wrong* shaping. The same source documents a real trap: **XB Shafigh Kurd "automatically adds a ئ (Yeh with Hamza above) before vowels … but since there aren't any actual ئs in the text, you have to add them when you switch fonts"** — i.e. font-specific behaviour can mutate what *looks* like correct text ([Wikipedia, *Kurdish typography*](https://en.wikipedia.org/wiki/Kurdish_typography)). This is why PRD §13.5 mandates *verifying* all Sorani letters render before locking the UI font.
- **Arabic shaping rules apply to the chrome too.** Arabic is a joining/cursive script needing more leading and forbidding naive letter-spacing (which "shatters cursive joins"); Persian/Sorani may want a Naskh model; numerals are locale-specific (Extended Arabic-Indic ۰-۹ for fa/ckb vs Arabic-Indic ٠-٩ for ar) ([W3C, *Arabic & Persian Layout Requirements (ALReQ)*](https://www.w3.org/TR/alreq/); [W3C, *Arabic Script Gap Analysis*](https://www.w3.org/TR/alreq-gap/)). These bind the design-system typography note, not this one, but they are the reason a single hard-coded "Arabic font" is insufficient across fa/ckb/ar.

### 8. Licence reality: KFGQPC fonts are *usable and redistributable but may not be modified*; the text and layout are separately licensed

The KFGQPC font licence grants permission "free of cost … to Use, Copy, and Distribute," but the font **"cannot be sold, modified, altered, translated, reverse engineered … or reproduced … without express written approval of King Fahd Glorious Quran Printing Complex"** ([Open Hub, *King Fahd Glorious Quran Printing Complex License*](https://openhub.net/licenses/KFGQPC)). For us this is a *good* fit and a *constraint*: redistributing the **unmodified** QCF/QPC fonts in a free, open asset repo is permitted, but we must **not** sub-set, re-hint, re-name, or otherwise alter the font files — which also happens to be exactly what fidelity demands (an altered Quran font is the nightmare R1 guards against). Note the layers carry *different* licences and must be tracked separately: the **Uthmani text** is Tanzil under CC BY 3.0 (PRD §11.1, R1), the **page-layout** dataset is QUL/open, and the **glyph fonts** are KFGQPC's own terms. Tanzil's font catalogue confirms KFGQPC HAFS Uthmanic Script and KFGQPC Uthman Taha Naskh alongside Unicode-shaping fonts (SIL **Scheherazade**, **me_quran**, **PDMS Saleem**) — the latter being the §4 "Unicode alternative" class, not the per-page glyph class ([Tanzil, *Quranic Fonts*](https://tanzil.net/docs/quranic_fonts)).

## Implications for Hifz Companion

Concrete, numbered, and scoped to the PRD's locked constraints (R1; §11; §13; §20).

1. **Render the muṣḥaf with KFGQPC per-page glyph fonts and never invoke the OS shaper for Quran text.** This is PRD §11.2 / R1 restated with its evidence: §1–§2 show Flutter's Arabic shaping is font-/platform-/backend-dependent and has shipped misplaced-diacritic and broken-ligature bugs (flutter#16886, #143975, #119805); §3 shows the ecosystem's answer is pre-baked per-word glyphs. The font *is* the typeset page; we draw glyph codes, we do not shape characters.

2. **Treat the glyph-code string and its page font as one atomic, version-pinned unit.** Per §4, a `code_v2` string is meaningless — even corrupting — without the exact matching `p{N}` font, and must never be parsed as, or substituted by, real Arabic text. Store the page's glyph payload and its font identity together; pin both by checksum (PRD §11.1.1). The opaque glyph codes are **not** the Quran text — keep the checksummed Uthmani text (Tanzil, R1) as the separate source of truth for search, audit, and any non-rendered use.

3. **Take page and line geometry only from the bundled layout dataset; group by `page-line`, never by verse, and never compute line breaks at runtime.** Per §5 and PRD §11.2/§6.1, a muṣḥaf line mixes words from multiple verses and editions differ in page count, so any runtime wrapping destroys fidelity. The 604-page structure, line types (ayah / sūra-name / basmala / centred), and word positions are fixed reference data (PRD §10.1 `page`/`line` tables).

4. **Draw every overlay (weak-line highlight, mutashābihāt anchor, error position, current-ayah) as coordinates *over* the immutable glyph layer — never by re-typesetting.** PRD R1 and §9.2/§11.2 require this; §5's "geometry-first, not text-first" framing is the supporting rationale. Overlays consume the same bundled line/word geometry as the layout, so they align to the printed page without ever touching the text.

5. **Make the visual-diff gate (PRD §20.2) the real guarantee, precisely because §1–§2 show shaping output varies by platform and backend.** Render all 604 pages on min-iOS *and* min-Android — both targets, because §2 documents iOS↔Android (#32734) and Skia↔Impeller (#147577) divergence on identical input — against reference muṣḥaf images within a tight pixel tolerance; a diff fails the build. Glyph fonts make this *deterministic enough to gate on* — the whole reason for choosing them over Unicode shaping. Add a manifest hash-check that all 604 page fonts are present and unmodified (PRD §11.3.2).

6. **Bundle/download the QCF fonts unmodified, and never sub-set or rename them.** §8: the KFGQPC licence permits redistribution of unmodified files but forbids modification without written approval — and modification would also break fidelity. The `qcf_quran` precedent (§6) shows ~604 fonts is viable fully offline; our packs arrive via the one-time checksum-verified download (PRD §11.1.1), not the binary, so the ~tens-of-MB font weight does not bloat the app install. Record the KFGQPC, Tanzil (CC BY 3.0), and QUL layout licences as three separate attribution entries.

7. **For UI chrome, choose one UI font with *verified* fa + ckb + ar coverage and prove Sorani before locking it (PRD §13.5).** Candidates: Vazirmatn (claims Kurdish from v27) or Noto Naskh Arabic (claims Central Kurdish) — but §7's evidence is that Sorani's extra letters (پ چ ژ گ ڤ ڕ ۆ ۇ ۊ ە ێ) and mandatory vowels are exactly where coverage silently fails, and that some Kurdish fonts mutate text on font-switch. Add a CI/golden screenshot that renders a Sorani pangram containing every extra letter in each candidate font and asserts no tofu; only then lock it. Bundle the UI font offline (no `google_fonts` runtime fetch — PRD §19.1).

8. **Keep the two pipelines architecturally separate in `/quran` vs the app UI layer.** Per the framing note: the `/quran` module (PRD §19.2) renders glyph-font pages and distrusts the shaper; the rest of the app renders localized chrome through the normal Flutter shaper with the §7 UI font. Do not let a generic "Arabic text" style leak the muṣḥaf into the shaped path, and do not pay the per-page-font machinery for ordinary labels. The UI chrome still owes the standard Arabic-script obligations (more leading, no letter-spacing, locale-correct numerals) — but those are owned by the design-system typography and RTL notes, cited here only to mark the boundary ([W3C, ALReQ](https://www.w3.org/TR/alreq/)).

9. **Architecture is muṣḥaf-swappable by construction (PRD R2), because layout+font+text are independent, co-versioned assets.** §3/§5 show edition geometry and glyph fonts are co-designed per edition; §8 shows the licences differ per layer. So model the muṣḥaf as a triple {text-checksum, layout-dataset, font-set} selected by `mushaf_id` (PRD §10.1 `mushaf` table) — a future riwāyah/layout plugs in as a new triple without an engine rewrite, exactly as PRD R2 requires.

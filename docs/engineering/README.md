# Hifz Companion — Engineering Documentation

This is the engineering doc set for **Hifz Companion**, a fully offline, free (built as *ṣadaqah jāriyah*) Flutter app that helps huffaz and serious students **retain** the Quran — an intelligent muṣḥaf-page revision scheduler dressed in the traditional sabaq / sabqi / manzil workflow, with no AI, no audio recognition, no backend, no accounts, and no telemetry. These documents turn the product spec ([PRD.md](../PRD.md)) and the deep research ([research/RESEARCH-FINDINGS.md](../../research/RESEARCH-FINDINGS.md)) into an implementable, auditable engineering specification — for a solo, AI-assisted build, and for the privacy-literate huffaz, teachers, scholars, and auditors who will read this repository openly on GitHub. Every technical decision below is final and recorded in the [tech-decision log](#tech-decision-log); individual docs apply decisions by referencing them as "Decision log: *topic*" and never re-litigate them. The standard is *iḥsān*: because the work is for Allah, "good enough because it's free" is not the bar — correctness is ([PRD §2](../PRD.md)).

## Non-negotiable engineering values

From [PRD §4, §19](../PRD.md) and the blueprint's engineering pillars, in priority order. Every doc in this set serves at least one of them; each is backed by its single strongest piece of evidence.

1. **Quran text fidelity is existential.** A single wrong diacritic or broken ligature ends the project, so the muṣḥaf is rendered only through bundled **KFGQPC per-page glyph fonts** — never the OS text shaper, the layer where broken ligatures and dropped/duplicated diacritics actually occur. Flutter's own Arabic shaping has shipped font-, platform-, and backend-dependent misplaced-diacritic and broken-ligature defects ([flutter/flutter #143975](https://github.com/flutter/flutter/issues/143975)), which is exactly why the whole serious Quran-tech ecosystem (Quran.com, Tarteel/QUL) renders pre-shaped per-page glyphs ([QUL: KFGQPC font](https://qul.tarteel.ai/resources/font/249)) and we draw glyph codes rather than shape characters. Enforced by [08-quran-data-and-immutable-rendering.md](08-quran-data-and-immutable-rendering.md) and [09-asset-packs-and-offline-integrity.md](09-asset-packs-and-offline-integrity.md).

2. **A deterministic, testable scheduling engine.** The scheduler is a pure-Dart package with zero I/O and an injected "today," so identical inputs yield an identical schedule ([PRD §7.12](../PRD.md)) and the FSRS-style math is golden- and property-testable. A pure-Dart package "contains pure business logic with no Flutter or framework dependencies, making it the easiest layer to test" and runs on `dart test` without a widget binding ([Flutter: Developing packages & plugins](https://docs.flutter.dev/packages-and-plugins/developing-packages)). Specified in [06-scheduling-engine.md](06-scheduling-engine.md) and [11-testing-strategy.md](11-testing-strategy.md).

3. **Fully offline; no backend, no telemetry.** The core muṣḥaf is **bundled in the signed app binary**, so the app is fully usable offline from first launch with **no network event in its critical path**; the only permitted network client is the **optional-pack downloader** (reciter audio, future alt-muṣḥaf), and after any such optional fetch the app runs in airplane mode forever ([PRD §17](../PRD.md)). The official Flutter offline-first pattern — "repositories combine different local and remote data sources … independently of the connectivity state" ([Flutter: Offline-first support](https://docs.flutter.dev/app-architecture/design-patterns/offline-first)) — degenerates cleanly to local-only when the remote branch is deleted, and CI bans every networking import outside the single optional-pack-downloader module. Specified in [01-architecture-overview.md](01-architecture-overview.md) and [09-asset-packs-and-offline-integrity.md](09-asset-packs-and-offline-integrity.md).

4. **Date and calendar correctness.** Dart has no date-only type — `DateTime` is always a proleptic-Gregorian instant and "does not provide internationalization" ([Dart: DateTime class](https://api.dart.dev/dart-core/DateTime-class.html)) — so the app imposes its own `CalendarDate` value type, keeps all interval/cycle math in Gregorian-serial integer days, and treats Hijri / Jalālī / Gregorian as display-only. This forecloses the DST "+1 day ≠ +24h" off-by-one class that returns a wrong "next due" date. Specified in [07-dates-calendars-and-correctness.md](07-dates-calendars-and-correctness.md).

5. **Crash-safe local persistence.** No teacher sign-off — a *sanad* act — is acknowledged before it is durably committed; every review is one transaction over a SQLite WAL store, which keeps the main database untouched until checkpoint and is robust to power loss ([SQLite: Write-Ahead Logging](https://sqlite.org/wal.html)). Specified in [05-persistence-and-encryption.md](05-persistence-and-encryption.md).

6. **No AI — local arithmetic only.** There is no ML model, no server inference, and no microphone; grading is human (self or on-device teacher), and the engine is the ~30 lines of published, MIT-licensed FSRS arithmetic vendored directly with no SDK and no runtime dependency ([open-spaced-repetition: FSRS](https://github.com/open-spaced-repetition/free-spaced-repetition-scheduler)). Specified in [06-scheduling-engine.md](06-scheduling-engine.md).

7. **Open-source and auditable.** Asset integrity is provable by pinned SHA-256 over an immutable, signed release channel, and the code is copyleft so the *waqf* intent is encoded in law; F-Droid's FLOSS-everything and banned-SDK rules ([F-Droid: Inclusion Policy](https://f-droid.org/en/docs/Inclusion_Policy/)) are verbatim the PRD's own offline/no-telemetry constraints, making auditability a build invariant, not a promise. Specified in [09-asset-packs-and-offline-integrity.md](09-asset-packs-and-offline-integrity.md) and [13-oss-repo-and-release.md](13-oss-repo-and-release.md).

## Rules that outrank everything

Two rules supersede every other decision in this set. When any guidance below conflicts with these, these win.

1. **The sacred text is never put at risk for any feature, performance win, or convenience.** Quran bytes are stored verbatim and checksummed; rendering goes through immutable glyph fonts only; layout comes from a fixed dataset, never runtime line-breaking; markers are overlays of coordinates, never re-typeset text ([PRD R1](../PRD.md)). No optimization, dependency, or shortcut that could alter a glyph, a diacritic, or a line break ships — ever. This is release-blocking and not subject to trade-off.

2. **The engine may only make a page *more* frequent, never less — and the app never says a page is "safe to drop."** The trust clamp `due_at = min(ideal_due, ceiling_due)` guarantees every page is re-recited at least once per chosen cycle no matter what the math computes ([PRD §7.6, §7.12](../PRD.md)). "Nothing decays silently" is the product's covenant with the ḥāfiẓ; any feature that would violate it is wrong by definition.

## Tech-decision log

The canonical, final decision log. Full statements (including alternatives considered) are applied and expanded in the linked docs; this table is the authoritative summary. Docs cite entries as "Decision log: *topic*".

|---|---|---|---|

## Citation convention

This set follows the citation rules in the [documentation blueprint](../_DOC-SET-BLUEPRINT.md) §2 exactly. Every significant claim carries an inline citation; engineering sources are official docs (Flutter/Dart/SQLite/Drift/Unicode/FSRS), standards (WCAG, Unicode UAX, NIST), and peer-reviewed SE evidence where a practice is claimed to improve quality (§3). **Citations are real and web-verified** — never fabricated; every URL in this set was confirmed to resolve during research, and unsourced "best practices" are rewritten as explicit assumptions or removed. Each numbered doc ends with a `## References` section listing only the sources it cites; [REFERENCES.md](REFERENCES.md) is the deduplicated master.

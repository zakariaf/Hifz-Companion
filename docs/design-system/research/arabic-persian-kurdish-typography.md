# Arabic-Script UI Typography — Persian, Arabic, and Kurdish-Sorani

**Topic:** Choosing and setting type for the app *chrome* (UI text — not the sacred Quran page): legibility of the Perso-Arabic script across Persian (fa), Arabic (ar), and Kurdish-Sorani (ckb); selecting a bundled UI font with verified Sorani coverage (Vazirmatn / Estedad / Rabar / NRT); sizing, line-height, and bidi setting; and keeping the UI typeface unmistakably distinct from the immutable QPC muṣḥaf type.
**Compiled:** 2026-06-16

This note is the evidence dossier behind `design-system/04-typography.md` and a sibling of `design-system/research/material3-flutter-design.md` and `accessibility-rtl-inclusive.md`. It deliberately draws a hard line: everything here concerns **localized interface text** (buttons, labels, dates, the heat-map legend, settings). The Quran itself is **never** typeset by these fonts or by the OS text shaper — it is rendered glyph-only from the bundled KFGQPC per-page fonts (PRD §11.2, R1). Theme §11 below makes that boundary explicit because conflating the two is the single most dangerous typographic mistake this app could make.

---

## What the evidence says

### 1. The three UI languages share one script but Sorani extends the Arabic letter set — a font is only viable if it covers *every* Sorani codepoint

Persian (fa), Arabic (ar), and Kurdish-Sorani (ckb) all render right-to-left in the cursive Perso-Arabic script, but they are **not** the same character repertoire. Standard Arabic is an abjad (short vowels optional); **Central Kurdish written in Arabic script is almost a true alphabet "in which vowels are given the same treatment as consonants"** and vowels are mandatory ([Wikipedia: Kurdish alphabets](https://en.wikipedia.org/wiki/Kurdish_alphabets); [Wikipedia: Kurdish typography](https://en.wikipedia.org/wiki/Kurdish_typography)). Sorani uses a **33–34 letter** Arabic-based alphabet that adds letters absent from core Arabic. The codepoints a UI font **must** carry for Sorani, beyond the standard Arabic set, are ([r12a/W3C: Sorani orthography notes](https://r12a.github.io/scripts/arab/ckb.html); [Wikipedia: Kurdish alphabets](https://en.wikipedia.org/wiki/Kurdish_alphabets)):

| Letter | Unicode | Role in Sorani |
|---|---|---|
| پ | U+067E | p |
| چ | U+0686 | ch |
| ژ | U+0698 | zh |
| ڤ | U+06A4 | v |
| ک | U+06A9 | k (Kurdish kaf, **not** U+0643) |
| گ | U+06AF | g |
| ڕ | U+0695 | strong/trilled r |
| ڵ | U+06B5 | velarized l |
| ۆ | U+06C6 | o (vowel) |
| ێ | U+06CE | ē (vowel) |
| ە | U+06D5 | æ/ə (vowel — ARABIC LETTER AE) |
| ھ | U+06BE | h (HEH DOACHASHMEE) |

These match the extra letters the PRD already flagged for ckb (پ چ ژ گ ڤ ڕ ڵ ۆ ێ ە — PRD §13.1). Two of them are notorious for breaking otherwise-good Arabic fonts: **U+06D5 (ە, AE)** is the dedicated Sorani vowel that the Kurdistan Region's standardization mandates *instead of* a heh+ZWNJ hack, and **U+06A9 (ک)** is the Kurdish kaf that the Kurdistan Region standardizes in place of the standard-Arabic kaf U+0643 ([r12a/W3C: Sorani notes](https://r12a.github.io/scripts/arab/ckb.html); [Wikipedia: Kurdish alphabets](https://en.wikipedia.org/wiki/Kurdish_alphabets)). A font that renders Persian and Arabic flawlessly can still drop `ە`, `ڕ`, `ڵ`, or `ۆ` to tofu (□) — so coverage must be **verified glyph-by-glyph**, not assumed from "supports Arabic."

### 2. Sorani's all-letters-mandatory orthography means encoding hygiene, not just glyph presence, must be tested

Because Sorani writes vowels as full letters, real-world Sorani text in the wild is full of encoding noise that a naïve renderer will mangle. The Kurdish typography record documents that **"Zero width non-joiner (U+200C) usage is non-standard but occurs a lot, most of the time this is due to poor conversions from non-Unicode to Unicode mapping,"** and that bad converters substitute **Teh Marbuta (ة) where the AE letter (ە) belongs** ([Wikipedia: Kurdish typography](https://en.wikipedia.org/wiki/Kurdish_typography)). The Sorani orthography notes add a second ambiguity: the *h* sound is encoded inconsistently as either U+06BE (HEH DOACHASHMEE) or U+0647 (ARABIC HEH), and the æ vowel inconsistently as U+06D5 vs. a U+0647+U+200C "hack" the community considers improper ([r12a/W3C: Sorani notes](https://r12a.github.io/scripts/arab/ckb.html)). For our app this is a **localization-data** discipline as much as a font one: the ckb ARB strings we author must be canonically encoded (U+06D5 for AE, U+06A9 for kaf, U+06BE for the chosen heh), and a CI check should reject stray ZWNJ and Teh-Marbuta-for-AE substitutions so the bundled font is never asked to render malformed input.

### 3. Vazirmatn covers Sorani in recent versions, is OFL-licensed, variable, and the strongest default — but coverage must be pinned to a verified version

Vazirmatn is a modern, low-contrast Perso-Arabic sans by Saber Rastikerdar, **9 weights with a true variable axis (wght 100–900)**, licensed **SIL Open Font License 1.1** ([GitHub: rastikerdar/vazirmatn](https://github.com/rastikerdar/vazirmatn); [Estedad/Vazirmatn specimens via Google Fonts](https://fonts.google.com/specimen/Vazirmatn)). Its language coverage has **expanded over time**: from roughly v27 onward Vazirmatn is described as supporting "Persian, Arabic, Iranian Azerbaijani, **Kurdish**, Pashto, Urdu, Gilaki, Uzbek, Kazakh and Balochi" ([Wikipedia: Central Kurdish / Vazirmatn coverage notes](https://en.wikipedia.org/wiki/Central_Kurdish)). The caution: older packaging and some mirrors still describe it only as "a Persian/Arabic font" with **no** Kurdish mention (e.g. [FreshPorts: vazirmatn](https://www.freshports.org/x11-fonts/vazirmatn/), pkg v33 description), and the repo README itself does not enumerate Kurdish codepoints ([GitHub: rastikerdar/vazirmatn](https://github.com/rastikerdar/vazirmatn)). The evidence-honest position: Vazirmatn is the right **default** for fa/ar and is a credible single-family choice for ckb, but the specific bundled version must be tested to render all twelve Sorani-specific codepoints in §1 (especially ە, ڕ, ڵ, ۆ, ێ) before it is locked — exactly the "verify all Sorani letters render before locking" gate the PRD already requires (§13.5).

### 4. Estedad is a viable OFL alternative; dedicated Kurdish families (Rabar, NRT) exist but carry licensing and style caveats

**Estedad** (Amin Abedi) is an Arabic–Latin sans, **9 weights plus a variable wght axis, OFL 1.1**, that "supports a wide range of Arabic and Latin codepoints" and has had explicit fixes to Kurdish glyphs (e.g. the HEH DOACHASHMEE isolated form, U+FBAA) ([GitHub: aminabedi68/Estedad](https://github.com/aminabedi68/Estedad); [GitHub: Estedad issue #17 / releases](https://github.com/aminabedi68/Estedad/issues/17)). It is a strong second choice and a sensible fallback face. The **dedicated Kurdish families** — **Rabar** and **NRT** — are the fonts Kurdish newspapers, education, and signage actually use ([Wikipedia: Kurdish typography](https://en.wikipedia.org/wiki/Kurdish_typography); [Kurdfonts: Rabar](https://www.kurdfonts.com/browse/rabar-kurdish-unicode-fonts)). Their advantage is native, idiomatic Sorani letterforms; their risk is **license clarity** — many Kurdish font packages are distributed as free downloads without a clean, embeddable OSS license, which is disqualifying for a free/auditable app that must ship fonts in its binary (PRD §13.5, §21.6). Recommendation hierarchy for this project: **Vazirmatn (verified) → Estedad (verified) → a Rabar/NRT-class font only if a clear OFL-equivalent license is confirmed.** Bundling, never CDN fetch, is mandatory regardless (PRD §13.5, §19.1).

### 5. For continuous UI reading the Naskh-derived structure is the legible baseline; geometric "Kufi" display faces are for headings only

Naskh is the script "commonly used in writing administrative documents and for transcribing books, including the Qur'an, **because of its easy legibility**," and it became "the basis for most types intended for continuous reading" precisely because it "differentiates various sounds through the use of diacritical points… which makes the script more easily legible" ([Wikipedia: Naskh (script)](https://en.wikipedia.org/wiki/Naskh_(script)); [Wikipedia: Arabic typography](https://en.wikipedia.org/wiki/Arabic_typography)). The practical takeaway for UI is not "use a literal Naskh book face" but "use a face whose **skeleton follows the Naskh proportions** the eye expects." Modern screen sans like Vazirmatn and Estedad are low-contrast humanist faces built on that familiar skeleton; decorative or strongly geometric Kufi-style display faces sacrifice the dot-and-tooth distinctions that carry legibility and should be reserved for short headings, if used at all ([Code Guru: best Arabic script for the web](https://codeguru.ae/blog/fonts-and-readability-best-arabic-script-for-the-web/)). This dovetails with the calm, non-decorative design pillar (PRD §1; blueprint §1 pillar 2).

### 6. Familiarity drives legibility — so the UI face should be the script readers already meet everywhere, and the muṣḥaf face should look traditional

An eye-tracking study by Nedeljković, Jovančić & Pušnik (2020, *Journal of Eye Movement Research* 13(2)) supports Zuzana Licko's maxim **"you read best what you read most,"** finding that legibility rests on a long-familiar letterform skeleton and that readers adapt to unfamiliar forms only with exposure ([Nedeljković et al., 2020, JEMR 13(2)](https://pmc.ncbi.nlm.nih.gov/articles/PMC7963459/)). Two design consequences follow. (a) The **UI** face should be a conventional, widely-seen screen Naskh-sans (Vazirmatn/Estedad) so first-time users read it effortlessly. (b) The **muṣḥaf** must keep its *traditional* QPC letterforms — huffaz have read that exact Madani page for years; "you read best what you read most" means the page's familiar shape is itself a memory cue, and re-typesetting it in a modern UI font would not only violate fidelity (R1) but actively *hurt* recall. This is the strongest evidence-based argument for the UI/sacred type split.

### 7. Size: Arabic-script UI body text should sit at ~16 px and up; do not shrink it to Latin "small" sizes

Field data on Arabic web text converges on **16 px (≈12 pt) as the floor for main content**, with 18 px for long-form and ~20 px on mobile; one summary reports readers "process Arabic text 20% faster when displayed at 16 points or larger," and a survey of Saudi Arabia's top-100 sites found 65% use only 12–16 px for body — below the recommended floor for readers with low vision or on mobile ([Code Guru: best Arabic script for the web](https://codeguru.ae/blog/fonts-and-readability-best-arabic-script-for-the-web/)). The mechanism is script-specific: Arabic letters are distinguished by **dots (1–3, above or below)** and small tooth shapes that collapse at small sizes ([Wikipedia: Naskh (script)](https://en.wikipedia.org/wiki/Naskh_(script))). Persian and Sorani add even more sub- and super-script marks (the small-v of ڕ/ڵ/ڤ/ێ, the vowel letters), so an Arabic-script UI cannot safely use the 12–14 px "secondary text" sizes common in Latin Material layouts without losing those distinguishing marks. In practice: set Material body sizes at the larger end and never go below ~14 sp for any readable label.

### 8. Line-height must be generous — Arabic ascenders/descenders and stacked marks need more leading than Latin

The W3C Arabic & Persian Layout Requirements note plainly that **"Arabic ascenders and descenders extend much further than those of the Latin script,"** which forces care with vertical spacing and mixed-script alignment ([W3C: Arabic & Persian Layout Requirements](https://www.w3.org/International/alreq/)); the gap-analysis adds that **"Arabic fonts require underlines positioned further from the baseline than Latin text due to longer descenders"** ([W3C: Arabic Script Gap Analysis](https://www.w3.org/TR/alreq-gap/)). a line-height of **1.5×** is the accessibility floor for body text (and the WCAG 2.2 text-spacing minimum), with the UAE government RTL design system applying that same ≥1.5 ratio to Arabic body content ([greadme: best font sizes for readability](https://www.greadme.com/blog/seo/best-font-sizes-for-readability-complete-guide); [UAE Design System: Typography](https://designsystem.gov.ae/guidelines/typography)). For Sorani the case is stronger still, because its mandatory vowel letters and small-v diacritics add vertical material on nearly every word. The conclusion: target **line-height ≈ 1.5–1.6** for Arabic-script body UI — looser than the ~1.2–1.4 a Latin layout might use — and verify that the chosen value never clips marks at the top of a line or descenders at the bottom.

### 9. Justification and letter-spacing rules are inverted vs. Latin: never letter-space joined script, and avoid kashida in UI

Because the script is cursive, **adjusting spacing between *joined* letters "creates undesirable results"** — spacing may only be added at non-joining gaps ([W3C: Arabic & Persian Layout Requirements](https://www.w3.org/International/alreq/)). Arabic justification traditionally uses kashida/tatweel (extending the connecting stroke, U+0640) and alternate letter widths rather than inter-word stretching, but **"excessive use of kashida or applying very long kashidas results in uneven color"** and CSS/most UI toolkits don't implement kashida well ([W3C: Arabic & Persian Layout Requirements](https://www.w3.org/International/alreq/); [W3C: Arabic Script Gap Analysis](https://www.w3.org/TR/alreq-gap/)). For our UI the safe rules are: **no `letterSpacing` on Arabic-script runs; left-justify off; use start-aligned (RTL "start" = right) ragged text**; never insert tatweel programmatically to pad labels.

### 10. Flutter has real Arabic-diacritic and line-height pitfalls that must be tested, not assumed

Flutter's text stack has documented Arabic-script bugs the project must guard against: **Arabic diacritics misbehave (clip/fail to stack) when text is split across `TextSpan`s** ([flutter/flutter #73108](https://github.com/flutter/flutter/issues/73108)); **trailing diacritics can wrap to a new line** in some versions ([flutter/flutter #105025](https://github.com/flutter/flutter/issues/105025)); and the `TextStyle.height` parameter **does not guarantee an exact line box for every script** — Arabic intervals can be unpredictably enlarged ([flutter/flutter #23875](https://github.com/flutter/flutter/issues/23875)). Mitigations: keep a localized label in a **single `Text`/`TextSpan`** (don't fragment a word to style part of it), set an explicit `TextStyle.height` and verify it per-locale with golden screenshots (already a CI gate — PRD §20.5), and let Flutter shape the **UI** text normally (this is fine — the OS-shaper ban is *only* for Quran glyphs, theme §11). Bidi isolation for embedded Latin/numbers uses Unicode FSI/PDI via Flutter's `Bidi`/`Directionality` (PRD §13.2).

### 11. The UI typeface must be visibly, deliberately different from the QPC muṣḥaf type — the most important distinction in the whole system

The Quran is **not** rendered with a UI font or the OS shaper. KFGQPC/QPC muṣḥaf fonts are **glyph-based**: "each glyph represents an entire word on the page," fonts are "designed on a page-by-page basis, with each font corresponding to a specific page," and **"most Muṣḥaf has 604 pages, so you'll need 604 font files to render the whole Quran"** ([QUL: Glyph-Based Fonts](https://qul.tarteel.ai/docs/glyph-based)). These fonts map pre-shaped words into the **Private Use Area**, which is exactly why they render correctly *without* OS-level shaping — the typeset page *is* the font ([QUL: Glyph-Based Fonts](https://qul.tarteel.ai/docs/glyph-based); [GitHub: nuqayah/qpc-fonts](https://github.com/nuqayah/qpc-fonts)). Two implications: (a) **Technically**, the muṣḥaf and the UI live in separate rendering pipelines — there is no shared `TextStyle`, and the UI font's metrics are irrelevant to the page; the app must never feed Quran text through the UI shaper (R1, the layer where ligatures break and diacritics drop). (b) **Aesthetically and for adab**, the two should look *unmistakably* different so a user never confuses chrome with scripture: the muṣḥaf keeps its traditional, high-contrast Naskh QPC letterforms; the UI uses a calm, lower-contrast modern sans (Vazirmatn/Estedad). Finding 6's "you read best what you read most" reinforces both directions of the split — the page must look traditional, the chrome must look like ordinary modern script. Sacred type is reverent and fixed; UI type is quiet and serves it.

## Implications for Hifz Companion

1. **Pin one verified UI family: Vazirmatn (OFL 1.1), Estedad as fallback.** Bundle in the binary, never `google_fonts`/CDN fetch (PRD §13.5, §19.1). Both are variable OFL faces, so we ship a small set of weights from one file.
2. **Gate Sorani coverage in CI.** Before locking the font version, programmatically assert that the bundled face renders all twelve Sorani-specific codepoints from theme §1 ( پ چ ژ ڤ ک گ ڕ ڵ ۆ ێ ە ھ) with no tofu, on both iOS and Android — this operationalizes the PRD's "verify all Sorani letters render before locking" (§13.5) and the RTL golden-screenshot gate (§20.5).
3. **Author ckb ARB strings with canonical encoding.** Use U+06D5 (ە) for AE, U+06A9 (ک) for kaf, the chosen heh (U+06BE), and **reject** stray U+200C and Teh-Marbuta-for-AE in a lint step (theme §2). Pair with native-speaker + scholarly term review still open in PRD §21.1.
4. **Set Arabic-script type larger and looser than a Latin layout would.** Body UI ≥ 16 sp (never below ~14 sp for readable labels); **line-height token ≈ 1.5–1.6**. Define this once as `type.lineHeight.arabic` and apply app-wide (themes §7, §8). Respect OS text-scale via `sp`/`textScaler` (PRD §18).
5. **No `letterSpacing` on Arabic-script runs; no programmatic kashida/tatweel; start-aligned ragged text.** Encode this as a rule in `04-typography.md` and the component theme defaults (theme §9).
6. **Keep each localized label in a single `Text`/`TextSpan`.** Don't fragment a word to style part of it (avoids the diacritic-clipping and wrap bugs, theme §10). Set explicit `TextStyle.height` and verify per-locale with golden screenshots.
7. **Treat UI fonts and the muṣḥaf as two separate pipelines with no shared style.** The UI font's metrics never touch the Quran; Quran text is glyph-only via the 604 QPC PUA fonts and never sees the OS shaper (theme §11; PRD §11.2, R1).
8. **Make the UI face visibly distinct from the QPC type as an adab requirement.** A calm modern sans for chrome; the traditional QPC Naskh for the page — so scripture is never visually mistaken for interface (theme §6, §11; design pillar "reverence first").
9. **Reserve any geometric/Kufi-style face for short headings only**, if used at all; body and label text stay on the legible Naskh-skeleton sans (theme §5).
10. **Headline tokens to define** in `04-typography.md`: `type.family.ui` (Vazirmatn), `type.family.uiFallback` (Estedad), `type.size.body` (≥16 sp), `type.size.label` (≥14 sp), `type.lineHeight.arabic` (1.5–1.6), `type.letterSpacing.arabic` (0). The muṣḥaf is **not** a `type.*` token — it is an asset-pack glyph font owned by the `/quran` module (PRD §19.2), and `04-typography.md` must say so explicitly.

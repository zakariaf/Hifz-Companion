# eng-add-persisted-model

Persisting a new domain fact in Hifz Companion means writing across a fixed chain of layers, each owned by one package, with the Drift/SQLite dependency held inside the `data` package alone. The chain is: an immutable value type in `models` (`Card`, `ReviewLog`, `Profile`, `ConfusionEdge`, `CycleConfig`, `LineBlock`) → its **Drift table class** with `STRICT`, `CHECK`, foreign-key, and index invariants encoded in the schema → a **DAO** in `data` that maps rows to those plain value types → the **repository transaction body** that opens exactly one `db.transaction` and commits before any state is observable. The auditable property this protects: **no Drift symbol crosses the `data` boundary** — the engine, features, and quran packages see only `models` value types, which is the same boundary that keeps the scheduling engine deterministic and golden-testable.

This skill governs the model + schema + DAO + repository-transaction surface, and the guided Drift migration that backs new columns. It is almost always paired with `eng-create-riverpod-store`: that skill owns the controller/`Notifier`/`StreamProvider` that *calls* the repository method this skill defines, while this skill owns the table, the DAO, and the transaction body itself.

## When to use

Use this skill when you:

- introduce a brand-new persisted domain record — a new `models` value type that needs a Drift table and round-trips through SQLite;
- add a field/column to an existing record and its mapping (pair it with the guided Drift migration in step 9);
- add a DAO read or write method, or the repository transaction body that backs it, for a new or existing record;
- evolve the schema: a `schemaVersion` bump, a committed JSON schema snapshot, a generated `stepByStep` migration, and its fixture test;
- add the append-only audit row for a review/teacher sign-off (the `review_log` insert inside `commitReview`).

Do **NOT** use this skill for:

- the **provider/`Notifier`/`StreamProvider`/composition-root wiring** and the controller's persist-then-republish command that *calls* the repository → use **eng-create-riverpod-store**.
- the **pure scheduling arithmetic** that produces the new `Card` state (`onReview`, the trust clamp `dueAt = min(ideal, ceiling)`, tracks, intervals, `R`/`S`/`D` math) — that is the engine; a DAO or repository that does interval math reads the wrong layer → use **domain-scheduling-engine-rules**.
- the **`pubspec.yaml`, `resolution: workspace` member, and the engine-purity / no-network / quran-isolation banned-import lints** that contain the `data` package → use **eng-create-package**.
- the **`CalendarDate` value type, the injected "today" clock, Hijri/Jalālī/Gregorian semantics, "elapsed days"** that the stored `due_at`/`last_review_at` serial-day integers *mean* → use **domain-calendars-and-hifzdate**.
- the **`.hifzbackup` export/import, the WAL-checkpointed snapshot, the set-union merge over `review_log`, one-tap erase** — the only sanctioned bulk touch of `review_log` → use **domain-backup-format**.
- normalizing a **recitation verdict into a `(grade, errorLines, source)` signal** (reveal-on-tap, the sacred-text guard, source-confidence) before it becomes a `review_log` row → use **domain-grading-pipeline**.
- the **read-only Quran reference tables** (`page`, `line`, `ayah`, `surah`, `mushaf`) and their checksum governance — these are never written at runtime and no DAO exposes a mutation → use **domain-mushaf-text-integrity** (and `domain-mutashabihat-system` for the scholar-reviewed `mutashabih_*` tables).

A `models` value type with no Drift import, a table with its invariants in the schema, a DAO confined to `data`, and one transaction per write: that is the whole shape. A view that touches a DAO, a stored `DateTime` for a scheduling day, an `UPDATE` on `review_log`, or a persisted juz-health roll-up is the wrong layer.

## The canonical pattern

1. **The value type lives in the pure `models` package, immutable, with no Drift import.** Records are immutable value types (`final` fields, `const` constructor where possible, `copyWith` for derivation; `freezed` optional) — `Card`, `ReviewLog`, `Profile`, `ConfusionEdge`, `CycleConfig`, `LineBlock`. `models` imports `dart:core`/`package:meta` only and is the bottom of the graph; a `package:drift` or `package:flutter` import in `models` is a compile error, not a style nit. `docs/engineering/01-architecture-overview.md` §2 (Layer 0 — Models: immutable value types, `dart:core`/`package:meta` only) and §3.1 (allowed-imports matrix); `docs/engineering/03-coding-standards.md` §5.1 (immutable domain types: `final`, `const`, `copyWith`).

2. **Names carry units and calendar semantics; closed sets are enums.** Full dictionary words, no abbreviations: `stabilityDays` not `s`, `cycleCeilingDays`, `dailyBudgetMinutes`, `targetRetention`. A scheduling day is a `CalendarDate` named as a day (`dueAt`, `lastReviewedDay`); a real instant is a `DateTime` named as one (`reviewedAtInstant`, `reminderFireInstant`) and is legal only at the notification/logging boundary — a `DateTime` named like a day is a review-blocking defect (the DST off-by-one class). Closed value sets (`track`, `grade`, `source`, `role`, `revelation`) are enums so invalid states are unrepresentable; booleans read as assertions (`isWeak`, `isManualLock`, `hasTeacherSignoff`). `docs/engineering/03-coding-standards.md` §1.1 (naming rules 1–5: full words, units in the name, `CalendarDate` vs `DateTime` marked by name *and* type, sacred-domain transliteration `mushaf`/`juz`/`hizb`/`surah`/`ayah`, booleans as assertions) and §5.1 (closed sets unrepresentable-invalid).

3. **The Drift table class lives in `data` only, with invariants in the schema — `STRICT`, `CHECK`, foreign keys, indices.** The table is authored as a Drift table class so `STRICT` typing, `CHECK (... IN (...))` for enumerable columns, range `CHECK`s (`d BETWEEN 1 AND 10`, `s >= 0`), `REFERENCES`/`ON DELETE CASCADE` foreign keys, and the `buildToday()` scan index (`card_due ON card(profile_id, track, due_at)`) are enforced by SQLite, not by application code — a corrupt row is unrepresentable at the storage layer. The schema is the PRD §10 data model split into read-only reference tables (Quran structure, scholar-reviewed mutashābihāt) and read-write user tables. `docs/engineering/05-persistence-and-encryption.md` §2 (Schema: `STRICT` tables, `CHECK` on enumerable columns and ranges, foreign keys, the load-bearing card invariant `CHECK (track = 'UNMEMORIZED' OR due_at IS NOT NULL)`, indices) and §1 (the `@DriftDatabase` table list; `PRAGMA foreign_keys = ON` re-asserted per connection); `docs/engineering/01-architecture-overview.md` §3.1 (`data` may import `package:drift`; the engine and features may not).

4. **No Drift symbol crosses the `data` boundary; the DAO maps rows to plain value types.** Drift and `package:sqlite3` are confined to `data` by a CI banned-import gate — exactly the mechanism that bans networking imports. The DAO maps rows to `models` value types; the engine, features, quran, and profiles packages never see a Drift `Companion`, `TableInfo`, or generated row class. A feature reaching for a raw query is a layering break that defeats the determinism boundary. `docs/engineering/05-persistence-and-encryption.md` §1 (Pitfalls: "We refuse a Drift import outside `/data`" — the `import_rules`/DCM banned-import gate); `docs/engineering/03-coding-standards.md` §6 (the package graph is the audit artifact; `data` imports Drift/SQLite, no networking) §7.2 (path-scoped import bans); `docs/engineering/01-architecture-overview.md` §3.1 (`engine`/`features` import `data` repositories, never a DAO's Drift types).

5. **Scheduling days are `CalendarDate` serial-day integers; true instants are UTC; no `REAL` for exact arithmetic.** `due_at` and `last_review_at` are stored as the `CalendarDate` serial-day **integer** owned by `domain-calendars-and-hifzdate`, never a `DateTime` — this forecloses the DST off-by-one that produces a wrong "next due". Wall-clock event times (`reviewed_at`, `created_at`, `last_confused_at`) are stored UTC ISO-8601. `docs/engineering/05-persistence-and-encryption.md` §2 (Specification: instants UTC, scheduling days as `CalendarDate` serial integers; Pitfalls: "We refuse to store `DateTime` instants as scheduling days") and the schema's `INTEGER` `due_at`/`last_review_at` columns; `docs/engineering/03-coding-standards.md` §1.1 rule 3 (the `CalendarDate`/`DateTime` boundary).

6. **The `review_log` is append-only — no `UPDATE`/`DELETE` DAO method exists.** The audit trail and the *sanad* record of teacher sign-offs is enforced by the **absence** of update/delete methods on its generated DAO, reviewed in CI. The only sanctioned bulk touch is export / one-tap erase (`domain-backup-format`). A lost write, a half-applied review, or a silently mutated log entry is a breach of the covenant "nothing decays silently", not a bug. `docs/engineering/05-persistence-and-encryption.md` §2 (Rationale: `review_log` append-only, "never updated or deleted by normal flows (export/erase only)"; Pitfalls: "We refuse any `UPDATE` or `DELETE` DAO method on `review_log`") and §3 (`commitReview` appends the immutable audit row first).

7. **Read/write is reached only through a repository method that opens ONE `db.transaction` and commits before republishing.** Every mutation a feature can trigger is one transaction — `commitReview` appends the `review_log` row, upserts the `card` (D/S/`due_at`/flags), may insert `line_block` rows, and may bump `confusion_edge` weights, all-or-nothing with auto-rollback on throw. The store runs in WAL with `synchronous = FULL` on the write connection; **every query inside the transaction is `await`-ed** (a missing `await` is a release-blocking review item — Drift closes the transaction and a later query corrupts data). When the write `Future` resolves, the change is durably on disk; the controller republishes only *after*. `docs/engineering/05-persistence-and-encryption.md` §3 (Crash safety: one `db.transaction` per review, WAL + `synchronous=FULL`, the canonical `commitReview` body, the `await` footgun, "We refuse persist-after-publish") and the cold-start `seedColdStart` outer transaction; `docs/engineering/01-architecture-overview.md` §4 (unidirectional flow — persist step 5 *before* republish step 6; "We refuse 'republish then persist'").

8. **Derived state is never persisted as a second source of truth; reference tables are never written.** Per-juz / per-ḥizb / per-page health roll-ups are **computed from `card` retrievability on read** (the min-leaning aggregate), never stored — one authority, no drift. The read-only reference tables (`page`, `line`, `ayah`, `surah`, `mushaf`, `mutashabih_group`, `mutashabih_member`) come from the bundled, checksum-verified asset pack; no DAO exposes a write to them. `*_json` columns (`settings_json`, `error_lines_json`, `ayah_refs_json`) carry small, schema-shaped, decode-validated data only — never health facts and never reconstructed Quran text. `docs/engineering/05-persistence-and-encryption.md` §2 (Rationale: reference data read-only because the Quran is immutable; "No derived state is stored as a second source of truth"; Pitfalls: "We refuse to persist derived health roll-ups", "We refuse to widen the `*_json` columns into health or Quran facts").

9. **Pair the change with a guided Drift migration; a shipped migration is never edited.** A new record/field needs a `schemaVersion` bump, a committed `drift_schemas/` JSON snapshot (`dart run drift_dev schema dump`), a generated `stepByStep` `.steps.dart` migration (`dart run drift_dev make-migrations`), and a fixture test that builds a populated v(n−1) DB, migrates, and asserts content **and** `PRAGMA integrity_check`. A shipped migration is append-only — editing it rewrites history for existing installs and is user-data loss. `eraseDatabaseOnSchemaChange` is DEBUG-only and gated so it never reaches a release build. Backup restore does **not** replay SQL migrations — the versioned payload is mapped to the current schema by the import pipeline. `docs/engineering/05-persistence-and-encryption.md` §4 (Migrations: `stepByStep`, committed schema snapshots, the CI-gate table, "A shipped migration is never edited", the fixture test with `integrity_check`, "We refuse `eraseDatabaseOnSchemaChange` in any shipping build").

10. **The engine is total and stays out of `data`; throwing is confined to this I/O boundary, typed, never swallowed.** The pure `engine/` never imports Drift and never throws — it returns a value for every input (a clamped interval, a low-confidence flag), so the §7.12 "identical inputs → identical schedule" guarantee stays golden-testable. The `data` package is one of the few I/O boundaries that *may* throw: define one sealed error type, surface it to the feature layer to handle exhaustively, type every `catch` (`on … catch (e)`, never bare `catch (_)`), and never swallow a write-path error or acknowledge a sign-off before the commit resolves. No `print`/`debugPrint`/logging of user data; no `!`/`late` shortcuts on persistence values. `docs/engineering/03-coding-standards.md` §5.2 (the engine is total — does not throw), §5.3–§5.4 (throwing only at I/O boundaries; one sealed error type; typed `catch`; "No swallowed errors on write paths"; a sign-off acknowledged only after durable commit), §5.5 (no `print`/logging of user data) and §6 (`engine/` imports no Drift; `data/` no networking).

11. **The store holds value types and asserts no Quran/health facts; user-facing strings live in `l10n`, RTL.** Records are value types only; any user-facing string a stored field maps to (a settings label, the sabaq/sabqi/manzil term-set, a verdict label) lives in the `l10n` package (`ar` template, `fa`/`ckb`), rendered RTL via `Directionality` with locale-appropriate numerals — never hard-coded in a record, a DAO, or a repository. No record encodes a streak/badge/score/confetti or a Quran/factual claim; the schema is sect-/madhhab-neutral. `docs/engineering/03-coding-standards.md` §1.1 rule 4 (sacred-domain transliteration; user-facing strings owned by ARB files, never hardcoded); `docs/engineering/01-architecture-overview.md` §3.1 (`l10n` owns ARB/intl). Adab/integrity non-negotiables are `domain-adab-and-religious-integrity`.

## Do / Don't

| Do | Don't |
|---|---|
| Put the value type in the pure `models` package, immutable (`final`, `const`, `copyWith`), `dart:core`/`package:meta` only | Put a record in `data`/`engine`/a feature, make it mutable, or import Drift/Flutter into `models` |
| Model closed sets (`track`, `grade`, `source`, `role`) as enums; booleans as assertions (`isWeak`) | Store a free `String` where an enum makes invalid states unrepresentable |
| Name units and semantics: `stabilityDays`, `cycleCeilingDays`, `dueAt: CalendarDate`, `reviewedAtInstant: DateTime` | `s`/`budget`/`target`, or a `DateTime` named like a scheduling day |
| Encode invariants in the schema: `STRICT`, `CHECK`, foreign keys, indices, `CHECK (track='UNMEMORIZED' OR due_at IS NOT NULL)` | Validate at every call site, use a non-`STRICT` table, or rely on app code for referential integrity |
| Keep the Drift table/DAO inside `data`; map rows to `models` value types | Let a Drift `Companion`/`TableInfo`/row class escape `data`, or query Drift from a view/engine |
| Store scheduling days as `CalendarDate` serial integers; instants as UTC | Store a `DateTime` instant for `due_at`/`last_review_at` (the DST off-by-one) |
| Append to `review_log`; expose no `UPDATE`/`DELETE` DAO on it | Add an update/delete method to the `review_log` DAO |
| Open ONE `db.transaction` per write; `await` every query in it; persist, then republish | Split a write across two transactions, miss an `await`, or republish before commit |
| Compute juz/page health from `card` on read (min-leaning) | Persist a derived health roll-up as a second source of truth |
| Type every `catch` (`on AssetIntegrityError catch (e)`); propagate write failure | Bare `catch (_) {}`, swallow a write error, `print` user data, or `!`/`late` a persistence value |
| Pair with an append-only guided Drift migration + a fixture test asserting `integrity_check` | Edit a shipped migration, skip the snapshot/test, or ship `eraseDatabaseOnSchemaChange` |

## Checklist

Before the change is done:

- [ ] Value type is in the pure `models` package, immutable (`final` fields, `const` constructor, `copyWith`), imports `dart:core`/`package:meta` only — no `package:drift`, no `package:flutter`.
- [ ] Closed value sets are enums; booleans read as assertions; names carry units and calendar semantics; scheduling days are `CalendarDate`, instants are `DateTime` named as instants (a `DateTime` named like a day is rejected).
- [ ] The Drift table class lives in `data` only; `STRICT`, `CHECK` (enum/range), foreign keys (`ON DELETE CASCADE` where a profile owns the row), and the query index are encoded in the schema; the card invariant `CHECK (track='UNMEMORIZED' OR due_at IS NOT NULL)` (or its analog) holds.
- [ ] No Drift/`sqlite3` symbol crosses the `data` boundary (banned-import gate green); the DAO maps rows to `models` value types; no engine/feature/quran package sees a Drift type.
- [ ] `due_at`/`last_review_at` are `CalendarDate` serial integers; event times (`reviewed_at`, `created_at`) are UTC; exact-arithmetic quantities are not silently coerced (`STRICT`).
- [ ] `review_log` exposes **no** `UPDATE`/`DELETE` DAO method (append-only, enforced by absence, reviewed in CI); the only bulk touch is export/erase.
- [ ] Read/write is reached through a repository method that opens exactly one `db.transaction`, `await`s every query inside it, and **commits before** the controller republishes; WAL + `synchronous=FULL`; no persist-after-publish, no debounced/"save later" write for a review or sign-off.
- [ ] Derived state (juz/page/ḥizb health, `R`) is computed from `card` on read (min-leaning), never persisted; reference tables (`page`/`line`/`ayah`/`surah`/`mushaf`/`mutashabih_*`) have no write DAO; `*_json` columns hold only small, decode-validated, non-Quran/non-health data.
- [ ] Throwing is confined to the `data` boundary, typed (one sealed error type, `on … catch`), never swallowed; a sign-off is acknowledged only after the commit resolves; no `print`/logging of user data; no `!`/`late` on persistence values; the engine stays Drift-free and total.
- [ ] Paired with a guided Drift migration: `schemaVersion` bump, committed `drift_schemas/` JSON snapshot, generated `stepByStep` `.steps.dart`, and a fixture test that migrates a populated v(n−1) DB and asserts content **and** `PRAGMA integrity_check`; the migration is append-only; `eraseDatabaseOnSchemaChange` is DEBUG-only.
- [ ] Round-trip is testable under plain `dart test`: value → row → value is identity, including the `CalendarDate`-serial and any unit conversions — see **eng-write-dart-test**.
- [ ] The record holds value types only; any user-facing string it maps to lives in `l10n` (`ar` template, `fa`/`ckb`), RTL via `Directionality`, locale numerals, sect-/madhhab-neutral; no streak/badge/score and no Quran/factual claim encoded in a record (PRD R3/C6).

This DAO + transaction body is the floor of the single write path: an optimistic republish before the WAL commit, an `UPDATE` on the append-only `review_log`, or a self-rating that reaches the top retention tier with no teacher sign-off would acknowledge a review the disk does not hold — a breach of the *sanad* covenant that "nothing decays silently." Encode the invariants in the schema so a corrupt row cannot exist; keep Drift inside `data` so the engine stays deterministic; and persist before you publish. When in doubt, add one more `CHECK` constraint rather than one more runtime validation.

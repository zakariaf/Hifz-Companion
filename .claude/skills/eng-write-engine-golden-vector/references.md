# references — eng-write-engine-golden-vector

The exact governing doc sections for this skill. Only real sections are listed; each line states the one thing to take from it.

## Primary — `docs/engineering/11-testing-strategy.md` (the vector & invariant test architecture)

- §3 (Engine golden vectors) — **Pin the FSRS arithmetic with a committed `FsrsVector(d_in, s_in, grade, elapsed, d_out, s_out, notes)` table asserted to `closeTo(_, 1e-6)`, never `==`.** Cover the curve, interval inversion, both stability branches, difficulty, the trust clamp, and the five cold-start seeds. Each row is computed once against an independent reference and frozen. Regenerate ONLY by an explicit, human-reviewed `--update-vectors`; CI only verifies — auto-bless makes the gate verify nothing. Cold-start seed table to pin exactly: Solid `D=3,S=60`→FAR, Shaky `D=5,S=14`→NEAR, Rusty `D=7,S=4`→active.
- §4 (Invariants as `glados` property tests) — **Every PRD §7.12 invariant is a `glados` property over generated `(Card, grade-sequence, today)` histories, with shrinking to the minimal counterexample.** The register: INV-1 `card.dueAt − today ≤ cycleCeilingDays(card, config)`; INV-2 every due FAR/manzil page in `buildToday`; INV-3 `Again ⇒ S'≤S ∧ track'≤track`; INV-4 two `buildToday` runs fingerprint-equal (fuzz OFF); INV-5 teacher `Again` overrides a prior self `Good`; INV-6 every memorized card has a finite non-null `dueAt` (the type forecloses "drop"/`null`). Refuse a fixed lucky seed; let `glados` explore and rely on shrinking.
- §2 (Engine with `package:test`) — **The pure engine is tested with standalone `package:test`, never `flutter_test`, never a widget binding.** `test()`/`expect()`/`group()`/`setUp()`, `_test.dart` files under `engine/test/`, run with `dart test`. `today` is constructed as a literal; nothing reads the wall clock. `DateTime.now()` is refused anywhere reachable from the engine; the day type is the integer `SerialDay`. A CI import rule fails the build if `engine/` ever imports Flutter — that is what keeps this the fastest tier.
- §1 (The test pyramid) — **These vectors and properties are the broad pure-Dart base.** Anything expressible as `(card, grade, today) → card'` is a unit/property test on `engine/`, never a `pumpWidget`. The vectors are the engine's analogue of `matchesGoldenFile`; the properties are the universal claims a point-vector cannot reach.
- §8 (CI shape → PRD gate mapping) — **Vectors and properties run in the `fast` job (`dart test engine/`), mapping to PRD §20 gate 3 (engine golden tests) and gate 4 (invariant property tests).** This is what makes "scheduler outputs match pinned fixtures (curve, intervals, trust clamp never exceeds ceiling, manzil never dropped, lapse demotes, cold-start seeds)" an auditable, machine-checked release contract.

## Primary — `docs/engineering/06-scheduling-engine.md` (what the vectors & properties assert)

- §8 (Determinism, weights, golden vectors) — **The anchor golden-vector table + the `glados` invariant list, verbatim.** Curve identity `retrievability(S,S)=0.9` (±1e-9); interval identity `interval(S,0.9)=S`; tier multipliers `interval(100,0.95)≈45`, `interval(100,0.97)≈27`; lapse clamp `postLapseStability ≤ s`. The cross-check vector pins one full review sequence against `dart-fsrs` (`enableFuzzing: false`, `kDefaultWeights45`). Refuse fixtures generated by the engine under test; refuse inlining the 19-weight vector (length-asserted `== kFsrsWeightCount`).
- §6 (The trust clamp) — **`due_at = min(ideal_due, ceiling_due)` — the single most-tested line; INV-1's subject.** SR may only pull a page *forward*. Refuse `max` where `min` belongs; refuse a ceiling that depends on anything non-deterministic; refuse ever exempting a page from the ceiling or telling a ḥāfiẓ a page is "safe to drop." Name this covenant (PRD §7.6) in the INV-1 assertion comment.
- §3 (The forgetting curve and the interval) — **`kDecay = -0.5`, `kFactor` computed from `kDecay` (= 19/81), `R(S,S)=0.9` by definition — the anchors a vector pins.** Intervals are the closed-form inversion; every interval is a pure multiple of the `r=0.90` interval. Refuse interval fuzzing (it breaks INV-4 determinism); declumping lives in the testable `loadBalance` peak-smoothing, never RNG.
- §4 (The review update: lapse vs success) — **The two stability branches + difficulty update + sacred-text guard that a vector's `d_out`/`s_out` must reproduce.** Success grows S; lapse applies `postLapseStability` (clamped `≤ S`), `D += kLapseDifficultyBump`, sets `weakFlag`. Self-rating (`kSelfConfidence = 0.5`) scales the applied S gain down; teacher = 1.0. A missed/altered word is never "Good" — caps the grade at Hard (R1).
- §5 (Phases, graduation, stakes-tiered retention, cold start) — **The exact cold-start seeds and `targetR` tiers a seed vector and INV-1 assert.** Solid `(D=3,S=60)`→FAR, Shaky `(D=5,S=14)`→NEAR, Rusty `(D=7,S=4)`→active; `targetR` New 0.90 → Near 0.94 → Far 0.95 (0.97 for prayer-critical/weak/lapsed). Priors deliberately *under*-estimate strength. `kNearMinS=9`, `kFarMinS=60` set `phaseOf`.
- §2 (Data model) — **`dueAt` is non-nullable for a memorized card — INV-6 enforced by the type, not copy-grep.** `Card` is immutable; `onReview` returns a new instance; `Track { unmemorized, near, far, newLesson }` and `Grade { again, hard, good, easy }` are the enums a generator draws from. A memorized page without a ceiling is unrepresentable.
- §1 (Vendored FSRS, pure Dart) — **The engine is a pure-Dart package with zero I/O — no Flutter, no DB, no clock, no `Random`; `today` injected as `SerialDay`.** Elapsed days is plain integer subtraction (immune to the DST off-by-one). `FACTOR` computed from `DECAY` — no magic numbers; an FSRS-6 bump is one reviewed edit. This purity is what makes the vectors and properties deterministic and meaningful.

## Supporting — `docs/PRD.md` §7.12 (the covenant the properties encode)

- §7.12 (Engine invariants) — **The five product-level invariants the `glados` properties make executable:** a memorized page's `due_at` is never later than its cycle ceiling; FAR/manzil due items are never silently dropped; the engine never implies a page is "safe to stop revising"; a teacher sign-off always supersedes self-rating and algorithmic state for that page; all math is pure Dart, deterministic, golden-tested (identical inputs → identical schedule). Gate 4 of §20 names "invariant tests (§7.12) as property-based checks."

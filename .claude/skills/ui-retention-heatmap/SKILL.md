# ui-retention-heatmap

The retention heat-map is the app's central emotional surface: a **cluster heat-map of the whole Quran** — all 604 pages grouped into the 30 juz the ḥāfiẓ already holds in mind — shaded so the entire corpus reads in one glance, leading the Progress screen with the muṣḥaf-shaped overview rather than a stats table. It is the literal embodiment of Pillar 4 (*honest about decay; nothing silently rots*): the green **recedes** to a muted neutral as a page softens, so the invisible is made visible — calmly, never as an alarm. The exact numbers (a page's retrievability, its next-due date, its history) live **behind a tap**, never on the face of the grid.

This surface carries the PRD's quiet hook, *"keep your Quran green"* (PRD §12.5). The pull is the intrinsic wish to keep one's own Quran whole, surfaced by honest situational awareness — never a streak the app can take away, a score to chase, or a red scoreboard for a missed day. Everything here is read-only state, offline, no-AI: the grid renders a model output, it never fetches, records, or infers.

## When to use

Use this skill when building or placing:

- the **whole-Quran retention heat-map** on the Progress screen — the 604-page / 30-juz cluster grid, *overview first → zoom (juz) → details-on-demand (page)* (`docs/design-system/08-data-visualization.md` §1);
- a **per-juz or per-page health visual** — a juz tile, a page cell, the min-leaning juz roll-up, the weakest-page badge (`docs/design-system/08-data-visualization.md` §6);
- the **page detail sheet** that opens behind a cell tap — retrievability range, next-due, short `review_log` history, stated in words + locale numerals (`docs/design-system/08-data-visualization.md` §1, §4, §5);
- the **upcoming-load forecast** or **simple review history** beside the map — informational planning aids, never a performance dashboard (`docs/design-system/08-data-visualization.md` §1; PRD §12.5);
- any **whole-Quran strength/decay visualization** that encodes page or juz health with the heat ramp.

Do **NOT** use this skill for — use the sibling instead:

- the **single page-card row** in the Today list (track chip + per-row decay indicator + headline) → use **ui-page-card**; this skill owns the *grid and its cells*, that one owns the *list row*.
- the **Today daily-session list container** (finite/capped, Far→Near→New ordering, all-done / catch-up states) → use **ui-daily-session-list**.
- the **recite/grade flow** a page taps into from the detail sheet (reveal-on-tap, Again/Hard/Good/Easy, teacher sign-off) → use **ui-recite-grade-flow**.
- the **engine math behind the cell** — `card.R` retrievability, FSRS D/S/R, `sourceConfidence`, the juz aggregate semantics, "never safe to drop" → use **domain-scheduling-engine-rules**; this skill *reads* the engine's result and *renders* it, it never re-derives a schedule or `R`.
- the **concrete ramp hex values, the four appearances, the contrast audit** → those live only in `docs/design-system/03-color-and-themes.md`; this skill references `color.heatmap.*` **by name** and never inlines a hex.
- **per-locale numerals, bidi isolation of a percentage/range run, and RTL grid mirroring mechanics** → use **eng-rtl-and-bidi-layout**; this skill only requires you *use* them.
- the **calm, non-coercive, no-scoreboard copy and adab** of every label, the no-celebration rule → use **domain-adab-and-religious-integrity**.

The heat-map is **honest situational awareness** — "where is my Quran today" — and nothing more. A heat-map that adds a streak, a points score, a "100% green!" celebration, a red overdue pile, or estimates exact strength from a shade is the wrong component.

## The canonical pattern

1. **The whole-Quran heat-map is the surface; numbers live behind a tap.** The Progress screen opens on the cluster grid — all 604 pages grouped into 30 juz — not a KPI table; the interaction is the canonical *overview first → zoom and filter → details-on-demand*: whole Quran → tap a juz → tap a page (`docs/design-system/08-data-visualization.md` §1, *Statement* + *In practice*; PRD §12.5). Tapping a page opens the detail sheet with retrievability, next-due, and a short history from `review_log` (PRD §10.2). Tiles meet `touch.min` (48dp) so a juz/page is tappable, with `space.*` gutters (not hairline gaps) so juz blocks read as the small-multiple unit (`docs/design-system/08-data-visualization.md` §1, §7; tokens owned by `docs/design-system/05-layout-spacing-touch.md`).

2. **Strength is a single-hue lightness ramp of the base green.** Page/juz health is a **sequential, single-hue lightness ramp** — `color.heatmap.strong → good → fair → weak → faded`, monotonic in relative luminance — where a strong page reads as the calm base green (`color.accent.green` family) and a decaying page fades toward a muted neutral (`docs/design-system/08-data-visualization.md` §2; `docs/design-system/03-color-and-themes.md` §5, ramp values owned there). Order is carried by **lightness, not hue**, so magnitude reads correctly in greyscale and under colour-vision deficiency. Reference the ramp **by token name only** — never inline a hex, never restate the audit.

3. **Decay is made visible, never alarming — no rainbow, no red scoreboard.** The decaying end is `color.heatmap.weak → faded`, a desaturated **muted neutral**; the grid carries **no saturated red and no rainbow/jet colormap** (`docs/design-system/08-data-visualization.md` §3; `docs/design-system/03-color-and-themes.md` §5/§6, no alarm-red for routine hifz state). A most-decayed or never-reviewed page reads as *quiet*, not *injured*. Pair the map with **maintenance framing, never loss framing**: a softened page is "ready for revision"; a gap is met by the calm catch-up plan, never a red overdue pile (`docs/design-system/08-data-visualization.md` §3; PRD §7.9). Never frame decay as loss ("you are losing your Quran") to wring out effort.

4. **An uncertain estimate looks uncertain — VSUP-style muting.** A retention value is a **prediction, not a measurement**, so a page seeded from a cold-start prior and never recited, or graded only by a low-confidence self-rating, renders **muted/greyed** toward `color.heatmap.faded`, and saturates toward full health-green only as confident reviews accumulate (`docs/design-system/08-data-visualization.md` §4, Value-Suppressing Uncertainty Palette). Two inputs drive the muting: (a) whether the page has had a real review at all, and (b) the `sourceConfidence` of its most recent grades — self-rating ≈ 0.5 vs. teacher sign-off = 1.0 (PRD §8.1). A single self-rating must **not** push a page to the most-saturated tier — confidence grows with confirmed reviews, mirroring the engine's conservative priors (PRD §7.10). The `R`, `sourceConfidence`, and priors come from the engine — **domain-scheduling-engine-rules** — never recomputed here.

5. **Every state is encoded redundantly — colour and number and label.** No cell or detail ever conveys its state by **colour alone**: each carries a **localized retrievability value** (a percentage/range in the locale numerals — Extended Arabic-Indic for fa/ckb, Arabic-Indic for ar, via `intl`, never raw ASCII) **and** a **plain text label** ("strong," "softening," "ready for revision"), with colour as reinforcement (`docs/design-system/08-data-visualization.md` §5; PRD §13.3; WCAG 2.2 SC 1.4.1). An optional **texture/pattern** on the decaying end gives a third colour-independent channel. Contrast posture follows the audit: `color.heatmap.strong` clears the **3:1** non-text floor (SC 1.4.11) as the glance-critical anchor; lower steps may sit below 3:1 because they appear only inside labelled, tap-through cells where the number and label carry the value (`docs/design-system/03-color-and-themes.md` §7; gated as a release check in `docs/design-system/09-accessibility-and-inclusivity.md`). The detail sheet states uncertainty in words — "estimated — not yet recited" or a retrievability *range* with its basis — never a false-precise single percent.

6. **The juz roll-up leans to the *weakest* page — the honest aggregate.** A juz tile's colour is a **min-leaning** roll-up of its pages, **never a mean** — one weak page is what fails you in ṣalāh, so a single rotting page must be able to "fail" its juz colour or surface a small **weakest-page badge** (`docs/design-system/08-data-visualization.md` §6; PRD §10.3). An averaging roll-up would quietly break the engine's invariant that no page is ever "safe to stop revising" at the chart layer even while the engine honours it (PRD §7.12). The detail sheet names the weakest page(s) so the user can go straight there. This is a load-bearing choice: never "improve" the roll-up into a mean for a greener map.

7. **Lay the grid out in muṣḥaf order, RTL — faithful small multiples.** The grid mirrors the structure the ḥāfiẓ already holds — **juz → pages, in muṣḥaf order, flowing start→end under `Directionality.rtl`** — so a hot spot's *spatial position is its location in the Quran* (`docs/design-system/08-data-visualization.md` §7; PRD §13.2). Build it as **faithful small multiples**: one consistent juz block (size, `space.*` gutter, internal page order) learned once and read thirty times. Never reorder the grid by weakness, and never swap the faithful grid for a clever chart (treemap, radial, packed bubbles) that breaks the learned juz→page unit or the RTL reading order. Use `EdgeInsetsDirectional` / logical start/end — never hardcode left/right (**eng-rtl-and-bidi-layout**).

8. **No glyphs and no decoration in the visualization layer.** The heat-map visualizes *state about* pages; it **never renders or decorates the sacred glyphs** — markers and overlays on the actual muṣḥaf page are governed separately and drawn over the immutable glyph layer, never here (`docs/design-system/08-data-visualization.md` §7; PRD R1; **domain-mushaf-text-integrity**). Green never appears as a celebratory flash, a badge on an ayah, or a tint over the QPC glyphs (`docs/design-system/03-color-and-themes.md` §2).

9. **The map is situational awareness — never a streak, score, or scoreboard.** There is **no** streak counter, "best day," points, level, or leaderboard on or near the heat-map; any continuity indicator is **opt-in, private, and non-punitive** (`docs/design-system/08-data-visualization.md` §8; PRD §12.5, R3, C6). A missed day changes the map honestly (some green recedes) but triggers **no shame UI** — the response is the calm catch-up plan. Completing a juz or session produces **no confetti, no celebratory green flash, no badge, no sound** — celebration of worship is gamification (`docs/design-system/08-data-visualization.md` §8; `docs/design-system/03-color-and-themes.md` §2; **domain-adab-and-religious-integrity**).

10. **Accessibility, RTL, and locale by construction.** Cells are graphical objects that must satisfy SC 1.4.11 at the strong anchor and SC 1.4.1 (colour never the sole channel) everywhere; numerals inside cells use locale numerals via `intl`, the Latin range/page-number runs are bidi-isolated (FSI/PDI) so they never reorder inside the RTL label, and the weakest-page badge sits at the logical **start** under `Directionality.rtl` (`docs/design-system/08-data-visualization.md` §5, §6, §7; PRD §13.3; **eng-rtl-and-bidi-layout**). A screen reader announces each cell's value + label in the active locale (`docs/design-system/09-accessibility-and-inclusivity.md`). The grid is identical across light/sepia/dark/night and across fa/ckb/ar; the framing copy is *transcreated* per locale, never literally translated, so "ready for revision" never reads as blame (`docs/design-system/08-data-visualization.md` §3; **domain-adab-and-religious-integrity**).

## Do / Don't

| Do | Don't |
|---|---|
| Lead the Progress screen with the muṣḥaf-shaped overview; exactness behind a tap (overview → juz → page) | Lead with a number-dense table or a KPI dashboard, or read exact strength off a shade |
| Encode health with the single-hue `color.heatmap.strong → faded` lightness ramp, monotonic in luminance | Encode health on a hue axis, a red→amber→green traffic light, or a rainbow/jet colormap |
| Render the decay end as a calm muted neutral (`color.heatmap.weak`/`faded`); frame as "ready for revision" | Render decay as a saturated alarming red, a failing scoreboard, or "you are losing your Quran" |
| Mute low-confidence / never-recited pages (VSUP); saturate only as confirmed reviews accumulate | Render a cold-start prior or a single self-rating as a confident, fully-saturated green |
| Encode every cell three ways: ramp colour + localized number/range + plain label (+ optional texture) | Let colour be the sole signal, or print a bare "weak" with no retrievability range and no due date |
| Roll juz health up **min-leaning**; surface a weakest-page badge; name the weak page in the detail | Roll a juz up by a mean (or any aggregate) that hides one rotting page behind a green tile |
| Lay 30 juz blocks in muṣḥaf order, start→end under `Directionality.rtl`, as faithful small multiples | Reorder the grid by weakness, or swap it for a treemap/radial/bubble chart that breaks juz→page order |
| Reference `color.heatmap.*` / `space.*` / `touch.min` by name; locale numerals via `intl`, bidi-isolated | Inline a hex/dp/pt, restate the contrast audit, or splice raw ASCII digits into a localized string |
| Keep the visualization layer glyph-free and decoration-free; read engine state only | Render or tint a Quran glyph here, or recompute `R` / `due_at` / the juz aggregate in the widget |
| Keep it situational awareness; any continuity indicator opt-in, private, non-punitive | Add a streak, points, leaderboard, "100% green!" flash, confetti, or a red mark for a missed day |

## Checklist

Before this heat-map / juz tile / cell / detail sheet is done:

- [ ] Progress opens on the **whole-Quran cluster grid** (604 pages / 30 juz), overview first; exact retrievability / due / history live in the page **detail sheet behind a tap**, never on the grid face (`docs/design-system/08-data-visualization.md` §1; PRD §12.5).
- [ ] Health is the **single-hue `color.heatmap.strong → good → fair → weak → faded` lightness ramp**, monotonic in luminance, referenced **by token name only** — no hue axis, no traffic light, no rainbow/jet, no inlined hex.
- [ ] The decay end is a **calm muted neutral** (`color.heatmap.weak`/`faded`), never a saturated alarm-red or a failing scoreboard; framing is maintenance ("ready for revision"), never loss, and a gap routes to the calm catch-up plan (PRD §7.9).
- [ ] Low-confidence inputs **mute** the cell (VSUP): a never-recited page and a self-rating-only page read less saturated than a teacher-confirmed one; a single self-rating cannot reach the top tier (`R` / `sourceConfidence` read from the engine, never recomputed — **domain-scheduling-engine-rules**; PRD §7.10, §8.1).
- [ ] Every cell/detail is **redundantly encoded** — ramp colour + a **localized** retrievability value/range (Extended Arabic-Indic fa/ckb, Arabic-Indic ar, via `intl`) + a **plain label** (+ optional decay texture); colour is never the sole channel (SC 1.4.1).
- [ ] `color.heatmap.strong` clears the **3:1** non-text floor (SC 1.4.11) as the glance anchor; lower steps appear only inside labelled, tap-through cells where number+label carry the value; the audit is not restated here (`docs/design-system/03-color-and-themes.md` §7; gated in `docs/design-system/09-accessibility-and-inclusivity.md`).
- [ ] The detail sheet states uncertainty **in words** ("estimated — not yet recited" / a retrievability *range* with its basis), never a false-precise single percent; the raw `R` / D / S is never shown.
- [ ] The juz tile colours **min-leaning** (never a mean); one weak page can fail the tile colour or surface a **weakest-page badge**; the detail names the weakest page(s) (`docs/design-system/08-data-visualization.md` §6; PRD §10.3, §7.12).
- [ ] The grid is **30 juz blocks in muṣḥaf order**, flowing start→end under `Directionality.rtl` as faithful small multiples (one learned juz block × 30); `EdgeInsetsDirectional` / logical start/end only — no hardcoded left/right; the weakest-page badge sits at the logical start.
- [ ] No Quran glyph is rendered or tinted in the visualization layer (markers live over the immutable glyph layer elsewhere — **domain-mushaf-text-integrity**); no `R` / `due_at` / juz aggregate is recomputed in the widget.
- [ ] **No gamification anywhere**: no streak/best-day/points/level/leaderboard near the map; any continuity indicator is opt-in, private, non-punitive; completing a juz/session produces no confetti, green flash, badge, or sound; a missed day shows honest recession with no shame UI (PRD §12.5, R3, C6; **domain-adab-and-religious-integrity**).
- [ ] Offline / no-AI / no-audio by construction: the grid renders streamed engine state only — it never fetches, records, infers, or recomputes a schedule; copy is **transcreated** per locale and follows **domain-adab-and-religious-integrity**.
- [ ] Widget + golden tests cover the grid, a min-leaning juz tile, a muted (uncertain) cell, and the detail sheet across **fa/ckb/ar** on the **real** bundled UI fonts (never `Ahem`), with a grayscale / deuteranope check that magnitude order still reads, and an `HttpOverrides` offline guard (**eng-write-dart-test**).

The heat-map makes one honest, calm claim — *where is my Quran today* — so nothing rots in silence. If it ever tempts you toward a streak, a score, a "100% green!" celebration, an alarm-red overdue pile, or a shade you read as an exact number, that is the non-negotiables being violated, not a missing feature. The standard is *iḥsān*, because the work is built free, *lillāh*.

# references — domain-backup-format

The exact governing doc sections for this skill. Only real sections are listed; each line states the one thing to take from it.

## Primary

- `docs/engineering/10-backup-format.md` §1 (Module boundary) — **`backup/` is a pure-Dart package over value types + bytes — never Drift, `sqlite3`, or networking.** Public API is `HifzBackup.export(BackupSnapshot, {passphrase})` / `import(Uint8List, {passphrase})`; the package is store-blind (it validates only the snapshot the shell handed it, so it cannot leak another profile's rows); the `BackupError` enum gives a distinct, user-mappable failure per stage, never a catch-all.
- `docs/engineering/10-backup-format.md` §2 (Default form) — **Default is plaintext, human-readable, versioned UTF-8 JSON; encryption is opt-in** — a deliberate, documented divergence from CycleVault's mandatory-encryption design, justified by a different threat model and the transfer-to-a-teacher use that *wants* readability. The export screen must state plainly that an unencrypted backup is readable by anyone who opens it; honesty about what the file is *is* the trust feature.
- `docs/engineering/10-backup-format.md` §3 (Container layout & versioning) — **The exact big-endian header** (magic `HIFZBK` + `0x1F` + format byte `0x01` + mode byte + UInt32 body length + 2 reserved zeros + 32-byte body SHA-256 + body; min 49 bytes) and the **normative restore-side parse order** with a distinct error at each step. Two version axes are checked separately — the cleartext **format version** (container grammar) first, the **`schemaVersion`** (data shape, inside the JSON) only after decrypt+decode — so an encrypted file's header leaks nothing.
- `docs/engineering/10-backup-format.md` §4 (Payload schema) — **The versioned JSON is a complete value-typed export of the per-profile user model and NOTHING else** — `profile`, `cycleConfig`, `cards`, `lineBlocks`, append-only `reviewLog`, `confusionEdges`, keyed by stable UUIDs (`profileId`/`logId`). Export only *truth* (D/S/`dueAt` + the log), never *derived* state (no health %, forecast, Today list, notification cache); reference Quran tables are excluded (only `mushaf {id, riwayah, name, checksumSha256}`); scheduling days are floating `"YYYY-MM-DD"`, instants are UTC; keys are sorted for a deterministic body + reproducible hash; forward-compat migrates older `schemaVersion` payloads, refuses newer-than-app.
- `docs/engineering/10-backup-format.md` §5 (Integrity) — **A SHA-256 of the body, present in BOTH modes, fail-closed before any decode/apply** — the same NIST FIPS 180-4 primitive (Dart-team `crypto`) the Quran asset packs are pinned with. It matters *most* in plaintext mode (no AEAD tag) to catch a truncated transfer as silently-partial history. State plainly: the hash is *corruption detection, not tamper resistance*; no MD5/SHA-1.
- `docs/engineering/10-backup-format.md` §6 (Optional encryption envelope) — **Argon2id (RFC 9106, memory-hard) → ChaCha20-Poly1305 (RFC 8439), the same AEAD family as the opt-in DB cipher.** KDF id + params + salt + nonce stored in the envelope; passphrase NFC-normalized then UTF-8; AAD = the binary header; **clamp Argon2 params to range BEFORE any derivation**; run off the UI isolate; AEAD failure is the single indistinguishable `wrongPasswordOrDamaged`; no recovery; the file key is **independent of the device DB key** (a portable file must open on another device).
- `docs/engineering/10-backup-format.md` §7 (Import: replace vs merge) — **Two explicit, separately-confirmed modes, each in ONE Drift transaction.** Replace = full restore (validate in memory → wipe + insert verbatim → re-clamp `dueAt` to this device's ceiling). Merge = content-addressed **set union** over the append-only `review_log` by `logId` (idempotent; never overwrite/delete a row; never duplicate a sign-off), then rebuild each touched card from the merged log; a cross-`mushaf_id`/checksum merge is refused, not coerced; the per-entity resolution table.
- `docs/engineering/10-backup-format.md` §8 (SQLite-dump variant & the WAL trap) — **A raw snapshot uses `VACUUM INTO` to a fresh file — never `cp` the live `-wal`/`-shm` family** (which drops uncheckpointed commits). The dump carries the same `schemaVersion` and §3 container; the documented JSON stays *primary* because the dump is not the mergeable, human-readable artifact the teacher-transfer use needs.
- `docs/engineering/10-backup-format.md` §9 (Export & erase flows) — **Export writes atomically to a temp file then hands it to the OS share sheet — the app performs no network transfer**; no plaintext is left for an encrypted export. **Erase** is one confirmed, irreversible action that deletes the `.sqlite` + `-wal` + `-shm` siblings (+ the secure-storage key if encryption was on) — right-to-be-forgotten by construction; the UI states the existing backup file is then the only remaining copy.
- `docs/engineering/10-backup-format.md` §10 (Test plan) — **The pure-Dart `package:test` matrix** run under ≥2 `TZ` values: round-trip identity, encrypted round-trip (emoji/CJK/combining-mark passphrases), committed golden files with header-offset asserts, integrity-rejection, truncation, wrong-passphrase, Argon2 param-clamp (KDF spy → zero calls), newer-format/schema (DB untouched), timezone invariance, merge idempotence + superset, trust-clamp-after-import, cross-muṣḥaf refusal, `VACUUM INTO` single-file, schema migration, crash-mid-import rollback, erase completeness.
- `docs/PRD.md` §16 (Backup & Data Portability) — **The product requirement this skill serves:** encrypted-optional export of all profiles/cards/logs/configs to local storage / share-sheet with **no** app network transfer; import with clear merge/replace semantics; transfer-to-a-teacher via the user moving the file by any means; one-action erase; format = documented, versioned JSON (or SQLite dump) so data is never trapped.

## Supporting

- `docs/PRD.md` §17 (Privacy & Security) — **No account, no login, no PII (a profile is a typed display name); no telemetry; all data on-device; backups are user-initiated local files.** Grounds the "plaintext default is honest here" call and the "state the file is readable" honesty rule; the only network use anywhere is the one-time public asset download.
- `docs/PRD.md` §10.2 / §10.3 (Data model + derived state) — **What is *truth* vs *derived*:** D/S/`dueAt` and the append-only `review_log` are authoritative; juz/ḥizb health roll-ups are *computed from `card` retrievability, not stored*. The backup carries the former and never the latter, so the engine recomputes on import.
- `docs/PRD.md` R1 / R2 (Religious integrity) — **R1:** Quran text fidelity is existential → the sacred text is never embedded in a backup (no unverified side-channel copy). **R2:** state the riwāyah / muṣḥaf edition → cards index a layout-specific `mushaf_id`+checksum, so a cross-muṣḥaf merge is refused, never coerced.
- `docs/engineering/05-persistence-and-encryption.md` §1 (The store: Drift over SQLite) — **The CI banned-import gate that confines `drift`/`sqlite3` to `/data`** is the same gate that walls them *out* of `backup/`; the package speaks value types and bytes only.
- `docs/engineering/05-persistence-and-encryption.md` §2 (Schema) — **`review_log` is append-only; no DAO exposes UPDATE/DELETE; backup/restore is the *only* sanctioned bulk path that touches it.** This is why merge must be a set-union over immutable, UUID-keyed log rows, never an overwrite.
- `docs/engineering/05-persistence-and-encryption.md` §3 (Crash safety) — **Drift transactions are atomic with auto-rollback on throw, and every query inside must be `await`-ed.** Replace and merge each run in one transaction so a mid-import failure leaves the live store byte-identical; never file-copy the live WAL family.
- `docs/engineering/05-persistence-and-encryption.md` §4 (Migrations) — **Backup restore does NOT replay SQL migrations**; the schema-version-stamped payload is mapped forward to the current schema by the import pipeline, reusing the same `schemaVersion` integer language as the DB.
- `docs/engineering/05-persistence-and-encryption.md` §5 (At-rest encryption, opt-in) — **The opt-in DB cipher (`sqlite3mc` ChaCha20-Poly1305) + the device-bound `flutter_secure_storage` key.** The backup envelope uses the *same AEAD family* for one crypto mental model, but its passphrase-derived key is deliberately **independent** of the device key so the file opens on another device.

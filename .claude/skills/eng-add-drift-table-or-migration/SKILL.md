# eng-add-drift-table-or-migration

Every change to Hifz Companion's local store — a new table, an added column, an index, a `CHECK`, or the migration that ships it — happens in the **one package that imports `drift`/`sqlite3` (`/data`)**, behind DAOs that hand the rest of the app plain value types. The store is the app's covenant made physical: the `review_log` is an **append-only audit trail and a teacher sign-off is a *sanad* act**, so a lost write, a half-applied review, or a silently mutated log row is not a bug but a breach of the promise that "nothing decays silently." Crash-safety (WAL + `synchronous=FULL`) and transactional integrity are the **mandatory floor**; at-rest encryption is an honest, **opt-in** defense-in-depth scoped to device theft, not a headline. A migration that has shipped is immutable — fixes go forward in a new version — and no migration merges without a fixture test that proves a populated `v(n−1)` database survives the upgrade with `PRAGMA integrity_check` still `ok`.

This skill governs the **schema, the DAOs, the migration, and the transaction body** — the `/data` layer only. The pure scheduling math the repository calls is the engine (**domain-scheduling-engine-rules**); the Riverpod controller and single-write-path *wiring* that invokes a repository method is **eng-create-riverpod-store**; what a stored "due date" *means* (Gregorian-serial `CalendarDate`, Hijri/Jalālī) is **domain-calendars-and-hifzdate**; the read-only reference data's checksum governance is **domain-asset-pack-integrity** and **domain-mushaf-text-integrity**; and the fixture/migration *test* itself is authored with **eng-write-dart-test**.

## When to use

Use this skill when you:

- add or alter a **Drift table class** — a new user table, a column, an index, a `CHECK`, a foreign key (`docs/engineering/05-persistence-and-encryption.md` §2);
- bump `schemaVersion` and write a **`stepByStep` migration** with its committed JSON schema snapshot (`docs/engineering/05-persistence-and-encryption.md` §4);
- write or extend a **DAO / repository method** that opens one `db.transaction` per review and commits before republishing (`docs/engineering/05-persistence-and-encryption.md` §3);
- wire the connection `setup`/`beforeOpen` pragmas (WAL, `synchronous=FULL`, `foreign_keys=ON`) or touch the **opt-in at-rest encryption** toggle and key lifecycle (`docs/engineering/05-persistence-and-encryption.md` §1, §5);
- fix a defect in an **already-shipped migration** — which is **still** a new, higher `schemaVersion` step, never a back-edit (`docs/engineering/05-persistence-and-encryption.md` §4).

Do **NOT** use this skill for:

- the **Riverpod controller / single-write-path wiring** that *calls* a repository method (the `Notifier`/`AsyncNotifier`, the composition-root `ProviderScope` overrides, the `StreamProvider` read model) → use **eng-create-riverpod-store** (it persists-then-republishes through the method *this* skill defines).
- the **pure scheduling arithmetic** the repository persists — the FSRS curve, `onReview`, the trust clamp, tracks, `due_at`/`cycleCeiling` — that is the engine and never imports Drift → use **domain-scheduling-engine-rules**.
- the **package `pubspec.yaml`, the `resolution: workspace` member, and the banned-import lints** that keep `drift`/`sqlite3` out of `/engine`, `/features`, `/quran` → use **eng-create-package**.
- normalizing a recitation verdict into the `ReviewInput`/`ReviewOutcome` the transaction writes → use **domain-grading-pipeline**.
- the **calendar value type** (`CalendarDate`, the serial-day integer, the injected clock) a `due_at` column stores → use **domain-calendars-and-hifzdate**.
- the **WAL-aware export/import (backup) format** that turns this store into a portable file — restore does *not* replay SQL migrations → use **domain-backup-format**.
- authoring the **migration fixture test / golden / property** that proves the upgrade is safe → use **eng-write-dart-test** (this skill *requires* that test; that skill writes it).

The schema is the floor everything else stands on: DAOs map rows to plain value types, the repository owns the transaction, the engine computes, the controller republishes. A feature reaching for a raw query, a column storing a `DateTime` instant as a scheduling day, or an `UPDATE` on `review_log` is the wrong layer.

## The canonical pattern

1. **Drift lives only in `/data`; everything else sees value types.** Author the table as a Drift table class in the `/data` package — the *only* package allowed to import `package:drift` / `package:sqlite3` (a CI banned-import gate enforces it, the same mechanism that quarantines networking). DAOs map rows to plain value types; no Drift symbol crosses into `/engine`, `/features`, or `/quran`, so the FSRS math stays golden-testable without a database. `docs/engineering/05-persistence-and-encryption.md` §1 (Decision: *Drift over SQLite*, `NativeDatabase` FFI backend; the boundary keeps determinism testable) and §1 Pitfalls (the `import_rules`/DCM banned-import gate); module boundaries in `eng-create-package`.

2. **Constraints live in the schema, not in Dart — and tables are `STRICT`.** Encode invariants in the table definition so a corrupt row is unrepresentable at the storage layer: enumerable columns get `CHECK (col IN (...))` (`track`, `grade`, `source`, `role`, `locale`), ranges get `CHECK (d BETWEEN 1 AND 10)`, referential integrity is a foreign key, and every table is `STRICT` so a column never silently coerces. The engine invariant "a memorized card always has a `due_at`" is defended here as `CHECK (track = 'UNMEMORIZED' OR due_at IS NOT NULL)`. `docs/engineering/05-persistence-and-encryption.md` §2 (Specification — the v1 DDL; constraints in the schema, not application code).

3. **Reference tables are read-only; the Quran is immutable.** The 604-page structure, line geometry, ayah refs, surah metadata, muṣḥaf descriptor, and the scholar-reviewed mutashābihāt dataset come from the bundled, checksum-verified asset pack and are **never written at runtime** — forbidden by construction, because no DAO exposes a write to them. Adding a user table that references one is fine; adding a runtime mutation of `page`/`line`/`ayah`/`surah`/`mushaf`/`mutashabih_*` is a violation of the first outranking rule. `docs/engineering/05-persistence-and-encryption.md` §2 (Rationale — reference data read-only because the Quran is immutable; checksum-governed per `domain-asset-pack-integrity`).

4. **`review_log` is append-only — there is no `UPDATE`/`DELETE` DAO method, ever.** The append-only property is enforced by the *absence* of those methods on the generated DAO, reviewed in CI; the only sanctioned bulk touch is export / one-tap erase. A new column on `review_log` is allowed; an `update`/`delete` path against it is a *sanad*-breach. `docs/engineering/05-persistence-and-encryption.md` §2 (Pitfalls — refuse any `UPDATE`/`DELETE` on `review_log`) and §2 (Rationale — the append-only audit trail, "never updated or deleted by normal flows").

5. **Scheduling days are `CalendarDate` serial integers, never `DateTime` instants.** `due_at` / `last_review_at` and any new scheduling-day column store the `CalendarDate` serial-day integer owned by **domain-calendars-and-hifzdate**; only true wall-clock event times (`reviewed_at`, `created_at`, `last_confused_at`) are stored UTC ISO-8601. An instant column used as a scheduling day reintroduces the DST off-by-one that produces a wrong "next due." `docs/engineering/05-persistence-and-encryption.md` §2 (Specification — UTC instants vs. `CalendarDate` serial days) and §2 (Pitfalls — refuse `DateTime` instants as scheduling days).

6. **One `db.transaction` per review; persist before publish; await every query.** Every mutation is exactly one `db.transaction` — committed atomically or rolled back on throw — and a review is several writes (append `review_log`, update the `card` row, maybe create `line_block`s, maybe bump `confusion_edge`s), so they are all-or-nothing. The repository's `Future` resolves only after the durable commit; the controller republishes *after*. **Every query inside the transaction is `await`-ed** — a missing `await` is a release-blocking review item (un-awaited queries can run after the transaction closes → data loss). `docs/engineering/05-persistence-and-encryption.md` §3 (Decision/Specification — `commitReview` one transaction; persist-before-publish) and §3 (Pitfalls — the await footgun; no persist-after-publish).

7. **WAL + `synchronous=FULL` + `foreign_keys=ON`, re-asserted on every open.** The write connection runs in WAL journal mode with `synchronous=FULL` (not `NORMAL` — a teacher sign-off must survive power loss), and `PRAGMA foreign_keys=ON` is set in `setup`/`beforeOpen` because pragmas are **per-connection, not persisted** in the file. A value set once and assumed persistent is a silent correctness hole. `docs/engineering/05-persistence-and-encryption.md` §1 (Specification — the fixed `setup` pragmas; FK pragma re-asserted in `beforeOpen`) and §3 (Rationale — `synchronous=FULL` not `NORMAL`).

8. **Schema evolution is guided `stepByStep`, append-only, with a committed snapshot per version.** Bump the integer `schemaVersion`, export a committed JSON schema snapshot (`drift_dev schema dump` → `drift_schemas/`), generate the step file + test scaffold (`drift_dev make-migrations`), and add one typed `fromNToM` callback. A **shipped migration is never edited** — editing rewrites history for existing installs and is user-data loss; a fix is a new higher-version step that corrects forward. Never `select`/`update` a not-yet-added column inside an older step. `docs/engineering/05-persistence-and-encryption.md` §4 (Decision/Specification — `stepByStep`, snapshots, the CI workflow table; append-only) and §4 (Pitfalls — refuse editing a shipped migration; refuse querying a column before its migration ran).

9. **Every migration ships with a release-blocking fixture test.** Build a populated `v(n−1)` database, migrate it, and assert content survived **and** `PRAGMA integrity_check` returns `ok` — this is the one code path that touches every user's entire hifz history, so it carries the strongest test discipline. `eraseDatabaseOnSchemaChange` is DEBUG-only and gated so it can never reach a release build. `docs/engineering/05-persistence-and-encryption.md` §4 (Specification — the fixture-test row of the CI table; `eraseDatabaseOnSchemaChange` DEBUG-only) and §4 (Pitfalls — refuse a migration without a passing `integrity_check` fixture test). The test is authored with **eng-write-dart-test** and runs in CI (`docs/engineering/11-testing-strategy.md` §1 table — DAO/migration tests in the unit tier; §8 — `fast` job).

10. **Restore does not replay SQL migrations; backups map forward.** A schema change is additive: if a new user-facing column should survive export/restore, the backup payload is version-stamped and the import pipeline maps any supported version onto the *current* schema — it never runs a SQL upgrade on the failure-intolerant import flow, and an older backup must always still restore. `docs/engineering/05-persistence-and-encryption.md` §4 (Decision — backup restore does not replay SQL migrations) and §2 (the `app_meta` `schema_version` singleton); the backup byte/JSON layout is **domain-backup-format**.

11. **Encryption is optional, opt-in, off by default — and never silently plaintext.** At-rest encryption is one declarative build-hook toggle (`hooks: user_defines: sqlite3: source: sqlite3mc`, ChaCha20-Poly1305) plus a `PRAGMA key` in `setup`; the mandatory floor is WAL + transactions, encryption is defense-in-depth for device theft, not a CycleVault-style mandatory/decoy posture. A **hard liveness guard** (`PRAGMA cipher;` empty ⇒ refuse to open) prevents shipping a plaintext store that only *looks* encrypted; the 32-byte raw key lives in `flutter_secure_storage` (`first_unlock_this_device`, non-syncing); key rotation never happens in place on a WAL database. `docs/engineering/05-persistence-and-encryption.md` §5 (Decision/Specification — the toggle, the key lifecycle, the `PRAGMA cipher` guard) and §5 (Pitfalls — refuse a silently-plaintext build; refuse in-place WAL re-keying; refuse decoy/duress isolation).

12. **No derived state is stored as a second source of truth.** Per-juz / per-ḥizb / per-page health roll-ups and `R` (retrievability) are **computed on read from `card` state** (the PRD's min-leaning aggregate), never persisted — one authority, no drift. A new column that caches a derived value is a second source of truth that goes stale. `docs/engineering/05-persistence-and-encryption.md` §2 (Rationale — no derived state stored; Pitfalls — refuse to persist derived health roll-ups; refuse to widen `*_json` columns into health/Quran facts).

## Do / Don't

| Do | Don't |
|---|---|
| Author the table/DAO in `/data`, the only package importing `drift`/`sqlite3` | Import `package:drift` or `package:sqlite3` from `/engine`, `/features`, or `/quran` |
| `STRICT` tables; `CHECK (... IN (...))` for enums; `CHECK` ranges; FKs with `ON DELETE CASCADE` | Validate only in Dart; `REAL`-coercing un-`STRICT` tables; orphan-able child rows |
| Keep reference tables (`page`/`line`/`ayah`/`surah`/`mushaf`/`mutashabih_*`) read-only by construction | Expose a runtime `INSERT`/`UPDATE`/`DELETE` DAO on any reference table |
| Treat `review_log` as append-only — `INSERT` only | Add an `UPDATE`/`DELETE` DAO method on `review_log` |
| Store scheduling days as `CalendarDate` serial integers; instants as UTC ISO-8601 | Store a `DateTime` instant as `due_at`/`last_review_at` (the DST off-by-one) |
| One `db.transaction` per review; `await` every query inside it; persist **before** republish | Split a review across transactions, drop an `await` inside `transaction(() async`, or publish before commit |
| Re-assert WAL + `synchronous=FULL` + `foreign_keys=ON` in `setup`/`beforeOpen` on every open | Assume a pragma persists in the file, or use `synchronous=NORMAL` on the write path |
| Bump `schemaVersion`, commit a snapshot, add one `stepByStep` callback, ship a fixture test | Edit a shipped migration, skip the snapshot, or `select` a not-yet-added column in an old step |
| Assert `PRAGMA integrity_check == ok` + content-survived on a populated `v(n−1)` fixture | Merge a migration with no fixture test, or trust it ran without an integrity check |
| Compute juz/page health and `R` on read (min-leaning) | Persist a derived health roll-up or `R` as a stored authority |
| Keep encryption opt-in with the `PRAGMA cipher;` liveness guard | Ship a silently-plaintext store, re-key in place on WAL, or build decoy/duress isolation |

## Checklist

Before this schema change / migration is done:

- [ ] The table/column/DAO lives in `/data` — the only package importing `package:drift`/`package:sqlite3`; no Drift symbol crosses into `/engine`, `/features`, or `/quran` (CI banned-import gate green).
- [ ] Table is `STRICT`; enumerable columns are `CHECK (col IN (...))`; ranges are `CHECK`ed; referential integrity is a foreign key (`ON DELETE CASCADE` for per-profile children); the memorized-card invariant `CHECK (track = 'UNMEMORIZED' OR due_at IS NOT NULL)` is preserved.
- [ ] Reference tables (`page`/`line`/`ayah`/`surah`/`mushaf`/`mutashabih_group`/`mutashabih_member`) stay read-only — no runtime write DAO; their bytes remain checksum-governed (**domain-asset-pack-integrity**).
- [ ] `review_log` keeps its append-only property — `INSERT` only, **no** `UPDATE`/`DELETE` DAO method; the only bulk touch is export / one-tap erase.
- [ ] Scheduling-day columns store `CalendarDate` serial integers (never `DateTime` instants); true event times store UTC ISO-8601 (`reviewed_at`, `created_at`, `last_confused_at`).
- [ ] Each mutation is exactly one `db.transaction`; **every** query inside is `await`-ed; the repository `Future` resolves only after the durable commit, and the controller republishes *after* (persist-before-publish, never optimistic).
- [ ] WAL + `synchronous=FULL` + `foreign_keys=ON` (+ `busy_timeout`) are set in `setup`/`beforeOpen` and re-asserted on every open; no `synchronous=NORMAL` on the write path.
- [ ] `schemaVersion` bumped; a JSON schema snapshot committed (`drift_dev schema dump`); one typed `stepByStep` callback added; **no** edit to any already-shipped migration (a fix is a new higher version that corrects forward); no `select`/`update` of a not-yet-added column in an older step.
- [ ] A fixture test builds a populated `v(n−1)` DB, migrates, and asserts content survived **and** `PRAGMA integrity_check == ok`; `eraseDatabaseOnSchemaChange` is DEBUG-only and never in a release build (test authored via **eng-write-dart-test**).
- [ ] No derived state (juz/page health, `R`) is persisted as a second source of truth — it is computed on read (min-leaning); `*_json` columns hold small, schema-shaped, decode-validated data only, never Quran text or health facts.
- [ ] If encryption is touched: it stays opt-in/off-by-default; the `PRAGMA cipher;` liveness guard refuses a silently-plaintext store; the raw 32-byte key is in `flutter_secure_storage` (`first_unlock_this_device`, non-syncing); rotation is export-to-freshly-keyed, never in-place WAL re-key; no decoy/duress isolation.
- [ ] If a new user-facing field should survive export/restore, the backup payload is bumped **additively** and an older backup still restores forward (**domain-backup-format**); restore does not replay SQL migrations.
- [ ] No streak/badge/score column, no Quran/health fact asserted in a column comment or value; every user-facing string still lives in `l10n` (`ar` template, `fa`/`ckb`, RTL), not the schema — sect-/madhhab-neutral.

The store is where the covenant becomes physical: a half-applied review, an `UPDATE` on the audit trail, a `DateTime` instant masquerading as a scheduling day, or a migration that silently drops a `review_log` row is "decay happening silently" — exactly what the app exists to prevent. Persist before you publish, keep the transaction whole, ship no migration without a passing `integrity_check`, and when in doubt make the constraint the database's job, not the call site's.

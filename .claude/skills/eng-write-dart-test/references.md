# references — eng-write-dart-test

The exact governing doc sections for this skill. Only real sections are listed; each line states the one thing to take from it.

## Primary — `docs/engineering/11-testing-strategy.md` (the test architecture & CI gates)

- §1 (The test pyramid) — **The shape is fixed: a broad pure-Dart base, a thin device-bound top.** Almost all coverage is unit/property on `engine` + DAOs; widget/golden for screens; `integration_test` for the four make-or-break journeys only. Refuse an inverted pyramid; refuse to test the engine through the UI — anything `(card, grade, today) → card'` is a unit/property test, never a `pumpWidget`.
- §2 (Engine with `package:test`) — **The pure `engine` is tested with standalone `package:test`, never `flutter_test`, never a widget binding.** `test()`/`expect()`/`group()`/`setUp()`, `_test.dart` files, run with `dart test`. `today` is constructed as a literal; nothing reads the wall clock. `DateTime.now()` is refused anywhere reachable from the engine; the day type is an integer `SerialDay`. `clock`/`fake_async` are reserved for the app layer around the engine.
- §3 (Engine golden vectors) — **Pin the FSRS arithmetic with a committed `(state, grade, elapsed) → (D, S, due offset)` table asserted to `closeTo(_, 1e-6)`, never `==`.** Cover the curve, interval inversion, both stability branches, difficulty, the trust clamp, and the five cold-start seeds. Regenerate only by an explicit, human-reviewed `--update-vectors`; CI only verifies. Cold-start table: Solid `D=3,S=60`→FAR, Shaky `D=5,S=14`→NEAR, Rusty `D=7,S=4`→active.
- §4 (Invariants as `glados` property tests) — **Every PRD §7.12 invariant is a `glados` property over generated `(Card, grades, today)` histories, with shrinking to the minimal counterexample.** The register: INV-1 `dueAt ≤ cycleCeiling`; INV-2 every due FAR/manzil page in `buildToday`; INV-3 `Again ⇒ S'≤S ∧ track'≤track`; INV-4 two runs fingerprint-equal (fuzz OFF); INV-5 teacher `Again` overrides self `Good`; INV-6 every memorized card has a finite `dueAt` (no "drop"/`null`). No fixed lucky seed.
- §5 (Muṣḥaf-fidelity goldens) — **Load the real KFGQPC + UI fonts via `FontLoader` (never Ahem); pin OS/Flutter version, DPR, surface size, theme; disable animations; `@Tags(['golden'])`.** `await expectLater(finder, matchesGoldenFile('goldens/...'))`. A dropped/shifted diacritic must move pixels and fail the build. Masters regenerated with `--update-goldens` locally only; CI never blesses. Ahem is forbidden here (it would render every glyph as a square).
- §6 (Widget, RTL & integration tests) — **Widget tests drive screens headlessly with in-memory fakes; RTL is captured per locale; the four journeys run the real stack.** Pump under `Directionality(textDirection: TextDirection.rtl)` for `ar`/`fa`/`ckb` with locale numerals/calendars (font-independent strategy allowed for layout goldens, forbidden for fidelity). `integration_test` runs after `IntegrationTestWidgetsFlutterBinding.ensureInitialized()` against real Drift/SQLite + assets. The journeys: J1 cold start, J2 a review, J3 teacher sign-off, J4 missed-day catch-up. Refuse a real DB in widget tests; refuse `pumpAndSettle()` on an indefinite indicator; refuse a fifth journey without a decision-log amendment.
- §7 (The no-network gate) — **Offline is a build invariant: keep the default 400-block and install an `HttpOverrides` that throws on any connection.** A stray network call becomes a loud, named failure. Only the single asset-downloader test opts out (resets `HttpOverrides.global` to a mock in its own `setUp`). Networking imports are banned outside the one downloader module; banned analytics/ads/backend SDKs fail the dependency allow-list.
- §8 (CI shape) — **Goldens run in a pinned, Linux-only job isolated by `--tags=golden`; the fast job runs analyze + `dart test engine/` + widget + coverage.** Pin an exact `flutter-version`. The PRD §20 gate→job mapping makes the release contract auditable. Refuse `channel: stable` without a pinned version on the golden job; refuse goldens on macOS/Windows in CI.
- §9 (Text-integrity gate) — **CI re-hashes every asset-pack file (text, 604 fonts, layout, mutashābihāt) with SHA-256 against the pinned manifest + the authoritative Tanzil hash.** This is the build-time half of R1; a journey/acceptance test may invoke the runtime verifier, but the integrity contract itself lives in `09`/the domain skill. Refuse a sidecar checksum, an unverified runtime byte, or a weaker hash than SHA-256.
- §10 (Coverage policy) — **Publish coverage for OSS auditability, strip generated files, set NO percentage gate.** A covered-but-unasserted line is worthless; the §4 invariants and §3 vectors are the real assurance. Strip `*.g.dart`/`*.drift.dart`/`*.freezed.dart`/`*/generated/*` from `lcov.info` before reporting.

## Supporting — `docs/engineering/03-coding-standards.md` (how the test code itself is written)

- §1 / §1.1 (Naming) — **Full dictionary words and unit-bearing names in tests too** (`stabilityDays`, `cycleCeilingDays`, not `s`/`budget`); terse `D`/`S`/`R` confined to one short transcription scope with a citation comment. `CalendarDate` (a floating day) vs `DateTime` (a real instant) is marked by name as well as type — a `DateTime` named like a day is a review-blocking defect, the exact class injected-`today` test fixtures avoid.
- §3 (Formatting) — **`dart format` is the only authority; CI fails a non-conforming file.** A `// dart format off` region is allowed *only* around a hand-laid test-vector table where alignment is the documentation, and then with a justifying comment.
- §4 (Comments & doc policy) — **Comment *why*, not *what*; every scheduling/science constant in a vector carries a citation comment** (e.g. `// DECAY = -0.5 from FSRS-4.5`). The two covenants are restated at their enforcement points; a test that asserts a covenant should name it (`// PRD §7.6: SR may only make a page MORE frequent`).
- §5 (Immutability & error handling) — **The engine is total (never throws), so its tests assert returned values, not caught exceptions;** throwing is confined to I/O boundaries (asset downloader, persistence, backup) where a test asserts the sealed error type. No bare `catch`, no `print`, no `!`/`late` on engine/persistence values inside a test.
- §6 (Library privacy & engine purity) — **The engine is tested through its public API only** (the same surface the app uses); no `@visibleForTesting` to expose internals, no `part`/`part of` to smuggle private members across a package boundary. The CI engine-purity import gate is what keeps the fastest tier framework-free.
- §7 / §7.2 (Tooling & import bans) — **`@Tags`/`--tags` isolation and the banned-import gates are config-driven and must not vanish in a lint refactor;** an `// ignore:` on a §7.2 architecture gate (networking/engine-purity) is a review blocker. The analyzer config a test runs under bans `print`, bare `catch`, and networking imports outside the downloader.
- §8 / §8.1 (Code review checklist) — **The PR checklist names the test-relevant covenants:** new behaviour covered by golden/property tests; the sacred-text guard (dropped word ≠ "Good") intact; trust clamp holds; new keys in all three ARB locales; no networking outside the downloader. A test PR self-annotates non-obvious hunks before review.

# ui-cold-start-placement

The cold-start placement flow is the onboarding step that turns "I am a ḥāfiẓ" into a seeded schedule **without grading 604 pages on day one**. In three calm passes it captures **coverage** (which juz the user holds, via fast juz-level taps), a **per-juz confidence** (Solid / Shaky / Rusty), and an **optional "when memorized"** date — and hands those to the engine, which seeds deliberately *conservative* priors `(D, S)` and converges on real grades within ~2–3 weeks. It must feel sub-20-minute, never a calibration grind, and it is genuinely make-or-break: a bad first impression here loses the user before the engine ever proves itself.

This flow is a **capture surface, not a scheduler**: it gathers `(juz held?, JuzConfidence, memorizedOn?)` and routes each held page through the engine's `coldStartCard` seeding. It never invents `(D, S)` itself, never shows the engine's internal numbers, and never frames a juz the user does *not* hold as a failure. Pillar 5 (*private & offline by feel*) and Pillar 7 (*servant to the teacher*) govern: the placement is the user's own honest self-assessment, kept on-device, and the conservative bias means the app would rather over-review than tell someone a page is solid when it is lost.

## When to use

Use this skill when building or placing:

- the **coverage-capture** step — the fast juz-level (or surah-level) "which do you hold?" tap grid; un-held juz stay `UNMEMORIZED` (`docs/PRD.md` §7.10 step 1, §12.1);
- the **per-juz confidence** rater — Solid / Shaky / Rusty for each held juz, which seeds initial `(D, S)` (`docs/PRD.md` §7.10 step 2);
- the **optional "when memorized"** input — the stale-time date that ages a juz's stability from that date (`docs/PRD.md` §7.10 step 3);
- the **placement summary / first-day confirmation** that completes onboarding and triggers the first generated day (`docs/PRD.md` §12.1 "Done → first day generated").

Do **NOT** use this skill for — use the sibling instead:

- the **cold-start seeding math** itself (`coldStartCard`, the `_coldStartSeed` `(D, S)` table, stale-time `R`-decay, the conservative-bias and calibration-pass invariants) → use **domain-scheduling-engine-rules**; this flow *calls* the seeder, it never re-derives `(D, S)`.
- the **first-launch bundled-core setup** that precedes coverage capture (the SHA-256 verify of the bundled core, `preparingMushaf`/`ready` states; amended 2026-06-18 — no download) → use **domain-asset-pack-integrity**; placement assumes the muṣḥaf is already verified locally.
- the **persisted `Card` / coverage rows and the single write path** that stores the seeded cards transactionally → use **eng-create-riverpod-store** + **eng-add-persisted-model**.
- the **onboarding feature scaffold** (the `lib/src/onboarding/` folder, the `GoRoute`, the dumb View ↔ ViewModel split) → use **eng-add-feature-module**.
- the **muṣḥaf "mark my memorized range" tool** in the reader that *also* feeds coverage later → use **ui-mushaf-page-view** (`docs/PRD.md` §12.3); this skill owns the onboarding capture, not the in-reader editor.
- **per-locale numerals on juz numbers, bidi isolation, and RTL mirroring of the grid** → use **eng-rtl-and-bidi-layout**.
- the **calm, non-coercive, never-shaming copy** of every label, the un-held framing, and the privacy line → use **domain-adab-and-religious-integrity** + `docs/design-system/11-voice-and-tone.md`.

The placement flow is a **fast, honest, one-time self-assessment** — not a test, not a graded onboarding quiz. If a step starts to feel like a calibration grind, demands the user grade pages, or shames an un-held juz, it is the wrong shape.

## The canonical pattern

1. **Three passes, each finite and capped — coverage, then confidence, then optional date.** Sequence the step as PRD §7.10 specifies: (1) **coverage capture** (which juz held), (2) **per-juz confidence** (Solid/Shaky/Rusty for held juz only), (3) **optional stale-time** ("when memorized"). Build it from the restrained M3 + Flutter widget set, RTL-native, and keep the whole thing sub-20-minute — no per-page grading, no calibration grind (`docs/PRD.md` §7.10, §12.1; `docs/design-system/07-components.md` intro — every component built from the restrained M3 set, RTL across fa/ckb/ar). The flow is a *finite, capped* sequence with a calm end, never an open feed (`docs/design-system/07-components.md` §1, finite-capped-list principle).

2. **Coverage capture is fast juz-level taps; un-held stays `UNMEMORIZED` with no penalty.** Render the 30 juz as a tap grid (a `GridView` of selectable cells, one tap = held/not-held), in muṣḥaf order (juz 1 at the **start/right** in RTL) so the spatial map matches the book the ḥāfiẓ carries (`docs/PRD.md` §7.10 step 1; mirror the heat-map's muṣḥaf-order grid, `docs/design-system/07-components.md` §8). A non-held juz is simply not selected — it stays `UNMEMORIZED`, drawn as a calm un-emphasized cell, **never** an alarm-red "missing" or "0%" tile (`docs/design-system/07-components.md` §8 *Anti-patterns*, no alarm red, no completion-% trophy; `docs/design-system/11-voice-and-tone.md` §4, never lead with fault). Each cell carries color **+ a number/label** (the juz number in locale numerals + a checked/selected glyph) — never selection by color alone (`docs/design-system/07-components.md` §8 *In practice*, redundant encoding; SC 1.4.1).

3. **Per-juz confidence is a three-way label pick, framed as self-report not a grade.** For each *held* juz, offer **Solid / Shaky / Rusty** as a small set of mutually-exclusive choices (an M3 `SegmentedButton` or a row of `FilledButton`s, single-selection, ≥56dp tall and `space.2` apart in the thumb zone, mirroring the grade band's sizing) (`docs/PRD.md` §7.10 step 2; `docs/design-system/07-components.md` §6 — `SegmentedButton` for a single mutually-exclusive choice, large thumb-zone buttons for a costly tap). These are **self-report words in the user's own register** (transcreated, not literally translated), worded as honest description ("I hold this firmly / it wobbles / it's gone rusty"), never as praise or a score (`docs/design-system/11-voice-and-tone.md` §2 *honest* attribute, §8 transcreation; `docs/PRD.md` §13.4). Pair each option's color family with its text label — never color alone (`docs/design-system/07-components.md` §3/§8, color-independence; SC 1.4.1).

4. **The confidence pick seeds priors through the engine — the UI never invents `(D, S)`.** The chosen `JuzConfidence` maps to the engine's seed via `coldStartCard` — **Solid → D=3, S=60 (FAR/manzil)**, **Shaky → D=5, S=14 (NEAR)**, **Rusty → D=7, S=4 (active revision)** — and the ViewModel calls the seeder; it does **not** hardcode the table or compute stability (`docs/PRD.md` §7.10 step 2; `docs/engineering/06-scheduling-engine.md` §5 — the `_coldStartSeed` table + `coldStartCard`, "phase is a pure function of stability"). The mapping lives in the engine so it stays pure, deterministic, and golden-tested; the flow passes the captured choice and `today` (the injected clock, never `DateTime.now()`) (`docs/engineering/06-scheduling-engine.md` §5; **eng-define-service-boundary** for the injected `CalendarDate` "today").

5. **The conservative bias is a copy-and-UX promise, not just engine policy — never let the UI imply false precision.** Priors deliberately *under*-estimate strength so the first real recitation can only pleasantly surprise upward (`docs/PRD.md` §7.10 step 4; `docs/engineering/06-scheduling-engine.md` §5 *Pitfalls* — "we refuse optimistic cold-start priors"). The UI therefore **never** shows the seeded `D`/`S`/`R`, a "readiness %," or a "you're 87% ready" score — those internal numbers are forbidden on every surface, exactly as on the page card (`docs/design-system/07-components.md` §2 *Anti-patterns*, never surface D/S/R or a percentage). Frame the result honestly: "we'll start by revising everything you hold once, and adjust as you recite" — not "your hifz is graded."

6. **The optional "when memorized" date is genuinely optional and rendered in the locale calendar.** Offer a skippable date input per juz (or a coarse "years ago / months ago" band) that the engine uses to age `S` via the forgetting curve from that date — a juz finished years ago is treated as needing reactivation (`docs/PRD.md` §7.10 step 3; `docs/engineering/06-scheduling-engine.md` §5 — the `memorizedOn` stale-time decay branch). The date is captured as the injected `CalendarDate` value type, displayed in the user's calendar (Hijri Umm al-Qurā / Solar-Hijri Jalālī / Gregorian) with locale numerals — never a raw `DateTime`, never an instant (**domain-calendars-and-hifzdate**; `docs/design-system/11-voice-and-tone.md` §2/§8 numeral mechanics). Skipping it is a first-class path, never a nag (`docs/design-system/11-voice-and-tone.md` §6 — invitation, never command).

7. **Single write path: the seeded cards persist transactionally before the first day is built.** When placement completes, route the captured `(held juz, JuzConfidence, memorizedOn?)` through a store/repository method that calls `coldStartCard` for each held page and **persists the cards transactionally before republishing in-memory state** — the View never writes `Card` rows directly, and there is no half-seeded state if the app is killed mid-onboarding (`docs/PRD.md` §7.12 — pure/deterministic engine; **eng-create-riverpod-store** single write path; **eng-add-persisted-model**). Only after the write commits does "Done → first day generated" fire (`docs/PRD.md` §12.1).

8. **The whole flow is RTL-native, locale-correct, and offline by construction.** The grid, the confidence rows, and the summary read start→end (right→left) via `EdgeInsetsDirectional`; juz numbers render in the locale set (Extended Arabic-Indic for fa/ckb, Arabic-Indic for ar), and any mixed Latin/numeral run is bidi-isolated (FSI/PDI) (`docs/PRD.md` §13.3; **eng-rtl-and-bidi-layout**). Every string comes from `AppLocalizations` (ar template, fa/ckb transcreated), the section/track/confidence vocabulary is a **regional term-set** awaiting native + scholarly review, and ckb's longer terms wrap rather than truncate (`docs/PRD.md` §13.4, §21; `docs/design-system/11-voice-and-tone.md` §8). No step touches the network — the core pack is already verified locally — and nothing is recorded, inferred, or sent anywhere (`docs/PRD.md` C1, C2; Pillar 5).

9. **Accessibility: each cell and option is a labelled, focusable, redundantly-encoded control.** A coverage cell announces "Juz ۱۳, held" / "Juz ۱۳, not held" via `Semantics`, toggles with the `isToggle`/selected flag, and shows a visible focus ring (`color.outline`) for switch-control/keyboard users; a confidence option announces its word and its meaning ("Shaky — needs regular revision") so a non-visual user places confidently (`docs/design-system/07-components.md` §6 *In practice* — focus ring + announced flags; §1/§8 semantics; SC 2.4.7, SC 1.3.1; **eng-rtl-and-bidi-layout** for the per-run locale tag). Hit targets are ≥48dp `touch.min` (`docs/design-system/07-components.md` §2; SC 2.5.5/2.5.8).

10. **No gamification, no shame, no authority claim anywhere in placement.** There is no progress streak, no "complete your profile!" badge, no completion-% trophy, and no celebration when the grid fills — onboarding ends on a calm informational surface, not confetti (`docs/design-system/07-components.md` §1 *Anti-patterns* / §8 *Anti-patterns*; `docs/PRD.md` R3, C6; **domain-adab-and-religious-integrity**). The app never tells the user their hifz is "strong" or "weak" as a verdict, never speaks *for* the Quran, and never overrides a future teacher sign-off — placement is the user's honest self-report that a teacher can later correct (`docs/design-system/11-voice-and-tone.md` §5 — never speak *for* the Quran, teacher outranks the machine; `docs/PRD.md` R6, §7.12).

## Do / Don't

| Do | Don't |
|---|---|
| Sequence exactly three passes — coverage → confidence → optional date — finite, capped, sub-20-min | Turn placement into a per-page grading quiz or a calibration grind |
| Capture coverage as fast juz-level taps in muṣḥaf order (juz 1 at start/right in RTL) | Make the user grade or even open 604 pages, or order the grid LTR |
| Draw an un-held juz as a calm un-selected cell (stays `UNMEMORIZED`) | Render an un-held juz as alarm-red "missing", a "0%", or a failure |
| Offer Solid/Shaky/Rusty as a single mutually-exclusive pick (`SegmentedButton`/large `FilledButton`s), ≥56dp, `space.2` | Allow multi-select, tiny targets, or a free-form slider that fakes precision |
| Word the three confidences as honest self-report, transcreated per locale | Word them as praise, a score, or literal-translate the register |
| Pass `JuzConfidence` to the engine's `coldStartCard`; let it own the `(D, S)` table | Hardcode D=3/S=60 etc. in the UI or compute stability in the View |
| Keep the conservative bias visible only as behavior ("revise everything once, then adjust") | Show the seeded D/S/R, a "readiness %", or a "you're 87% ready" score |
| Make "when memorized" skippable and render it in the locale calendar via `CalendarDate` | Require the date, or store/display a raw `DateTime`/instant |
| Persist the seeded cards transactionally through the store *before* generating day one | Let the View write `Card` rows, or leave a half-seeded state on a mid-flow kill |
| Use the injected "today" clock for seeding and stale-time decay | Call `DateTime.now()` in the flow |
| RTL by geometry, locale numerals, bidi-isolated, strings from `AppLocalizations` | Hard-code left/right, splice ASCII digits, or ship a hardcoded literal |
| End on a calm informational summary | Celebrate a filled grid with confetti, a streak, a badge, or a completion trophy |

## Checklist

Before this placement flow is done:

- [ ] The flow runs exactly three passes — coverage capture → per-juz Solid/Shaky/Rusty → optional "when memorized" — finite and capped, built from the restrained M3/Flutter set, and feels sub-20-minute with **no** per-page grading or calibration grind.
- [ ] Coverage is fast juz-level taps in a `GridView` in muṣḥaf order (juz 1 at the **start/right** in RTL); each cell carries a locale-numeral label **+** a selected glyph (never color alone); an un-held juz stays `UNMEMORIZED`, drawn calm and un-emphasized, **never** alarm-red / "missing" / "0%".
- [ ] Per-juz confidence is a single mutually-exclusive pick (Solid/Shaky/Rusty) on ≥56dp targets `space.2` apart, each pairing a color family with a transcreated text label; worded as honest self-report, never praise or a score.
- [ ] The confidence choice is passed to the engine's `coldStartCard` (which owns the `_coldStartSeed` `(D, S)` table: Solid→D3/S60/FAR, Shaky→D5/S14/NEAR, Rusty→D7/S4/active) — the UI **never** hardcodes the seeds or computes stability.
- [ ] No seeded `D`/`S`/`R`, no "readiness %", no "you're N% ready" score appears on any placement surface; the conservative bias shows only as behavior ("we'll revise everything once, then adjust").
- [ ] "When memorized" is genuinely optional/skippable (no nag), captured as a `CalendarDate`, rendered in the user's calendar (Hijri/Jalālī/Gregorian) with locale numerals; the engine ages `S` from that date via `coldStartCard`'s `memorizedOn` branch.
- [ ] Seeding uses the **injected** "today" clock — no `DateTime.now()` anywhere in the flow.
- [ ] On completion, the seeded `Card`s persist **transactionally through the store/repository before** republishing state and before "Done → first day generated"; the View never writes rows; a mid-flow kill leaves no half-seeded state.
- [ ] RTL-native by geometry (`EdgeInsetsDirectional`, logical start/end); juz numbers in locale numerals (Extended Arabic-Indic fa/ckb, Arabic-Indic ar), mixed runs bidi-isolated (FSI/PDI); all strings from `AppLocalizations` (ar template, fa/ckb transcreated), term-sets regional and review-pending, ckb wraps not truncates.
- [ ] Each cell/option is `Semantics`-labelled with its state ("Juz ۱۳, held" / "Shaky — needs regular revision"), toggle/selected flag, and a visible focus ring (`color.outline`); hit targets ≥48dp `touch.min`; readable in grayscale/deuteranope.
- [ ] No streak/badge/completion-trophy/confetti anywhere; the flow ends on a calm informational summary, not a celebration; no verdict on the user's hifz, no speaking *for* the Quran, and nothing that could override a future teacher sign-off.
- [ ] Offline / no-AI / no-audio by construction: placement touches no socket (core pack already verified), records nothing, infers nothing; copy and adab follow **domain-adab-and-religious-integrity** + `docs/design-system/11-voice-and-tone.md`.
- [ ] Widget + golden tests cover the coverage grid (held/un-held), each confidence pick, the skip-date path, and the seed→store→first-day commit, in all three locales on the **real** bundled UI fonts (never `Ahem`), with `androidTapTargetGuideline`/`labeledTapTargetGuideline` and a deuteranope check (**eng-write-dart-test**).

The placement flow is the user's first, honest "here is what I hold" — captured gently, seeded conservatively, kept on this phone, and corrected by real recitation and, in time, a teacher. If a step ever tempts you toward a grade, a readiness score, a streak, an alarm-red "missing" juz, or a claim about the user's hifz, that is a non-negotiable being violated, not a missing feature. The standard is *iḥsān*, because the work is built free, *lillāh*.

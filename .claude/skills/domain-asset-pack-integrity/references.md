## Primary

- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß1 (Pack hosting) ‚Äî **Pin an exact GitHub immutable-Release tag baked into the binary, never `latest`.** A 404 on a pinned asset means "keep using the verified local copy," never "go fetch something else"; no first-party server/API/CDN exists; the pack taxonomy (core / reciter-audio / alt-mu·π£·∏•af) and the per-file manifest (`sha256` + `source` + `license`) live here.
- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß2 (First-launch core setup + optional-pack download) ‚Äî **The bundled core: `installCorePack` loads bundled bytes ‚Üí verify-each ‚Üí build-DB (no network).** Optional packs use **one whitelisted `/assets` downloader**, a plain HTTPS GET with no auth/cookies/identifiers, download-to-temp-then-promote, the `dio` controls (`CancelToken`, timeouts), and the honest preparing/ready / awaitingDownload/interrupted state table; "offline when an optional pack is requested is not a failure to scold."
- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß3 (Runtime SHA-256 verification) ‚Äî **The fail-closed state machine, exactly like Subresource Integrity.** Chunked `startChunkedConversion` (never `readAsBytes`); the expected digest comes from the signed binary, never a sidecar; match ‚Üí re-fetch once ‚Üí refuse to render Quran text; SHA-256 only, no MD5/SHA-1, no soft path, no unbounded retry; the avalanche property catches a single altered diacritic.
- `docs/PRD.md` ¬ß11.1 + ¬ß11.1.1 (Asset source + Download integrity) ‚Äî **The binary ships pinned SHA-256 + pinned release version; a pack is rejected and re-fetched unless its hash matches exactly, and the app refuses to render Quran text from any unverified asset.** Downloads are HTTPS GET from GitHub Releases / CDN carrying only a public asset URL; the asset-pack table (core/reciter/alt-mu·π£·∏•af, sources, when downloaded).
- `docs/PRD.md` ¬ß1 / C1 (Hard Constraints) ‚Äî **Offline after setup; the only network use is fetching static, public, checksum-verified open data ‚Äî no backend, accounts, sync, telemetry, remote config, or API we run.** This is the outranking rule the whole skill serves.

## Supporting

- `docs/engineering/01-architecture-overview.md` ¬ß3.1 (Packages and allowed imports) ‚Äî **`assets` is the only module that opens a socket** (`package:http` / `dart:io HttpClient`), quarantined and holding no user data; the verified bytes build the `data` reference DB; `engine`/`models` import no Flutter and no I/O.
- `docs/engineering/01-architecture-overview.md` ¬ß6 (The offline guarantee, made auditable) ‚Äî **"Fully offline" is a build invariant, not a promise:** network quarantined to one module, dependency allow-list, banned-import lint, test-time `HttpOverrides` that throws; content-hash pinning over GitHub immutable releases, **not** TLS cert pinning; no push (server-required), no `url_launcher`/WebView.
- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß4 (Reproducible builds & provenance) ‚Äî **Attestation / `attest-build-provenance` / minisign are CI- and audit-time controls only; the mobile app verifies nothing but the pinned digest** ‚Äî no PKI, no Sigstore client, no attestation parsing in the app.
- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß5 (Transport: TLS without cert pinning) ‚Äî **Require TLS 1.2+/1.3 via the platform trust store; never pin GitHub's cert/public key, never `badCertificateCallback => true`.** Integrity is the content hash, not the transport; pinning a third-party cert is an OWASP anti-pattern and a rotation-outage risk.
- `docs/engineering/09-asset-packs-and-offline-integrity.md` ¬ß6 (Proving "offline forever") ‚Äî **The two-layer CI gate + airplane-mode acceptance test;** from first launch (the core is bundled) every core Quran feature works with the radio off, permanently ‚Äî the optional-pack downloader is the only outbound path.

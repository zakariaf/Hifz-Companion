# references — ui-numerals-calendar-text

The exact governing doc sections for this skill. Only real sections are listed; each line states the one thing to take from it. This skill is the **rendering primitive** that turns a bare integer or a `CalendarDate` into a localized text run — shaped into the locale's digit set (Extended Arabic-Indic ۰۱۲ for fa/ckb, Arabic-Indic ٠١٢ for ar) and the chosen calendar (Jalālī / Hijri Umm al-Qurā / Gregorian), via `intl`, isolated for bidi, never raw ASCII. It *renders* values; computing them (the engine), laying them out (RTL geometry), and storing the preference (the store) are other layers.

## Primary

- `docs/design-system/12-localization-and-rtl.md` §4 (Numerals render in the locale's own digit set, always via `intl`) — **The numeral contract:** Extended Arabic-Indic (U+06F0–U+06F9, `۰۱۲۳۴۵۶۷۸۹`) for fa/ckb, Arabic-Indic (U+0660–U+0669, `٠١٢٣٤٥٦٧٨٩`) for ar, produced by `intl` formatters bound to the active locale — "never by hand-substituting ASCII digits." One `type.numeral` token; a locale switch "reshapes every number on screen with no string edits"; verify per surface because of the documented `intl` date-vs-number digit inconsistency. The Quran's own printed ayah numbers are the immutable glyph layer and are **not** `intl`-reshaped.

- `docs/engineering/12-localization-rtl-accessibility-impl.md` §5 (Locale numerals and regional terminology) — **The `numberFormatFor(locale)` implementation:** pin the numbering system explicitly with a Unicode `-u-nu-` extension — `fa-u-nu-arabext` / `ckb-u-nu-arabext` (Extended Arabic-Indic), `ar-u-nu-arab` (Arabic-Indic) — because `intl`'s Eastern-digit emission "is implemented in dates, but not in `NumberFormat`" and bare `ar` vs sublocales disagree. ASCII digits are **never** concatenated into a localized string; a number is formatted, then injected into an ICU placeholder. Calendar dates render through this **same** numeral path, downstream of the conversion owned by `07`.

- `docs/engineering/07-dates-calendars-and-correctness.md` §4 (Hijri / Jalālī / Gregorian are display-only, behind one presentation helper) — **The `CalendarPresenter` boundary:** the three calendars are produced **only at the display boundary** by one helper that maps a `CalendarDate` + chosen `CalendarSystem` + locale to a localized string, using pure-Dart offline packages (`shamsi_date` BSD-3 for Jalālī, `hijri` BSD-2 for Umm al-Qurā; Gregorian via `intl` `DateFormat`). **Numerals are remapped downstream of the calendar** (month/era from the package's localized tables since `intl`'s `DateFormat` is Gregorian-only; digits via the numeral path). Views never construct a calendar object; calendar choice is an explicit Settings value, not inferred from `Locale.current`. "We refuse a font-CDN or network calendar source."

## Supporting

- `docs/design-system/12-localization-and-rtl.md` §5 (Calendars are user-selectable — display over a single stored instant) — **The "render-only, never mutate" rule:** Solar-Hijri/Jalālī (default fa), Hijri Umm al-Qurā, and Gregorian are user-selectable; the calendar choice "is a pure *display transform*, so changing it never mutates engine state or due-dates — it only re-renders." Defaults are per-locale CLDR/constant data (Jalālī for fa, Umm al-Qurā lead for ar, Saturday week-start) — not inference.

- `docs/engineering/07-dates-calendars-and-correctness.md` §2 (All interval, cycle, and ceiling math is integer day arithmetic) — **Why the calendar can be a pure label:** day-distance is calendar-invariant — "a 'due in 7 days' interval is seven days whether the user reads it in Jalālī, Hijri, or Gregorian." So the rendered calendar affects only the label, never the schedule; this skill renders the label and touches nothing the engine computed.

- `docs/engineering/07-dates-calendars-and-correctness.md` §6 (Hijri honesty: a civil approximation, never a religious authority) — **The Hijri caveat:** a Hijri date is the **Umm al-Qurā civil** date, labelled as such, with a standing one-line honesty note, **never** presented as the authoritative date of Ramaḍān/ʿĪd/any observance; out-of-range conversion falls back to a Gregorian label rather than throwing; nothing keys a deadline to a Hijri date being exact; madhhab/sect-neutral, no fiqh ruling, no sect-specific sighting claim.

- `docs/design-system/12-localization-and-rtl.md` §3 (Mixed-direction text is isolated with Unicode FSI/PDI) — **Isolating the embedded run:** every embedded opposite-direction run — a page number, "Juz 7," a date, a percentage — is wrapped in Unicode isolation so the Bidi algorithm cannot reorder it ("page 7 of 30" → "30 of 7"), a screen-reader-order fix as well as a visual one. The embedded value is a formatted, isolated **placeholder** in an ARB message, never a hard-spliced substring; isolation wraps the *token*, not the surrounding word.

- `docs/engineering/12-localization-rtl-accessibility-impl.md` §4 (One bidi-isolation helper for every mixed-script run) — **Which isolate to call:** route every mixed run through the one helper (`isolate` = FSI…PDI for unknown direction; `isolateLtr`/`isolateRtl` when the run's direction is known) built on `intl`'s `Bidi`/`Unicode`. Prefer the **known-direction** `isolateLtr` for a digit run over first-strong `FSI`, because FSI mis-detects on leading punctuation. The helper is **owned by eng-rtl-and-bidi-layout**; this skill calls it around the formatted number.

- `docs/engineering/12-localization-rtl-accessibility-impl.md` §6 (Plurals: ICU `plural` and Arabic's six categories) — **Counts done right:** every count-bearing string uses an ICU `plural` message, and the count is **formatted to locale numerals** then placed in the plural form — so number agreement and digit shaping are both correct. Arabic's six CLDR categories (`zero`/`one`/`two`/`few`/`many`/`other`) are a translation contract; a naive `"$count pages"` is grammatically wrong and un-localized.

- `docs/design-system/12-localization-and-rtl.md` §8 (The Quran is never localized — only the chrome is) — **Where the toolkit stops:** localizing the app changes "buttons, labels, dates, numerals, and term-sets — never a single glyph of the page." The riwāyah/surah/juz *labels* are ARB strings and their *numbers* are `intl`-shaped, but the page content, ayah markers, and sajda signs are the fixed glyph layer — this skill's `intl` reshaping must never reach across that line.

- `docs/engineering/12-localization-rtl-accessibility-impl.md` §1 (The ARB / `gen_l10n` pipeline) — **Where the formatted number goes:** the formatted, isolated value is injected into a typed `AppLocalizations` ARB message via an `@placeholders` slot — never a hardcoded literal — so a missing key is a compile error and no user-facing string bypasses the pipeline. The pure `engine/` package has no strings and imports nothing from this layer.

## Sibling skills

- **eng-add-localized-string** — adds the ARB key, `@placeholders`, ICU `plural`/`select`, and *manzil/dhor* term-set strings the formatted numbers and dates are injected into (fa/ckb/ar, ckb provisional).

# ui-numerals-calendar-text

The numeral-and-date **rendering primitive** every other chrome surface leans on: how a bare integer (a page number, a juz index, a heat-map percentage, a daily-budget minute count) and a `CalendarDate` (next-due, last-reviewed) become a localized text run — shaped into the reader's own digit set and the user's chosen calendar, isolated for bidi, and never a raw ASCII digit spliced into a sentence. It owns the `numberFormatFor(locale)` numeral path and the `CalendarPresenter` display boundary; the surfaces that consume them (the page card, the Today list, the settings pickers, the heat-map) call this path rather than re-implementing it.

This is the lowest layer of the "RTL-native & multilingual" pillar (Pillar 6): correct numerals, calendars, and digit shaping are not a polish pass but a first-class, always-on invariant, because all three shipping locales are RTL and a number in the wrong block (`٤٥٦` shown to a Persian reader, or `456` anywhere in chrome) is a defect a native reader sees instantly. A chrome surface that builds `"Page " + n.toString()`, hardcodes a Gregorian grid, or lets a calendar package's Latin digits reach a widget is the bug this skill exists to prevent.

## When to use

Use when building or placing:
- any **number** in chrome — a page number, juz/ḥizb index, retention percentage on the heat-map, daily-budget minute count, day-count, sign-off count
- any **date** the user reads — next-due, last-reviewed, a history entry — rendered in the chosen calendar
- a **"Page N · Juz M"** (or "Juz N", "صفحة N") mixed run where a locale-numeral token sits inside RTL copy
- the **numeral-system** selector (Extended Arabic-Indic ۰۱۲ for fa/ckb · Arabic-Indic ٠١٢ for ar) and the **calendar-system** selector (Jalālī · Hijri Umm al-Qurā · Gregorian) — the *rendering* half
- the shared `numberFormatFor(locale)` formatter and the `CalendarPresenter` that maps a `CalendarDate` → localized date string
- any place a count goes into a plural message (the count must be locale-shaped *and* placed in the right CLDR plural form)

Do NOT use this skill for → use the named skill instead:
- the `CalendarDate` value type, the injected "today", integer day-math, or computing a `due_at` → use **domain-calendars-and-hifzdate** (this skill *renders* a `CalendarDate`; it never computes one)
- the RTL geometry (logical start/end insets), the directional-icon mirror policy, or the bidi-isolation **helper itself** → use **eng-rtl-and-bidi-layout** (this skill *calls* `isolate*`; that skill owns it)
- adding/renaming the ARB key, the `@placeholders`, the ICU `plural`, or the *manzil/dhor* term-set strings → use **eng-add-localized-string**
- the single-choice Settings **control** (radiogroup, persist-before-republish, 48dp rows) → use **ui-settings-picker** (this skill is the numeral/date rendering that picker's options use)
- numbers **printed on the muṣḥaf page** — ayah numbers, juz/ḥizb markers, sajda signs → use **ui-mushaf-page-view** / **domain-mushaf-text-integrity** (the immutable glyph layer is never re-shaped by `intl`)
- the long-lived notifier that persists the numeral/calendar **preference** → use **eng-create-riverpod-store**

This skill turns a value into a *rendered run*. Computing the value (the engine), laying it out (RTL geometry), and storing the preference (the store) are other layers' jobs.

## The canonical pattern

1. **Numbers render in the locale's digit set, always via `intl` — never hand-substituted.** Persian and Kurdish-Sorani show **Extended Arabic-Indic** (U+06F0–U+06F9, `۰۱۲۳۴۵۶۷۸۹`); Arabic shows **Arabic-Indic** (U+0660–U+0669, `٠١٢٣٤٥٦٧٨٩`). These are two distinct, non-interchangeable Unicode blocks — a `fa` UI showing `٤٥٦` is a defect a Persian reader notices. Produce every chrome number through `intl`'s `NumberFormat` bound to the active locale; never string-replace ASCII digits and never `"Page " + n.toString()`. `docs/design-system/12-localization-and-rtl.md` §4 (numerals via `intl`; `type.numeral` discipline; the two blocks are distinct and locale-driven) and `docs/engineering/12-localization-rtl-accessibility-impl.md` §5 (`numberFormatFor(locale)`).

2. **Pin the numbering system on the locale tag — don't trust sublocale defaults.** `intl`'s Eastern-digit emission is historically inconsistent between date and number formatting, and bare `ar` vs country sublocales disagree, so the numbering system is pinned explicitly with a Unicode `-u-nu-` extension: `fa-u-nu-arabext` / `ckb-u-nu-arabext` (Extended Arabic-Indic), `ar-u-nu-arab` (Arabic-Indic). This is the single `numberFormatFor(locale)` helper every numeric surface calls. `docs/engineering/12-localization-rtl-accessibility-impl.md` §5 (pin `-u-nu-arabext` / `-u-nu-arab`; never rely on sublocale defaults; the `numberFormatFor` switch) and `docs/design-system/12-localization-and-rtl.md` §4 (the documented `intl` date-vs-number caveat — verify per surface).

3. **Dates are computed in the engine as a `CalendarDate`, rendered here by one `CalendarPresenter`.** The three calendars — **Solar-Hijri/Jalālī** (default for `fa`), **Hijri Umm al-Qurā**, **Gregorian** — are produced only at the display boundary by one presenter that maps a `CalendarDate` + the user's `CalendarSystem` choice + the locale to a localized string, using pure-Dart, offline, BSD-licensed packages (`shamsi_date` for Jalālī, `hijri` for Umm al-Qurā; Gregorian via `intl` `DateFormat`). The engine never sees a calendar; a view never constructs a `Jalali`/`HijriCalendar` itself — it asks the presenter. `docs/engineering/07-dates-calendars-and-correctness.md` §4 (the one `CalendarPresenter`; pure-Dart offline packages; views never build a calendar object) and `docs/design-system/12-localization-and-rtl.md` §5 (three user-selectable calendars; the choice is a display transform).

4. **Numerals are remapped downstream of the calendar conversion — never inside it.** The calendar packages emit **Latin digits** in their formatter output; the helper converts the date *first* (month name + era from the calendar package's localized tables, since `intl`'s `DateFormat` is Gregorian-only for non-Gregorian calendars), *then* passes the numeric fields through `numberFormatFor(locale)`. The numeral transform sits **after** the calendar conversion, so a date never reaches a widget carrying ASCII digits. `docs/engineering/07-dates-calendars-and-correctness.md` §4 ("numerals are remapped downstream of the calendar"; month/era from the package, digits from `intl`) and `docs/engineering/12-localization-rtl-accessibility-impl.md` §5 (calendar dates render through this same numeral path, downstream of the conversion owned by `07`).

5. **The calendar is a display transform — rendering never mutates an instant or a `due_at`.** Switching the calendar (or the numeral set) re-renders dates over a single, unambiguous stored value and never perturbs engine state, a stored UTC instant, or a `due_at`. A "due in 7 days" interval is seven calendar days in Jalālī, Hijri, and Gregorian alike — calendar choice changes only the *label*. `docs/design-system/12-localization-and-rtl.md` §5 (calendars are "a display concern over a single, unambiguous stored instant"; changing the calendar "never mutates engine state or due-dates — it only re-renders") and `docs/engineering/07-dates-calendars-and-correctness.md` §2 (day-distance is calendar-invariant; the calendar affects only the displayed label).

6. **Every mixed Latin/number run inside RTL copy is FSI/PDI-isolated.** A locale-numeral token embedded in localized text — "Page ۲۵۳", "Juz ۷", a date, a percentage — is wrapped in Unicode isolation so the Bidi algorithm cannot reorder it ("page 7 of 30" → "30 of 7"), which is a screen-reader-order fix as much as a visual one. The number is a formatted, isolated **placeholder** inside an ARB message, never a hard-spliced substring; prefer the known-direction `isolateLtr` over first-strong `FSI` for a digit run. The isolation **helper** is owned by **eng-rtl-and-bidi-layout**; this skill *calls* it around the formatted number. `docs/design-system/12-localization-and-rtl.md` §3 (FSI/PDI isolation of every embedded run; isolate the token, not the surrounding word) and `docs/engineering/12-localization-rtl-accessibility-impl.md` §4 (the one isolation helper; prefer `isolateLtr` for a known-direction digit run).

7. **Counts are locale-shaped AND placed in the correct CLDR plural form.** A count-bearing string ("N pages due", "N days in your catch-up plan") goes through an ICU `plural` message *and* the count is shaped to locale numerals — both must be right, because Arabic has six CLDR plural categories (`zero`/`one`/`two`/`few`/`many`/`other`). The formatted number goes into the plural form; a naive `"$count pages"` is grammatically wrong and un-localized. `docs/engineering/12-localization-rtl-accessibility-impl.md` §6 (ICU `plural`; Arabic's six categories a translation contract; the count is formatted to locale numerals then placed in the plural form) and §5 (ASCII-digit concatenation banned).

8. **Hijri is a civil approximation, labelled and caveated — never a religious authority.** Wherever a Hijri date is shown it is the **Umm al-Qurā civil** date, labelled as such, accompanied by the standing one-line honesty note, and never presented as the authoritative date of Ramaḍān, ʿĪd, or any observance. An out-of-range conversion falls back to a Gregorian label rather than throwing; nothing keys a deadline to a Hijri date being exact (the daily reminder keys off the local civil day). `docs/engineering/07-dates-calendars-and-correctness.md` §6 (Hijri honesty: civil approximation, labelled "Umm al-Qurā", standing caveat, no observance promise, range-guarded fallback, madhhab/sect-neutral). Run any Hijri/observance copy past **domain-adab-and-religious-integrity**.

9. **The muṣḥaf is never localized — the toolkit stops at the chrome.** `numberFormatFor` and the `CalendarPresenter` apply only to chrome (Today, Progress, Settings, dates). The Quran's own printed ayah numbers, juz/ḥizb markers, and sajda signs are part of the **immutable glyph layer** and are never re-rendered by `intl` to "match the UI numeral set". `docs/design-system/12-localization-and-rtl.md` §4 (the Quran's ayah numbers are the immutable glyph layer, not `intl`-reshaped) and §8 (the Quran is never localized — only the chrome is; the localization toolkit stops at the chrome). The glyph-layer integrity is owned by **domain-mushaf-text-integrity**.

10. **Offline / no-AI — both calendar packages are bundled pure Dart; nothing infers, nothing phones home.** `shamsi_date` and `hijri` are pure-Dart, BSD-licensed, and bundle offline; date and numeral rendering touch no network and no model, and defaults are per-locale constants (Jalālī for `fa`, Umm al-Qurā lead for `ar`, the locale's numeral block, Saturday week-start from CLDR) — not inference. `docs/engineering/07-dates-calendars-and-correctness.md` §4 ("we refuse a font-CDN or network calendar source"; both packages are bundled pure Dart) and `docs/design-system/12-localization-and-rtl.md` §5 (per-locale calendar/week-start defaults are CLDR/constant data, not inference).

## Do / Don't

| Do | Don't |
|---|---|
| Format every chrome number through `numberFormatFor(locale)` (`intl` `NumberFormat`) | `"Page " + n.toString()`, string-replace ASCII digits, or pass a raw `int` into `Text`/ARB |
| Pin the numbering system: `fa/ckb-u-nu-arabext`, `ar-u-nu-arab` | Trust the sublocale default (`intl` disagrees between dates and numbers) |
| Show Extended Arabic-Indic (۰۱۲) for fa/ckb and Arabic-Indic (٠١٢) for ar | Ship the Arabic block for Persian/Kurdish, or one numeral set for all three locales |
| Render dates through the one `CalendarPresenter` (Jalālī/Hijri/Gregorian) | Construct a `Jalali`/`HijriCalendar` in a view, or hardcode a Gregorian grid / English month names |
| Remap numerals **downstream** of the calendar conversion | Let a calendar package's Latin digits reach a widget; or remap before converting |
| Treat the calendar/numeral switch as a pure re-render over unchanged data | Mutate a stored instant, a `due_at`, or any engine state from a rendering path |
| Isolate every embedded number/date run with FSI/PDI (prefer `isolateLtr` for digits) | Concatenate an opposite-direction run raw, or assume "Flutter handles bidi" |
| Put the formatted count into an ICU `plural` form (six categories for `ar`) | Ship `"$count pages"` — wrong grammar and un-localized digits |
| Label Hijri "Umm al-Qurā", carry the honesty caveat, fall back to Gregorian out of range | Present a Hijri date as the authoritative date of Ramaḍān/ʿĪd, or let a conversion throw |
| Keep `intl` reshaping to chrome only | Re-render the muṣḥaf page's ayah numbers / juz markers to "match the UI numerals" |

## Checklist

Before this rendering is done:

- [ ] Every chrome number is produced by `numberFormatFor(locale)` (`intl` `NumberFormat`) — **no** `n.toString()` splice, **no** raw `int` into `Text`/ARB, **no** ASCII-digit string-replace (`docs/design-system/12-localization-and-rtl.md` §4; `docs/engineering/12-localization-rtl-accessibility-impl.md` §5).
- [ ] The numbering system is pinned on the tag: `fa-u-nu-arabext` / `ckb-u-nu-arabext` (Extended Arabic-Indic ۰۱۲), `ar-u-nu-arab` (Arabic-Indic ٠١٢) — never a sublocale default; a per-locale numeral golden asserts fa/ckb render U+06F0-range and ar renders U+0660-range glyphs (`docs/engineering/12-localization-rtl-accessibility-impl.md` §5).
- [ ] Dates render through the one `CalendarPresenter` (Jalālī default for fa · Hijri Umm al-Qurā · Gregorian) over a `CalendarDate` from the engine; a view never builds a `Jalali`/`HijriCalendar` (`docs/engineering/07-dates-calendars-and-correctness.md` §4).
- [ ] Numerals are remapped **downstream** of the calendar conversion (month/era from the calendar package, digits via `numberFormatFor`); no Latin digits reach a widget on a Jalālī/Hijri date (`docs/engineering/07-dates-calendars-and-correctness.md` §4; `docs/engineering/12-localization-rtl-accessibility-impl.md` §5).
- [ ] Switching calendar or numeral set is a **pure re-render** — no stored instant, `due_at`, or engine state is mutated (`docs/design-system/12-localization-and-rtl.md` §5; `docs/engineering/07-dates-calendars-and-correctness.md` §2).
- [ ] Every mixed Latin/number run ("Page ۲۵۳ · Juz ۷", a date, a %) is FSI/PDI-isolated as a formatted placeholder (prefer `isolateLtr` for digits), never a hard-spliced substring (`docs/design-system/12-localization-and-rtl.md` §3; `docs/engineering/12-localization-rtl-accessibility-impl.md` §4; helper owned by **eng-rtl-and-bidi-layout**).
- [ ] Count strings go through an ICU `plural` message with the count shaped to locale numerals; Arabic supplies all six CLDR categories (`docs/engineering/12-localization-rtl-accessibility-impl.md` §6, §5).
- [ ] Hijri dates are labelled **Umm al-Qurā**, carry the standing honesty caveat, never assert an observance date, and fall back to a Gregorian label out of range (`docs/engineering/07-dates-calendars-and-correctness.md` §6; conscience-check via **domain-adab-and-religious-integrity**).
- [ ] `intl` reshaping is **chrome-only**: the muṣḥaf page's ayah numbers, juz/ḥizb markers, and sajda signs stay the immutable glyph layer, never `intl`-reshaped (`docs/design-system/12-localization-and-rtl.md` §4, §8; **domain-mushaf-text-integrity**).
- [ ] Fully offline / no-AI: rendering uses bundled pure-Dart `shamsi_date` / `hijri`, no network, no model, no telemetry; defaults are per-locale constants (Jalālī for fa, Umm al-Qurā lead for ar) and work in airplane mode forever (`docs/engineering/07-dates-calendars-and-correctness.md` §4; `docs/design-system/12-localization-and-rtl.md` §5).
- [ ] Tests: a numeral golden per locale (fa/ckb Extended Arabic-Indic, ar Arabic-Indic); a Jalālī and a Hijri round-trip/format check; a bidi-isolation golden on "Page N · Juz M"; an assertion that a calendar switch leaves the stored instant unchanged (**eng-write-dart-test**).

This rendering primitive decides *how a number or date looks*, never *what the engine computed* or *what the Quran says*. The two outranking lines: a Hijri date is a civil courtesy label with a caveat, never a religious deadline (it stays madhhab/sect-neutral and issues no sighting ruling); and the `intl` toolkit stops at the chrome — the muṣḥaf's printed numerals are immutable scripture, not UI digits. If a date label drifts toward asserting an observance, or a numeral starts being applied to the glyph page, run it past **domain-adab-and-religious-integrity** and **domain-mushaf-text-integrity**.
